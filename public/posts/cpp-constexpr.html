컴파일 타임 vs 런타임

대부분의 코드는 런타임에 실행된다. 그런데 값이 이미 소스에 박혀 있다면 굳이 실행 시점에 계산할 필요가 없다. 컴파일 시점에 미리 계산해 두면 다음과 같은 이점이 있다.

이점설명
런타임 비용 0실행 파일에 결과 값만 박힌다. CPU 사이클을 쓰지 않는다.
상수 표현식 문맥 사용배열 크기, 템플릿 인자, switch case, static_assert 등 컴파일 타임 값이 필요한 곳에 쓸 수 있다.
컴파일러 진단 강화잘못된 입력은 빌드 에러로 잡힌다. 런타임 디버깅이 줄어든다.
constexpr int Square(int x) { return x * x; }

int arr[Square(4)];                 // OK — 16짜리 배열
static_assert(Square(5) == 25);     // 컴파일 타임 검증

const vs constexpr 차이

둘 다 "변하지 않는다"는 뜻으로 보이지만 의미는 전혀 다르다.

키워드의미초기화 시점
const변수 값을 변경하지 못함런타임 또는 컴파일 타임 모두 가능
constexpr컴파일 타임에 평가 가능반드시 컴파일 타임에 결정
int GetValue();              // 런타임 함수

const int a = GetValue();    // OK — 런타임 상수
constexpr int b = GetValue(); // 에러 — 컴파일 타임에 알 수 없음

const int c = 10;            // OK
constexpr int d = 10;        // OK — 둘 다 컴파일 타임 상수

함수에 붙는 의미도 다르다. const는 멤버 함수가 객체 상태를 바꾸지 않음을 뜻하지만, constexpr은 그 함수를 컴파일 타임에 호출할 수 있다는 뜻이다.

constexpr 변수

constexpr 변수는 반드시 컴파일 타임에 평가 가능한 값으로 초기화해야 한다. 한번 초기화되면 변경할 수 없다.

constexpr int kMaxImageWidth = 4096;
constexpr double kPi = 3.14159265358979;
constexpr const char* kAppName = "ImagingLib";
C++17부터 inline constexpr constexpr 변수는 암묵적으로 inline 속성을 가진다. 헤더에 정의해도 ODR 위반이 발생하지 않으므로, 헤더 only 라이브러리의 상수로 안심하고 사용할 수 있다.

constexpr 함수의 발전사

constexpr 함수는 표준 개정마다 점점 더 많은 것을 허용해 왔다.

표준허용 사항
C++11단 한 줄의 return문만. 지역 변수, if, 반복문 모두 금지.
C++14지역 변수, if/switch, for/while 모두 허용. 사실상 일반 함수처럼 작성 가능.
C++17constexpr 람다, if constexpr 추가.
C++20virtual constexpr, try-catch, 컴파일 타임 동적 할당, std::vector/std::string constexpr 지원.
// C++11 — 재귀로만 가능
constexpr int Factorial11(int n) {
    return n <= 1 ? 1 : n * Factorial11(n - 1);
}

// C++14 이후 — 자연스러운 반복문
constexpr int Factorial14(int n) {
    int result = 1;
    for (int i = 2; i <= n; ++i) {
        result *= i;
    }
    return result;
}

if constexpr — 컴파일 타임 분기

if constexpr는 조건이 컴파일 타임에 평가되며, 거짓인 가지는 컴파일조차 되지 않는다. 덕분에 한 함수 안에서 서로 다른 타입에 대해 호환되지 않는 코드를 사용할 수 있다.

#include <type_traits>
#include <iostream>

template<class T>
void Print(T v)
{
    if constexpr (std::is_integral_v<T>)
    {
        std::cout << "int: " << v;
    }
    else if constexpr (std::is_floating_point_v<T>)
    {
        std::cout << "float: " << v;
    }
    else
    {
        std::cout << "other";
    }
}

C++17 이전에는 SFINAE나 함수 오버로드로 같은 효과를 내야 했지만, if constexpr 덕분에 코드가 훨씬 직관적이 된다.

consteval — 무조건 컴파일 타임

constexpr 함수는 가능하면 컴파일 타임에 평가되고, 그렇지 않으면 런타임에 실행된다. 반면 C++20의 consteval은 반드시 컴파일 타임에 호출되어야 하며, 런타임에 호출하면 컴파일 에러가 난다.

consteval int Cube(int x) { return x * x * x; }

constexpr int a = Cube(3);   // OK — 27

int n = 4;
int b = Cube(n);             // 에러 — 런타임 인자는 불가
consteval은 "즉시 함수(immediate function)" 매크로의 타입 안전한 대체로 자주 쓴다. 전처리기 없이도 "반드시 컴파일 타임에 결과가 나오는" 동작을 강제할 수 있다.

constinit — 정적 초기화 강제

C++20의 constinit은 정적/스레드 저장 기간 변수의 초기화를 컴파일 타임에 강제한다. static initialization order fiasco(여러 번역 단위 간 정적 변수 초기화 순서가 정의되지 않아 발생하는 버그)를 막는 도구다.

constexpr int ComputeFlag() { return 0xDEAD; }

constinit int g_flag = ComputeFlag();  // 컴파일 타임 초기화 보장

void Update()
{
    g_flag = 0xBEEF;   // OK — 런타임에 변경 가능
}
키워드컴파일 타임 평가이후 변경
const선택불가
constexpr강제불가
consteval강제 (함수)—
constinit초기화만 강제가능

실용 예시

컴파일 타임 피보나치

constexpr long long Fib(int n)
{
    long long a = 0, b = 1;
    for (int i = 0; i < n; ++i)
    {
        long long t = a + b;
        a = b;
        b = t;
    }
    return a;
}

static_assert(Fib(10) == 55);
static_assert(Fib(20) == 6765);

컴파일 타임 lookup table

이미지 처리에서 자주 쓰는 감마 보정 테이블을 빌드 시점에 만들어 두면, 런타임 초기화 코드가 사라진다.

#include <array>
#include <cmath>

constexpr std::array<uint8_t, 256> MakeGammaLut(double gamma)
{
    std::array<uint8_t, 256> lut{};
    for (int i = 0; i < 256; ++i)
    {
        // C++26 전까지 std::pow는 constexpr이 아니라 직접 구현 필요
        double v = static_cast<double>(i) / 255.0;
        double p = 1.0;
        for (int k = 0; k < 10; ++k) { p *= v; }   // 단순 근사
        lut[i] = static_cast<uint8_t>(p * 255.0);
    }
    return lut;
}

constexpr auto kGammaLut = MakeGammaLut(2.2);
lookup table은 constexpr의 단골 사용처 픽셀 변환, CRC 테이블, 비트 카운트, 사인/코사인 근사 등 "처음 한 번만 만들면 되는" 데이터는 빌드 시점에 만들어 두면 메모리는 그대로 쓰면서 시작 시간을 줄일 수 있다.

컴파일 타임 문자열 해싱 (FNV-1a)

constexpr uint32_t Fnv1a(const char* s)
{
    uint32_t h = 2166136261u;
    while (*s)
    {
        h ^= static_cast<uint8_t>(*s++);
        h *= 16777619u;
    }
    return h;
}

constexpr uint32_t kBlurId = Fnv1a("GaussianBlur");

void Dispatch(uint32_t id)
{
    switch (id)
    {
        case Fnv1a("GaussianBlur"): /* ... */ break;
        case Fnv1a("MedianFilter"): /* ... */ break;
    }
}

흔한 함정

constexpr 함수가 런타임 호출되는 것을 못 알아챈다 constexpr 함수는 가능하면 컴파일 타임에 평가되지만, 인자가 런타임 값이면 조용히 런타임 함수로 동작한다. 반드시 컴파일 타임 평가를 보장하고 싶다면 결과를 constexpr 변수에 받거나 consteval로 선언한다.
  • constexpr 함수 안에서 호출하는 모든 함수도 constexpr이어야 한다. 일반 라이브러리 함수를 무심코 부르면 컴파일 타임 평가가 깨진다.
  • if (constexpr_var)는 일반 런타임 분기다. if constexpr (cond)(공백 위치)와 혼동하면 의미가 완전히 달라진다.
  • consteval 함수는 함수 포인터로 받을 수 없다. 항상 즉시 호출되어야 한다.
  • C++20부터 std::vector를 constexpr 컨텍스트에서 쓸 수 있지만, 결과 컨테이너 자체를 constexpr 변수로 보존할 수는 없다(런타임 힙으로 풀어줘야 한다).

언제 constexpr를 추가할까

상황권장
헤더 only 라이브러리의 상수inline constexpr 변수
입력이 종종 컴파일 타임에 알려진 수학 함수constexpr 함수
매크로 대체 (반드시 컴파일 타임)consteval 함수
전역 객체 초기화 순서가 걱정될 때constinit
모든 함수에 constexpr를 붙이지는 말자 무분별한 constexpr 남용은 헤더 의존성을 늘리고 컴파일 시간을 증가시킨다. "컴파일 타임에 호출될 가능성이 실제로 있는" 함수에만 붙이는 것이 좋다.

요약하면 constexpr는 "런타임 비용을 빌드 시점으로 옮기는 도구"다. 잘 쓰면 성능과 안전성을 동시에 얻고, 잘못 쓰면 빌드만 느려진다. 새 코드에서는 상수와 순수 함수에 우선 적용하고, 필요할 때 consteval/constinit으로 의도를 더 강하게 표현하는 식으로 단계적으로 도입하면 된다.
