코루틴이란?

일반 함수는 호출 → 실행 → return → 종료의 단방향 흐름이다. 한번 return하면 함수의 지역 상태는 사라진다.

코루틴은 호출 → 일부 실행 → suspend → 재개 → 다시 suspend → ... → 종료의 흐름을 가진다. 즉, 함수 중간에서 멈췄다가 나중에 다시 그 자리에서 이어서 실행할 수 있는 함수다. 멈춰 있는 동안 지역 변수와 실행 위치는 코루틴 프레임에 보존된다.

C++20에 추가된 코루틴 키워드는 세 개다.

키워드역할
co_await현재 코루틴을 suspend하고 awaitable의 결과를 기다린다
co_yield값을 외부로 넘기고 suspend한다 (제너레이터 패턴)
co_return코루틴을 종료한다 (값 반환 가능)

함수 본문 안에 이 셋 중 하나라도 등장하면 그 함수는 자동으로 코루틴이 된다. 별도 선언 키워드는 없다.

다른 언어 코루틴과의 비교

코루틴 자체는 새로운 개념이 아니다. Python, JavaScript, Kotlin 등 여러 언어에 이미 있다. 다만 C++의 코루틴은 그 구현 방식이 다르다.

언어키워드특성
Pythonyield, async/await인터프리터가 프레임 관리, 사용 매우 간단
JavaScriptasync/awaitPromise 기반, 런타임이 스케줄링
Kotlinsuspend컴파일러가 CPS 변환, 코루틴 컨텍스트 제공
C++20co_await 등stackless, promise_type 사용자 정의 필요

C++ 코루틴은 stackless다. 호출 스택 전체를 보존하지 않고, 코루틴 함수의 지역 상태와 재개 지점만 담은 코루틴 프레임을 (보통) 힙에 할당한다. 덕분에 가볍지만, 코루틴 안에서 호출한 일반 함수의 스택 프레임은 보존되지 않는다.

적합한 용도는 비동기 I/O, 제너레이터, lazy 시퀀스, 상태 머신 단순화 등이다.

라이브러리 친화적 언어 기능

C++ 코루틴의 가장 큰 특이점은 표준이 키워드와 컴파일러 변환 규칙만 정의한다는 점이다. 실제로 suspend할 때 무엇을 하고 어떤 객체를 반환할지는 모두 사용자가 작성한 promise_type이 결정한다.

"라이브러리 친화적 언어 기능" Python처럼 바로 쓸 수 있는 형태가 아니다. 표준은 빌딩 블록만 제공하고, 사용 가능한 추상화는 라이브러리(또는 사용자)가 만들어야 한다. 이 때문에 처음 배울 때 추상화 비용이 크다. C++23부터 std::generator가 추가되어 비로소 표준에서 쓸 만한 코루틴 타입이 하나 생겼다.

promise_type

코루틴의 동작을 정의하는 타입이다. 코루틴의 반환 타입 안에 promise_type이라는 이름의 중첩 클래스로 정의해야 한다. 컴파일러는 이 이름을 약속처럼 찾는다.

멤버역할
get_return_object()코루틴 호출자에게 돌려줄 반환 객체를 만든다
initial_suspend()코루틴 시작 시 즉시 suspend할지 결정 (보통 lazy면 suspend_always)
final_suspend()코루틴 종료 시 suspend할지 결정 (반드시 noexcept)
unhandled_exception()코루틴 본문에서 빠져나온 예외 처리
return_void() / return_value(v)co_return 처리. 둘 중 하나만 정의
yield_value(v)co_yield가 호출될 때 동작 정의
final_suspend는 반드시 noexcept 그렇지 않으면 컴파일 에러가 난다. 표준이 강제하는 규칙이다.

awaitable / awaiter

co_await expr이 적힐 때, expr은 awaitable이어야 한다. awaitable로부터 awaiter가 만들어지고, 컴파일러는 awaiter의 세 멤버를 호출한다.

멤버호출 시점의미
await_ready()co_await 직후true면 suspend 안 함 (이미 결과 있음)
await_suspend(handle)suspend 직전핸들을 어딘가에 등록하고 나중에 재개시킨다
await_resume()재개 직후반환값이 co_await 표현식의 결과가 된다

표준이 제공하는 트리비얼 awaiter도 있다.

std::suspend_always sa;  // await_ready() == false. 항상 suspend
std::suspend_never  sn;  // await_ready() == true.  절대 suspend 안 함

간단한 generator 직접 만들기

다음은 co_yield로 값을 하나씩 넘겨주는 최소 Generator 구현이다. 핵심만 남기고 부수적인 부분은 생략했다.

#include <coroutine>
#include <utility>

template<typename T>
struct Generator
{
    struct promise_type
    {
        T value;

        Generator get_return_object()
        {
            return Generator{ handle::from_promise(*this) };
        }
        std::suspend_always initial_suspend()       { return {}; }
        std::suspend_always final_suspend() noexcept { return {}; }
        void unhandled_exception() { std::terminate(); }

        // co_yield v -> yield_value(v)
        std::suspend_always yield_value(T v)
        {
            value = std::move(v);
            return {};
        }
        void return_void() {}
    };

    using handle = std::coroutine_handle<promise_type>;
    handle h_;

    explicit Generator(handle h) : h_(h) {}
    ~Generator() { if (h_) h_.destroy(); }
    Generator(const Generator&) = delete;
    Generator(Generator&& o) noexcept : h_(std::exchange(o.h_, {})) {}

    bool next()      { h_.resume(); return !h_.done(); }
    const T& value() const { return h_.promise().value; }

    // ... 반복자 인터페이스는 생략
};

실제 사용은 다음과 같다.

Generator<int> range(int from, int to)
{
    for (int i = from; i < to; ++i)
        co_yield i;
}

int main()
{
    auto g = range(1, 5);
    while (g.next())
        std::cout << g.value() << ' ';
}
실행 결과
1 2 3 4

핵심 흐름은 이렇다. range() 호출은 코루틴 프레임을 만들고 initial_suspend()가 suspend_always를 반환하므로 본문 한 줄도 실행하지 않은 채 호출자에게 돌아온다. next()가 resume()을 부르면 co_yield i까지 실행되고 다시 suspend된다. done()이 true가 되면 끝이다.

co_await 사용 예

co_await은 비동기 작업을 동기 코드처럼 적게 해준다. 가상의 비동기 read를 기다리는 형태는 다음과 같다.

Task<std::string> FetchAsync(Url url)
{
    auto bytes = co_await Http::Get(url);   // suspend until done
    auto json  = co_await Parse(bytes);     // suspend again
    co_return json.AsString();
}
std::future에 co_await을 직접 못 쓴다 표준은 std::future에 awaiter 인터페이스를 정의하지 않았다. 따라서 co_await some_future는 그대로 컴파일되지 않는다. cppcoro, folly, Boost.Asio 같은 라이브러리를 쓰거나 직접 awaiter를 작성해야 한다.
임시 awaitable의 수명 co_await GetAwaitable()처럼 임시 객체를 await할 때, 그 임시 객체는 suspend 동안 살아있어야 한다. C++20 표준은 이 수명을 보장하지만, awaitable이 외부 자원의 참조를 들고 있을 경우 그 자원도 함께 살아있어야 한다.

C++23 std::generator

C++23부터 <generator> 헤더에 std::generator<T>가 추가됐다. 사용자 promise_type을 작성하지 않고 바로 쓸 수 있다.

#include <generator>

std::generator<int> fibonacci()
{
    int a = 0, b = 1;
    while (true)
    {
        co_yield a;
        std::tie(a, b) = std::pair{ b, a + b };
    }
}

int main()
{
    int i = 0;
    for (int v : fibonacci())
    {
        std::cout << v << ' ';
        if (++i == 8) break;
    }
}
실행 결과
0 1 1 2 3 5 8 13
코루틴을 처음 쓴다면 std::generator부터 직접 promise_type을 짜기 전에 std::generator로 lazy 시퀀스부터 익히자. 컴파일러 지원만 갖춰지면 가장 부담 없는 출발점이다.

언제 코루틴을 쓸까

적합한 용도는 분명하다.

  • 비동기 I/O — 네트워크, 파일. 콜백 지옥을 평탄한 코드로 바꾼다
  • 무한/lazy 시퀀스 — 피보나치, 무한 입력 스트림 등
  • 상태 머신 단순화 — 명시적 상태 변수 대신 코드 흐름으로 표현

반대로 단점도 분명하다.

  • 디버깅이 어렵다. 스택 트레이스가 suspend 지점에서 끊긴다
  • 컴파일러/표준 라이브러리 지원에 격차가 있다
  • 학습 곡선이 가파르다. promise_type, awaiter, 핸들 수명 모두 알아야 한다
  • 코루틴 프레임이 힙에 할당되어 메모리/할당 비용이 든다 (HALO 최적화 가능하지만 보장 X)

현실 가이드

표준 라이브러리만으로는 코루틴을 실용적으로 쓰기 어렵다. 현장에서는 보통 다음 셋 중 하나를 택한다.

선택지설명
cppcoroLewis Baker의 라이브러리. task, generator, async_mutex 등 제공
Boost.Asio비동기 네트워크 I/O. co_await 통합 잘 되어 있음
folly::coroFacebook의 코루틴 유틸리티. 대규모 코드베이스에서 검증됨
C++23 std::generator제너레이터 한정. 표준만으로 가능한 가장 현실적인 출발점

산업용 코드베이스에 도입할 때는 신중해야 한다. 도구 체인이 통일되어 있어야 하고, 디버거/스택 트레이스 도구가 코루틴을 다룰 수 있는지 먼저 확인할 가치가 있다.

흔한 함정

코루틴 프레임 누수 반환 타입이 코루틴 핸들을 소유하면서 소멸자에서 destroy()를 부르지 않으면 프레임이 그대로 누수된다. RAII로 핸들 수명을 반드시 묶어야 한다. final_suspend()가 suspend_always면 코루틴이 끝나도 자동으로 파괴되지 않는다는 점도 함께 기억하자.
dangling 참조 코루틴은 람다 캡처처럼 외부 변수를 참조할 수 있다. 그런데 코루틴이 suspend된 사이 그 외부 변수가 소멸하면 재개 후 dangling 참조에 접근하게 된다. 특히 코루틴 함수의 매개변수를 참조로 받지 말 것. 매개변수 참조의 수명은 호출자에서 관리되므로 suspend 시점을 넘기지 못할 가능성이 크다.
co_await의 임시 객체 수명 co_await foo()처럼 임시 awaitable을 await하면 그 임시는 suspend 동안 살아있어야 한다. 임시 안에 외부 참조가 있으면 그 외부 자원도 같은 기간 살아있어야 한다.
final_suspend는 noexcept 강제 final_suspend() noexcept가 아니면 컴파일이 통과하지 않는다. 코루틴 종료 시점에 예외를 다시 던질 길이 없기 때문이다.
