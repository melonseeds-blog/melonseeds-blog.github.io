RAII란?

Resource Acquisition Is Initialization의 약자다. "리소스 획득은 곧 초기화"라는 뜻으로, 객체의 생성자에서 리소스를 획득하고 소멸자에서 해제하는 패턴이다. Bjarne Stroustrup이 C++ 초기에 제안했고, 이제는 C++ Core Guidelines의 가장 핵심적인 원칙이 됐다.

이 패턴이 가능한 이유는 C++가 결정론적 소멸(deterministic destruction)을 보장하기 때문이다. 스택 객체는 스코프를 벗어나는 그 순간 반드시 소멸자가 호출된다. 예외가 발생해 스택이 풀려도(stack unwinding) 마찬가지다.

void Foo()
{
    std::vector<int> v(1000);  // 생성자: 메모리 할당
    DoSomething(v);             // 예외가 발생하더라도...
}                               // 소멸자: 메모리 자동 해제 (보장됨)
가비지 컬렉션 언어와의 차이 Java/C#은 GC가 비결정적 시점에 메모리만 회수한다. 파일/락/소켓 같은 비메모리 리소스는 try-finally나 using으로 따로 처리해야 한다. C++는 RAII 하나로 모든 리소스를 통일한다.

왜 RAII인가

수동 해제(new/delete, fopen/fclose) 코드의 문제점을 보자.

// 안티 패턴 — 수동 해제
ErrorCode ProcessFile(const char* path)
{
    FILE* fp = std::fopen(path, "rb");
    if (!fp) return ErrorCode::FileOpenFailed;

    char* buffer = new char[1024];

    if (std::fread(buffer, 1, 1024, fp) != 1024)
    {
        delete[] buffer;     // 잊기 쉬움
        std::fclose(fp);     // 잊기 쉬움
        return ErrorCode::ReadFailed;
    }

    DoSomething(buffer);     // 여기서 예외가 던져지면 누수!

    delete[] buffer;
    std::fclose(fp);
    return ErrorCode::Ok;
}

문제는 세 가지다.

  • 예외 누수 — DoSomething이 예외를 던지면 delete[]와 fclose가 호출되지 않는다.
  • 분기 누락 — early return, break, goto가 늘어날 때마다 해제 코드를 매번 복제해야 한다.
  • 가독성 저하 — 본질 로직보다 정리 코드가 더 길어진다. try/catch + finally 흉내는 코드를 더럽힌다.

RAII로 다시 쓰면 이렇다.

ErrorCode ProcessFile(const std::string& path)
{
    std::ifstream file(path, std::ios::binary);  // 생성자가 open
    if (!file) return ErrorCode::FileOpenFailed;

    std::vector<char> buffer(1024);              // 자동 메모리 관리
    file.read(buffer.data(), buffer.size());

    DoSomething(buffer);  // 예외? 상관없다. 둘 다 알아서 정리됨.
    return ErrorCode::Ok;
}                          // 스코프 끝 — file, buffer 모두 자동 해제

표준 라이브러리의 RAII 예

표준 라이브러리 자체가 RAII로 가득 차 있다.

도구관리 리소스비고
std::unique_ptr힙 메모리 (단독 소유)이동 전용, 오버헤드 0
std::shared_ptr힙 메모리 (공유 소유)참조 카운트 (원자적)
std::lock_guard뮤텍스가장 단순, 이동 불가
std::unique_lock뮤텍스지연 락/조건변수와 사용
std::scoped_lock (C++17)뮤텍스 여러 개데드락 방지 락 순서
std::ifstream / ofstream파일 핸들소멸자에서 close
std::threadOS 스레드join 안 하면 terminate (RAII 위반)
std::jthread (C++20)OS 스레드소멸자에서 자동 join + 협력 취소
std::vector / string힙 메모리모든 STL 컨테이너가 RAII
std::thread는 RAII 위반의 대표 사례 스레드 객체가 소멸될 때 join()이나 detach()가 호출되지 않은 상태면 std::terminate()로 프로그램이 즉시 종료된다. C++20부터는 std::jthread를 쓰자. 소멸자에서 자동 join을 보장한다. 
// 락 RAII — 가장 흔한 사용 예
std::mutex mtx;

void ThreadSafeIncrement(int& counter)
{
    std::lock_guard<std::mutex> lock(mtx);  // 생성자: lock
    ++counter;
}                                            // 소멸자: unlock (예외 안전)

사용자 정의 RAII 클래스 만들기

표준 라이브러리에 없는 리소스를 다룰 때는 직접 RAII 클래스를 만든다. 핵심 5가지를 모두 챙겨야 한다.

class FileHandle
{
public:
    // 1. 생성자에서 리소스 획득
    explicit FileHandle(const char* path, const char* mode)
        : m_fp(std::fopen(path, mode))
    {
        if (!m_fp)
        {
            throw std::runtime_error("file open failed");
        }
    }

    // 2. 소멸자에서 리소스 해제 (절대 throw 금지)
    ~FileHandle()
    {
        if (m_fp)
        {
            std::fclose(m_fp);
        }
    }

    // 3. 복사 금지 — 이중 close 방지
    FileHandle(const FileHandle&) = delete;
    FileHandle& operator=(const FileHandle&) = delete;

    // 4. 이동 지원 — 소유권 이전
    FileHandle(FileHandle&& other) noexcept
        : m_fp(other.m_fp)
    {
        other.m_fp = nullptr;
    }

    FileHandle& operator=(FileHandle&& other) noexcept
    {
        if (this != &other)
        {
            if (m_fp) std::fclose(m_fp);
            m_fp = other.m_fp;
            other.m_fp = nullptr;
        }
        return *this;
    }

    // 5. 원시 핸들 접근 (필요할 때만)
    FILE* Get() const { return m_fp; }

private:
    FILE* m_fp;
};
Rule of Zero / Rule of Five 가능하면 멤버 전체를 RAII 타입으로 만들어 특수 멤버 함수를 하나도 정의하지 않는 게 최선이다(Rule of Zero). 직접 리소스를 들고 있어야 한다면 5개(소멸자, 복사 생성/대입, 이동 생성/대입)를 모두 정의하거나 명시적으로 = delete한다.

C 스타일 리소스 RAII로 감싸기

산업 SDK는 거의 다 C API다. 카메라 핸들, GPU 메모리, OS 핸들 같은 것들을 RAII로 감싸야 한다.

빠른 방법: unique_ptr + custom deleter

전용 클래스를 만들기 귀찮을 때, std::unique_ptr의 두 번째 템플릿 인자에 deleter를 지정하면 한 줄로 RAII가 된다.

// FILE* 감싸기
auto fileDeleter = [](FILE* fp) { if (fp) std::fclose(fp); };
std::unique_ptr<FILE, decltype(fileDeleter)> file(
    std::fopen("data.bin", "rb"), fileDeleter);

// CUDA 메모리
auto cudaDeleter = [](void* p) { if (p) cudaFree(p); };
std::unique_ptr<float, decltype(cudaDeleter)> gpuBuf(
    [] { float* p; cudaMalloc(&p, 1024 * sizeof(float)); return p; }(),
    cudaDeleter);

// Windows HANDLE
auto handleDeleter = [](HANDLE h)
{
    if (h && h != INVALID_HANDLE_VALUE) CloseHandle(h);
};
std::unique_ptr<void, decltype(handleDeleter)> hFile(
    CreateFile(L"file.txt", GENERIC_READ, 0, nullptr,
               OPEN_EXISTING, FILE_ATTRIBUTE_NORMAL, nullptr),
    handleDeleter);

카메라 핸들 예 (산업 SDK 컨텍스트)

// Vieworks SDK 가정 — HCAM_OpenDevice / HCAM_Close
struct CameraDeleter
{
    void operator()(CameraHandle* h) const noexcept
    {
        if (h) HCAM_Close(h);
    }
};

using CameraPtr = std::unique_ptr<CameraHandle, CameraDeleter>;

CameraPtr OpenCamera(int deviceId)
{
    CameraHandle* raw = nullptr;
    if (HCAM_OpenDevice(deviceId, &raw) != HCAM_OK)
    {
        return nullptr;
    }
    return CameraPtr(raw);  // 자동 close 보장
}

scope guard 패턴

매번 클래스를 만들기엔 과한 일회성 정리 작업이 있다. 이때 scope guard를 쓴다. 람다를 들고 있다가 소멸자에서 호출하는 임시 RAII 객체다.

template<typename F>
class ScopeExit
{
public:
    explicit ScopeExit(F&& f) : m_func(std::move(f)), m_active(true) {}
    ~ScopeExit() { if (m_active) m_func(); }

    ScopeExit(const ScopeExit&) = delete;
    ScopeExit& operator=(const ScopeExit&) = delete;

    void Dismiss() noexcept { m_active = false; }

private:
    F m_func;
    bool m_active;
};

template<typename F>
ScopeExit<F> MakeScopeExit(F&& f)
{
    return ScopeExit<F>(std::forward<F>(f));
}

사용 예.

void GrabFrame(CameraHandle* cam)
{
    HCAM_StartAcquisition(cam);
    auto guard = MakeScopeExit([cam] { HCAM_StopAcquisition(cam); });

    // 이후 어떤 경로로 빠져나가든 StopAcquisition 보장
    if (!ReadFrame(cam)) return;
    Process(cam);
}
표준화 동향 Andrei Alexandrescu가 2000년에 처음 소개한 패턴이 표준 위원회까지 들어왔다. std::experimental::scope_exit(LFTS v3) 또는 Boost.Scope를 쓸 수 있다. 한 줄짜리 정리 작업에 매우 유용하다.

예외 안전성 보장 수준

Abrahams가 정리한 세 가지 보장 수준이 있다.

수준의미달성 방법
basic예외 발생 시 누수 없음, 객체는 유효하지만 상태는 불명RAII 사용 (출발점)
strong실패 시 호출 전 상태로 롤백 (트랜잭션 의미)copy-and-swap idiom
no-throw절대 예외를 던지지 않음noexcept 함수, swap, 소멸자

RAII는 basic 보장의 출발점이다. RAII 없이는 누수를 막을 수 없으므로 basic 보장조차 어렵다. strong 보장이 필요하면 copy-and-swap 패턴을 쓴다.

// copy-and-swap idiom — strong 보장
class Image
{
public:
    Image& operator=(Image other) noexcept  // 값으로 받음 → 복사 생성
    {
        Swap(*this, other);  // no-throw swap
        return *this;
    }                        // other 소멸자가 옛 리소스 정리

    friend void Swap(Image& a, Image& b) noexcept
    {
        using std::swap;
        swap(a.m_data, b.m_data);
        swap(a.m_width, b.m_width);
        swap(a.m_height, b.m_height);
    }
private:
    std::unique_ptr<uint8_t[]> m_data;
    int m_width;
    int m_height;
};

흔한 함정

소멸자에서 예외 던지기

C++11부터 소멸자는 기본적으로 noexcept(true)다. 소멸자에서 예외가 새어 나오면 std::terminate()가 호출돼 프로그램이 죽는다. 특히 stack unwinding 중에 또 예외가 던져지면 정의 동작이 보장되지 않는다.

소멸자에서 throw 금지 실패할 수 있는 정리 작업이 있다면, 별도 Close() / Commit() 메서드를 두어 사용자가 명시적으로 호출하게 한다. 소멸자는 fallback으로 동작하되 예외를 잡아서 로깅만 하고 끝낸다.

생성자 도중 예외

생성자가 도중에 예외를 던지면 해당 객체의 소멸자는 호출되지 않는다. 단, 이미 생성이 완료된 멤버들의 소멸자는 호출된다. 따라서 멤버 자체를 RAII 타입으로 두면 안전하다.

class Bad
{
    int* m_a;
    int* m_b;
public:
    Bad()
        : m_a(new int[100])
        , m_b(new int[100])  // 여기서 throw → m_a 누수!
    {}
    ~Bad() { delete[] m_a; delete[] m_b; }
};

class Good
{
    std::vector<int> m_a;
    std::vector<int> m_b;  // 여기서 throw → m_a는 자동 정리
public:
    Good() : m_a(100), m_b(100) {}
};

std::thread join 누락

스코프를 벗어나기 전에 반드시 join 또는 detach 잊으면 소멸자에서 terminate다. C++20에서는 std::jthread로 해결됐으니 가능하면 그걸 쓰자.

얕은 복사로 인한 이중 해제

RAII 클래스의 복사 정책을 명시하라 포인터 멤버를 단순 복사하면 두 객체가 같은 리소스를 가리키고, 둘 다 소멸자에서 해제하려다 이중 해제로 크래시 난다. 복사를 막거나(= delete), 깊은 복사를 구현하거나, shared_ptr로 공유하라.

정적/전역 객체의 소멸 순서

Static Initialization Order Fiasco 번역 단위가 다른 전역 객체끼리는 생성/소멸 순서가 정의되지 않는다. 전역 Logger를 전역 Camera 소멸자에서 호출하면 이미 죽은 Logger를 만질 수 있다. Meyers의 함수 정적 객체(Construct on First Use)로 해결하자.

class Image
{
public:
    Image(int width, int height, PixelFormat fmt);
    ~Image();  // CPU/GPU 메모리 모두 자동 해제

    Image(const Image&) = delete;             // 큰 이미지 실수 복사 방지
    Image& operator=(const Image&) = delete;
    Image(Image&&) noexcept;
    Image& operator=(Image&&) noexcept;

    ErrorCode EnsureCpu();
    ErrorCode EnsureGpu();
    ErrorCode Clone(Image& outImage) const;    // 명시적 복사

private:
    std::unique_ptr<uint8_t[]> m_cpuData;
    void* m_gpuData;  // CUDA 메모리 — 소멸자에서 cudaFree
    int m_width;
    int m_height;
};

CameraSession — Acquisition RAII

class CameraSession
{
public:
    explicit CameraSession(Camera& cam) : m_cam(cam)
    {
        m_cam.StartAcquisition();
    }
    ~CameraSession()
    {
        try { m_cam.StopAcquisition(); }
        catch (...) { /* 로깅만, swallow */ }
    }
    CameraSession(const CameraSession&) = delete;
    CameraSession& operator=(const CameraSession&) = delete;
private:
    Camera& m_cam;
};

// 사용 — 어떤 경로로 빠져나가도 StopAcquisition 보장
void Capture(Camera& cam)
{

RAII 체크리스트 새 리소스를 만질 때 자문하자. (1) 이 리소스를 들고 있는 객체의 소멸자는 자동으로 정리되는가? (2) 복사 정책은 명확한가? (3) 이동은 가능한가? (4) 생성 도중 실패해도 누수 없는가? 네 가지에 모두 "예"라고 답할 수 있으면 RAII가 제대로 적용된 것이다.
