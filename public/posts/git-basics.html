Git이란?

버전 관리가 필요한 이유

코드를 수정하다가 이전 상태로 돌아가고 싶을 때, 여러 사람이 동시에 같은 파일을 수정할 때, 어떤 변경이 언제 누구에 의해 이루어졌는지 추적할 때 — 이 모든 상황에서 버전 관리 시스템(VCS)이 필요하다.

Git의 핵심 개념

Git은 분산형 버전 관리 시스템(DVCS)이다. 각 개발자가 전체 히스토리를 로컬에 갖고 있어 네트워크 없이도 작업할 수 있다.

개념설명
Working Directory실제 파일이 있는 작업 폴더
Staging Area커밋할 변경사항을 준비하는 영역 (git add)
Repository커밋된 스냅샷이 저장되는 곳 (.git 폴더)
RemoteGitHub, GitLab 등 원격 저장소
Git vs GitHub Git은 버전 관리 도구이고, GitHub는 Git 저장소를 호스팅하는 서비스다. Git 없이 GitHub를 쓸 수 없지만, GitHub 없이 Git은 쓸 수 있다.

초기 설정

Git 설치 확인

$ git --version
git version 2.43.0.windows.1

설치가 안 되어 있다면 git-scm.com에서 다운로드한다.

사용자 정보 설정

커밋에 기록될 이름과 이메일을 설정한다. --global 옵션은 시스템 전체에 적용된다.

# 이름 설정
$ git config --global user.name "Hong Gildong"

# 이메일 설정
$ git config --global user.email "hong@example.com"

# 설정 확인
$ git config --list
user.name=Hong Gildong
user.email=hong@example.com
기본 편집기 변경 Git의 기본 편집기는 Vim이다. VS Code로 바꾸려면:
git config --global core.editor "code --wait"

기본 워크플로우

저장소 만들기

1

git init — 새 저장소 생성

프로젝트 폴더에서 Git 저장소를 초기화한다. .git 폴더가 생성된다.

$ mkdir my-project
$ cd my-project
$ git init
Initialized empty Git repository in /home/user/my-project/.git/
2

git clone — 기존 저장소 복제

이미 원격에 있는 저장소를 로컬로 가져온다.

$ git clone https://github.com/user/repo.git
Cloning into 'repo'...
done.

add, commit, push 흐름

Git의 기본 작업 흐름은 수정 → add → commit → push 4단계이다.

1

파일 수정 / 생성

코드를 작성하거나 수정한다.

$ echo "Hello Git" > hello.txt
2

git add — 스테이징

변경사항을 Staging Area에 올린다.

# 특정 파일 추가
$ git add hello.txt

# 모든 변경 파일 추가
$ git add .
3

git commit — 커밋

스테이징된 변경사항을 저장소에 기록한다.

$ git commit -m "feat: add hello.txt"
[main abc1234] feat: add hello.txt
1 file changed, 1 insertion(+)
4

git push — 원격에 업로드

로컬 커밋을 원격 저장소에 반영한다.

$ git push origin main
To https://github.com/user/repo.git
abc1234..def5678 main -> main

git add 전후 상태 비교

Before (git add 전)
$ git status
On branch main
Untracked files:
  (use "git add <file>..." to include)
        hello.txt

nothing added to commit
After (git add 후)
$ git status
On branch main
Changes to be committed:
  (use "git restore --staged <file>...")
        new file:   hello.txt
git add . 사용 시 주의 .env, node_modules/ 등 민감하거나 불필요한 파일이 포함될 수 있다. .gitignore 파일을 먼저 설정하자.

브랜치

브랜치 기본 명령어

브랜치는 독립적인 작업 공간이다. 기능 개발, 버그 수정 등을 별도 브랜치에서 진행하고, 완료 후 메인에 병합한다.

# 브랜치 목록 확인
$ git branch
* main
feature/login

# 새 브랜치 생성
$ git branch feature/signup

# 브랜치 이동
$ git checkout feature/signup
Switched to branch 'feature/signup'

# 생성 + 이동 (한 번에)
$ git checkout -b feature/signup

# 또는 최신 방식
$ git switch -c feature/signup

브랜치 병합 (Merge)

작업이 끝난 브랜치를 main에 병합한다.

# main 브랜치로 이동
$ git checkout main

# feature 브랜치 병합
$ git merge feature/signup
Merge made by the 'ort' strategy.
signup.html | 25 +++++++++++++++
1 file changed, 25 insertions(+)

# 병합 완료 후 브랜치 삭제
$ git branch -d feature/signup
Deleted branch feature/signup (was abc1234).

충돌 해결

같은 파일의 같은 부분을 두 브랜치에서 수정하면 충돌(conflict)이 발생한다.

# 충돌 발생 시 파일 내용
<<<<<<< HEAD
현재 브랜치의 내용
=======
병합하려는 브랜치의 내용
>>>>>>> feature/signup
1

충돌 파일 열기

<<<<<<<, =======, >>>>>>> 마커를 찾아 원하는 내용으로 수정한다.

2

수정 후 커밋

$ git add .
$ git commit -m "fix: resolve merge conflict"
VS Code에서 충돌 해결 VS Code는 충돌이 발생한 부분을 시각적으로 표시해준다. "Accept Current Change", "Accept Incoming Change", "Accept Both Changes" 버튼으로 쉽게 해결할 수 있다.

자주 쓰는 명령어

상태 확인

# 현재 상태 확인
$ git status

# 커밋 히스토리 (한 줄씩)
$ git log --oneline
def5678 feat: add hello.txt
abc1234 initial commit

# 변경사항 확인
$ git diff

되돌리기

# 스테이징 취소 (add 취소)
$ git restore --staged hello.txt

# 파일 수정 취소 (워킹 디렉토리 복원)
$ git restore hello.txt

# 마지막 커밋 메시지 수정
$ git commit --amend -m "fix: correct message"

# 특정 커밋으로 되돌리기 (이력 유지)
$ git revert abc1234
git reset --hard는 신중하게! git reset --hard는 변경사항을 완전히 삭제한다. 복구가 어려우므로 git revert를 먼저 고려하자.

원격 저장소

# 원격 저장소 연결
$ git remote add origin https://github.com/user/repo.git

# 원격 목록 확인
$ git remote -v
origin https://github.com/user/repo.git (fetch)
origin https://github.com/user/repo.git (push)

# 원격 변경사항 가져오기
$ git pull origin main

# fetch + merge 분리
$ git fetch origin
$ git merge origin/main

.gitignore 설정

추적하지 않을 파일/폴더를 지정한다. 프로젝트 루트에 .gitignore 파일을 만든다.

# .gitignore 예시

# 환경 변수
.env
.env.local

# 의존성 폴더
node_modules/
vendor/

# 빌드 결과물
dist/
build/
*.exe
*.o

# OS 파일
.DS_Store
Thumbs.db

# IDE 설정
.vscode/
.idea/
