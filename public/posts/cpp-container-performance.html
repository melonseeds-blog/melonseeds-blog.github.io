시퀀스 컨테이너 메모리 레이아웃

컨테이너의 성능은 결국 메모리 어디에 어떻게 놓이는가로 결정된다. 같은 O(n) 순회라도 메모리 레이아웃에 따라 실측 성능은 10배 이상 갈린다. 먼저 각 컨테이너가 어떻게 데이터를 저장하는지 그림으로 살펴본다.

vector — 단일 연속 버퍼

힙에 단 하나의 연속된 배열을 잡고, 용량(capacity)이 부족하면 더 큰 버퍼를 새로 할당해 옮긴다. 이 재할당이 일어날 때마다 모든 요소가 복사·이동되지만, 1.5배 또는 2배씩 늘리는 정책 덕분에 push_back은 amortized O(1)이 된다.

vector<int> v = {10, 20, 30, 40}; // capacity=8 힙: ┌──┬──┬──┬──┬──┬──┬──┬──┐ │10│20│30│40│ │ │ │ │ └──┴──┴──┴──┴──┴──┴──┴──┘ └─── size=4 ───┘└capacity 여유 4┘ ↑ data() 시작 주소 — 모두 인접

deque — 청크(블록) 단위

고정 크기 청크 여러 개를 가리키는 포인터 배열(map)을 가진다. 양 끝에 공간이 부족하면 새 청크만 할당하면 되므로 push_front / push_back 모두 O(1)이다. 다만 임의 위치 삽입은 청크 경계까지 밀어야 하므로 비싸다.

deque<int> — 청크 크기 4 가정 포인터 맵: ┌──────┬──────┬──────┬──────┐ │ ptr0 │ ptr1 │ ptr2 │ ptr3 │ └──┬───┴──┬───┴──┬───┴──────┘ ↓ ↓ ↓ [_,_, [10,20, [40,50, _,5] 30,_] _,_] 청크A 청크B 청크C (앞) (중간) (뒤)

list — 이중 연결 리스트

각 노드를 힙에 따로 할당한다. 노드는 {prev, value, next}를 가지므로 int 하나를 저장하는 데도 포인터 2개(64비트에서 16바이트)가 추가로 든다. 노드끼리의 메모리 위치는 전혀 인접하지 않다.

list<int> l = {10, 20, 30}; // 노드는 힙에 산재 주소 0x1000: [prev=null │ 10 │ next=0x4080] 주소 0x4080: [prev=0x1000 │ 20 │ next=0x2A10] 주소 0x2A10: [prev=0x4080 │ 30 │ next=null] 각 노드는 메모리 어디에든 흩어져 있음 (캐시 비친화)

forward_list — 단일 연결 리스트

다음 노드 포인터만 가진다. list 대비 노드당 8바이트(64비트)를 절약한다. 역방향 순회와 size()가 없어 메모리에 민감한 임베디드 환경 정도에서나 의미 있다.

array — 컴파일 타임 고정 크기

크기가 템플릿 파라미터로 박혀 있고 힙 할당이 전혀 없다. 스택에 통째로 올릴 수 있어 가장 빠르다. 단, 크기가 컴파일 타임에 결정되어야 한다.

std::array<int, 4> a = {10, 20, 30, 40}; 스택 프레임: ┌──┬──┬──┬──┐ │10│20│30│40│ ← 함수 호출 시 자동 정리 └──┴──┴──┴──┘ 힙 할당 0회, 메타데이터 0바이트

캐시 친화도 — 왜 vector가 빠른가

현대 CPU는 메모리에서 1바이트만 가져오지 않는다. 한 번에 캐시 라인(보통 64바이트) 단위로 읽어온다. 게다가 하드웨어 prefetcher가 "다음 캐시 라인도 어차피 읽겠지"라고 예측해 미리 가져와 둔다.

캐시 라인과 prefetch

vector는 모든 요소가 인접해 있다. v[0]을 읽을 때 v[1]~v[15](int 기준)가 같이 캐시에 올라온다. 다음 반복에서 v[1]은 이미 캐시 안에 있다. L1 hit는 ~1ns, 메인 메모리 접근은 ~100ns. 100배 차이다.

list는 노드가 메모리에 흩어져 있다. 각 노드 접근마다 캐시 미스가 발생할 수 있고, prefetcher는 다음 노드 주소를 예측할 수 없다. 결과적으로 같은 O(n) 순회라도 list가 vector보다 5~20배 느린 경우가 흔하다.

cache miss는 복잡도 표에 없는 비용이다 big-O 분석은 모든 메모리 접근 비용을 동일하게 본다. 실제로는 L1 hit ↔ DRAM 접근의 차이가 100배가 넘는다. "list가 중간 삽입 O(1)이라 vector보다 빠를 것"이라는 추론이 자주 빗나가는 이유다.

sequential vs random access

같은 양의 데이터를 읽더라도 접근 패턴에 따라 처리량이 크게 갈린다.

접근 패턴예시특징
Sequentialvector 순회prefetch 효과 최대, 거의 L1 속도
Strided2차원 배열 열 방향stride가 캐시 라인보다 크면 미스 폭증
Random / pointer chasinglist 순회매 접근마다 미스 가능성, prefetch 무력

실무 벤치마크 시나리오

아래는 실제 코드와 의사 측정 결과다. 수치는 환경에 따라 달라지므로 절대값보다 상대 비율에 주목한다.

시나리오 1 — 100만 개 정수 push_back

// 핵심 루프 (의사 코드)
constexpr int kCount = 1'000'000;

vector<int> v;
v.reserve(kCount);                  // 재할당 0회
for (int i = 0; i < kCount; ++i) v.push_back(i);

deque<int> dq;
for (int i = 0; i < kCount; ++i) dq.push_back(i);

list<int> lst;
for (int i = 0; i < kCount; ++i) lst.push_back(i);
측정 결과 (대략적, MSVC release x64)
vector (reserve) : 2 ms vector (no reserve) : 8 ms ← 재할당 ~20회 deque : 18 ms list : 62 ms ← 노드 100만 개 힙 할당

reserve() 한 줄이 4배 빨라지게 만든다. list가 느린 결정적 이유는 매 push_back이 힙 할당이라는 점이다.

시나리오 2 — 중간 삽입 1만 번 (vector vs list 역전 지점)

"중간 삽입은 list가 빠르다"는 통념을 깨는 부분이다. list는 삽입 자체는 O(1)이지만 위치를 찾는 비용이 크고, 캐시 미스로 순회 자체가 느리다.

// 정렬 상태 유지하며 무작위 값 삽입
template <typename Container>
void insertSorted(Container& c, int value)
{
    auto it = std::lower_bound(c.begin(), c.end(), value);  // list는 random access 불가 → 선형
    c.insert(it, value);
}
측정 결과 — N개 컨테이너에 1만 회 삽입
N=100 vector: 0.4 ms list: 1.8 ms N=1,000 vector: 3 ms list: 9 ms N=10,000 vector: 38 ms list: 96 ms N=100,000 vector: 410 ms list: 1100 ms

대부분의 실무 N(수만 이하)에서는 vector가 list보다 일관되게 빠르다. 캐시 친화적인 메모리 이동(memmove)이 포인터 추적보다 싸기 때문이다.

list가 이기는 조건은 좁다 (1) 삽입 위치 iterator를 이미 가지고 있고, (2) 요소 타입이 매우 크고(복사 비용 ↑), (3) 컨테이너가 아주 클 때. 이 셋을 동시에 만족하지 못하면 vector를 쓰자.

시나리오 3 — 순회 후 합산

long long sum = 0;
for (int x : v)   sum += x;   // vector
for (int x : lst) sum += x;   // list, 동일 요소 수
N = 1,000,000 합산
vector : 0.6 ms list : 6.4 ms ← 약 10배, 전부 캐시 미스 비용

small vector 최적화

std::vector는 항상 힙에 데이터 버퍼를 잡는다. 컨테이너에 보통 몇 개밖에 안 들어가는 경우(예: 함수 인자 목록, 짧은 경로) 매번 힙 할당이 낭비다.

small vector(small buffer optimization, SBO)는 객체 안에 작은 인라인 버퍼를 두고, 용량이 그걸 넘을 때만 힙으로 빠진다. llvm::SmallVector, boost::container::small_vector, folly::small_vector가 대표적이다.

// 개념적 구조
template <typename T, size_t N>
class small_vector
{
    union
    {
        T m_inline[N];     // 인라인 버퍼 (스택 또는 객체 내부)
        T* m_heap;         // 힙으로 넘어간 후의 포인터
    };
    size_t m_size;
    size_t m_capacity;
};

// 사용 예 — 보통 4개 이하인 경우 힙 할당 0회
small_vector<int, 4> sv;
sv.push_back(1);
sv.push_back(2);          // 아직 인라인 버퍼
sv.push_back(3);
sv.push_back(4);
sv.push_back(5);          // 이때 처음 힙으로 넘어감
std::string은 이미 SBO를 한다 대부분의 표준 라이브러리는 짧은 문자열을 객체 안에 직접 저장한다(SSO). short string은 힙 할당이 일어나지 않는다. std::vector는 표준이 SBO를 강제하지 않아 직접 안 한다.

연관 컨테이너 — map / unordered_map / flat_map

구분 std::map std::unordered_map std::flat_map (C++23)
내부 구조Red-Black TreeHash Table + 버킷정렬된 두 vector
메모리 레이아웃노드별 힙 할당버킷 + 노드 힙 할당연속 버퍼 2개
탐색O(log n)O(1) 평균, O(n) 최악O(log n)

flat_map은 자주 조회하지만 거의 변하지 않는 데이터에 이상적이다. 룩업 테이블, 설정값 맵, 작은 사전 등이 후보다. 삽입이 비싸도 한번 채워두면 그 뒤의 조회·순회가 캐시 덕에 매우 빠르다.

set 계열도 동일 set / unordered_set / flat_set이 같은 트레이드오프로 짝을 이룬다. 키만 필요하면 set 계열, 키-값이 필요하면 map 계열.

reserve / shrink_to_fit 활용 팁

reserve — 재할당 제거

최종 크기를 알거나 추정 가능하면 반드시 미리 잡는다. 시나리오 1에서 봤듯 4배까지 빨라진다.

// 안 좋은 예 — 재할당이 log2(n)번 일어남
vector<Pixel> pixels;
for (int i = 0; i < width * height; ++i)
    pixels.push_back(loadPixel(i));

// 좋은 예 — 단 한 번 할당
vector<Pixel> pixels;
pixels.reserve(width * height);
for (int i = 0; i < width * height; ++i)
    pixels.push_back(loadPixel(i));
reserve와 resize는 다르다 reserve(n)은 capacity만 늘리고 size는 그대로다. resize(n)은 size 자체를 n으로 만들고 기본 생성된 요소로 채운다. push_back과 함께 쓸 때는 reserve가 맞다.

shrink_to_fit — 메모리 회수

벡터는 capacity를 자동으로 줄여주지 않는다. 큰 데이터를 처리한 뒤 size가 작아져도 capacity는 그대로다. 명시적으로 회수해야 한다.

vector<double> buffer;
buffer.reserve(10'000'000);
// ... 대량 처리 ...
buffer.clear();              // size=0이지만 capacity는 그대로
buffer.shrink_to_fit();      // 가능하면 capacity도 0으로

// 강제로 비우는 관용구 (swap trick, C++03 시절)
vector<double>().swap(buffer);
shrink_to_fit는 강제가 아니다 표준은 "비구속적 요청(non-binding request)"이라고 명시한다. 구현이 무시할 수 있다. 정말 메모리를 해제하고 싶다면 swap 관용구를 쓰는 편이 확실하다.

iterator 무효화 규칙

재할당이 일어나면 vector의 모든 iterator·포인터·참조가 무효화된다. 반복 중 push_back은 위험하다.

컨테이너insert 시 무효화erase 시 무효화
vector재할당 시 전부지운 위치 이후 전부
deque모든 iterator (참조는 살아남는 경우 있음)모든 iterator
list / forward_list없음지운 노드 자신만

iterator·참조 안정성이 절대적으로 필요하면 list를 고려할 만한 거의 유일한 이유가 된다.
