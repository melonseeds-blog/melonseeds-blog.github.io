이 문서의 목적 이미지 획득은 머신비전의 시작이지만, 실무에서는 후단 알고리즘 품질을 결정하는 가장 중요한 단계이기도 합니다. operator 사용법보다 설계 판단·튜닝·실패 원인·디버깅 관점에서 정리한 문서입니다.

개요

이미지 획득은 모든 머신비전 시스템의 시작점입니다. HALCON에서는 다양한 카메라, frame grabber, image acquisition interface를 operator 호출로 추상화합니다.

하지만 실무에서는 "이미지를 가져온다"가 전부가 아닙니다. 이미지 획득 단계에서 이미 다음 요소가 결정됩니다.

  • 해상도
  • 노출 시간
  • frame rate
  • trigger timing
  • motion blur
  • noise
  • dynamic range
  • synchronization
  • multi-camera timing
  • buffer 지연

후단 알고리즘이 아무리 좋아도 acquisition이 불안정하면 검사 시스템은 안정화되지 않습니다.

기본 구조

HALCON의 이미지 획득 기본 흐름은 다음입니다.

open_framegrabber
→ grab_image 또는 grab_image_async
→ close_framegrabber

2.1 장치 열기

장치 열기 단계에서는 acquisition interface, device, port, camera type, color space, image size 등을 설정합니다.

실무적으로 중요한 항목:

  • interface 이름
  • device index 또는 serial number
  • camera file 또는 GenICam parameter
  • color/mono 설정
  • field/frame 설정
  • external trigger 사용 여부
  • image width/height/start position
  • bits per pixel
주의점 멀티 카메라 시스템에서는 "몇 번째 장치"로 열기보다 serial number 기반으로 명시하는 것이 안전합니다. OS나 드라이버 인식 순서가 바뀌면 index 기반 연결은 다른 카메라를 잡을 수 있습니다.

2.2 이미지 획득

HALCON에서 대표적인 획득 방식은 다음입니다.

방식 의미 특징
grab_image 동기 획득 호출 후 이미지가 들어올 때까지 대기
grab_image_async 비동기 획득 이전/현재 acquisition과 병렬 처리 가능
continuous grabbing 연속 취득 streaming 기반 처리
triggered grabbing 외부 trigger 기반 라인/장비 sync에 적합

grab_image와 grab_image_async

3.1 grab_image

동기 방식입니다. 코드 흐름은 단순하지만 acquisition 동안 프로그램이 대기합니다.

적합한 경우:

  • 속도 요구가 낮음
  • 테스트/개발 단계
  • 단일 이미지 처리
  • trigger 없는 수동 검사

한계:

  • 처리 시간과 acquisition 시간이 직렬화됨
  • frame rate가 높은 시스템에서 병목
  • trigger timing과 processing timing이 섞여 분석 어려움

3.2 grab_image_async

비동기 방식입니다. 카메라가 다음 이미지를 취득하는 동안 CPU는 이전 이미지를 처리할 수 있습니다.

start async acquisition
while running:
    get acquired image
    process previous image
    request next image

장점:

  • throughput 증가
  • processing/acquisition overlap
  • real-time 시스템에 적합

주의점:

  • buffer 관리 필요
  • 오래된 frame을 처리할 위험
  • trigger와 timestamp 확인 필요
  • timeout 처리 필요
  • 처리 시간이 acquisition 주기보다 길면 frame drop 또는 queue delay 발생
실무 포인트 비동기 구조에서 가장 중요한 질문은 "현재 처리하는 이미지가 언제 촬영된 이미지인가?"입니다. 검사 시스템에서는 frame timestamp, trigger ID, encoder position 등을 함께 관리하는 것이 좋습니다.

카메라 파라미터

4.1 Exposure

노출 시간은 밝기와 motion blur를 동시에 결정합니다.

  • 길면 밝아지지만 blur 증가
  • 짧으면 blur 감소하지만 조명 세기 필요
  • moving object에서는 exposure가 검사 정밀도에 직접 영향

실무 기준:

허용 blur[pixel] = object speed[pixel/ms] × exposure[ms]

정밀 측정에서는 blur가 1 pixel 이하, 가능하면 subpixel 수준으로 관리되어야 합니다.

4.2 Gain

Gain은 신호를 증폭하지만 noise도 함께 증폭합니다.

  • gain 증가 → 밝기 증가
  • gain 증가 → random noise 증가
  • threshold/edge stability 악화 가능

원칙:

  1. 조명 세기를 먼저 확보
  2. exposure를 motion 조건 내에서 조정
  3. gain은 마지막 보정 수단으로 사용

4.3 Trigger

Trigger는 장비와 카메라 촬영 타이밍을 맞추는 핵심입니다.

유형:

  • free-run
  • software trigger
  • hardware trigger
  • encoder trigger
  • strobe synchronization
Trigger 실무 체크리스트
  • trigger polarity
  • trigger delay
  • exposure start 기준
  • strobe on timing
  • camera ready signal
  • missed trigger handling
  • debouncing
  • multi-camera sync

4.4 Frame Rate와 Throughput

카메라 frame rate는 단순히 초당 촬영 수가 아닙니다. 전체 시스템 throughput은 다음 중 가장 느린 요소가 결정합니다.

exposure time
+ readout time
+ transfer time
+ processing time
+ communication / IO time

GigE/USB 카메라는 대역폭 제한도 고려해야 합니다.

Multi-Camera Acquisition

멀티 카메라에서는 각 카메라 handle을 독립적으로 관리합니다.

중요한 설계 포인트:

  • 카메라별 serial mapping
  • trigger 동기화
  • exposure/gain parameter 동기화
  • timestamp 기준 정렬
  • image buffer 개별 관리
  • bandwidth 분산

실패 사례

증상 원인
특정 카메라 frame drop USB/GigE bandwidth 부족
좌우 카메라 이미지 시간 차 trigger sync 불량
장치 순서 바뀜 index 기반 open
가끔 timeout trigger missing 또는 buffer overflow

외부 장치와의 연동

실제 장비에서는 카메라만 존재하지 않습니다.

  • PLC
  • motion controller
  • robot
  • encoder
  • strobe controller
  • IO board
  • frame grabber

HALCON operator로 이미지를 가져오더라도, 시스템 설계에서는 trigger source와 결과 output timing을 함께 고려해야 합니다.

Unsupported Device 처리

HALCON interface가 직접 지원하지 않는 장치라도 다음 방식으로 처리할 수 있습니다.

  • 외부 SDK로 buffer 획득 후 HALCON image 생성
  • memory pointer 기반 image generation
  • file 기반 임시 저장 후 read_image
  • custom acquisition wrapper 작성

이 경우 중요한 것은 memory ownership입니다.

  • buffer lifetime
  • copy 여부
  • stride/pitch
  • pixel format
  • channel order
  • thread safety

Acquisition 단계 디버깅

이미지 처리 알고리즘 문제처럼 보여도 실제 원인은 acquisition인 경우가 많습니다.

확인 순서

  1. 원본 image 저장
  2. histogram 확인
  3. saturated pixel 비율 확인
  4. focus/blur 확인
  5. trigger timing 확인
  6. frame drop 여부 확인
  7. gain/noise 확인
  8. 조명 균일도 확인

저장해야 할 로그

  • frame ID
  • timestamp
  • exposure
  • gain
  • trigger ID
  • camera temperature
  • acquisition status
  • processing time
  • result

핵심 정리

  • Acquisition은 후처리 이전의 품질을 결정합니다.
  • grab_image_async는 속도 개선에 유리하지만 frame freshness 관리가 필수입니다.
  • exposure/gain/trigger는 알고리즘 파라미터보다 먼저 안정화해야 합니다.
  • 멀티 카메라에서는 serial mapping과 timestamp 관리가 중요합니다.
  • 원본 이미지와 acquisition metadata를 함께 저장해야 디버깅이 가능합니다.
