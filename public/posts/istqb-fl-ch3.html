3.1 정적 테스팅 기초

정적 테스팅(Static Testing)은 소프트웨어를 실행하지 않고 코드, 명세, 아키텍처 등의 작업 산출물을 검토하는 테스팅 활동이다. 수동(리뷰)이나 도구(정적 분석)를 사용해 수행하며, 품질 개선과 결함 식별이 목적이다.

정적 테스팅은 베리피케이션(Verification)과 밸리데이션(Validation) 모두에 적용할 수 있다. SDLC 초기 단계부터 적용 가능하므로 조기 테스팅 원리를 실현하는 핵심 수단이다.

정적 분석(Static Analysis)이란? 테스트 케이스 없이 도구를 사용해 코드나 문서를 분석하는 기법이다. CI(지속적 통합) 프레임워크에 통합하는 경우가 많으며, 코드 결함 식별뿐 아니라 유지보수성과 보안 평가에도 활용한다. 맞춤법 검사기, 가독성 도구도 정적 분석의 예이다.

3.1.1 검사 가능한 작업 산출물 (FL-3.1.1)

정적 테스팅으로 거의 모든 작업 산출물을 검토할 수 있다. 대표적인 예시는 다음과 같다:

  • 요구사항 명세서 (기능/비기능 요구사항)
  • 소스 코드
  • 테스트 계획서, 테스트 케이스, 테스트 차터
  • 제품 백로그 항목, 사용자 스토리
  • 프로젝트 문서, 계약서, 모델

읽고 이해할 수 있는 모든 작업 산출물은 리뷰 대상이 될 수 있다. 단, 정적 분석의 경우 검토 기준이 되는 구조(모델, 공식 문법 등)가 필요하다.

정적 테스팅에 적합하지 않은 산출물 사람이 해석하기 어려운 것이나 도구로 분석해서는 안 되는 산출물(예: 법적으로 문제가 되는 타사의 실행 코드)은 정적 테스팅에 적합하지 않다.

3.1.2 정적 테스팅의 가치 (FL-3.1.2)

  • SDLC 초기 단계에서 결함을 식별 (조기 테스팅 원리 실현)
  • 동적 테스팅으로 식별할 수 없는 결함 발견 (도달 불가능한 코드, 설계 패턴 결함 등)
  • 작업 산출물의 품질 평가와 신뢰 구축
  • 이해관계자 간 공통된 이해 형성 및 의사소통 개선
  • 프로젝트 후반 결함 수정 비용 절감으로 전체 프로젝트 비용 감소
  • 일부 코드 결함은 동적 테스팅보다 정적 분석이 더 효율적
시험 포인트: 정적 테스팅의 이점 "SDLC 후반에 결함을 더 많이 식별하므로 비용이 적게 든다"는 오답이다. 정적 테스팅은 초기에 결함을 식별해서 비용을 줄이는 것이 핵심이다.

3.1.3 정적 테스팅 vs 동적 테스팅 비교

정적 테스팅과 동적 테스팅은 서로를 보완하는 관계이다. 둘 다 결함 식별이 목적이지만, 다음과 같은 차이가 있다:

비교 항목 정적 테스팅 (Static Testing) 동적 테스팅 (Dynamic Testing)
소프트웨어 실행 실행하지 않음 반드시 실행함
결함 식별 방식 결함을 직접 식별 장애(Failure)를 일으킨 후 연관 결함을 후속 분석으로 찾음
적용 대상 실행 가능 + 비실행 작업 산출물 모두 실행 가능한 작업 산출물만
도달하기 어려운 코드 거의 실행되지 않는 코드 경로의 결함을 더 쉽게 발견 도달하기 어려운 코드 결함 발견이 어려움
측정 품질 특성 코드 실행에 의존하지 않는 특성 (예: 유지보수성) 코드 실행에 의존적인 특성 (예: 성능 효율성)
대표 활동 리뷰, 정적 분석 단위 테스트, 통합 테스트, 시스템 테스트

정적 테스팅으로 더 쉽게 찾을 수 있는 결함

  • 요구사항 결함: 불일치, 모호성, 모순, 누락, 부정확성, 중복
  • 설계 결함: 비효율적인 데이터베이스 구조, 모듈화 불량
  • 코딩 결함: 정의되지 않은 값, 선언되지 않은 변수, 도달 불가능한 코드, 중복 코드, 과도한 복잡성
  • 표준 위반: 코딩 표준 명명 규칙 미준수
  • 잘못된 인터페이스 명세: 매개변수의 개수, 유형, 순서 불일치
  • 보안 취약성: 버퍼 오버플로우 등
  • 테스트 베이시스 커버리지 차이: 인수 조건에 대한 테스트 누락

3.2 피드백과 리뷰 프로세스

3.2.1 조기 피드백의 이점 (FL-3.2.1)

이해관계자의 피드백을 조기에 자주 받으면 다음과 같은 이점이 있다:

  • 잠재적인 품질 문제를 조기에 파악
  • 요구사항에 대한 오해 방지
  • 요구사항 변경을 조기에 이해하고 구현
  • 이해관계자에게 가장 중요하고, 리스크에 가장 긍정적인 영향을 미치는 기능에 집중 가능
피드백이 부족하면? 이해관계자 참여가 적으면 큰 비용이 드는 재작업, 납기일 지연, 심한 경우 프로젝트 완전 실패가 발생할 수 있다.

3.2.2 리뷰 프로세스 활동 (FL-3.2.2)

ISO/IEC 20246 표준에 기반한 리뷰 프로세스는 다음 5가지 활동으로 구성된다:

1
계획 (Planning)
리뷰 범위 정의: 목적, 리뷰 대상 작업 산출물, 평가할 품질 특성, 집중할 영역, 완료 조건, 적용 표준, 공수, 리뷰 일정 등
2
리뷰 착수 (Review Initiation)
모든 참여자가 작업 산출물에 접근 가능한지, 역할과 책임을 이해하는지, 필요한 것들을 받았는지 확인
3
개별 리뷰 (Individual Review)
리뷰어가 리뷰 기법(체크리스트 기반, 시나리오 기반 등)을 적용해 작업 산출물을 평가하고 이상 사항, 권장 사항, 의문 사항을 식별/기록
4
논의 및 분석 (Issue Communication & Analysis)
이상 사항에 대해 분석/논의하여 상태, 담당자, 필요 조치를 판단. 일반적으로 리뷰 회의에서 진행. 후속 리뷰가 필요할 수 있음
5
수정 및 보고 (Fixing & Reporting)
결함 보고서를 작성해 후속 조치 추적. 완료 조건 충족 시 작업 산출물 승인. 리뷰 결과 보고
시험 포인트: 리뷰 프로세스 순서 계획 → 리뷰 착수 → 개별 리뷰 → 논의 및 분석 → 수정 및 보고 순서를 반드시 기억하자. 작업 설명과 활동을 매칭하는 문제가 자주 출제된다.

3.2.3 리뷰 역할과 책임 (FL-3.2.3)

역할 책임
관리자 (Manager) 리뷰할 내용을 결정하고, 리뷰에 필요한 사람과 시간 등 자원을 제공한다
저자 (Author) 리뷰 대상 작업 산출물을 작성하고 수정한다
중재자 (Moderator / Facilitator) 리뷰 회의의 효과적인 운영 담당: 중재, 시간 관리, 안전한 리뷰 환경 조성
리뷰 리더 (Review Leader) 리뷰 참여자 결정, 시간/장소 협의 등 리뷰에 대한 전반적인 책임
검토자 (Reviewer) 리뷰를 수행한다. 프로젝트 참여자, 주제 전문가, 기타 이해관계자가 될 수 있음
서기 (Scribe / Recorder) 이상 사항을 수집하고 결정 사항, 새로운 이상 사항 등 리뷰 정보를 기록한다
시험 포인트: 역할 구분 관리자 = "리뷰 대상 결정 + 자원 제공", 리뷰 리더 = "전반적 책임 + 참여자 결정", 중재자 = "회의 운영 + 중재". 이 세 역할의 차이가 자주 출제된다.

3.2.4 리뷰 유형 비교

리뷰 유형은 공식성 수준에 따라 4가지로 구분된다. 같은 작업 산출물도 처음에 비공식적으로, 나중에 공식적으로 리뷰할 수 있다.

유형 공식성 주요 목적 진행자 역할 특징
비공식 리뷰
(Informal Review)		 가장 낮음 이상 사항 식별 없거나 자유 정의된 프로세스 없음, 공식 결과 문서 불요
워크쓰루
(Walkthrough)		 낮음~중간 품질 평가, 신뢰 구축, 리뷰어 교육, 합의 도출, 이상 사항 발견 저자가 리더 개별 리뷰는 선택사항 (필수가 아님)
기술 리뷰
(Technical Review)		 중간~높음 기술 문제 합의, 결정, 이상 사항 식별, 품질 평가 중재자가 리더 기술 자격 갖춘 리뷰어가 수행
인스펙션
(Inspection)		 가장 높음 이상 사항 최대한 많이 찾기, 품질 평가, 메트릭 수집 훈련된 중재자 프로세스를 철저히 따름, 저자가 리뷰 리더나 서기가 될 수 없음, 메트릭 수집하여 SDLC 개선
시험 포인트: 워크쓰루 vs 기술 리뷰 vs 인스펙션 워크쓰루는 저자가 리더, 기술 리뷰는 중재자가 리더이다. 인스펙션은 가장 공식적이며 저자가 리뷰 리더/서기 역할을 맡을 수 없다는 점이 핵심이다. 또한 인스펙션만 메트릭(metrics) 수집을 명시적으로 수행한다.

3.2.5 리뷰의 성공 요소 (FL-3.2.5)

  • 명확한 목표와 측정 가능한 완료 조건 정의 (참가자 평가 목적은 절대 안 됨)
  • 목표 달성에 적합한 리뷰 유형 선택
  • 집중력 유지를 위한 작은 단위의 리뷰
  • 이해관계자와 저자에게 피드백 제공
  • 참가자에게 충분한 준비 시간 제공
  • 관리층의 지원
  • 리뷰가 조직 문화의 일부가 되도록 함
  • 참가자에게 역할 수행을 위한 적절한 교육 제공
  • 회의에 퍼실리테이션(facilitation) 적용
시험 주의: "장애" vs "결함" 리뷰에서는 장애(Failure)가 아닌 결함(Defect)을 찾게 된다. 소프트웨어를 실행하지 않으므로 장애가 발생하지 않기 때문이다. "식별한 장애를 인정하고 환영해야 한다"는 오답이다 (장애가 아니라 결함이 맞다).

핵심 용어 정리

Ch3 정적 테스팅에서 반드시 알아야 할 핵심 용어들이다.

Static Testing
정적 테스팅 - 소프트웨어를 실행하지 않고 작업 산출물을 평가하는 테스팅
Dynamic Testing
동적 테스팅 - 소프트웨어를 실행하여 수행하는 테스팅
Static Analysis
정적 분석 - 도구를 사용해 코드/문서를 실행 없이 분석하는 기법
Review
리뷰 - 작업 산출물을 수동으로 검토하는 정적 테스팅 활동
Anomaly
이상 사항 - 리뷰 중 식별된 결함, 권장 사항, 의문 사항 등
Informal Review
비공식 리뷰 - 정의된 프로세스 없이 수행하는 리뷰
Walkthrough
워크쓰루 - 저자가 리더가 되어 진행하는 리뷰
Technical Review
기술 리뷰 - 기술 자격을 갖춘 리뷰어가 수행하고 중재자가 리더인 리뷰
Inspection
인스펙션 - 가장 공식적인 리뷰 유형, 프로세스를 철저히 따름
Formal Review
공식 리뷰 - 정의된 프로세스와 문서화된 결과를 갖는 리뷰

샘플 문제

실제 시험과 유사한 형식의 문제로 학습 내용을 점검한다.

문제 1 (FL-3.1.2)
다음 중 정적 테스팅의 이점이 아닌 것은?
a. 소프트웨어 수명 주기(SDLC) 후반에 결함을 더 많이 식별하므로 결함 관리 비용이 적게 든다
b. 일반적으로 정적 테스팅 중 식별한 결함을 수정하는 것이 동적 테스팅 중 식별한 결함을 수정하는 것보다 비용이 훨씬 적게 든다
c. 동적 테스팅만으로는 식별하지 못할 수 있는 코딩 결함을 식별할 수 있다
d. 요구사항과의 차이 및 요구사항 내 불일치를 식별할 수 있다
정답: a
SDLC 후반에 결함을 발견하면 오히려 비용이 더 많이 든다. 정적 테스팅의 핵심 이점은 SDLC 초기에 결함을 식별하여 수정 비용을 줄이는 것이다. b, c, d는 모두 정적 테스팅의 실제 이점이다.
문제 2 (FL-3.2.4)
조직에서 수행하고 있는 리뷰가 다음과 같은 특징을 가지고 있다:
- 서기 역할을 하는 사람이 있다
- 주요 목적은 품질을 평가하는 것이다
- 작업 산출물 작성자가 회의를 주도한다
- 개별 준비 단계가 있다
- 리뷰 보고서를 작성한다

다음 중 사용하고 있는 리뷰 유형일 가능성이 가장 높은 것은?
a. 비공식 리뷰 (Informal Review)
b. 워크쓰루 (Walkthrough)
c. 기술 리뷰 (Technical Review)
d. 인스펙션 (Inspection)
정답: b
핵심 단서는 "작업 산출물 작성자(저자)가 회의를 주도"한다는 점이다. 워크쓰루에서는 저자가 리더가 된다. 기술 리뷰에서는 중재자가 리더이고, 인스펙션에서는 저자가 리뷰 리더나 서기가 될 수 없다. 서기가 있고, 품질 평가가 목적인 점도 워크쓰루와 일치한다.
문제 3 (FL-3.2.3)
리뷰에 다음과 같은 역할이 있다:
1. 서기    2. 리뷰 리더    3. 중재자    4. 관리자

그리고 리뷰에서 맡은 책임은 다음과 같다:
A. 리뷰 회의의 효과적인 진행과 편안한 리뷰 환경을 보장한다
B. 리뷰 회의에서 결정사항, 식별한 새로운 이상 현상과 같은 리뷰 정보를 기록한다
C. 리뷰 대상을 결정하고 리뷰에 참여할 인력, 리뷰 시간 등 자원을 제공한다
D. 리뷰 진행 시기, 장소 협의 등 리뷰에 대한 전반적인 책임을 진다

다음 중 위의 역할과 책임을 가장 적절하게 연결한 것은?
a. 1A, 2B, 3D, 4C
b. 1A, 2C, 3B, 4D
c. 1B, 2D, 3A, 4C
d. 1B, 2D, 3C, 4A
정답: c
1(서기) = B(리뷰 정보 기록), 2(리뷰 리더) = D(전반적 책임, 시기/장소 협의), 3(중재자) = A(효과적 진행, 편안한 환경 보장), 4(관리자) = C(리뷰 대상 결정, 자원 제공).
