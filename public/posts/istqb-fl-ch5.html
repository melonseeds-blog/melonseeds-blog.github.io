5.1 테스트 계획

5.1.1 테스트 계획서의 목적과 내용

테스트 계획서(Test Plan)는 테스트 프로젝트의 테스트 목적, 자원, 프로세스를 설명하는 문서다. 테스트 계획서는 팀원과 이해관계자 간의 의사소통 수단이자, 테스팅이 정책 및 전략을 준수함을 보여주는 역할을 한다.

테스트 계획서에 포함되는 일반적인 내용은 다음과 같다:

  • 테스팅 정황 - 테스트 범위, 테스트 목적, 테스트 베이시스
  • 가정 및 제약 사항 - 테스트 프로젝트의 전제 조건
  • 이해관계자 - 역할, 책임, 테스팅 관련성, 채용 및 훈련 요구사항
  • 의사소통 - 의사소통 방법, 빈도, 문서 양식
  • 리스크 목록 - 제품 리스크, 프로젝트 리스크
  • 테스트 접근법 - 테스트 레벨, 유형, 기법, 시작/완료 조건, 메트릭 등
  • 예산 및 일정

5.1.2 반복주기/릴리스 계획에 대한 테스터의 기여

반복적 SDLC에서는 릴리스 계획과 반복 주기 계획 두 가지가 이루어진다.

릴리스 계획에 참여하는 테스터는 다음 활동을 수행한다:

  • 테스트 가능한 사용자 스토리와 인수 조건 작성 지원
  • 프로젝트 및 품질 리스크 분석 참여
  • 사용자 스토리 관련 테스트 노력 추정
  • 테스트 접근법 결정 및 릴리스 테스팅 계획

반복 주기 계획에 참여하는 테스터는 사용자 스토리의 리스크 분석, 테스트 용이성 판단, 업무 분류, 테스트 노력 추정, 기능/비기능 측면 식별 및 구체화를 수행한다.

5.1.3 시작 조건과 완료 조건

구분 시작 조건 (Entry Criteria) 완료 조건 (Exit Criteria)
정의 활동 수행을 위한 전제 조건 활동 종료를 선언하기 위해 달성해야 하는 사항
예시 자원 가용성(인력, 도구, 환경, 예산), 테스트웨어 가용성(테스트 베이시스, 요구사항, 테스트 케이스), 테스트 대상 초기 품질(스모크 테스트 합격) 달성 커버리지 수준, 미해결 결함 수, 결함 밀도, 계획 테스트 실행 완료 여부, 리그레션 테스트 자동화 여부
애자일 용어 준비의 정의 (Definition of Ready) 완료의 정의 (Definition of Done)
시간/예산 소진도 완료 조건이 될 수 있다 이해관계자들이 추가 테스트 없이 배포하는 리스크를 검토하고 수용했다면, 다른 완료 조건이 충족되지 않더라도 테스트를 종료할 수 있다.

5.1.4 추정 기법

테스트 노력 추정은 테스트 목적 달성에 필요한 작업량을 예측하는 것이다. 큰 작업은 여러 작은 작업으로 나눈 후 추정하면 정확도가 높아진다.

기법 분류 설명
비율 기반 추정 메트릭 기반 이전 프로젝트에서 수집한 비율(예: 개발:테스트 = 3:2)로 새 프로젝트의 테스트 노력 추정. 개발 노력 600MD이면 테스트 노력은 400MD.
외삽법 메트릭 기반 현재 프로젝트 초기 데이터를 수집하고 수학적 모델로 남은 작업 예측. 반복적 SDLC에 적합.
와이드밴드 델파이 전문가 기반 전문가가 독립적으로 추정 후 논의를 반복해 합의 도달. 애자일의 플래닝 포커가 이 기법의 변형.
3점 추정 전문가 기반 낙관(a), 최빈(m), 비관(b) 3개 추정치 사용. E=(a+4*m+b)/6, SD=(b-a)/6
3점 추정 계산 예시 a=6, m=9, b=18인 경우: E=(6+4x9+18)/6 = 10, SD=(18-6)/6 = 2. 최종 추정치는 10 +/- 2 맨아워(8~12 사이).

5.1.5 테스트 케이스 우선순위 지정

테스트 케이스의 실행 순서를 정하는 주요 방법은 다음 세 가지다:

  • 리스크 기반 우선순위 지정 - 리스크 분석 결과에 따라 가장 중요한 리스크를 다루는 테스트를 먼저 실행
  • 커버리지 기반 우선순위 지정 - 가장 높은 커버리지를 달성하는 테스트를 먼저 실행 (추가 커버리지 우선순위 변형도 있음)
  • 요구사항 기반 우선순위 지정 - 이해관계자가 정의한 요구사항 우선순위에 따라 결정

실무에서는 테스트 케이스 간 종속성이 있을 수 있어, 우선순위가 낮은 테스트를 먼저 실행해야 하는 경우도 있다. 자원 가용성도 고려해야 한다.

5.1.6 테스트 피라미드

테스트 피라미드는 테스트의 세분화 수준(granularity)이 다를 수 있음을 보여주는 모델로, 팀의 테스트 자동화와 노력 할당을 지원한다.

E2E / UI 테스트 (느림, 적은 수, 높은 의존도)
통합(서비스) 테스트 (중간 속도, 중간 수)
단위(컴포넌트) 테스트 (빠름, 많은 수, 독립적)

아래층일수록 테스트 수가 많고, 빠르며, 독립적이다. 위로 갈수록 테스트는 복잡하고 느리지만 큰 기능을 확인할 수 있다.

피라미드 핵심 포인트 아래층(단위 테스트)은 자동화 비율이 높고 실행 속도가 빠르다. 위층(E2E)은 실행 시간이 길고 시스템 의존도가 높아 적은 수로 운영한다. 적절한 커버리지를 위해 하위층 테스트를 충분히 확보하는 것이 핵심이다.

5.1.7 테스팅 사분면

브라이언 마릭이 정의한 테스팅 사분면은 애자일 소프트웨어 개발에서 테스트 레벨, 유형, 활동, 기법을 체계적으로 분류하는 모델이다. 테스트가 비즈니스/기술 중 무엇에 대한 것인지, 그리고 팀 지원/제품 평가 중 어떤 목적인지에 따라 4개의 사분면으로 구분한다.

Q1: 기술 측면 + 팀 지원
컴포넌트 테스트, 컴포넌트 통합 테스트. 자동화해야 하며 CI 프로세스에 포함.
Q2: 비즈니스 측면 + 팀 지원
기능 테스트, 사용자 스토리 테스트, API 테스팅, 시뮬레이션. 인수 조건 확인. 수동 또는 자동화 가능.
Q3: 비즈니스 측면 + 제품 평가
탐색적 테스팅, 사용성 테스팅, 사용자 인수 테스팅. 사용자 중심, 주로 수동 실행.
Q4: 기술 측면 + 제품 평가
스모크 테스트, 비기능 테스트(사용성 제외). 주로 자동화.

5.2 리스크 관리

5.2.1 리스크의 정의와 속성

리스크(Risk)란 발생 시 부정적인 영향을 미칠 수 있는 잠재적인 사건, 위험, 위협 또는 상황을 말한다. 리스크는 두 가지 요소로 특징지어진다:

  • 리스크 발생 가능성(Likelihood) - 리스크가 발생할 확률 (0 < 확률 < 1)
  • 리스크 영향(Impact) - 발생했을 때 생기는 피해의 크기

리스크 수준(Risk Level) = 발생 가능성 x 영향도. 리스크 수준이 높을수록 조치의 중요성이 커진다.

리스크 영향도와 발생 가능성은 독립적이다 영향도가 높다고 발생 가능성도 높은 것은 아니다. 두 요소는 서로 독립적으로 평가해야 한다. 이것은 시험에 자주 출제되는 포인트다.

5.2.2 제품 리스크 vs 프로젝트 리스크

구분 제품 리스크 (Product Risk) 프로젝트 리스크 (Project Risk)
관련 대상 제품 품질 특성 (ISO 25010) 프로젝트 관리 및 제어
예시 누락/잘못된 기능, 부정확한 계산, 런타임 오류, 열악한 아키텍처, 느린 응답 시간, 보안 취약점, 열악한 UX 작업 산출물 인도 지연, 부정확한 추정, 예산 축소, 기술 부족, 인력 갈등, 도구 지원 부족, 제3자 인도 실패
영향 사용자 불만족, 매출/신뢰/평판 손실, 높은 유지보수 비용, 법적 처벌, 심각한 경우 신체적 피해 프로젝트 일정, 예산, 범위에 영향

5.2.3 제품 리스크 분석

제품 리스크 분석은 리스크 식별과 리스크 평가로 구성된다.

  • 리스크 식별 - 브레인스토밍, 워크숍, 인터뷰, 원인-결과 다이어그램 등으로 포괄적 리스크 목록 생성
  • 리스크 평가 - 식별한 리스크를 분류하고, 발생 가능성/영향/수준을 결정하고, 우선순위를 정하고, 조치 방법 제안. 정량적(발생 가능성 x 영향) 또는 정성적(리스크 행렬) 접근법 사용

제품 리스크 분석 결과는 다음과 같이 활용된다:

  • 수행할 테스트 범위 결정
  • 테스트 레벨 및 테스트 유형 제안
  • 사용할 테스트 기법과 커버리지 수준 결정
  • 업무별 필요 테스트 노력 추정
  • 중요 결함을 빨리 식별하기 위한 테스트 우선순위 지정

5.2.4 제품 리스크 제어

제품 리스크 제어는 리스크 완화와 리스크 모니터링으로 구성된다.

리스크에 대응하는 방법에는 다음이 있다:

  • 리스크 완화(Mitigation) - 테스팅 등으로 리스크 수준을 낮추는 활동
  • 리스크 수용(Acceptance) - 리스크를 있는 그대로 받아들이기
  • 리스크 전가(Transfer) - 리스크를 다른 당사자에게 전가
  • 대안 계획(Contingency) - 리스크 발생 시 대응할 비상 계획 수립

테스팅으로 리스크를 완화하기 위한 구체적 조치:

  • 적절한 경험과 기술을 갖춘 테스터 선정
  • 적절한 수준의 테스팅 독립성 적용
  • 리뷰 및 정적 분석 수행
  • 적절한 테스트 기법 및 커버리지 수준 적용
  • 영향받는 품질 특성을 다루는 테스트 유형 적용
  • 리그레션 테스팅 포함 동적 테스팅 수행

5.3 테스트 모니터링, 제어, 완료

5.3.1 테스팅에 사용하는 메트릭

테스트 메트릭은 진행 상황, 품질, 효과를 보여주기 위해 수집한다. 주요 메트릭 분류:

분류예시
프로젝트 진행 상황작업 완료율, 자원 사용률, 테스트 노력 투입률
테스트 진행 상황테스트 케이스 구현 진행률, 환경 준비 진행률, 합격/불합격 테스트 수, 실행 시간
제품 품질가용성, 응답 시간, 평균 장애 시간(MTTF)
결함발견/수정 결함 수와 우선순위, 결함 밀도, 결함 발견 비율
리스크잔여 리스크 수준
커버리지요구사항 커버리지, 코드 커버리지
비용테스팅 비용, 조직의 품질 비용

5.3.2 테스트 보고서

테스트 보고서는 두 종류가 있다:

테스트 진행 상황 보고서(Test Progress Report) - 정기적으로 작성(매일/매주). 포함 내용:

  • 테스팅 기간 및 진행 상황 (예정 대비 상태)
  • 진행 방해 요소와 대응 방법
  • 테스트 메트릭
  • 신규 및 변경된 리스크
  • 다음 주기에 예정된 테스팅

테스트 완료 보고서(Test Completion Report) - 테스트 활동 종료 시 작성. 포함 내용:

  • 테스트 요약, 테스팅 및 제품 품질 평가
  • 테스트 계획과의 편차
  • 완화되지 않은 리스크, 수정되지 않은 결함
  • 테스팅 관련 교훈

5.3.3 테스팅 상황 전달 방법

테스트 상황을 전달하는 방법은 다양하다:

  • 팀원 및 이해관계자와의 대화
  • 대시보드 - CI/CD 대시보드, 태스크 보드, 번다운 차트
  • 전자 통신 채널 - 이메일, 채팅
  • 온라인 문서
  • 공식 테스트 보고서

분산된 팀은 더 공식적인 의사소통 방법이 적합하며, 이해관계자마다 관심 정보가 다르므로 그에 맞게 조정해야 한다.

5.4 형상 관리

형상 관리(Configuration Management)는 테스트 계획서, 전략서, 테스트 케이스, 테스트 스크립트, 결과, 로그, 보고서 등의 작업 산출물을 형상 항목으로 식별, 제어, 추적하는 지침을 제공한다.

형상 관리의 핵심 원칙:

  • 형상 항목이 테스팅 용으로 승인되면 베이스라인(Baseline)이 된다
  • 베이스라인 수정은 공식적인 변경 제어 프로세스를 통해서만 가능
  • 이전 베이스라인으로 되돌리면 이전 테스트 결과를 재현할 수 있다

형상 관리가 테스팅을 지원하는 방법:

  • 모든 형상 항목에 고유 식별자 부여, 버전 관리, 변경 추적, 연관성 식별로 추적성 유지
  • 식별된 모든 문서와 소프트웨어 항목이 테스트웨어 내에서 명확하게 참조
DevOps와 형상 관리 지속적 통합(CI), 지속적 전달(CD), 지속적 배포 파이프라인에는 보통 자동화된 형상 관리가 포함되어 있다.

5.5 결함 관리

테스트의 주요 목적 중 하나가 결함 식별이므로 잘 확립된 결함 관리 프로세스가 필요하다. 보고된 이상 사항은 실제 결함일 수도, 거짓-양성(false positive)이나 변경 요청일 수도 있다.

결함 관리 프로세스의 작업 흐름:

  1. 이상 사항 기록
  2. 분석 및 분류
  3. 적절한 대응책 결정 (수정, 유지 등)
  4. 결함 보고 종료

결함 보고서 필수 항목

동적 테스팅 중 작성하는 결함 보고서에 포함해야 하는 항목:

항목내용
고유 식별자각 결함을 고유하게 식별하는 ID

날짜/작성자관찰 날짜, 보고 주체 조직, 작성자(역할 포함)

핵심 용어 정리

5장에서 반드시 숙지해야 할 핵심 용어들이다.

Test Plan
테스트 계획서 - 테스트 목적, 자원, 프로세스를 설명하는 문서
Entry Criteria
시작 조건 - 활동 수행을 위한 전제 조건 (= Definition of Ready)
Exit Criteria
완료 조건 - 활동 종료 선언을 위한 달성 사항 (= Definition of Done)
Risk Level
리스크 수준 - 발생 가능성 x 영향도로 산출
Product Risk
제품 리스크 - 제품 품질 특성과 관련된 리스크
Project Risk
프로젝트 리스크 - 프로젝트 관리/제어와 관련된 리스크
Risk-based Testing
리스크 기반 테스팅 - 리스크 분석과 제어를 기반으로 테스트 활동을 선택/우선순위 지정
Risk Mitigation
리스크 완화 - 조치를 실행해 리스크 수준을 낮추는 활동
Test Pyramid
테스트 피라미드 - 테스트 세분화 수준과 자동화 할당을 보여주는 모델
Testing Quadrants
테스팅 사분면 - 비즈니스/기술, 팀 지원/제품 평가 축으로 테스트 유형 분류
Defect Report
결함 보고서 - 결함의 식별, 재현, 처리를 위한 문서
Test Monitoring
테스트 모니터링 - 진행 상황 평가를 위한 정보 수집 활동
Test Control
테스트 제어 - 모니터링 정보를 바탕으로 효과적 테스팅을 위한 조치 제공
Configuration Management
형상 관리 - 작업 산출물을 형상 항목으로 식별, 제어, 추적하는 활동
Test Approach
테스트 접근법 - 테스트 레벨, 유형, 기법, 시작/완료 조건 등의 총체적 방법론

연습 문제

ISTQB FL 샘플 시험 문제 중 5장 관련 문제를 선별했다. 정답을 확인하기 전에 먼저 풀어보자.

문제 1 (FL-5.1.4 K3)
당신의 팀은 3점 추정 기법을 사용해 리스크가 높은 새로운 기능에 대한 테스트 노력을 추정하고 있다. 다음과 같은 추정이 이루어졌다.
- 가장 낙관적인 추정: 2M/H
- 가장 유력한 추정: 11M/H
- 가장 비관적인 추정: 14M/H

최종 추정치는 얼마인가?
  • a. 9M/H
  • b. 14M/H
  • c. 11M/H
  • d. 10M/H
정답: d
E = (a + 4*m + b) / 6 = (2 + 4*11 + 14) / 6 = (2 + 44 + 14) / 6 = 60/6 = 10M/H
문제 2 (FL-5.2.4 K2)
리스크 분석 결과 다음 리스크가 식별되고 평가되었다.
- 리스크: 보고서 생성에 너무 많은 시간이 걸린다
- 리스크 발생 가능성: 중간; 리스크 영향도: 높음
- 리스크 대응 방법: 시스템 테스팅 중에 독립적인 테스트팀이 성능 효율성 테스팅을 수행하고, 최종 사용자 표본 집단을 선별해 릴리스 전에 알파/베타 테스팅을 수행한다.

다음 중 분석한 리스크에 대해 대응 방법으로 제안된 것은?
  • a. 리스크 수용 (Risk acceptance)
  • b. 비상 대책 (Contingency plan)
  • c. 리스크 완화 (Risk mitigation)
  • d. 리스크 전가 (Risk transfer)
정답: c
성능 효율성 테스팅과 알파/베타 테스팅을 수행하는 것은 테스팅을 통해 리스크 수준을 낮추려는 활동이므로 리스크 완화(Risk mitigation)에 해당한다.
문제 3 (FL-5.5.1 K3)
개발자로부터 다음과 같은 결함 보고서를 받았다. 테스트 보고서에 설명된 이상 현상이 재현 불가능하다고 한다.

"애플리케이션 멈춤 - 새로운 사용자 생성 화면의 이름 필드에 특수문자를 입력하면 애플리케이션이 멈춥니다. 로그에는 치명적인 오류 알림이 포함되어 있습니다. 테스트 케이스 TC-1305에 따르면, 애플리케이션은 제공된 입력값을 인식하고 사용자를 생성해야 합니다."

개발자에게 유용했을 이 테스트 보고서에서 누락된 중요한 정보는 무엇인가?
  • a. 기대 결과와 실제 결과
  • b. 참고자료 및 결함 상태
  • c. 테스트 환경 및 테스트 항목
  • d. 우선순위 및 심각도
정답: c
개발자가 이상 현상을 재현하지 못한 이유는 테스트 환경 및 테스트 항목(버전 정보 등)이 누락되었기 때문이다. 환경 정보가 있어야 동일한 조건에서 재현을 시도할 수 있다.
