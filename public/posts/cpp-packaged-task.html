std::packaged_task란?

std::packaged_task<R(Args...)>는 호출 가능한 객체(callable)와 future를 한 묶음으로 제공하는 클래스 템플릿이다. 호출 가능한 객체를 래핑하면서, 그 결과를 받을 수 있는 std::future<R>을 함께 만들어 준다. 즉 "실행할 함수"와 "실행 결과를 비동기로 받을 통로"가 한 객체에 포장된다.

#include <future>
#include <thread>

std::packaged_task<int(int, int)> task([](int a, int b) {
    return a + b;
});

std::future<int> fut = task.get_future();
std::thread t(std::move(task), 3, 4);  // task는 move-only

int result = fut.get();  // 7
t.join();

핵심은 get_future()로 미래 결과 통로를 먼저 떼어 두고, task 자체는 원하는 시점에 원하는 스레드에서 호출하면 된다는 점이다. 이 분리가 다음 절의 차이를 만든다.

std::async와의 차이

std::async는 "실행 정책 + 자동 실행"을 한꺼번에 처리한다. 반면 packaged_task는 언제, 어디서 실행할지 사용자가 결정한다. 작업을 객체로 만들어 큐에 넣고 워커가 꺼내 실행하는 패턴에 적합하다.

비교std::asyncstd::packaged_task
실행 시점호출 즉시 (정책에 따라)사용자가 직접 호출
실행 위치런타임 결정 또는 호출 스레드임의 스레드 / 큐
future 획득반환값으로 획득get_future()
재사용불가reset()으로 재사용 가능
적합한 패턴일회성 비동기 호출스레드 풀, 태스크 큐

packaged_task의 특징

  • move-only — 복사 생성자/대입 연산자가 삭제되어 있다. 항상 std::move로 옮겨야 한다.
  • 한 번만 호출 가능 — operator()를 호출하면 결과가 future에 채워지고, 두 번째 호출은 예외를 던진다.
  • valid()로 호출 가능 여부 확인.
  • reset()으로 새 공유 상태를 만들면 다시 호출할 수 있다. 이때 새 future를 다시 get_future()로 받아야 한다.
std::packaged_task<int()> task([] { return 42; });
auto f1 = task.get_future();
task();          // 첫 호출 OK
// task();       // 두 번째 호출 → std::future_error throw

task.reset();    // 공유 상태 새로 생성
auto f2 = task.get_future();
task();          // 다시 호출 가능

std::function과의 차이

둘 다 callable을 감싸지만 목적과 제약이 다르다.

비교std::functionstd::packaged_task
역할callable 저장 + 호출callable + future 결합
복사가능 (CopyConstructible 요구)불가 (move-only)
결과 회수직접 반환값으로future 통해
호출 횟수제한 없음1회 (reset 전까지)

태스크 큐를 단순히 std::queue<std::function<void()>>로 만들면 호출은 되지만 반환값을 어떻게 받을지가 문제가 된다. 그래서 packaged_task로 결과 통로를 만들어 두고, 큐에는 std::function<void()> 형태의 얇은 호출 래퍼만 넣는 조합이 자연스럽다.

태스크 큐 패턴

핵심 아이디어는 단순하다. 워커 스레드들이 큐에서 작업을 꺼내 실행한다. 자료구조는 다음과 같다.

  • std::queue<...> — 작업 보관
  • std::mutex — 큐 보호
  • std::condition_variable — 워커가 작업을 기다림

문제는 packaged_task의 시그니처가 고정이라 다양한 반환 타입의 작업을 같은 큐에 넣을 수 없다는 점이다. 해결책은 큐 원소를 std::function<void()>로 통일하고, 결과는 호출자에게 반환된 future로 따로 받는 것이다.

간단한 TaskQueue 구현

#include <queue>
#include <mutex>
#include <condition_variable>
#include <future>
#include <functional>
#include <thread>
#include <memory>

class TaskQueue
{
public:
    template<class F, class... Args>
    auto Submit(F&& f, Args&&... args)
        -> std::future<std::invoke_result_t<F, Args...>>
    {
        using R = std::invoke_result_t<F, Args...>;

        auto task = std::make_shared<std::packaged_task<R()>>(
            std::bind(std::forward<F>(f), std::forward<Args>(args)...)
        );
        std::future<R> fut = task->get_future();

        {
            std::lock_guard<std::mutex> lock(m_mutex);
            m_tasks.emplace([task] { (*task)(); });
        }
        m_cv.notify_one();
        return fut;
    }

    void Stop()
    {
        {
            std::lock_guard<std::mutex> lock(m_mutex);
            m_stop = true;
        }
        m_cv.notify_all();
    }

    void WorkerLoop()
    {
        while (true)
        {
            std::function<void()> job;
            {
                std::unique_lock<std::mutex> lock(m_mutex);
                m_cv.wait(lock, [this] { return m_stop || !m_tasks.empty(); });
                if (m_stop && m_tasks.empty()) return;
                job = std::move(m_tasks.front());
                m_tasks.pop();
            }
            job();
        }
    }

private:
    std::queue<std::function<void()>> m_tasks;
    std::mutex m_mutex;
    std::condition_variable m_cv;
    bool m_stop = false;
};

왜 shared_ptr로 감싸는가

std::function은 자신이 담는 타깃이 복사 가능해야 한다. 그러나 packaged_task는 move-only다. 람다 캡처에 직접 넣으면 그 람다도 move-only가 되어 std::function에 들어가지 못한다. std::shared_ptr로 감싸면 람다가 복사되어도 내부의 packaged_task는 같은 인스턴스를 공유한다.

shared_ptr 우회 트릭 move-only 객체를 복사 가능한 컨테이너에 넣어야 할 때 자주 쓰는 패턴이다. 약간의 힙 할당 비용은 있지만 코드가 깔끔해진다.
C++23: std::move_only_function C++23에서 도입된 std::move_only_function은 move-only 타깃을 받을 수 있다. 큐 원소를 std::move_only_function<void()>로 바꾸면 shared_ptr 우회 없이 packaged_task를 직접 캡처해 넣을 수 있다.

왜 std::invoke_result_t인가

임의의 callable과 인자에 대해 반환 타입을 정확히 추론하기 위해서다. F가 함수 포인터든 람다든 멤버 함수 포인터든, std::invoke_result_t<F, Args...>가 일관되게 동작한다. (C++14 이전의 std::result_of는 deprecated.)

결과 회수 패턴

Submit()이 future를 반환하므로, 호출자는 원하는 시점에 fut.get()으로 결과를 기다리거나 wait_for로 폴링할 수 있다.

TaskQueue queue;
std::thread worker(&TaskQueue::WorkerLoop, &queue);

auto f1 = queue.Submit([](int x) { return x * x; }, 7);
auto f2 = queue.Submit([] { return std::string("done"); });

int sq = f1.get();              // 49
std::string msg = f2.get();     // "done"

queue.Stop();
worker.join();

결과가 필요 없는 fire-and-forget 작업이라면 future를 무시해도 된다. 다만 future를 받지 않거나 즉시 폐기하면 packaged_task가 채워 둔 결과는 그대로 사라진다.

워커 스레드를 여러 개 만들면 자연스럽게 스레드 풀이 된다. 다음 글에서 본격적으로 다룬다.

이미지 파이프라인 적용

카메라에서 들어오는 프레임마다 처리 작업을 큐에 넣는 단순한 형태다.

TaskQueue queue;
// (워커 스레드 N개 시작 ...)

std::vector<std::future<ProcessedFrame>> futures;
while (auto frame = camera.Grab())
{
    futures.push_back(
        queue.Submit([f = std::move(frame)] {
            return ProcessFrame(f);
        })
    );
}

// 일괄 대기
for (auto& fut : futures)
{
    auto result = fut.get();
    // result 사용 ...
}

결과 순서가 중요하지 않다면 future 컨테이너 없이 fire-and-forget도 가능하다. 단 처리 완료 시점을 외부 콜백으로 따로 알려야 한다.

흔한 함정

packaged_task는 복사할 수 없다 값으로 전달하거나 람다에 값 캡처하면 컴파일 에러다. 항상 std::move로 넘기거나 shared_ptr로 감싸야 한다.
두 번 호출하면 future_error 한 번 실행한 packaged_task는 더 이상 호출할 수 없다. 재사용하려면 reset() 후 get_future()를 다시 받아야 한다.
std::function에 packaged_task를 직접 넣으면 컴파일 에러 move-only이기 때문이다. shared_ptr로 감싸거나 C++23 move_only_function을 사용한다.
packaged_task가 호출 전에 폐기되면 broken_promise 큐에 넣은 작업이 실행되기 전에 큐가 파괴되거나 예외로 작업이 버려지면, 대기 중인 future는 std::future_error(broken_promise)를 던진다. Stop() 시 남은 작업을 어떻게 처리할지 명확히 정해 두어야 한다.
