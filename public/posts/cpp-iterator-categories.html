이터레이터 5종 카테고리

C++ 표준은 이터레이터를 능력에 따라 5단계로 분류한다. 위로 갈수록 더 많은 연산을 지원하며, 상위 카테고리는 하위 카테고리의 모든 능력을 포함한다.

카테고리핵심 능력지원 연산대표 예시
Input한 번 읽고 전진*it(read), ++it, ==, !=istream_iterator
Output한 번 쓰고 전진*it = v(write), ++itostream_iterator, back_inserter
Forward여러 번 읽기/전진Input + 다회 통과 가능forward_list, unordered_set
Bidirectional역방향 이동Forward + --itlist, map, set
Random Access임의 위치 점프Bidirectional + +=, -=, it[n], <, -vector, deque, array, 포인터

능력 비교표

연산InputOutputForwardBidirRandom

*it = v 쓰기XOO*O*O*
++itOOOOO
--itXXXOO

it1 < it2XXXXO
다회 통과XXOOO

* const_iterator가 아닌 경우에 한해 쓰기 가능.

컨테이너별 제공 카테고리

컨테이너이터레이터 카테고리
vector, array, deque, stringRandom Access
list, map, set, multimap, multisetBidirectional

deque도 Random Access지만… deque는 내부적으로 여러 청크로 나뉘어 있어 vector처럼 메모리가 연속되지 않는다. 인덱스 접근은 O(1)이지만 캐시 효율은 vector보다 떨어진다.

C++17/20 분류 보강

C++20에서는 기존 5종 위에 contiguous iterator가 추가됐다. 메모리가 연속이라는 사실을 타입 수준에서 보장한다.

표준변경점
C++17 이전5종 카테고리 + iterator_category 태그
C++17알고리즘 병렬 정책(execution::par) 도입, 이터레이터 요건 일부 완화
C++20contiguous_iterator_tag 추가, std::ranges에서 새 컨셉 기반 분류(input_iterator, random_access_iterator 등)

contiguous는 random access의 부분집합이며, vector::iterator나 raw pointer가 여기에 해당한다. 이 정보 덕분에 std::copy가 내부적으로 memcpy로 최적화될 수 있다.

C++20 ranges 컨셉 std::contiguous_iterator<T>, std::random_access_iterator<T>처럼 컨셉으로 직접 제약할 수 있다. 기존 태그 기반 SFINAE보다 컴파일 에러가 훨씬 친절하다.

iterator_traits와 tag dispatch

std::iterator_traits<It>는 임의의 이터레이터 타입에서 카테고리, 값 타입, 차이 타입 등을 추출하는 메타함수다. 알고리즘이 이터레이터의 능력에 따라 다른 구현을 고를 때 핵심 도구가 된다.

#include <iterator>

template <typename It>
void Inspect(It it)
{
    using Category = typename std::iterator_traits<It>::iterator_category;
    using ValueType = typename std::iterator_traits<It>::value_type;
    using DiffType = typename std::iterator_traits<It>::difference_type;
    // Category is std::random_access_iterator_tag for vector::iterator.
}

표준 라이브러리는 흔히 tag dispatch 패턴으로 분기한다. 같은 이름의 함수를 카테고리 태그로 오버로드해 컴파일 타임에 골라낸다.

// Tag dispatch example: distance().
template <typename It>
auto MyDistance(It first, It last, std::random_access_iterator_tag)
{
    return last - first;          // O(1).
}

template <typename It>
auto MyDistance(It first, It last, std::input_iterator_tag)
{
    typename std::iterator_traits<It>::difference_type n = 0;
    while (first != last)
    {
        ++first;
        ++n;
    }
    return n;                     // O(n).
}

template <typename It>
auto MyDistance(It first, It last)
{
    using Cat = typename std::iterator_traits<It>::iterator_category;
    return MyDistance(first, last, Cat{});
}

컨테이너별 무효화 규칙

이터레이터 무효화(invalidation)는 컨테이너 변경으로 인해 기존 이터레이터·포인터·참조가 더 이상 유효하지 않게 되는 현상이다. 무효화된 이터레이터를 역참조하거나 비교하면 정의되지 않은 동작(UB)이다.

UB는 컴파일러가 경고하지 않는다 무효화된 이터레이터를 쓰는 코드가 디버그 빌드에서는 멀쩡히 돌다가 릴리스 빌드에서 크래시하는 경우가 흔하다. 규칙을 외우는 것 외에 안전 장치가 없다.

vector — 재할당의 공포

연산무효화 범위
push_back, emplace_backcapacity 초과 시 전부 무효화, 아니면 end()만
insert(pos, …)재할당 시 전부, 아니면 pos 이후 전부
erase(pos)pos 이후 전부 무효화
reserve(n) (n > capacity)전부 무효화
resize(n) (n > capacity)전부 무효화
clear()전부 무효화 (end는 유효)

deque — 가장 까다롭다

연산이터레이터참조/포인터
양 끝 push/pop전부 무효화유효 유지
중간 insert/erase전부 무효화전부 무효화
deque의 함정 push_front 한 번에도 모든 이터레이터가 무효화된다. 단 기존 요소의 참조와 포인터는 유효하다는 점이 특이하다.

list / forward_list — 노드 기반

노드 기반 컨테이너는 지워진 노드를 가리키는 이터레이터만 무효화된다. 다른 노드는 영향 없다.

연산무효화 범위
insert, push_*, splice없음
erase(it)it만
clear()전부 (end는 유효)

map / set / multimap / multiset — 트리 기반

역시 노드 기반. insert는 무효화 없음, erase는 지운 노드만.

unordered_* — 해시 기반(rehash 주의)

연산이터레이터참조/포인터
insert (rehash 발생)전부 무효화유효
insert (rehash 없음)유효유효
erase(it)it만해당 요소만
rehash 발생 조건 size() > bucket_count() * max_load_factor()일 때. 미리 reserve()를 호출해 두면 rehash로 인한 무효화를 막을 수 있다.

const_iterator vs iterator

const_iterator는 가리키는 요소를 수정할 수 없는 이터레이터다. 이터레이터 자체는 증감 가능하다.

std::vector<int> v = {1, 2, 3};

auto it = v.begin();        // vector<int>::iterator
auto cit = v.cbegin();      // vector<int>::const_iterator

*it = 10;                   // OK.
// *cit = 10;               // Error: read-only element.
++cit;                      // OK: iterator itself is mutable.

// const 컨테이너에서는 begin()도 const_iterator를 반환.
const std::vector<int> cv = {1, 2, 3};
auto it2 = cv.begin();      // const_iterator

iterator는 const_iterator로 암묵 변환된다. 반대 방향 변환은 불가하다. 함수 인자로 const_iterator를 받으면 양쪽 다 받을 수 있어 유연하다.

reverse_iterator 동작 원리

reverse_iterator는 내부적으로 일반 이터레이터를 감싸는 어댑터다. 핵심은 base()와의 한 칸 어긋남이다.

이터레이터가리키는 위치
v.rbegin()마지막 요소(v.end() - 1)
v.rend()첫 요소 바로 앞(v.begin() - 1 개념)
rit.base()rit이 가리키는 요소의 한 칸 뒤
std::vector<int> v = {1, 2, 3, 4, 5};

auto rit = v.rbegin();      // points to 5
std::cout << *rit;          // 5
std::cout << *rit.base();   // UB: base() == v.end()

++rit;                      // points to 4
std::cout << *rit.base();   // 5  (one past the logical position)
reverse_iterator로 erase할 때 주의 list의 erase는 forward iterator를 받는다. 역순 순회 중 삭제하려면 list.erase(std::next(rit).base()) 같은 변환이 필요하다.

실무: erase 루프의 흔한 실수

잘못된 패턴

// BAD: vector erase invalidates it and beyond.
std::vector<int> v = {1, 2, 3, 4, 5};
for (auto it = v.begin(); it != v.end(); ++it)
{
    if (*it % 2 == 0)
    {
        v.erase(it);          // it is now invalid.
        // ++it on invalid iterator -> UB.
    }
}

올바른 패턴 (vector / deque)

erase는 다음 유효 위치를 반환한다. 반환값을 받아 다시 대입하는 것이 정석이다.

// GOOD: reuse the returned iterator.
for (auto it = v.begin(); it != v.end(); )
{
    if (*it % 2 == 0)
    {
        it = v.erase(it);     // erase returns the next valid iterator.
    }
    else
    {
        ++it;
    }
}

list의 경우도 동일

list::erase 역시 다음 이터레이터를 반환한다. 다만 list는 다른 노드의 이터레이터가 무효화되지 않으므로 미리 다음 위치를 저장해 두는 패턴도 흔하다.

// list-only trick: save next before erase.
std::list<int> lst = {1, 2, 3, 4, 5};
for (auto it = lst.begin(); it != lst.end(); )
{
    auto next = std::next(it);
    if (*it % 2 == 0)
    {
        lst.erase(it);
    }
    it = next;
}

가장 깔끔한 방법: erase-remove 관용구

// C++17 and earlier.
v.erase(std::remove_if(v.begin(), v.end(),
                       [](int x) { return x % 2 == 0; }),
        v.end());

// C++20: std::erase_if.
std::erase_if(v, [](int x) { return x % 2 == 0; });
가능하면 std::erase_if(C++20) 한 줄이면 끝나고 컨테이너 종류와 무관하게 동작한다. 오래된 코드베이스가 아니라면 erase-remove 대신 이쪽을 쓰자.

직접 구현: std::advance

이터레이터 카테고리에 따라 분기하는 가장 고전적인 예제. random access면 O(1), 그 외에는 O(n)으로 동작해야 한다.

#include <iterator>

namespace MyStd
{
    // Random access: jump in one step.
    template <typename It, typename Distance>
    void AdvanceImpl(It& it, Distance n, std::random_access_iterator_tag)
    {
        it += n;
    }

    // Bidirectional: step by step, can go backward.
    template <typename It, typename Distance>
    void AdvanceImpl(It& it, Distance n, std::bidirectional_iterator_tag)
    {
        if (n >= 0)
        {
            while (n--) ++it;
        }
        else
        {
            while (n++) --it;
        }
    }

    // Input/forward: forward only.
    template <typename It, typename Distance>
    void AdvanceImpl(It& it, Distance n, std::input_iterator_tag)
    {
        while (n--) ++it;
    }

    template <typename It, typename Distance>
    void Advance(It& it, Distance n)
    {
        using Cat = typename std::iterator_traits<It>::iterator_category;
        AdvanceImpl(it, n, Cat{});
    }
}

핵심은 호출자가 카테고리를 신경 쓰지 않아도 컴파일러가 최적의 구현을 선택한다는 점이다. vector의 이터레이터를 넘기면 +=가, list를 넘기면 루프가, forward_list를 넘기면 전진 전용 루프가 호출된다. 만약 음수 n을 forward iterator에 넘기면 컴파일은 되지만 무한 루프에 빠지므로 표준은 이를 UB로 규정한다.

C++20 이후 스타일 같은 분기를 if constexpr + 컨셉으로 작성하면 함수 하나로 끝난다. 다만 기존 라이브러리 코드와 호환을 유지하려면 tag dispatch 패턴이 여전히 표준이다.
