C++의 인터페이스 — 추상 클래스

C++에는 Java나 C#의 interface 키워드가 없다. 대신 모든 멤버 함수가 순수 가상 함수(pure virtual)인 추상 클래스(ABC, Abstract Base Class)로 같은 목적을 이룬다.

언어인터페이스 표현
Java / C#interface IFoo { ... }
C++순수 가상 함수만 가진 추상 클래스
가상 소멸자는 필수다 다형 객체를 베이스 포인터로 delete할 때 가상 소멸자가 없으면 정의되지 않은 동작(UB)이 된다. 인터페이스에는 반드시 virtual ~Base() = default;를 선언한다.

기본 인터페이스 작성

이미징 라이브러리의 IProcessBlock을 예로 든다. 모든 처리 블록은 이 인터페이스를 구현한다.

// IProcessBlock.h
class IProcessBlock
{
public:
    virtual ~IProcessBlock() = default;  // 필수
    virtual ErrorCode Process(const Image& in, Image& out) = 0;
    virtual std::string Name() const = 0;
};

// GaussianBlur.h
class GaussianBlur : public IProcessBlock
{
public:
    ErrorCode Process(const Image& in, Image& out) override;
    std::string Name() const override { return "GaussianBlur"; }
    void SetKernelSize(int k) { m_kernelSize = k; }

private:
    int m_kernelSize = 3;
};
override 키워드를 항상 쓰자 시그니처를 잘못 적어도 컴파일러가 잡아준다. 더 이상 재정의를 막고 싶으면 final을 붙인다. 
class FastBlur final : public IProcessBlock { ... };  // 더 이상 상속 불가

class Base
{
public:
    virtual void Foo() final;  // 자식은 Foo 재정의 불가
};

NVI 패턴

Non-Virtual Interface. public 함수는 non-virtual로 두고, 그 안에서 private virtual을 호출한다. 부모가 사전 검증·로깅·락 같은 공통 로직을 강제할 수 있다.

class IProcessBlock
{
public:
    virtual ~IProcessBlock() = default;

    // public — non-virtual. 공통 로직을 부모가 통제한다.
    ErrorCode Process(const Image& in, Image& out)
    {
        if (in.IsEmpty()) return ErrorCode::InvalidImage;
        SPDLOG_DEBUG("Process begin: {}", Name());

        const auto rc = DoProcess(in, out);

        SPDLOG_DEBUG("Process end: {} -> {}", Name(), (int)rc);
        return rc;
    }

    virtual std::string Name() const = 0;

private:
    // 자식은 이것만 구현한다.
    virtual ErrorCode DoProcess(const Image& in, Image& out) = 0;
};
NVI를 권하는 이유 Herb Sutter와 Scott Meyers가 권장하는 표준 관용구다. 자식이 검증을 빠뜨려도 부모가 막아주고, 나중에 측정·로깅·재시도 같은 공통 로직을 한 곳에 추가할 수 있다.
구분장점단점
일반 가상 함수단순, 직관적공통 로직을 자식이 잊을 수 있음
NVI공통 로직 강제, 테스트 용이래퍼 한 단계가 추가됨

Strategy 패턴

알고리즘을 인터페이스로 분리해 런타임에 교체할 수 있게 한다. 데이터 플로우 블록 에디터처럼 사용자가 알고리즘을 고르는 구조에 자연스럽게 어울린다.

class IThresholdStrategy
{
public:
    virtual ~IThresholdStrategy() = default;
    virtual ErrorCode Apply(const Image& in, Image& out) = 0;
};

class OtsuThreshold : public IThresholdStrategy { ... };
class AdaptiveThreshold : public IThresholdStrategy { ... };

class ThresholdBlock : public IProcessBlock
{
public:
    void SetStrategy(std::unique_ptr<IThresholdStrategy> s)
    {
        m_strategy = std::move(s);
    }

private:
    ErrorCode DoProcess(const Image& in, Image& out) override
    {
        return m_strategy->Apply(in, out);
    }
    std::unique_ptr<IThresholdStrategy> m_strategy;
};

Template Method 패턴은 NVI와 같은 구조다. 부모가 알고리즘의 큰 골격을 잡고, 자식이 세부 단계만 구현한다.

CRTP — 정적 다형성

Curiously Recurring Template Pattern. 자식 타입을 부모 템플릿 인자로 전달해 컴파일 타임에 디스패치한다. virtual 호출 비용이 없다.

template <class Derived>
class ProcessBase
{
public:
    void Run(const Image& in, Image& out)
    {
        // virtual 없이 자식의 RunImpl 호출
        static_cast<Derived*>(this)->RunImpl(in, out);
    }
};

class FastSobel : public ProcessBase<FastSobel>
{
public:
    void RunImpl(const Image& in, Image& out) { /* ... */ }
};
CRTP의 트레이드오프 가상 호출 오버헤드가 사라지고 인라인 최적화가 잘 된다. 대신 다형성이 컴파일 타임에 고정되어 런타임 교체가 불가능하다. hot path나 매우 짧은 함수에 어울리고, 데이터 플로우 블록처럼 동적 교체가 필요한 곳에는 부적합하다.

다형성 시점런타임컴파일 타임런타임 (닫힌 집합)
호출 비용vtable 간접 호출거의 0 (인라인)점프 테이블
런타임 교체가능불가가능 (정해진 타입 내)
플러그인 확장가능불가불가 (타입 추가 시 재컴파일)
적합한 곳플러그인, 큰 함수hot path, 작은 함수닫힌 메시지 집합

std::variant + std::visit

C++17의 std::variant는 정해진 몇 개 타입 중 하나를 담는다. 닫힌 집합일 때 가상 함수의 대안이 된다.

#include <variant>

struct Resize { int w; int h; };
struct Crop   { int x; int y; int w; int h; };
struct Rotate { double angleDeg; };

using Op = std::variant<Resize, Crop, Rotate>;

ErrorCode Apply(const Op& op, Image& img)
{
    return std::visit([&](auto&& o) -> ErrorCode {
        using T = std::decay_t<decltype(o)>;
        if constexpr (std::is_same_v<T, Resize>) return DoResize(img, o.w, o.h);
        else if constexpr (std::is_same_v<T, Crop>) return DoCrop(img, o);
        else return DoRotate(img, o.angleDeg);
    }, op);
}
variant가 빛나는 경우 타입이 추가될 일이 거의 없는 닫힌 집합(파일 포맷의 토큰, 메시지 종류 등)에 효과적이다. 힙 할당이 없고, 컴파일러가 모든 케이스를 알아 최적화도 잘 된다.

타입 소거(Type Erasure)

인터페이스를 상속하지 않은 객체들도 같은 타입으로 다룰 수 있게 한다. std::function이 대표 예시다. Sean Parent의 토크 "Inheritance is the Base Class of Evil"에서 다듬어진 패턴이다.

class AnyDrawable
{
public:
    template <class T>
    AnyDrawable(T x) : m_self(std::make_shared<Model<T>>(std::move(x))) {}

    void Draw(Canvas& c) const { m_self->Draw(c); }

private:
    struct Concept
    {
        virtual ~Concept() = default;
        virtual void Draw(Canvas&) const = 0;
    };
    template <class T>
    struct Model : Concept
    {
        T data;
        Model(T x) : data(std::move(x)) {}
        void Draw(Canvas& c) const override { data.Draw(c); }  // 덕 타이핑
    };
    std::shared_ptr<const Concept> m_self;
};

비용은 가상 호출 + 힙 할당으로 일반 인터페이스와 비슷하다. 대신 외부 타입(예: 서드파티 라이브러리)을 수정하지 않고도 같은 컨테이너에 담을 수 있다.

인터페이스 분리 원칙(ISP)

한 인터페이스가 너무 많은 책임을 갖지 않도록 잘게 나눈다. 클라이언트가 사용하지 않는 메서드에 의존하지 않게 만든다.

// 별로다 — 모든 책임이 한 곳에
class IProcessBlock
{
public:
    virtual ErrorCode Process(...) = 0;
    virtual std::string Name() const = 0;
    virtual nlohmann::json Serialize() const = 0;
    virtual void SetParameter(const std::string&, ...) = 0;
};

// 좋다 — 작은 인터페이스로 분리
class INameable    { public: virtual std::string Name() const = 0; ... };
class IConfigurable { public: virtual void SetParameter(...) = 0; ... };
class ISerializable { public: virtual nlohmann::json Serialize() const = 0; ... };

class IProcessBlock : public INameable
{
public:
    virtual ErrorCode Process(const Image&, Image&) = 0;
};

DLL 경계의 인터페이스

C++ ABI는 표준이 없다. 컴파일러 버전·런타임이 다르면 가상 함수 테이블의 구조도 달라질 수 있다. DLL 경계에서 C++ 추상 클래스를 그대로 노출하면 클라이언트가 같은 컴파일러로 빌드해야 한다는 제약이 생긴다.

방식장점단점
C++ 추상 클래스 export편리컴파일러/STL 버전 일치 필요
C ABI (extern "C" 함수)호환성 최고객체지향 표현 직접 구현
COM 스타일 (IUnknown 유사)버전 호환, 인터페이스 추가 가능설계 부담
함수 포인터 테이블 구조체단순, C 호환vtable을 손으로 만드는 격
산업용 SDK의 관행 대부분의 상용 SDK(Vieworks, Cognex, MIL 등)는 C 인터페이스를 export하고 그 위에 C++ 래퍼를 같이 제공한다. 사용자는 어떤 컴파일러로든 링크할 수 있고, 라이브러리 측은 ABI 안정성을 보장한다.

플러그인 가능 구조

IProcessBlock 기반의 데이터 플로우는 자연스럽게 플러그인 구조로 확장된다. 블록 팩토리를 이름으로 등록해두고, 그래프 직렬화 시 이름으로 다시 만든다.

class BlockRegistry
{
public:
    using Factory = std::function<std::unique_ptr<IProcessBlock>()>;

    void Register(const std::string& name, Factory f)
    {
        m_factories[name] = std::move(f);
    }

    std::unique_ptr<IProcessBlock> Create(const std::string& name) const
    {
        auto it = m_factories.find(name);
        return it != m_factories.end() ? it->second() : nullptr;
    }

private:
    std::unordered_map<std::string, Factory> m_factories;
};

// 블록 등록
registry.Register("GaussianBlur",
    []{ return std::make_unique<GaussianBlur>(); });

// JSON 그래프 로딩
auto block = registry.Create(jsonNode["type"].get<std::string>());
다음 글로 이어진다 외부 DLL에서 블록을 동적으로 로드하면 진짜 플러그인이 된다. 다음 글에서 플러그인 아키텍처와 동적 라이브러리 로딩, 버전 호환성을 다룬다.

흔한 함정

가상 소멸자 누락 베이스 포인터로 delete할 때 자식 소멸자가 호출되지 않는다. 메모리 누수와 자원 누수, UB로 이어진다. 인터페이스의 첫 줄에 virtual ~I() = default;를 박아두자.
객체 슬라이싱(slicing) Base b = *derived; 처럼 베이스 타입 값으로 복사하면 자식 부분이 잘려나간다. 다형 객체는 항상 포인터나 참조로 다룬다.
생성자/소멸자에서 가상 함수 호출 생성·소멸 중에는 정적 디스패치로 동작한다. 자식 버전이 호출되지 않는다. 초기화/정리 로직은 별도 메서드로 빼서 객체가 완성된 뒤에 호출한다.
override 안 쓰면 조용히 새 함수가 된다 const 빠뜨리거나 인자 타입이 한 글자 다르면, 재정의가 아니라 새 가상 함수가 추가될 뿐이다. override를 항상 쓰자.
private 상속을 인터페이스 구현으로 쓰지 말자 private 상속은 "is-implemented-in-terms-of" 관계다. 인터페이스 구현은 public 상속이 정답이다. 혼동하면 클라이언트가 베이스 타입으로 다형 호출을 못 하게 된다.
