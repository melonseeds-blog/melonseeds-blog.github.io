GitHub Pages 배포 시 404 에러 해결

2026-03-24 3 min read
한줄 요약
  • GitHub 무료 플랜에서 private repo의 Pages는 지원되지 않아 404가 발생한다.
  • repo를 public으로 되돌리고, 비공개 콘텐츠는 AES-GCM 클라이언트 암호화로 보호하면 된다.

문제 상황

문제

GitHub Pages로 운영 중이던 블로그의 repository 가시성을 public에서 private으로 변경했더니, 사이트 전체에서 404 에러가 발생했다. 모든 페이지가 접근 불가 상태가 되었다.

404
There isn't a GitHub Pages site here.

원인 분석

원인

GitHub 무료 플랜(Free)에서는 private repository의 GitHub Pages를 지원하지 않는다.

GitHub Pages를 private repo에서 사용하려면 GitHub Pro, Team, 또는 Enterprise 플랜이 필요하다. 무료 플랜 사용자가 repo를 private으로 전환하면 Pages 배포가 자동으로 비활성화되어 404 에러가 발생한다.

해결 방법

해결

repository를 다시 public으로 되돌리고, 비공개가 필요한 콘텐츠는 AES-GCM 암호화를 적용하여 클라이언트 측에서 보호하는 방식으로 전환했다.

소스코드는 공개되지만, 암호화된 콘텐츠는 비밀번호 없이는 복호화할 수 없으므로 실질적인 비공개 효과를 얻을 수 있다.

시도 과정

1단계: Repository를 private으로 변경

블로그 소스코드를 비공개하기 위해 Settings에서 repository 가시성을 private으로 변경했다.

2단계: 사이트 전체 404 발생

변경 직후 블로그의 모든 페이지에서 404 에러가 발생. GitHub Pages가 완전히 비활성화되었다.

3단계: Pages 재활성화 시도 — 실패

Settings > Pages에서 다시 활성화를 시도했지만, 무료 플랜에서는 private repo의 Pages를 켤 수 없었다.

4단계: public 복구 + AES-GCM 암호화 적용

repo를 public으로 되돌리고, 비공개 콘텐츠는 AES-256-GCM 클라이언트 암호화로 보호. 사이트 정상 복구 완료.

해결 완료

public repo + AES-GCM 암호화로 보안과 배포를 동시에 해결

교훈: GitHub Pages 무료 플랜의 제약을 기억하자 GitHub Pages는 무료 플랜에서 public repo만 지원한다. 비공개가 필요한 콘텐츠는 repo 가시성이 아닌 클라이언트 측 암호화(AES-GCM 등)로 보호하는 것이 올바른 접근이다. 서버 없이 정적 사이트에서도 강력한 암호화를 적용할 수 있다.
