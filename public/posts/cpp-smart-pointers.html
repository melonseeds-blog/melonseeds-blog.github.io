왜 스마트 포인터인가

raw new/delete는 세 가지 고질병을 가진다. 메모리 누수, 이중 해제, 그리고 예외 안전성 문제다. 다음 코드는 모두 위험하다.

// 1. 누수 — 예외가 던져지면 delete 호출 안 됨
void Process()
{
    Image* img = new Image();
    DoSomething(img);   // 여기서 throw 되면 누수
    delete img;
}

// 2. 이중 해제 — 두 번 delete
Image* a = new Image();
Image* b = a;
delete a;
delete b;   // UB

// 3. 소유권 모호 — 누가 delete 해야 하나?
Image* CreateImage();   // 호출자가 delete? 아니면?

스마트 포인터는 RAII로 이 문제를 해결한다. 객체의 수명이 스코프와 묶이고, 소유권이 타입으로 명시된다. 비용은 사실상 0이거나 (unique_ptr) 작은 참조 카운트 (shared_ptr)다.

소유권 개념

스마트 포인터를 이해하는 핵심은 "누가 객체를 해제할 책임이 있는가"라는 질문이다. C++은 세 가지 의미론을 타입으로 표현한다.

의미론타입설명
단독 소유unique_ptr<T>한 시점에 한 명만 소유. 소멸 시 자동 해제.
공유 소유shared_ptr<T>여러 곳에서 공유. 마지막 하나가 사라질 때 해제.
비소유 관찰weak_ptr<T> / T* / T&참조만 함. 해제 책임 없음.

std::unique_ptr<T>

가장 많이 써야 하는 스마트 포인터다. 단독 소유이며 복사가 불가능하고 이동(move)만 가능하다.

#include <memory>

auto img = std::make_unique<Image>(1920, 1080);   // C++14 권장
// std::unique_ptr<Image> img(new Image(1920, 1080)); // 가능하지만 권장 X

auto img2 = img;             // 컴파일 에러 — 복사 불가
auto img3 = std::move(img);  // OK — 소유권 이전 (img는 nullptr)

// 함수에서 반환 — 자연스러운 factory 패턴
std::unique_ptr<Image> CreateImage()
{
    return std::make_unique<Image>();
}

비용은 0이다

unique_ptr은 raw 포인터와 동일한 크기다. 런타임 오버헤드도 없다. "안전한 raw 포인터"라고 생각하면 된다.

배열 특수화

// 단일 객체
auto p = std::make_unique<int>(42);

// 배열 — T[] 형태로 명시해야 delete[] 호출됨
auto arr = std::make_unique<int[]>(100);
arr[0] = 1;
arr[99] = 100;
배열에 unique_ptr<T> (T[] 아님)을 쓰면 UB std::unique_ptr<int> p(new int[10]) 같은 코드는 소멸 시 delete가 호출된다. delete[]가 아니다. 미정의 동작이다. 반드시 std::unique_ptr<int[]>를 쓰자.

Custom deleter — C 리소스 RAII

unique_ptr의 deleter는 타입의 일부다. C 스타일 리소스를 RAII로 감싸는 데 매우 유용하다.

// FILE* RAII
auto fileDeleter = [](FILE* f) { if (f) std::fclose(f); };
std::unique_ptr<FILE, decltype(fileDeleter)> file(
    std::fopen("data.bin", "rb"), fileDeleter);

// Win32 HANDLE RAII
auto handleDeleter = [](HANDLE h) { if (h) CloseHandle(h); };
std::unique_ptr<void, decltype(handleDeleter)> handle(
    CreateFileW(L"a.txt", GENERIC_READ, 0, nullptr,
                OPEN_EXISTING, 0, nullptr),
    handleDeleter);

// 구식 OpenCV IplImage* RAII
auto iplDeleter = [](IplImage* p) { if (p) cvReleaseImage(&p); };
std::unique_ptr<IplImage, decltype(iplDeleter)> ipl(
    cvCreateImage(cvSize(640, 480), IPL_DEPTH_8U, 1), iplDeleter);
pImpl 이디엄에도 unique_ptr이 정석 헤더에서 구현 세부를 숨기는 pImpl 패턴은 std::unique_ptr<Impl> m_impl;로 표현한다. 단, 소멸자는 cpp 파일에서 정의해야 한다 (Impl이 incomplete type일 수 없으므로).

std::shared_ptr<T>

참조 카운트 기반의 공유 소유 포인터다. 여러 shared_ptr이 같은 객체를 가리킬 수 있고, 마지막 하나가 사라지면 객체가 해제된다.

auto a = std::make_shared<Image>(1920, 1080);   // 카운트 = 1
auto b = a;   // 복사 OK — 카운트 = 2
{
    auto c = a;   // 카운트 = 3
}                 // c 소멸 — 카운트 = 2
// b, a 소멸 시 카운트 0 → 객체 해제

a.use_count();   // 현재 카운트 확인 (디버그용)

제어 블록(control block) 구조

shared_ptr은 두 가지 메모리를 가리킨다. 객체 자체와, 객체 옆에 떠 있는 제어 블록이다. 제어 블록 안에는 strong count, weak count, deleter, allocator가 들어 있다.

// shared_ptr<Image>(new Image) ─ 두 번 할당 shared_ptr ┌──────────────┐ ┌──────────────┐ ┌────────┐ │ strong cnt │ │ │ │ ptr │──┤ weak cnt │ ────► │ Image │ │ ctrl │──┤ deleter │ │ (객체) │ └────────┘ │ allocator │ │ │ └──────────────┘ └──────────────┘ 제어 블록 (heap) 객체 (heap, 별도 할당) // make_shared<Image>() ─ 한 번 할당 shared_ptr ┌─────────────────────────────────┐ ┌────────┐ │ strong cnt │ weak cnt │ deleter │ │ ptr │──┤─────────────────────────────────┤ │ ctrl │──┤ Image (객체) │ └────────┘ └─────────────────────────────────┘ 제어 블록 + 객체가 하나의 메모리 덩어리

make_shared vs new

비교std::make_shared<T>()std::shared_ptr<T>(new T)
할당 횟수1번 (객체 + 제어 블록 한꺼번에)2번 (객체 따로, 제어 블록 따로)
캐시 친화성좋음 (인접한 메모리)낮음 (분리된 위치)
예외 안전성안전특정 상황에서 누수 가능
weak_ptr 메모리객체 메모리도 weak가 살아있는 한 못 풀음strong=0이면 객체 즉시 해제
custom deleter지원 X지원 O
make_shared의 숨은 단점 제어 블록과 객체가 한 덩어리이므로, strong count가 0이 되어도 weak count가 남아 있으면 객체 메모리도 풀리지 않는다. 큰 객체에 weak_ptr이 오래 살아남으면 의도치 않은 메모리 점유가 발생한다.

비용

  • 크기: raw 포인터 2개 (객체 ptr + 제어 블록 ptr)
  • 복사/소멸 시: atomic 참조 카운트 증감 (멀티스레드 안전)
  • atomic 연산은 일반 ++보다 비싸다 — 핫 루프에서 무분별한 복사는 피하자

enable_shared_from_this

객체 내부에서 자기 자신의 shared_ptr을 얻고 싶을 때 쓴다. this로 새 shared_ptr을 만들면 제어 블록이 따로 생겨 이중 해제가 일어난다.

class Worker : public std::enable_shared_from_this<Worker>
{
public:
    void Schedule()
    {
        // this를 직접 shared_ptr로 감싸면 안 됨!
        auto self = shared_from_this();   // 제어 블록 공유
        thread_pool.Post([self]() { self->Run(); });
    }

    void Run() { /* ... */ }
};

// 반드시 shared_ptr로 생성해야 함
auto w = std::make_shared<Worker>();
w->Schedule();

Aliasing constructor

"객체 X의 수명을 따르지만, 실제로는 X의 멤버 Y를 가리키고 싶다"는 경우를 위한 생성자다.

struct Header { /* ... */ };
struct Packet { Header header; /* ... */ };

auto pkt = std::make_shared<Packet>();

// pkt의 수명에 묶여 있지만, header를 가리키는 shared_ptr
std::shared_ptr<Header> hdr(pkt, &pkt->header);

std::weak_ptr<T>

비소유 관찰자다. shared_ptr에서 만들어지지만 참조 카운트(strong)에는 영향을 주지 않는다. 객체가 살아있는지 확인하고, 살아있으면 임시로 shared_ptr을 얻는 용도다.

auto sp = std::make_shared<Image>();
std::weak_ptr<Image> wp = sp;

if (wp.expired())
{
    // 객체가 이미 해제됨
}

if (auto locked = wp.lock())   // strong shared_ptr 임시 획득
{
    locked->Process();   // 안전하게 사용
}                              // locked 소멸 — 카운트 원복

순환 참조 끊기

shared_ptr의 가장 흔한 함정은 순환 참조다. Parent와 Child가 서로를 shared_ptr로 가리키면 카운트가 절대 0이 되지 않아 영원히 누수된다.

// 잘못된 예 — 순환 참조로 누수
struct Child;
struct Parent
{
    std::shared_ptr<Child> child;
};
struct Child
{
    std::shared_ptr<Parent> parent;   // 문제!
};

// 올바른 예 — 자식 → 부모는 weak_ptr
struct Parent
{
    std::shared_ptr<Child> child;
};
struct Child
{
    std::weak_ptr<Parent> parent;     // 끊는다
};
일반 규칙 부모 → 자식은 strong, 자식 → 부모는 weak. 트리/그래프 구조에서 "위로 가는" 링크는 거의 항상 weak_ptr이다.

함수 매개변수 가이드 (Herb Sutter)

"스마트 포인터 함수에 어떻게 넘기지?"는 신입이 가장 자주 헤매는 부분이다. Herb Sutter의 가이드라인이 사실상 표준이다.

// 1. sink — 소유권을 가져간다 (이전)
void Take(std::unique_ptr<Image> img);
// 호출: Take(std::move(myUnique));

// 2. share — 소유권을 함께 가진다 (참조 카운트 증가)
void Share(std::shared_ptr<Image> img);
// 호출: Share(myShared);

// 3. 그냥 사용 — 소유권 무관, 단지 빌려서 본다
void Inspect(const Image& img);     // null 가능성 없음
void InspectMaybe(const Image* img); // nullptr 가능

// 4. shared_ptr 생명주기를 안 늘려도 된다면
void Use(const Image& img);          // ← 정답
// 다음은 모두 안티패턴
// void Use(const std::shared_ptr<Image>& img);   불필요한 복잡성
// void Use(std::shared_ptr<Image> img);          불필요한 atomic 비용
shared_ptr&는 거의 쓰지 마라 함수가 단순히 객체를 사용만 한다면 const T&이면 충분하다. 인자로 shared_ptr 자체가 필요한 경우는 정말로 카운트를 늘려야 할 때 (저장, 비동기 캡처) 뿐이다.

흔한 함정

같은 raw 포인터로 shared_ptr 두 개 만들기

Image* raw = new Image();
std::shared_ptr<Image> a(raw);
std::shared_ptr<Image> b(raw);   // 위험!

// a와 b는 서로 다른 제어 블록을 가짐
// 둘 다 strong cnt = 1
// a 소멸 → delete raw
// b 소멸 → delete raw 다시 → 이중 해제 (UB)
이중 해제 — 절대 raw 포인터로 두 번 감싸지 마라 이미 shared_ptr이 있으면 그것을 복사하라. raw로부터 새로 만드는 것이 아니다. 이 실수를 원천 차단하려면 std::make_shared를 항상 쓰는 것이 가장 안전하다.

멀티스레드에서 같은 shared_ptr 동시 read/write

std::shared_ptr<Image> g_image;   // 전역

// Thread A
g_image = std::make_shared<Image>();   // write

// Thread B
auto local = g_image;                  // read — 데이터 레이스!
참조 카운트는 atomic이지만, shared_ptr 객체 자체는 아니다 같은 shared_ptr 변수를 여러 스레드가 동시에 읽고 쓰면 데이터 레이스가 발생한다. 보호하려면 std::atomic<std::shared_ptr<T>> (C++20) 또는 mutex를 써야 한다. 각 스레드가 자신만의 복사본을 가지고 그 위에서 읽기만 한다면 안전하다.

make_shared + 큰 객체 + 오래 사는 weak_ptr

큰 객체는 make_shared 대신 new를 고려하라 make_shared는 객체와 제어 블록을 한 덩어리로 할당하기 때문에, weak_ptr이 살아있는 한 객체 메모리도 회수되지 않는다. 100MB짜리 이미지를 make_shared로 만들고 어디선가 weak_ptr이 남아 있으면 100MB가 못 풀린다. 이런 경우는 std::shared_ptr<T>(new T)가 낫다.

unique_ptr<T>에 배열 담기

위에서 다뤘듯 std::unique_ptr<int> p(new int[10])은 UB다. 배열은 반드시 std::unique_ptr<int[]>를 써야 한다. 또는 그냥 std::vector를 쓰자 — 거의 모든 경우 더 낫다.

이미지 라이브러리 적용 예

실무에서 어떻게 적용되는지 ImagingLib 컨텍스트에서 짧게 본다.

cv::Mat — 자체 참조 카운트

cv::Mat은 내부에 자체 참조 카운트를 가진다. 픽셀 데이터가 공유되며 복사가 얕은(shallow) 복사다. 따라서 shared_ptr<cv::Mat> 같은 래핑은 거의 의미 없고 오히려 카운트를 두 번 관리하게 된다. cv::Mat은 그대로 값으로 다루면 된다.

카메라 핸들 — unique_ptr + custom deleter

// Vieworks SDK 핸들 RAII
struct CameraDeleter
{
    void operator()(VWK_HANDLE h) const noexcept
    {
        if (h) VWK_CloseDevice(h);
    }
};

using CameraHandle = std::unique_ptr<
    std::remove_pointer_t<VWK_HANDLE>,
    CameraDeleter>;

CameraHandle OpenCamera(int index)
{
    VWK_HANDLE h = nullptr;
    if (VWK_OpenDevice(index, &h) != VWK_OK)
    {
        return nullptr;
    }
    return CameraHandle(h);
}

GPU 메모리 — unique_ptr + cudaFree

struct CudaDeleter
{
    void operator()(void* p) const noexcept { cudaFree(p); }
};

using CudaPtr = std::unique_ptr<void, CudaDeleter>;

CudaPtr AllocateDeviceMemory(size_t bytes)
{
    void* devPtr = nullptr;
    cudaMalloc(&devPtr, bytes);
    return CudaPtr(devPtr);
}
실전 결론 대부분의 경우 std::unique_ptr로 충분하다. 데이터 플로우 블록의 출력을 여러 후속 블록이 공유한다거나, 비동기 작업이 객체 수명을 잡고 있어야 할 때만 std::shared_ptr을 쓰자. weak_ptr은 옵저버 패턴과 순환 끊기 두 가지 용도가 거의 전부다.
