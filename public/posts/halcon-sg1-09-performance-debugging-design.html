성능 최적화의 기본 원칙

HALCON 시스템 성능은 operator 하나만 바꿔서 해결되는 경우보다 pipeline 구조를 바꿔야 좋아지는 경우가 많습니다.

전체 처리 시간:

acquisition
+ transfer
+ preprocessing
+ ROI/matching
+ segmentation
+ feature extraction
+ decision
+ display/logging
+ communication
의외의 병목 실제 현장에서는 display, image copy, logging, UI update가 병목인 경우도 많습니다.

ROI 최적화

가장 먼저 봐야 할 최적화입니다.

2.1 Search Space 줄이기

  • matching search ROI 제한
  • angle range 제한
  • scale range 제한
  • expected position 기반 ROI
  • product recipe 기반 ROI

2.2 Processing Domain 줄이기

  • reduce_domain
  • crop_domain
  • local ROI processing
  • multi-stage coarse-to-fine
이중 효과 ROI 최적화는 속도뿐 아니라 false positive 감소에도 효과가 큽니다.

Pyramid / Resolution 전략

이미지 해상도는 정확도와 속도를 동시에 좌우합니다.

3.1 Downscale이 가능한 경우

  • 존재 여부 검사
  • 대략 위치 검출
  • 큰 defect 검출
  • coarse matching

3.2 원본 해상도가 필요한 경우

  • 정밀 edge 측정
  • 작은 defect
  • OCR/code module
  • subpixel fitting
  • fine alignment

실무 패턴:

저해상도에서 coarse detection
→ 원본 해상도 ROI에서 fine measurement

Model / Handle 재사용

반복 생성하면 느리고 구조도 불안정해집니다.

재사용 대상:

  • shape model
  • measure object
  • metrology model
  • data code model
  • OCR classifier
  • calibration map
  • ROI template
  • DL model

Recipe 변경 시에만 다시 load/create하는 구조가 좋습니다.

Memory와 Copy

HALCON은 object 기반이지만 내부적으로 copy가 발생할 수 있습니다.

주의할 지점:

  • image format conversion
  • crop/copy 반복
  • color channel split 반복
  • large tuple append
  • display용 image 생성
  • 외부 SDK buffer 변환

실시간 시스템에서는 image copy를 줄이는 것이 중요합니다.

Multi-thread / Async Pipeline

6.1 Pipeline 분리

Thread 1: acquisition
Thread 2: processing
Thread 3: UI/display
Thread 4: logging/communication

주의:

  • frame order
  • buffer lifetime
  • race condition
  • result-image matching
  • thread-safe handle 사용 여부

6.2 Async Acquisition

비동기 acquisition은 처리량을 높이지만, 지연 frame 문제를 만들 수 있습니다.

관리해야 할 것:

  • frame ID
  • timestamp
  • trigger ID
  • queue size
  • dropped frame count
  • latest frame policy

Debugging 원칙

문제가 생겼을 때 최종 NG 결과만 보면 원인을 찾기 어렵습니다. 항상 pipeline 중간 결과를 저장해야 합니다.

7.1 기본 디버깅 순서

Raw image
→ ROI
→ Preprocessed image
→ Segmentation / Edge
→ Candidate objects
→ Feature values
→ Decision

7.2 저장해야 하는 정보

  • raw image
  • processed image
  • overlay image
  • parameter set
  • model version
  • recipe name
  • acquisition metadata
  • feature values
  • final decision
  • processing time

실패 원인 분류

8.1 Acquisition 문제

증상:

  • 전체 알고리즘이 가끔 흔들림
  • 밝기/contrast가 frame마다 다름
  • blur 발생
  • 특정 위치에서만 불량

원인:

  • exposure/gain
  • trigger timing
  • lighting
  • focus
  • vibration
  • camera bandwidth

8.2 ROI/Alignment 문제

증상:

  • 가끔 검사 대상이 ROI 밖으로 나감
  • 잘못된 부품을 검사
  • 위치 편차가 큰 sample에서 실패

대응:

  • 기준 matching 강화
  • ROI margin 조정
  • 다중 기준점 사용
  • fallback search ROI 추가

8.3 Segmentation 문제

증상:

  • blob area가 흔들림
  • defect가 붙거나 끊김
  • threshold가 sample마다 맞지 않음

대응:

  • 조명 개선
  • dynamic threshold
  • background correction
  • morphology 재검토
  • color/texture feature 추가

8.4 Measuring 문제

증상:

  • 치수가 불안정
  • 특정 edge를 잘못 잡음
  • offset이 일정하게 존재

대응:

  • edge profile 확인
  • measure ROI 축소
  • transition/select 재설정
  • fuzzy measure
  • calibration 검증

8.5 Matching 문제

증상:

  • score 낮음
  • false match
  • 위치가 튐
  • 특정 각도에서 실패

대응:

  • model ROI 정제
  • search range 제한
  • pyramid level 조정
  • min score 재설정
  • contrast/polarity 확인

실무 설계 패턴

9.1 안정적인 검사 기본형

1. raw image acquisition
2. global alignment
3. local ROI transform
4. local preprocessing
5. segmentation/measurement
6. feature extraction
7. rule-based decision
8. overlay/log 저장

9.2 위치 편차가 큰 시스템

coarse matching
→ fine matching
→ ROI transform
→ inspection

9.3 정밀 측정 시스템

calibration load
→ alignment
→ measure ROI transform
→ subpixel edge extraction
→ world coordinate conversion
→ tolerance decision

9.4 외관 검사 시스템

illumination normalization
→ defect enhancement
→ candidate extraction
→ feature filtering
→ classifier/DL
→ final rule

9.5 다품종 Recipe 시스템

product ID
→ recipe load
→ camera parameter set
→ model/ROI/threshold load
→ inspection
→ result with recipe version

Parameter 관리

중상급 시스템에서는 parameter가 코드에 하드코딩되면 유지보수가 어렵습니다.

관리할 parameter:

  • camera exposure/gain
  • ROI geometry
  • matching parameter
  • threshold
  • morphology size
  • measure parameter
  • calibration file
  • tolerance
  • output mapping

권장:

  • recipe 파일
  • version 관리
  • parameter 변경 로그
  • offline tuning tool
  • golden sample dataset

Validation

알고리즘 개발 후 반드시 validation set으로 확인해야 합니다.

필요 sample:

  • 정상 양품
  • 한계 양품
  • 대표 불량
  • 조명 편차
  • 위치 편차
  • 회전 편차
  • 초점 편차
  • 표면 편차
  • lot 편차

평가 지표:

  • false reject
  • false accept
  • repeatability
  • processing time
  • score/feature margin
  • robustness under variation

현장 튜닝 순서

문제가 생겼을 때 무작정 threshold를 조정하지 말고 다음 순서로 봅니다.

  1. 원본 이미지 품질
  2. 조명/노출/초점
  3. ROI/alignment
  4. preprocessing
  5. segmentation/edge
  6. feature 분포
  7. decision threshold
  8. 예외 처리
안 좋은 패턴 이 순서를 지키면 SW parameter로 HW 문제를 덮는 상황을 줄일 수 있습니다. 반대로 하면 단기에는 통과돼도 lot이 바뀌는 순간 무너집니다.

핵심 정리

  • 최적화는 ROI, search range, 재사용, pipeline 분리부터 봐야 합니다.
  • 디버깅은 최종 결과가 아니라 중간 결과를 저장해야 합니다.
  • 대부분의 불안정성은 acquisition, ROI, segmentation 단계에서 발생합니다.
  • parameter는 recipe/version 기반으로 관리해야 합니다.
  • validation dataset 없이 현장 감으로 threshold를 잡으면 장기 안정성이 떨어집니다.
