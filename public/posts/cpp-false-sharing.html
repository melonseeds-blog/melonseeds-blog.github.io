CPU 캐시 기본

현대 CPU는 메인 메모리(RAM)가 너무 느리기 때문에 코어 안에 작고 빠른 메모리를 두고 데이터를 임시로 보관한다. 이게 캐시다. 보통 L1 / L2 / L3 세 단계로 계층화되어 있다.

계층크기 (대략)접근 비용 (사이클)특징
레지스터수십 byte0코어 내부, 즉시 사용
L1 캐시32~64 KB3~5코어별 전용
L2 캐시256 KB ~ 1 MB10~15코어별 또는 소수 코어 공유
L3 캐시수~수십 MB30~50모든 코어 공유
메인 메모리수 GB200~300매우 느림

핵심은 캐시가 캐시 라인(cache line)이라는 단위로 메모리를 읽고 쓴다는 점이다. 보통 64 byte다. 단 1 byte를 읽어도 그 byte가 속한 64 byte 라인 전체가 캐시에 올라온다.

캐시 일관성 — MESI 프로토콜 멀티코어 시스템에서 같은 메모리 영역이 여러 코어 캐시에 동시에 존재할 수 있다. MESI는 각 캐시 라인의 상태를 Modified / Exclusive / Shared / Invalid 네 가지로 추적해 일관성을 유지한다. 한 코어가 라인을 수정하면 다른 코어가 가진 같은 라인은 Invalid로 떨어지고, 다음 접근 시 다시 가져와야 한다.

false sharing이란

서로 다른 스레드가 서로 다른 변수를 수정하는데, 그 변수들이 우연히 같은 캐시 라인에 있어 발생하는 성능 함정이다. 데이터 의존성이 전혀 없는데도 캐시 일관성 프로토콜이 그 라인을 계속 코어 사이로 옮기느라 폭주한다.

흐름은 이렇게 된다.

  1. 스레드 1이 코어 A에서 변수 X를 수정 → 라인이 A의 캐시에 Modified 상태로 들어간다.
  2. 스레드 2가 코어 B에서 같은 라인의 변수 Y를 수정하려 함 → A의 라인은 Invalid가 되고, B가 라인을 가져간다.
  3. 다시 스레드 1이 X를 수정 → B의 라인이 Invalid가 되고 A가 다시 가져온다.
  4. 이 핑퐁이 반복되면서 사실상 매번 캐시 미스가 발생한다.
캐시 라인 64 byte (int 4개, 16 byte 정렬 가정 시 c[0]~c[3])
c[0]
스레드 1 수정
c[1]
스레드 2 읽기 → invalid
c[2]
c[3]
스레드 1이 c[0]을 수정하면 라인 전체가 dirty가 된다. 스레드 2가 자기 변수 c[1]만 읽으려 해도 같은 라인이라 invalid 상태에서 다시 가져와야 한다.

결과적으로 스레드 수를 늘리면 빨라져야 할 작업이 오히려 느려진다. 수십 배 차이가 나는 경우도 흔하다.

전형적 발생 패턴

가장 흔한 패턴은 스레드별 카운터를 배열로 인접하게 배치하는 경우다.

// 나쁜 예 — 인접한 카운터들이 같은 캐시 라인에 들어간다
struct Counters
{
    int c[8];  // int 8개 = 32 byte → 같은 64 byte 라인 안에 모두 들어감
};

Counters g_counters;

void Worker(int tid)
{
    for (int i = 0; i < 1'000'000; ++i)
    {
        g_counters.c[tid] += 1;  // 각 스레드가 자기 인덱스만 증가
    }
}

겉보기에는 각 스레드가 완전히 독립된 변수를 다루지만, c[0]부터 c[7]까지 모두 같은 캐시 라인에 있어 스레드끼리 라인을 핑퐁하게 된다.

진단 방법

의심 단서 — 스레드 수 늘렸는데 안 빨라진다 병렬화 가능한 작업인데 스레드 1개로 돌렸을 때보다 8개로 돌렸을 때가 오히려 느리거나 비슷하면 false sharing을 강하게 의심해야 한다. CPU 사용률은 높은데 throughput이 안 나온다면 거의 확실하다.
측정 도구 Linux의 perf c2c는 캐시 라인 핑퐁을 직접 추적해주는 가장 강력한 도구다. Intel VTune의 Memory Access 분석에는 false sharing 전용 항목이 있다. Windows에서는 VTune 또는 AMD uProf를 쓴다.

회피 방법

패딩(padding)

변수마다 캐시 라인 크기만큼 채워서 서로 다른 라인에 들어가도록 강제한다.

struct alignas(64) PaddedCounter
{
    int value;
    char pad[64 - sizeof(int)];  // 라인 끝까지 채움
};

std::array<PaddedCounter, 8> counters;

void Worker(int tid)
{
    for (int i = 0; i < 1'000'000; ++i)
    {
        counters[tid].value += 1;  // 각 카운터가 별개 라인
    }
}

std::hardware_destructive_interference_size (C++17)

플랫폼별 캐시 라인 크기를 컴파일 타임에 알려준다. 64라고 하드코딩하지 않아도 된다.

#include <new>

struct alignas(std::hardware_destructive_interference_size) Counter
{
    int value;
};

Counter counters[8];  // 각 원소가 별개 캐시 라인

스레드별 지역 변수 (가장 권장)

가장 깔끔한 방법은 애초에 공유하지 않는 것이다. 각 워커가 스택에 지역 변수를 두고 누적한 뒤, 작업 끝에 한 번만 공유 변수에 합산한다.

std::atomic<long long> g_total{0};

void Worker(int tid, int iterations)
{
    long long localSum = 0;  // 스레드 스택에 존재 — 캐시 공유 없음
    for (int i = 0; i < iterations; ++i)
    {
        localSum += DoWork(i);
    }
    g_total += localSum;  // 마지막에 한 번만 동기화
}
패딩보다 지역 변수가 우선이다 패딩은 어쩔 수 없이 공유해야 할 때의 차선책이다. "워커 안에서 누적, 끝에 합산" 패턴으로 풀 수 있는 문제라면 그쪽이 더 빠르고, 코드도 단순하다.

true sharing과 cache friendly

true sharing은 같은 변수를 여러 스레드가 의도적으로 접근하는 경우다. 동기화가 필요한 진짜 공유다. 작은 데이터를 자주 수정하면 어쩔 수 없이 비용이 발생한다.

구분false sharingtrue sharing
실제 공유 데이터없음 (서로 다른 변수)있음 (같은 변수)
의도공유 의도 없음의도적 공유
해결패딩, 분리배치 처리, 락 단위 키우기
비용피할 수 있는 비용본질적 비용

true sharing은 더 큰 단위로 작업을 묶어 동기화 횟수를 줄이는 식으로 완화한다. 한편 std::hardware_constructive_interference_size는 그 반대로 같이 두면 좋은 데이터의 권장 크기를 알려준다. 자주 함께 접근되는 멤버는 같은 라인에 두는 편이 캐시에 유리하다.

벤치마크 예시

// 나쁜 예 — 같은 라인에 8개 카운터
struct BadCounters
{
    int c[8];
};

// 좋은 예 — 각 카운터가 자기 라인 차지
struct alignas(64) GoodCounter
{
    int value;
};
GoodCounter counters[8];
측정 결과 (4코어, 각 스레드 1천만 회 증가)
BadCounters : 1820 ms GoodCounter[] : 240 ms speedup : 7.6x

4코어 머신에서 3~10배 차이는 흔하다. 코어 수가 많을수록 라인 핑퐁 비용이 더 커지므로 차이가 더 벌어진다.

이미지 처리 컨텍스트 적용

이미지 처리에서는 큰 이미지를 영역으로 나눠 여러 스레드가 동시에 처리하는 패턴이 흔하다. 이때 각 스레드의 결과를 누적하는 변수가 같은 캐시 라인에 있으면 성능이 무너진다.

// 나쁜 예 — histogramSums가 같은 라인에 빽빽하게 배치
double histogramSums[NumThreads];

ParallelFor(0, image.GetHeight(), [&](int row, int tid)
{
    for (int col = 0; col < image.GetWidth(); ++col)
    {
        histogramSums[tid] += image.At(row, col);  // false sharing
    }
});
// 좋은 예 — 로컬 변수에 누적, 끝에 합산
std::atomic<double> totalSum{0.0};

ParallelFor(0, image.GetHeight(), [&](int row, int /*tid*/)
{
    double localSum = 0.0;  // 스레드 스택
    for (int col = 0; col < image.GetWidth(); ++col)
    {
        localSum += image.At(row, col);
    }
    totalSum += localSum;
});

밝기 합, 히스토그램, 결함 카운팅, 통계 측정 — 이미지 처리에서 reduction 패턴은 어디에나 있다. 모두 같은 원칙이 적용된다.

흔한 함정

data-oriented design 의도라도 멀티스레드에선 역효과 작은 멤버를 빽빽하게 배치한 구조체는 단일 스레드 캐시 효율에는 좋지만, 여러 스레드가 인접 원소를 수정하면 false sharing의 온상이 된다.
std::vector<int>를 인접 인덱스로 쓰는 패턴 N개 스레드가 v[tid]를 빈번히 수정하면 BadCounters와 똑같이 false sharing이 발생한다. 결과 누적은 별도 구조로 빼야 한다.
모든 변수에 패딩을 박지 말 것 false sharing 회피한답시고 작은 변수마다 64 byte를 채우면 메모리 사용량이 폭증하고 단일 스레드 캐시 히트율이 떨어진다. 진짜 핫스팟에만 적용한다.
컴파일러는 자동으로 라인 정렬을 안 해준다 멀티스레드 시나리오를 컴파일러는 모른다. alignas와 패딩을 명시적으로 지정해야 한다.

다른 캐시 친화 기법

false sharing은 캐시와 성능 이야기의 한 부분일 뿐이다. 실무에서는 다음 주제들도 함께 다룬다.

  • Data-Oriented Design — 데이터 접근 패턴을 우선해 자료구조를 설계
  • SoA vs AoS — Struct of Arrays / Array of Structs. 어떤 필드를 같이 쓰느냐에 따라 선택
  • 메모리 접근 패턴 — 순차(sequential) 접근이 무작위 접근보다 압도적으로 빠르다
  • Prefetching — 다음에 쓸 데이터를 CPU에 미리 알려주기 (__builtin_prefetch)
이 시리즈에서는 깊이 들어가지 않는다 각 주제는 책 한 권 분량이다. 이 글에서는 false sharing이 캐시 친화 코드의 한 조각이라는 점만 짚어두고, 다른 글들에서 자연스럽게 연결한다.
