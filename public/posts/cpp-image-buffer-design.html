이미지 버퍼 클래스가 풀어야 하는 문제

산업용 비전 라이브러리에서 이미지 버퍼 클래스는 가장 먼저 결정해야 하는 핵심 타입이다. 단순한 "픽셀 데이터 wrapper"처럼 보이지만, 실제로는 여러 직교적인 요구를 동시에 만족해야 한다.

문제구체 사례
메모리 소유권누가 release를 책임지는가 — 클래스 소멸자, 사용자, 외부 SDK?
다양한 픽셀 포맷Mono8, Mono12 packed, Mono16, Bayer RG/BG, BGR8, RGBA8, Float32
메모리 위치일반 CPU 힙, GPU 디바이스 메모리, Pinned 호스트 메모리, Mapped 메모리
ROI / sub-image원본의 일부분만 처리할 때 깊은 복사할지 얕은 참조할지
외부 SDK 버퍼 wrap카메라 SDK가 자체 버퍼에 채워준 데이터를 소유권 안 가져가고 처리
멀티스레드 안전성참조 카운트 원자성, 복사 비용, 한 이미지를 여러 블록이 동시에 읽기

이 모든 요구를 한 클래스로 해결하려고 하면 인터페이스가 비대해지고, 거꾸로 너무 많이 쪼개면 데이터 플로우가 파편화된다. 균형점을 찾는 것이 설계의 핵심이다.

참조 사례 — cv::Mat / HObject / Vision Pro

유명 라이브러리들이 이 문제를 어떻게 풀었는지 한 줄씩 정리한다.

  • OpenCV cv::Mat — 헤더(width/height/step/data ptr)와 데이터 본체를 분리하고, 참조 카운트 기반 얕은 복사. 복사 비용 O(1), clone()으로 명시적 깊은 복사.
  • HALCON HObject — 이미지/리전/XLD를 하나의 추상 타입으로 묶어 데이터 플로우 관점에서 통일. 내부적으로 광범위한 참조 카운트.
  • Cognex Vision Pro CogImage8Grey 계열 — 픽셀 포맷별로 클래스 분리, ROI는 별도 root/region 객체. C# 친화적 reference semantics.
  • MIL MIL_ID — opaque handle로 추상화, ABI를 C 핸들 수준으로 안정화.

공통 패턴은 두 가지다. (1) 헤더와 데이터를 분리한다. (2) 데이터는 참조 카운트로 공유한다.

설계 결정 트리

value vs reference semantics

항목value semantics (deep copy)reference semantics (shallow copy)
복사 비용O(N) — 전체 픽셀 복사O(1) — 헤더만 복사
예측 가능성높음 (수정이 다른 객체에 영향 없음)낮음 (의도치 않은 공유 가능)
ROI 표현원본을 잘라낸 새 메모리같은 데이터 + 다른 offset/stride
대표 예std::vectorcv::Mat, std::shared_ptr

표준 컨테이너 vs 직접 관리

std::vector<uint8_t>로 픽셀을 들고 있으면 RAII가 거저 따라온다. 다만 aligned 할당이 까다롭고(SIMD/GPU는 보통 32 byte 또는 64 byte 정렬 필요), 외부 SDK 버퍼를 wrap하는 시나리오를 표현하기 어렵다. 산업용 비전 라이브러리에서는 직접 관리가 일반적이다.

shared_ptr 참조 카운트 vs unique_ptr

이미지가 데이터 플로우 그래프에서 여러 블록의 입력으로 동시에 사용되는 시나리오에서는 공유 소유권이 자연스럽다. 따라서 픽셀 데이터 본체는 std::shared_ptr<uint8_t[]> 또는 사용자 정의 control block을 쓰는 것이 일반적이다.

aligned 할당

SIMD(SSE/AVX)와 CUDA의 cudaMemcpy 모두 정렬된 포인터에서 더 빠르다. std::aligned_alloc(C++17), MSVC의 _aligned_malloc 등을 사용한다. 해제는 반드시 짝이 맞는 함수로 한다.

aligned 메모리 해제 짝 _aligned_malloc은 _aligned_free로, std::aligned_alloc은 std::free로 해제한다. delete[]로 해제하면 UB. shared_ptr에 담을 때 반드시 커스텀 deleter를 같이 등록한다.

기본 Image 클래스 (value-like, deep copy)

가장 단순한 출발점은 value semantics 버전이다. RAII로 메모리를 들고, 복사 시 깊은 복사를 한다. 이해가 쉽고 동작이 예측 가능하다.

// Image.h
#pragma once
#include <cstdint>
#include <cstddef>

namespace ImagingLib::Core
{
    enum class PixelFormat
    {
        Mono8,
        Mono16,
        BGR8,
        RGBA8,
        BayerRG8,
        Float32
    };

    class Image
    {
    public:
        Image() noexcept = default;
        Image(int width, int height, PixelFormat format);
        ~Image();

        // Deep copy semantics.
        Image(const Image& other);
        Image& operator=(const Image& other);

        // Move transfers ownership.
        Image(Image&& other) noexcept;
        Image& operator=(Image&& other) noexcept;

        int          Width()  const noexcept { return m_width; }
        int          Height() const noexcept { return m_height; }
        PixelFormat  Format() const noexcept { return m_format; }
        size_t       Stride() const noexcept { return m_stride; }
        uint8_t*     Data()         noexcept { return m_data; }
        const uint8_t* Data()  const noexcept { return m_data; }
        bool         IsEmpty() const noexcept { return m_data == nullptr; }

    private:
        int         m_width  = 0;
        int         m_height = 0;
        PixelFormat m_format = PixelFormat::Mono8;
        size_t      m_stride = 0;     // bytes per row, including alignment padding.
        uint8_t*    m_data   = nullptr;
    };
}

주요 포인트는 다음과 같다.

  • Width/Height 같은 단순 getter는 noexcept 값 반환 — ErrorCode 불필요.
  • Stride는 정렬 패딩을 포함한 행당 바이트 수. width * bpp와 다를 수 있다.
  • 5가지 특수 멤버를 모두 명시 — Rule of Five.
  • 할당은 std::aligned_alloc으로 64 byte 정렬, 소멸자에서 짝 맞는 free 호출.
Stride가 width와 다른 이유 예를 들어 폭 1023 Mono8 이미지를 64 byte 정렬로 만들면 stride는 1024가 된다. 픽셀 접근은 data + y * stride + x * bpp 공식으로 한다. width * bpp로 계산하면 행 단위로 어긋난다.

참조 카운트 버전 (cv::Mat 스타일, shallow copy)

데이터 플로우 그래프에서 한 이미지가 여러 블록의 입력이 되는 경우, value 복사는 비용이 크다. 헤더와 데이터를 분리하고 데이터 본체에 참조 카운트를 붙이면 복사가 O(1)이 된다.

// Header is light, data body is shared.
class Image
{
public:
    Image() noexcept = default;
    Image(int width, int height, PixelFormat format);

    // Shallow copy — only refcount bumps.
    Image(const Image& other) noexcept = default;
    Image& operator=(const Image& other) noexcept = default;

    // Explicit deep copy.
    Image Clone() const;

    int          Width()  const noexcept { return m_width; }
    size_t       Stride() const noexcept { return m_stride; }
    uint8_t*     Data()         noexcept { return m_dataBlock.get() + m_offset; }
    const uint8_t* Data()  const noexcept { return m_dataBlock.get() + m_offset; }

private:
    int         m_width  = 0;
    int         m_height = 0;
    PixelFormat m_format = PixelFormat::Mono8;
    size_t      m_stride = 0;
    size_t      m_offset = 0;          // for ROI: offset into shared block.
    std::shared_ptr<uint8_t[]> m_dataBlock;  // refcounted body.
};
얕은 복사인데 사용자가 깊은 복사를 기대하는 함정 Image b = a; b.SetPixel(0,0,255); 가 a를 변경한다. 문서로 명시하고, 깊은 복사가 필요하면 Clone()을 호출하도록 강제한다. cv::Mat과 같은 모델을 따른다고 명시하면 사용자 학습 비용이 줄어든다.

"copy-on-write"는 이론적으로 매력적이지만, 매 write마다 참조 카운트를 검사하고 락을 걸어야 한다. 멀티스레드 환경에서 비용이 커서 cv::Mat을 포함한 대부분의 라이브러리가 채택하지 않는다.

pImpl 패턴 적용

SDK를 DLL/SO로 배포하는 순간 ABI 안정성이 중요해진다. 헤더에 std::shared_ptr, std::vector 등 표준 컨테이너 멤버가 노출되면 표준 라이브러리 버전이 바뀔 때 ABI가 깨질 수 있다. pImpl(pointer to implementation)이 표준 해법이다.

// Image.h — public header, ABI-stable.
#pragma once
#include <memory>

namespace ImagingLib::Core
{
    enum class PixelFormat;

    class Image
    {
    public:
        Image() noexcept;
        Image(int width, int height, PixelFormat format);
        ~Image();

        Image(const Image& other);
        Image& operator=(const Image& other);
        Image(Image&& other) noexcept;
        Image& operator=(Image&& other) noexcept;

        int          Width()  const noexcept;
        int          Height() const noexcept;
        PixelFormat  Format() const noexcept;
        const uint8_t* Data() const noexcept;

    private:
        class Impl;
        std::unique_ptr<Impl> m_impl;
    };
}
// Image.cpp — implementation, free to change.
class Image::Impl
{
public:
    int         width  = 0;
    int         height = 0;
    PixelFormat format = PixelFormat::Mono8;
    size_t      stride = 0;
    std::shared_ptr<uint8_t[]> dataBlock;
};

Image::Image() noexcept : m_impl(std::make_unique<Impl>()) {}
Image::~Image() = default;          // Impl is complete here.
int Image::Width() const noexcept { return m_impl->width; }
// ...
pImpl의 효과 내부 구현을 바꿔도 헤더 재컴파일 불필요 → 컴파일 시간 단축. DLL 경계에서 표준 라이브러리 ABI가 노출되지 않음 → 호환성 안정. 단점은 추가 간접 참조 1회와 동적 할당 1회. 이미지 처리처럼 픽셀당 비용이 지배적인 도메인에서는 무시할 만한 수준이다.

CPU/GPU 메모리 추상화

CUDA 파이프라인을 도입하기 시작하면 데이터가 어디에 있는지 추적해야 한다. 무지성 매번 GPU↔CPU 복사를 하면 PCIe 대역폭이 병목이 된다. 데이터가 머무는 위치를 enum으로 들고, 필요할 때만 옮긴다.

위치의미비고
Cpu일반 호스트 힙aligned_alloc
Pinnedpage-locked 호스트cudaMallocHost — H2D 전송 빠름
Gpu디바이스 메모리cudaMalloc
enum class MemoryLocation { Cpu, Pinned, Gpu };

class Image
{
public:
    MemoryLocation Location() const noexcept;

    // Ensure the data is reachable from the requested side.
    // Triggers an internal copy only if the location does not match.
    ErrorCode EnsureCpu();
    ErrorCode EnsureGpu();

    // Raw pointers — caller must call EnsureXxx first.
    const uint8_t* CpuData() const noexcept;
    const uint8_t* GpuData() const noexcept;
};

핵심 아이디어는 lazy 동기화다. 파이프라인이 GPU에서 시작해 GPU 블록만 거쳐 GPU에서 끝나면 CPU 왕복이 한 번도 일어나지 않는다. 사용자가 결과를 화면에 띄울 때 비로소 EnsureCpu가 호출되어 한 번 복사한다.

GPU 비동기 실행 후 CPU 접근 위험 cudaMemcpyAsync 또는 커널 launch 직후 CPU에서 데이터를 읽으면 미완료 상태일 수 있다. EnsureCpu 내부에서 반드시 cudaStreamSynchronize 또는 이벤트 wait를 해야 한다. 동기화를 빠뜨리면 디버깅이 매우 어려운 데이터 레이스가 생긴다.

ROI / sub-image

처리할 영역이 이미지 전체가 아닌 경우, 사본을 만들지 않고 원본의 일부분만 참조하는 ROI 객체가 자연스럽다. 참조 카운트와 결합하면 거의 공짜다.

struct Rect { int x, y, width, height; };

// Returns a new Image header that shares the underlying data block.
// The data pointer is shifted by (y * stride + x * bpp);
// stride remains the parent's stride.
Image Image::GetRoi(const Rect& rect) const;
ROI는 데이터를 공유한다 ROI에 쓰면 부모 이미지에도 반영된다. ROI 안의 stride는 부모와 같으므로 행 사이에 "다른 픽셀"이 끼어 있다는 점을 처리 코드가 인지해야 한다. SIMD/GPU 커널이 행 단위로 stride를 받도록 짜야 한다.

외부 SDK 버퍼 wrap

Vieworks 같은 카메라 SDK는 자체 메모리 풀에 이미지를 채워준다. 우리 라이브러리가 이 메모리를 복사하지 않고 그대로 처리하려면, 소유권을 가져가지 않는 wrap 모드가 필요하다.

// Wrap an externally owned buffer. The caller-supplied deleter is invoked
// when the last Image referencing this block is destroyed. Pass an empty
// deleter to indicate that the buffer must outlive every Image instance.
static Image Image::WrapExternal(
    uint8_t* data,
    int width,
    int height,
    PixelFormat format,
    size_t stride,
    std::function<void(uint8_t*)> deleter);
소유권을 가져갈지 명시 deleter를 등록하면 우리가 소유권을 가져간 것이다. 빈 deleter면 외부가 책임진다. 이 둘을 헷갈리면 double free 또는 dangling pointer가 발생한다. 카메라 SDK에서 받은 버퍼는 보통 SDK가 풀로 재사용하므로 빈 deleter + 짧은 수명이 정답이다.

에러 처리 — ErrorCode 패턴 통합

프로젝트 컨벤션을 그대로 따른다. 단순 getter는 값을 반환하고, 메모리 할당이나 복사 같은 실패 가능 동작은 ErrorCode를 반환한다.

// Simple getters — no ErrorCode.
int          Width()  const noexcept;
int          Height() const noexcept;
bool         IsEmpty() const noexcept;

// Actions that can fail — return ErrorCode.
ErrorCode    AllocBuffer(int width, int height, PixelFormat format);
ErrorCode    CopyTo(Image& dst) const;
ErrorCode    EnsureGpu();

내부 구현에서는 std::bad_alloc 같은 예외를 잡아 ErrorCode로 변환한다. 예외가 API 경계를 넘어가지 않도록 한다.

흔한 함정

얕은 복사 의도 미공유 cv::Mat 스타일을 채택했으면서 문서에 "deep copy"처럼 적어두면 사용자가 혼란을 겪는다. 명시적으로 "shallow copy + Clone()" 모델임을 첫 페이지에 적는다.
aligned 메모리에 잘못된 deallocator _aligned_malloc으로 받은 포인터를 delete[]로 해제하면 UB. shared_ptr<uint8_t[]>에 담을 때 반드시 커스텀 deleter를 등록한다.
GPU 메모리를 CPU 포인터처럼 접근 cudaMalloc으로 받은 포인터를 memcpy나 일반 dereference하면 segfault. EnsureCpu를 거쳐 호스트 포인터를 얻는 경로를 강제한다.
DLL 경계에 표준 컨테이너 노출 헤더에 std::vector, std::string이 멤버로 있으면 표준 라이브러리 버전/플래그가 다른 클라이언트와 ABI가 깨진다. pImpl로 숨긴다.

Image 외 컨테이너 — Region, XLD Contour

비전 라이브러리의 데이터 플로우는 Image만 다루지 않는다. 같은 설계 원칙이 다른 컨테이너에도 적용된다.

  • Region — 픽셀 마스크. 메모리 절약을 위해 보통 RLE(run-length encoding)로 인코딩한다. 픽셀이 100만 개라도 연결된 영역이면 수십 byte로 표현된다.
  • XLD Contour — 서브픽셀 정밀도 윤곽선. std::vector<Point2f>로 들고 있고, 위상(open/closed) 메타를 추가한다.

이들을 같은 base 인터페이스(예: IFlowData) 아래 노출하면 데이터 플로우 블록이 입출력을 통일된 방식으로 다룰 수 있다.

다음 글로 이어지는 지점 Image, Region, XLD를 어떤 인터페이스로 묶을지 — IProcessBlock과 IFlowData의 관계, 플러그인 경계에서의 타입 erasure 등은 다음 "C++ 인터페이스 패턴" 글에서 이어 다룬다.
