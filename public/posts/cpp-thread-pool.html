왜 스레드 풀인가

매번 작업이 들어올 때마다 std::thread를 새로 만드는 건 비싸다. OS 시스템 콜로 커널 스레드를 생성하고, 스택(보통 1MB)을 할당하고, 스케줄러에 등록하고, 끝나면 다시 join 비용을 치러야 한다. 짧은 작업 1000개를 처리한다고 매번 thread를 만들면, 정작 작업 시간보다 생성/소멸 비용이 더 커지기 쉽다.

스레드 풀은 워커 스레드 N개를 미리 만들어 두고 재사용한다. 작업은 큐에 넣기만 하면 되고, 노는 워커가 알아서 꺼내 처리한다. 핵심은 두 가지다.

  • 스레드 생성 비용 분할상환 — 한 번 만든 워커가 수많은 작업을 처리한다.
  • 동시 실행 수 제어 — 코어 수에 맞춘 N개로 context switch 부하를 줄인다.
하드웨어 코어 수와 워커 수의 균형 std::thread::hardware_concurrency()로 논리 코어 수를 얻을 수 있다. CPU 바운드 작업이면 코어 수만큼, I/O 대기가 많은 작업이면 코어 수보다 더 많이 두는 편이 처리량이 높다. 두 경우의 최적값은 다르다.

최소한의 스레드 풀을 만든다면 다음 4가지가 필요하다.

구성 요소역할
워커 스레드 N개큐에서 작업을 꺼내 실행하는 무한 루프 스레드

Producer-Consumer 패턴

스레드 풀은 본질적으로 Producer-Consumer 패턴이다. 작업을 큐에 넣는 쪽이 Producer, 꺼내 실행하는 쪽이 Consumer다.

큐가 비어 있을 때 워커는 condition_variable::wait()로 잠들어 있다가, Producer가 작업을 넣고 notify_one()을 호출하면 깨어나 작업을 꺼낸다. 바쁜 대기(busy-wait) 없이 효율적으로 동기화된다.

ThreadPool 클래스 구현

핵심 기능만 담은 단순한 ThreadPool. submit()은 어떤 호출 가능 객체든 받아 std::future로 결과를 돌려준다.

#include <vector>
#include <queue>
#include <thread>
#include <mutex>
#include <condition_variable>
#include <future>
#include <functional>
#include <memory>

class ThreadPool {
    std::vector<std::thread> workers_;
    std::queue<std::function<void()>> tasks_;
    std::mutex mtx_;
    std::condition_variable cv_;
    bool stop_ = false;

public:
    explicit ThreadPool(size_t n = std::thread::hardware_concurrency())
    {
        if (n == 0) n = 1;  // hardware_concurrency()가 0을 반환할 수 있음
        for (size_t i = 0; i < n; ++i)
        {
            workers_.emplace_back([this] { workerLoop(); });
        }
    }

    ~ThreadPool()
    {
        {
            std::lock_guard<std::mutex> lock(mtx_);
            stop_ = true;
        }
        cv_.notify_all();
        for (auto& w : workers_) w.join();
    }

    // 임의의 호출 가능 객체를 받아 future로 결과 반환
    template<class F, class... Args>
    auto submit(F&& f, Args&&... args)
        -> std::future<std::invoke_result_t<F, Args...>>
    {
        using R = std::invoke_result_t<F, Args...>;
        auto task = std::make_shared<std::packaged_task<R()>>(
            std::bind(std::forward<F>(f), std::forward<Args>(args)...));
        std::future<R> fut = task->get_future();
        {
            std::lock_guard<std::mutex> lock(mtx_);
            if (stop_) throw std::runtime_error("submit on stopped pool");
            tasks_.emplace([task] { (*task)(); });
        }
        cv_.notify_one();
        return fut;
    }

private:
    void workerLoop()
    {
        for (;;)
        {
            std::function<void()> job;
            {
                std::unique_lock<std::mutex> lock(mtx_);
                cv_.wait(lock, [this] { return stop_ || !tasks_.empty(); });
                if (stop_ && tasks_.empty()) return;
                job = std::move(tasks_.front());
                tasks_.pop();
            }
            job();  // 락을 풀고 실행
        }
    }
};

구현의 핵심 포인트.

  • packaged_task를 shared_ptr로 감싸는 이유는 std::function이 복사 가능 호출 객체를 요구하기 때문이다 (packaged_task는 move-only).
  • cv_.wait()의 술어는 spurious wakeup을 막는다. stop_이 true이거나 큐에 작업이 있을 때만 깨어난다.
  • 소멸자에서 stop_ = true 후 notify_all() — 모든 워커가 깨어나 큐가 빌 때까지 처리하고 종료한다 (graceful shutdown).
  • job()은 락을 푼 뒤 실행한다. 실행 중에 락을 잡고 있으면 워커들이 줄 서서 기다린다.
graceful vs immediate shutdown 위 구현은 큐에 남은 작업까지 모두 처리하고 종료한다. 즉시 중단이 필요하면 stop_ 체크를 더 자주 하거나, 별도의 cancel 플래그를 두고 워커가 중간에 빠져나가도록 설계해야 한다.

고급 기능 옵션

Work stealing

워커마다 자기 큐를 두고, 자기 큐가 비면 다른 워커의 큐에서 작업을 훔쳐 온다. 단일 글로벌 큐의 mutex 경합을 줄인다. Intel TBB가 대표적이며, Java의 ForkJoinPool도 같은 방식이다. 구현 자체는 lock-free deque가 필요해 단순한 풀보다 훨씬 복잡하다.

Priority queue

작업에 우선순위를 부여해 중요한 일을 먼저 처리한다. std::queue 대신 std::priority_queue를 쓰고, 작업과 priority를 함께 저장하면 된다. 다만 starvation(낮은 우선순위 작업이 영영 실행되지 않음)에 주의해야 한다.

결과 일괄 모으기

ThreadPool pool(4);
std::vector<std::future<int>> futures;

for (int i = 0; i < 100; ++i)
{
    futures.push_back(pool.submit([i] { return i * i; }));
}

int total = 0;
for (auto& f : futures) total += f.get();  // 순서대로 대기

풀 크기 가이드라인

작업 종류권장 워커 수이유
CPU 바운드코어 수 또는 코어 수 - 1너무 많으면 context switch 오버헤드. 메인 스레드용 1개 양보
I/O 바운드코어 수의 2~10배대기 중인 워커 동안 다른 워커가 일함
혼합측정해서 결정rule of thumb은 출발점일 뿐
최종은 측정으로 결정한다 실제 작업 부하와 시스템에서 throughput과 latency를 측정해 정해야 한다. 책상 계산은 출발점일 뿐이고, 같은 코드도 머신과 OS, 캐시 상태에 따라 최적값이 달라진다.

이미지 acquisition 파이프라인 적용

산업 비전 SDK에서 흔한 구조다. 카메라가 프레임을 grab하고, 워커들이 처리(필터, 측정, 결함 검출 등)하고, 결과는 display 스레드가 화면에 그린다.

여기서 중요한 건 backpressure다. 카메라가 30fps로 프레임을 던지는데 처리가 50ms 걸리면, 큐가 무한정 쌓여 메모리가 터진다. 해결책은 bounded queue(최대 크기 제한)다.

// 큐가 가득 차면 가장 오래된 프레임 버리기 (실시간 우선)
ErrorCode EnqueueFrame(std::shared_ptr<Image> frame)
{
    std::lock_guard<std::mutex> lock(mtx_);
    if (tasks_.size() >= kMaxQueueSize)
    {
        tasks_.pop();  // oldest drop
    }
    tasks_.push([frame] { ProcessFrame(*frame); });
    cv_.notify_one();
    return ErrorCode::Success;
}
drop oldest vs producer block 실시간 화면 표시는 oldest drop이 맞다 (오래된 프레임은 의미 없음). 검사 결과를 모두 저장해야 하는 inspection이라면 producer를 막아 grab 속도를 늦추거나, 별도 디스크 버퍼로 빼야 한다.

C++17/20 도구로 단순화

std::scoped_lock (C++17)

여러 mutex를 동시에 잠글 때 deadlock 없이 처리한다. 단일 mutex에도 사용 가능하다.

std::scoped_lock lock(mtx1_, mtx2_);  // 두 락 동시 획득, 순서 자동 조정

std::jthread + std::stop_token (C++20)

소멸 시 자동 join + 협력적 취소 토큰. 풀 구현이 훨씬 깔끔해진다.

std::vector<std::jthread> workers_;

workers_.emplace_back([this](std::stop_token st) {
    while (!st.stop_requested())
    {
        // 작업 처리...
    }
});

// 소멸자에서 자동으로 stop_requested()=true 후 join
// 별도의 stop_ 플래그와 join 루프가 필요 없음
std::jthread는 C++20 Visual Studio 2022 (MSVC v143) 기준으로 /std:c++20 옵션에서 사용 가능하다. C++23에서는 std::move_only_function이 추가되어 std::function의 "복사 가능 요구" 제약을 풀어준다 — packaged_task를 더 자연스럽게 큐에 넣을 수 있다.

표준 라이브러리/외부 라이브러리 대안

C++ 표준에는 아직 thread pool이 없다 (C++26 후보로 논의 중). 큰 프로젝트라면 검증된 라이브러리를 쓰는 편이 안전하다.

흔한 함정

중첩 submit 데드락 워커 안에서 다시 같은 풀에 submit()하고 그 결과를 future.get()으로 기다리면 데드락 위험이 있다. 모든 워커가 자식 작업의 완료를 기다리는데, 자식 작업을 실행할 워커가 남아있지 않으면 영원히 멈춘다. 의존성이 있으면 별도 풀을 쓰거나 continuation 패턴을 사용하자.
공유 자원에 대한 data race 스레드 안전하지 않은 자원(전역 변수, OpenCV의 일부 함수, 파일 핸들 등)에 여러 워커가 동시 접근하면 정의되지 않은 동작이다. 자원별로 mutex로 보호하거나, 워커마다 자기 사본을 두자.
풀 소멸 시 잔여 작업 누락 소멸자에서 stop_만 세우고 큐를 비우지 않으면, 큐에 남은 작업의 promise가 영영 fulfill되지 않아 future.get()이 영원히 대기한다. 소멸 정책(잔여 작업 모두 처리 vs 즉시 중단 + 예외로 future broken)을 명확히 하자.
submit한 future 무시하면 결과 폐기 반환된 std::future를 받지 않으면 결과는 그냥 사라진다 (예외도 함께 사라진다). "결과는 필요 없고 실행만 하고 싶다"면 의도적으로 무시하되, 워커 안에서 던진 예외는 어디로도 가지 않으니 주의하자.

워커 스레드 안에서 작업이 예외를 던져도 packaged_task가 잡아 future의 예외 상태로 저장한다. 풀 자체는 영향을 받지 않으므로, 호출자가 future.get()에서 예외를 받아 처리하면 된다. 단, future를 무시했다면 예외도 함께 무시된다는 점은 위에 언급한 그대로다.
