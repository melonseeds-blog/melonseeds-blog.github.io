Q1 ~ Q10

Q1Ch1 테스팅의 기초FL-1.1.1 (K2)
다음 중 테스팅의 목적에 대한 설명으로 올바르지 않은 것은?
  • A작업 산출물(요구사항, 설계, 코드 등)을 평가하는 것
  • B테스트 대상의 품질에 대한 자신감을 획득하는 것
  • C장애를 유발하고 결함을 식별하는 것
  • D소프트웨어에 결함이 없음을 보장하는 것
정답: D

원리 1: 테스팅은 결함의 존재를 밝히지만 결함이 없음을 증명하지는 않습니다. A, B, C는 모두 실러버스 1.1.1에 명시된 테스트 목적이며, D는 완벽한 테스팅이 불가능하므로 목적이 될 수 없습니다.

참조: 실러버스 1.1.1, 원리 1 & 2
Q2Ch1 테스팅의 기초FL-1.1.2 (K2)
동적 테스팅 중 특정 입력값에서 시스템이 비정상 종료되었다. 개발팀이 해당 장애의 원인을 분석한 결과, 배열 인덱스 범위 초과 코드를 발견하여 수정하였다. 이후 수정된 코드로 동일한 테스트를 다시 수행하여 문제가 해결되었음을 확인했다.
위 시나리오에서 "배열 인덱스 범위 초과 코드를 발견하여 수정" 활동은 무엇에 해당하는가?
  • A테스팅 (Testing)
  • B디버깅 (Debugging)
  • C확인 테스팅 (Confirmation testing)
  • D리그레션 테스팅 (Regression testing)
정답: B

결함을 찾고, 분석하고, 제거하는 활동은 디버깅(B)입니다. 테스팅(A)은 장애를 유발하는 활동이고, 확인 테스팅(C)은 수정 후 재검증, 리그레션 테스팅(D)은 부작용 확인입니다.

참조: 실러버스 1.1.2
Q3Ch1 테스팅의 기초FL-1.2.3 (K2)
개발자가 시간 압박 속에서 날짜 계산 로직을 구현하면서, 윤년 조건을 잘못 코딩하였다. QA팀은 이 코드를 검토하면서 조건문의 오류를 발견했다. 하지만 이 결함은 아직 실행되지 않아 사용자에게 문제를 일으키지 않은 상태이다.
위 시나리오에서 "시간 압박", "잘못 코딩된 조건문", "사용자에게 문제를 일으키지 않은 상태"는 각각 무엇에 해당하는가?
  • A결함 → 오류 → 장애
  • B오류 → 장애 → 결함
  • C근본 원인(오류의 원인) → 결함 → 아직 장애가 아님
  • D장애 → 결함 → 오류
정답: C

근본 원인(시간 압박) → 오류 → 결함(잘못된 코드) → 아직 장애 아님. C가 정확한 순서입니다. 결함이 실행되어야 장애가 되므로, 아직 실행되지 않은 상태는 장애가 아닙니다.

참조: 실러버스 1.2.3
Q4Ch1 테스팅의 기초FL-1.3.1 (K2)
프로젝트에서 로그인 모듈에 대해 수만 가지의 사용자명/비밀번호 조합을 테스트하고 싶지만, 현실적으로 모든 조합을 테스트하는 것은 불가능하다. 따라서 테스트팀은 동등분할과 경계값 분석을 사용하여 테스트 케이스를 줄이기로 했다.
위 상황과 가장 직접적으로 관련된 테스팅 원리는?
  • A완벽한 테스팅은 불가능하다
  • B결함은 집중된다
  • C테스트 효과는 줄어든다 (살충제 패러독스)
  • D조기 테스팅으로 시간과 비용을 절약할 수 있다
정답: A

모든 조합을 테스트할 수 없어 기법으로 축소하는 것은 원리 2 "완벽한 테스팅은 불가능하다(A)"에 해당합니다. B(결함 집중)은 파레토 원리, C(살충제 패러독스)는 같은 테스트 반복 시 효과 감소, D(조기 테스팅)는 초기 결함 발견 비용 효율입니다.

참조: 실러버스 1.3 원리 2
Q5Ch1 테스팅의 기초FL-1.3.1 (K2)
"베리피케이션과 함께 밸리데이션도 수행해야 한다"는 내용과 가장 관련 깊은 테스팅 원리는?
  • A테스팅은 결함의 존재를 밝히는 활동이다
  • B결함은 집중된다
  • C테스팅은 정황에 의존적이다
  • D결함 부재의 궤변
정답: D

D. 결함 부재의 궤변(원리 7): 베리피케이션으로 결함을 모두 수정해도, 밸리데이션 없이는 사용자 요구를 충족하지 못할 수 있습니다.

Q6Ch1 테스팅의 기초FL-1.4.1 (K2)
다음 활동들을 테스트 프로세스의 올바른 순서로 나열한 것은?
i. 테스트 환경 구축 및 테스트 데이터 준비
ii. 테스트 베이시스를 분석하여 테스트 컨디션 식별
iii. 테스트 케이스에서 기대 결과와 실제 결과 비교
iv. 테스트 컨디션을 커버하는 테스트 케이스 도출
  • Ai → ii → iv → iii
  • Bii → iv → i → iii
  • Civ → ii → iii → i
  • Dii → i → iv → iii
정답: B

테스트 프로세스 순서: 테스트 분석(ii) → 테스트 설계(iv) → 테스트 구현(i) → 테스트 실행(iii)

Q7Ch1 테스팅의 기초FL-1.5.2 (K1)
전체 팀 접근법(Whole Team Approach)에 대한 설명으로 올바르지 않은 것은?
  • A모든 팀원이 품질에 대한 책임을 진다
  • B테스터가 비즈니스 담당자의 적절한 인수 테스트 작성을 지원한다
  • C안전이 치명적인 시스템에서는 항상 전체 팀 접근법을 사용해야 한다
  • D같은 공간 사용(co-location)은 의사소통과 상호작용을 용이하게 한다
정답: C

C가 틀림: 안전이 치명적인 시스템에서는 높은 수준의 테스트 독립성이 필요할 수 있어 전체 팀 접근법이 부적절할 수 있습니다. A, B, D는 모두 올바른 설명입니다.

참조: 실러버스 1.5.2
Q8Ch1 테스팅의 기초FL-1.5.3 (K2)
독립적 테스팅의 단점으로 올바른 것은?
  • A저자와 다른 인지 편향으로 다른 유형의 결함을 식별한다
  • B개발자가 품질에 대한 책임감을 잃을 수 있다
  • C이해관계자의 가정을 검증하고 반증할 수 있다
  • D작업 산출물의 품질이 향상된다
정답: B

B: 독립적 테스터가 있으면 개발자가 품질 책임감을 잃을 수 있습니다. A, C, D는 모두 독립적 테스팅의 장점입니다.

Q9Ch2 SDLC와 테스팅FL-2.1.1 (K2)
순차적 개발 모델과 반복적/점진적 개발 모델에서의 테스팅 차이에 대한 설명으로 올바른 것은?
  • A순차적 모델에서는 각 반복 주기마다 리그레션 테스팅이 자동으로 수행된다
  • B반복적 모델에서는 동적 테스팅을 SDLC 초기에 수행하기 어렵다
  • C순차적 모델에서는 가벼운 작업 산출물과 테스트 자동화를 선호한다
  • D반복적 모델에서는 각 반복 주기마다 모든 테스트 레벨에서 정적/동적 테스팅을 수행할 수 있다
정답: D

D: 반복적 모델에서는 각 반복마다 동작하는 제품 증분이 생성되므로 모든 테스트 레벨에서 정적/동적 테스팅이 가능합니다. A는 순차적 모델에 반복 주기 개념이 없어 틀리고, B와 C는 각 모델의 특성이 뒤바뀌었습니다.

참조: 실러버스 2.1.1
Q10Ch2 SDLC와 테스팅FL-2.1.5 (K2)
다음 중 시프트 레프트(Shift-left) 접근법의 실천 사례가 아닌 것은?
  • A요구사항 작성 단계에서 리뷰를 수행한다
  • B코드 작성 전에 테스트 케이스를 작성한다
  • C컴포넌트 테스팅 중에 비기능 테스팅(성능 테스트)을 수행한다
  • D시스템 테스팅 완료 후에 인수 테스팅 기준을 정의한다
정답: D

시프트 레프트는 테스팅 활동을 가능한 한 일찍 수행하는 접근법입니다. 시스템 테스팅 완료 후에야 인수 테스트 기준을 정의하는 것은 늦은 단계 활동이므로 시프트 레프트에 해당하지 않습니다.

참조: 실러버스 2.1.5 시프트 레프트 접근법

DevOps가 테스팅에 미치는 영향으로 올바르지 않은 것은?

당신의 팀은 전자상거래 시스템을 개발하고 있다. 결제 모듈이 외부 PG(Payment Gateway) 시스템과 통신하는 부분에서 데이터 전송 오류가 빈번하게 발생하고 있다.
위 문제를 발견하기에 가장 적합한 테스트 레벨은?

C. 시스템 통합 테스팅은 외부 시스템(PG사)과의 인터페이스를 테스트하는 레벨입니다. 컴포넌트 테스팅(A)은 개별 모듈, 컴포넌트 통합 테스팅(B)은 내부 모듈 간 인터페이스, 인수 테스팅(D)은 비즈니스 요구 충족 여부 확인입니다.

Q14Ch2 SDLC와 테스팅FL-2.1.6 (K2)
회고(Retrospective)의 목적에 대한 설명으로 가장 적절한 것은?

Q15Ch3 정적 테스팅FL-3.1.3 (K2)
정적 테스팅과 동적 테스팅의 차이에 대한 설명으로 올바르지 않은 것은?

  • B정적 테스팅은 비-실행 작업 산출물에도 적용할 수 있다

  • C요구사항 커버리지
  • D개발자별 코드 생산성 (LOC/day)
정답: D

코드 생산성(LOC/day)은 개발 메트릭이지 테스팅 메트릭이 아닙니다.

참조: 실러버스 5.3.1 테스팅 메트릭
Q36Ch5 테스트 관리FL-5.5.1 (K3)
결함 보고서에 포함되어야 하는 정보가 아닌 것은?
  • A기대 결과와 실제 결과
  • B결함의 심각도와 우선순위
  • C결함을 수정한 코드의 상세 내용
  • D결함 재현 절차
정답: C

수정 코드의 상세 내용은 개발자의 수정 활동에 해당하며, 결함 보고서에 포함되지 않습니다.

참조: 실러버스 5.5.1 결함 보고서
Q37Ch5 테스트 관리FL-5.4.1 (K2)
형상 관리(Configuration Management)가 테스팅을 지원하는 방법으로 올바른 것은?
  • A테스트웨어의 모든 항목에 대해 고유하게 식별하고, 버전 관리하고, 변경을 추적한다
  • B테스트 케이스의 우선순위를 자동으로 결정한다
  • C결함의 근본 원인을 분석한다
  • D테스트 추정 기법을 선택하는 데 도움을 준다
정답: A

형상 관리는 테스트웨어를 포함한 모든 작업 산출물의 식별, 버전 관리, 변경 추적을 통해 일관성과 무결성을 보장합니다.

참조: 실러버스 5.4.1 형상 관리
Q38Ch6 테스트 도구FL-6.1.1 (K2)
다음 중 테스트 실행 도구의 주요 기능에 해당하는 것은?
  • A결함을 기록하고 추적하는 기능
  • B미리 작성된 스크립트를 사용하여 자동으로 테스트를 반복 실행하는 기능
  • C테스트 계획과 진행 상황을 관리하는 기능
  • D소스 코드의 정적 분석을 수행하는 기능
정답: B

B: 테스트 실행 도구는 스크립트를 사용해 자동으로 테스트를 반복 실행합니다. A는 결함 관리 도구, C는 테스트 관리 도구, D는 정적 분석 도구입니다.

참조: 실러버스 6장
Q39Ch6 테스트 도구FL-6.2.1 (K1)
테스트 자동화의 잠재적 리스크로 올바른 것은?
  • A리그레션 테스트의 반복 실행이 쉬워진다
  • B테스트 커버리지가 향상될 수 있다
  • C테스트웨어 유지 관리를 위한 충분한 노력이 할당되지 않을 수 있다
  • D수동 테스팅보다 빠르게 실행할 수 있다

A, B, D는 장점. C는 자동화의 잠재적 리스크: 자동화 테스트웨어는 지속적인 유지 관리가 필요한데 노력이 부족할 수 있음.

참조: 실러버스 6장 테스트 도구

  • A정적 분석 도구 + 결함 관리 도구
  • BCI/CD 도구 + 테스트 실행 도구 + 테스트 관리 도구

CI/CD 파이프라인: CI/CD 도구(빌드/배포 자동화) + 테스트 실행 도구(자동 테스트 수행) + 테스트 관리 도구(결과 보고/추적).
