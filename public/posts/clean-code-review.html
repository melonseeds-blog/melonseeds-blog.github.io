"클린 코드"는 소프트웨어 장인 정신의 바이블이라 불리는 책이다. 저자 로버트 C. 마틴(일명 Uncle Bob)은 수십 년간의 개발 경험을 바탕으로, 읽기 좋고 유지보수하기 쉬운 코드를 작성하는 원칙과 패턴을 체계적으로 정리했다.

단순히 "동작하는 코드"를 넘어 "깨끗한 코드"가 왜 중요한지, 그리고 어떻게 달성할 수 있는지를 구체적인 예제와 함께 설명한다. Java 코드 위주이지만, 언어를 불문하고 적용 가능한 보편적인 원칙을 담고 있다.

핵심 내용

1 의미 있는 이름 (Meaningful Names) — 변수, 함수, 클래스의 이름만 보고도 역할을 알 수 있어야 한다. d 대신 elapsedTimeInDays, process() 대신 validateUserInput()처럼 의도를 드러내는 이름을 사용하라.
2 함수는 작게 (Small Functions) — 함수는 한 가지 일만 해야 한다. 20줄을 넘기지 않는 것이 이상적이며, 추상화 수준을 일관되게 유지해야 한다. 함수가 길어지면 더 작은 함수로 분리하라.
3 주석보다 코드 (Code over Comments) — 주석은 코드로 표현하지 못하는 것을 보충할 때만 사용한다. 코드 자체가 명확하면 주석이 필요 없다. 나쁜 주석은 없는 것보다 해롭다.
4 에러 처리 (Error Handling) — 오류 코드보다 예외를 사용하라. null을 반환하거나 전달하지 마라. Try-Catch 블록을 별도 함수로 분리하여 정상 흐름과 오류 처리를 분명히 나눠라.
5 테스트 코드 (Unit Tests) — 테스트 코드는 실제 코드만큼 중요하다. TDD(테스트 주도 개발)의 세 가지 법칙을 따르고, 테스트당 assert 하나, 테스트당 개념 하나를 유지하라. 깨끗한 테스트가 깨끗한 코드를 가능하게 한다.

좋은 이름의 조건

저자는 이름 짓기에 대해 여러 규칙을 제시한다. 특히 다음 네 가지가 핵심이다.

  • 의도를 분명히 밝혀라 — 이름이 존재 이유, 수행 기능, 사용 방법을 설명해야 한다
  • 그릇된 정보를 피하라 — 널리 쓰이는 약어나 다른 의미가 있는 단어를 피해라
  • 검색하기 쉬운 이름을 사용하라 — 한 글자 변수나 상수는 검색이 어렵다
  • 클래스는 명사, 메서드는 동사 — 역할에 맞는 품사를 사용하라

함수 작성의 규칙

클린 코드에서 함수는 프로그램의 기본 단위다. 저자가 강조하는 규칙은 다음과 같다.

  • 작게 만들어라 — if/else/while 블록 안은 한 줄이 이상적이다
  • 한 가지만 해라 — 함수 이름으로 설명되는 한 가지 작업만 수행해야 한다
  • 인수를 줄여라 — 이상적인 인수 개수는 0개, 최대 3개까지
  • 부수 효과를 일으키지 마라 — 함수가 약속한 것 외의 변경을 하지 마라

인상적인 구절

"깨끗한 코드는 잘 쓴 문장처럼 읽힌다. 좋은 소설처럼 깨끗한 코드는 해결할 문제의 긴장을 명확히 드러낸다."
— Robert C. Martin, Clean Code, Chapter 1

이 구절은 코드가 단순히 기계를 위한 명령어가 아니라, 사람이 읽는 글이라는 점을 일깨워준다. 코드 리뷰를 할 때마다 이 문장이 떠오른다.

"나쁜 코드에 주석을 달지 마라. 새로 짜라."
— Brian W. Kernighan & P. J. Plaugher (Clean Code에서 인용)

주석으로 복잡한 코드를 설명하려는 유혹에 빠질 때, 이 말이 브레이크 역할을 한다. 코드를 리팩토링하는 것이 주석을 다는 것보다 근본적인 해결이다.

실무 적용 포인트

변수명 개선

책을 읽기 전에는 ret, tmp, data 같은 이름을 습관적으로 사용했다. 이제는 해당 변수가 무엇을 담고 있는지를 이름에 반영하려 노력한다. 예를 들어 ret 대신 validationResult, tmp 대신 swapBuffer처럼 역할을 명확히 한다.

함수 분리

200줄짜리 함수를 작성하던 습관을 고치기 시작했다. 하나의 큰 함수를 여러 개의 작은 함수로 나누면, 각 함수의 역할이 명확해지고 디버깅이 훨씬 쉬워진다.

코드 리뷰 기준

팀 내 코드 리뷰에서 "이 함수 이름만으로 무슨 일을 하는지 알 수 있나요?"라는 질문을 자주 하게 되었다. 클린 코드의 원칙이 자연스럽게 리뷰 기준이 된다.

보이스카우트 규칙을 실천하자 "캠프장은 처음 왔을 때보다 더 깨끗하게 해놓고 떠나라." 코드를 수정할 때마다 주변 코드를 조금씩 개선하면, 시간이 지날수록 코드베이스 전체가 깨끗해진다.

총평

클린 코드는 단순한 코딩 스타일 가이드가 아니다. 소프트웨어 개발자로서의 태도와 책임에 대한 책이다. "동작하면 됐지"라는 마인드에서 벗어나, 코드를 읽는 다른 개발자(미래의 나 포함)를 배려하는 습관을 기르게 해준다.

다만, 모든 원칙을 교조적으로 따를 필요는 없다. 실무에서는 상황에 따라 타협이 필요하다. 중요한 것은 "왜 이렇게 작성하는가"를 항상 의식하는 것이다. 0.5점을 깎은 이유는 Java 중심의 예제가 다른 언어 사용자에게는 다소 거리감이 있기 때문이다.

코드를 "작성"하는 시간보다 "읽는" 시간이 압도적으로 많다는 사실을 깨닫게 해주는 필독서
