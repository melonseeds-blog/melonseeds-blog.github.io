왜 move semantics인가

C++03 시절에는 큰 객체를 함수 반환값으로 받거나 컨테이너에 넣을 때마다 깊은 복사가 일어났다. 1MB 짜리 버퍼를 가진 객체를 반환하면 1MB를 통째로 복사한 뒤 원본을 파괴하는, 명백한 낭비다.

// C++03 — 복사 비용 발생
std::vector<int> makeBigVector()
{
    std::vector<int> v(1000000, 42);
    return v;  // 임시 복사본 생성 후 원본 파괴 (이론상)
}

std::vector<int> result = makeBigVector();  // 또 한 번 복사?

C++11에서 도입된 이동 의미론은 "어차피 곧 사라질 객체"의 자원을 새 객체로 훔쳐오는 방식으로 이 문제를 해결한다. 포인터만 옮기면 끝이므로 비용은 사실상 0이다.

value category 이해

C++의 모든 표현식은 두 축으로 분류된다. 이름이 있는가(identity)와 이동 가능한가(movable).

카테고리이름 있음이동 가능예시
lvalueOX변수 x, obj.member
xvalueOOstd::move(x)의 결과
prvalueXO리터럴 42, 임시 객체 T()
glvaluelvalue + xvalue (이름 있음)주소 얻을 수 있음
rvaluexvalue + prvalue (이동 가능)곧 만료될 값
핵심만 외우자 실무에서는 lvalue(이름 있는 변수)와 rvalue(임시값/std::move 결과) 둘만 구분해도 충분하다. xvalue/prvalue는 표준 문서를 읽을 때만 신경 쓰면 된다. 
int x = 10;             // x: lvalue, 10: prvalue
int& lref = x;          // lvalue reference
int&& rref = 42;        // rvalue reference (42는 prvalue)
int&& rref2 = std::move(x);  // std::move(x)는 xvalue

// &x        OK    (lvalue는 주소 가능)
// &42       에러  (prvalue는 주소 불가)

rvalue reference (T&&)

C++11에서 새로 추가된 참조 종류다. T&가 lvalue에만 바인딩되는 것과 달리 T&&는 rvalue에만 바인딩된다.

void Foo(int&  v) { /* lvalue 받음 */ }
void Foo(int&& v) { /* rvalue 받음 */ }

int x = 10;
Foo(x);              // → int& 호출
Foo(20);             // → int&& 호출
Foo(std::move(x));   // → int&& 호출
매개변수 안에서 T&&는 lvalue다 함수 매개변수로 받는 순간 그 매개변수는 이름이 생기므로 lvalue로 취급된다. 다음 함수로 이동시키려면 다시 std::move해야 한다.

std::move

이름과 달리 아무것도 옮기지 않는다. 단순히 lvalue를 rvalue로 캐스팅하는 함수일 뿐이다. 실제 이동은 이동 생성자/대입 연산자가 호출되어야 일어난다.

// std::move 구현은 사실상 이게 전부
template <class T>
constexpr std::remove_reference_t<T>&& move(T&& x) noexcept
{
    return static_cast<std::remove_reference_t<T>&&>(x);
}

의미는 "이 객체에서 자원을 가져가도 좋다"는 표시다. 표시를 받은 함수가 실제로 자원을 옮길지는 별개의 문제다.

std::move된 객체의 상태 표준은 "valid but unspecified" 상태라고 정의한다. 즉 파괴와 재대입은 안전하지만, 그 외 동작의 결과는 정의되지 않는다. std::move 후에 그 객체의 값을 읽거나 사용하지 마라.

이동 생성자 / 이동 대입 연산자

자원을 가진 클래스라면 다음과 같이 정의한다. 핵심은 원본의 포인터를 훔치고 원본을 무효화하는 것이다.

class Buffer
{
    int*   m_data;
    size_t m_size;

public:
    // 이동 생성자
    Buffer(Buffer&& other) noexcept
        : m_data(other.m_data), m_size(other.m_size)
    {
        other.m_data = nullptr;  // 원본 무효화
        other.m_size = 0;
    }

    // 이동 대입 연산자
    Buffer& operator=(Buffer&& other) noexcept
    {
        if (this != &other)
        {
            delete[] m_data;          // 기존 자원 해제
            m_data = other.m_data;
            m_size = other.m_size;
            other.m_data = nullptr;
            other.m_size = 0;
        }
        return *this;
    }

    ~Buffer() { delete[] m_data; }
};
noexcept가 중요한 이유 std::vector는 재할당(push_back으로 capacity 초과 시) 시 기존 원소를 새 메모리로 옮긴다. 이때 이동 생성자가 noexcept로 표시된 경우에만 이동을 사용하고, 그렇지 않으면 안전을 위해 복사로 폴백한다. 강한 예외 보장을 유지하기 위해서다. 이동 함수에는 반드시 noexcept를 붙이자.

Rule of 0 / 3 / 5

특수 멤버 함수를 어떤 조합으로 정의할지에 대한 가이드다.

규칙정의 대상적용 시점
Rule of 0아무것도 정의하지 않음모든 멤버가 RAII 타입(vector, unique_ptr 등) — 가장 권장
Rule of 3복사 생성자 + 복사 대입 + 소멸자C++03 시절. 자원을 직접 관리할 때
Rule of 5위 3개 + 이동 생성자 + 이동 대입C++11 이후. 자원을 직접 관리할 때
5개 중 하나라도 정의했다면 나머지도 처리하자 명시적 = default나 = delete로라도 의도를 분명히 한다. 한쪽만 정의하면 컴파일러가 다른 쪽을 자동 생성하지 않는 경우가 있어 미묘한 버그가 생긴다. 
// Rule of 0 — 가장 좋다
class Image
{
    std::vector<uint8_t> m_pixels;  // 알아서 이동/복사됨
    int                  m_width;
    int                  m_height;
    // 특수 멤버 함수 일체 정의 불필요
};

// Rule of 5 — 자원을 직접 잡을 때
class RawBuffer
{
    int* m_data;
public:
    RawBuffer(const RawBuffer&);              // 복사 생성
    RawBuffer& operator=(const RawBuffer&);   // 복사 대입
    RawBuffer(RawBuffer&&) noexcept;          // 이동 생성
    RawBuffer& operator=(RawBuffer&&) noexcept; // 이동 대입
    ~RawBuffer();                              // 소멸자
};

완벽 전달(perfect forwarding)

호출자가 넘긴 인자의 value category(lvalue/rvalue)를 그대로 보존해서 다른 함수로 넘기는 기법이다. std::forward와 forwarding reference(흔히 universal reference)를 함께 쓴다.

forwarding reference vs rvalue reference T&&가 템플릿 매개변수나 auto에 쓰이면 forwarding reference, 그 외 일반 함수의 매개변수면 rvalue reference다. 형태는 같아도 의미가 다르다. 
// T&&가 템플릿이면 forwarding reference
template <class T>
void Wrapper(T&& arg)
{
    Inner(std::forward<T>(arg));  // 카테고리 보존 전달
}

// 일반 함수면 그냥 rvalue reference
void Foo(int&& arg);  // rvalue만 받음

std::make_unique의 구현이 완벽 전달의 대표 예시다.

template <class T, class... Args>
std::unique_ptr<T> make_unique(Args&&... args)
{
    return std::unique_ptr<T>(new T(std::forward<Args>(args)...));
}

// 사용
auto p = std::make_unique<Image>(1920, 1080);  // 인자가 그대로 Image 생성자로 전달

vector::emplace_back도 같은 패턴이다. 인자를 받아 컨테이너 안에서 직접 객체를 생성하므로 임시 객체조차 만들지 않는다.

RVO / NRVO (copy elision)

컴파일러는 함수 반환 시 복사/이동 자체를 생략할 수 있다. 반환할 객체를 호출자의 메모리에 곧바로 만든다.

  • RVO(Return Value Optimization) — 이름 없는 임시 객체 반환
  • NRVO(Named RVO) — 이름 있는 지역 변수 반환
  • C++17부터 prvalue 반환은 guaranteed copy elision — 컴파일러가 무조건 생략한다
Image Load(const std::string& path)
{
    Image img(path);
    // ... 처리 ...
    return img;  // NRVO 적용 — 복사도 이동도 일어나지 않음
}

Image i = Load("a.png");  // 그냥 Load 안에서 i가 직접 생성됨
return std::move(local)은 안티패턴 지역 변수를 반환할 때 std::move를 쓰면 NRVO가 깨져서 오히려 느려진다. 컴파일러는 그냥 return local;일 때 elision을 적용한다. 친절을 베풀려다 일을 망치는 케이스다.

흔한 함정

매개변수 T&&를 그냥 전달

void Sink(std::string s);

void Forward(std::string&& s)
{
    Sink(s);             // BAD: s는 lvalue 취급 → 복사
    Sink(std::move(s));  // OK: 명시적 이동
}

반환문에서 std::move

Image Make()
{
    Image img;
    return std::move(img);  // BAD: NRVO 깨짐
    // return img;          // OK: 컴파일러가 elision
}

최신 컴파일러(GCC, Clang)는 -Wpessimizing-move로 경고한다.

move 후 재사용

std::string s = "hello";
std::string t = std::move(s);
std::cout << s;  // DANGER: s의 내용은 정의되지 않음

noexcept 누락

class Buffer
{
public:
    Buffer(Buffer&&);  // noexcept 없음

std::vector<Buffer> v;
v.reserve(100);
// reserve 시 재할당이 일어나면 → 복사 생성자 호출 (이동 안 함)
