1. 테스팅 기초 개념

Pair 1: 오류(Error) vs 결함(Defect) vs 장애(Failure)

3가지 비교

오류 (Error)

  • 사람이 저지르는 실수
  • 결함의 원인
  • 시간적 압박, 피로, 경험 부족 등으로 발생
  • 코딩뿐 아니라 요구사항 작성 시에도 발생

결함 (Defect)

  • 코드/문서에 존재하는 결함(버그)
  • 오류의 결과물
  • 결점, 버그, 결함 모두 동의어
  • 정적 테스팅으로 직접 발견 가능

장애 (Failure)

  • 실행 시 나타나는 오동작
  • 사용자가 관측하는 현상
  • 결함으로 인해 발생하지만, 환경 조건에 의해서도 발생
  • 동적 테스팅으로 유발
핵심 체인:
오류 (Error)
사람의 실수
➡
결함 (Defect)
산출물의 버그
➡
장애 (Failure)
실행 시 오동작
시험 함정: "결함은 항상 장애를 유발한다" → 거짓! 실행되지 않은 코드의 결함은 장애를 일으키지 않는다. 또한 장애가 반드시 결함에 의해서만 발생하는 것도 아니다(환경 조건 등).
연습 Q1
개발자가 피곤한 상태에서 경계값 처리 코드를 잘못 작성했다. 이 상황에서 "오류"에 해당하는 것은?
정답: b)
오류(Error)는 사람이 저지르는 실수 자체를 의미한다. a)는 결함(Defect), c)는 장애(Failure)에 해당한다.

Pair 2: 베리피케이션(Verification) vs 밸리데이션(Validation)

VS

베리피케이션 (Verification)

  • "올바르게 만들었는가?"
  • Are we building the product right?
  • 명세/설계 충족 여부 확인
  • 내부 관점, 사양 대비 확인

밸리데이션 (Validation)

  • "올바른 것을 만들었는가?"
  • Are we building the right product?
  • 사용자/이해관계자 필요 충족 확인
  • 외부 관점, 사용자 요구 대비 확인
기억법: Veri-fication = 사양에 맞게 올바르게(right way) 만들었나 / Vali-dation = 사용자에게 가치(value) 있는 걸 만들었나
시험 함정: 두 단어가 비슷해 보이지만 관점이 완전히 다르다. 테스팅은 베리피케이션뿐 아니라 밸리데이션도 포함한다.
연습 Q2
사용자 인수 테스팅(UAT)에서 이해관계자가 시스템이 비즈니스 요구를 충족하는지 확인한다. 이 활동은?
정답: b)
밸리데이션은 시스템이 사용자/이해관계자의 요구를 충족하는지 확인하는 활동이다. UAT는 대표적인 밸리데이션 활동이다.

Pair 3: 테스팅(Testing) vs 디버깅(Debugging)

VS

테스팅 (Testing)

  • 결함으로 인한 장애를 유발하거나 결함을 직접 발견
  • 주로 테스터의 활동
  • 동적 테스팅: 장애 유발
  • 정적 테스팅: 결함 직접 식별

디버깅 (Debugging)

  • 장애의 원인(결함)을 찾고 수정
  • 주로 개발자의 활동
  • 장애 재현 → 분석 → 결함 수정
  • 테스팅과는 별개의 활동
핵심 차이: 테스팅 = 결함 발견, 디버깅 = 결함 수정. 테스팅이 끝난 후 디버깅이 이어지며, 디버깅 후에는 확인 테스팅이 필요하다.

Pair 4: 테스팅(Testing) vs 품질 보증(QA)

VS

테스팅 (Testing)

  • 제품 지향적
  • 교정(corrective) 접근법
  • 품질 제어(QC)의 주요 활동
  • 결함 식별에 초점

품질 보증 (QA)

  • 프로세스 중심
  • 예방(preventive) 접근법
  • 개발 및 테스팅 프로세스 모두에 적용
  • 프로젝트 참여 모두의 책임
핵심 차이: 테스팅은 QA의 일부이다. 테스팅은 제품 품질에, QA는 프로세스 품질에 초점을 맞춘다.
시험 함정: "테스팅과 QA는 같은 것이다" → 거짓! 테스팅은 교정적, QA는 예방적 접근법이다.
연습 Q3
다음 중 품질 보증(QA) 활동에 해당하는 것은?
정답: b)
QA는 프로세스 구현과 개선에 초점을 맞춘 예방적 접근법이다. a), c), d)는 모두 테스팅(품질 제어) 활동에 해당한다.

2. 프로세스와 테스트 레벨

Pair 5: 확인 테스팅 vs 회귀 테스팅

VS

확인 테스팅 (Confirmation)

  • 수정한 결함이 제대로 고쳐졌는지 확인
  • 이전에 실패했던 테스트 케이스를 재실행
  • 결함 자체에 초점
  • 처음 테스트한 사람이 수행하는 것이 바람직

회귀 테스팅 (Regression)

  • 변경으로 인한 의도치 않은 부작용 확인
  • 변경된 부분 외의 다른 영역도 테스트
  • 시스템 전체에 초점
  • 자동화에 매우 적합한 대상
핵심 차이: 확인 = "고친 게 잘 고쳐졌나?", 회귀 = "고치면서 다른 곳이 깨지지 않았나?"
시험 함정: "결함 수정 후 해야 하는 테스팅은?" → 둘 다 필요! 확인 테스팅과 회귀 테스팅 모두 수행해야 한다.
연습 Q4
로그인 기능의 결함을 수정한 후, 결제 기능이 정상 작동하는지 확인하는 테스팅은?
정답: b)
결제 기능은 로그인 결함 수정과 직접 관련 없는 다른 영역이다. 변경으로 인한 의도치 않은 부작용을 확인하는 것은 회귀 테스팅이다.

Pair 6: 컴포넌트 테스팅 vs 컴포넌트 통합 테스팅

VS

컴포넌트 테스팅

  • 개별 모듈을 단독으로 테스트
  • 단위 테스팅이라고도 함
  • 일반적으로 개발자가 수행
  • 테스트 하네스/단위 테스트 프레임워크 사용

컴포넌트 통합 테스팅

  • 모듈 간 인터페이스와 상호작용 테스트
  • 단위 통합 테스팅이라고도 함
  • 상향식, 하향식, 빅뱅 등 통합 전략에 따라 달라짐
  • 모듈이 함께 동작하는지 확인
핵심 차이: 컴포넌트 = 모듈 하나, 컴포넌트 통합 = 모듈 사이

Pair 7: 시스템 테스팅 vs 시스템 통합 테스팅

VS

시스템 테스팅

  • 전체 시스템의 기능/비기능 테스트
  • 엔드투엔드 동작 확인
  • 독립된 테스트팀이 수행 가능
  • 시스템 명세와 관련

시스템 통합 테스팅

  • 다른 시스템/외부 서비스와의 연동 테스트
  • 인터페이스 호환성 확인
  • 운영 환경과 유사한 테스트 환경 사용
  • 외부 시스템과의 상호작용에 초점
핵심 차이: 시스템 = 우리 시스템 내부 전체, 시스템 통합 = 우리 시스템과 외부 시스템 사이
연습 Q5
결제 시스템이 PG사(Payment Gateway) API와 정상적으로 통신하는지 확인하는 테스팅은?
정답: c)
외부 서비스(PG사 API)와 우리 시스템의 인터페이스를 테스트하는 것은 시스템 통합 테스팅이다.

Pair 8: 시프트 레프트(Shift-Left) 접근법

개념 정리

시프트 레프트란?

  • 테스팅을 SDLC 초기 단계로 이동
  • "왼쪽(초기)으로 이동"이라는 의미
  • 조기 테스팅 원리(원리 3)와 직결
  • 초기에 비용/노력이 증가하나 전체적으로 절감

시프트 레프트의 예시

  • 초안이 가용한 즉시 작업 산출물 리뷰
  • 코드 작성 전 단위 테스트 작성 (TDD)
  • 동적 테스팅 전 정적 분석 수행
  • 컴포넌트 레벨에서 비기능 테스팅 시작
시험 함정: "시프트 레프트는 시스템 테스팅을 컴포넌트 테스팅보다 먼저 수행한다" → 거짓! 시프트 레프트는 테스트 레벨 순서를 바꾸는 것이 아니라, 각 레벨의 활동을 더 일찍 시작하는 것이다. 또한 SDLC 후반 테스팅을 생략한다는 의미가 아니다.

3. 정적 테스팅

Pair 9: 정적 테스팅(Static) vs 동적 테스팅(Dynamic)

VS

정적 테스팅

  • 코드를 실행하지 않음
  • 리뷰, 정적 분석
  • 결함을 직접 식별
  • 비-실행 산출물(요구사항, 설계 문서)에도 적용 가능
  • 유지보수성 등 코드 실행 불필요 품질 특성 측정

동적 테스팅

  • 코드를 실행함
  • 테스트 케이스 실행
  • 장애를 유발한 후 후속 분석으로 결함 식별
  • 실행 가능한 산출물에만 적용 가능
  • 성능 효율성 등 코드 실행 필요 품질 특성 측정
정적 테스팅이 찾기 좋은 결함: 요구사항 모순/누락, 설계 결함, 코딩 표준 위반, 도달 불가능한 코드, 인터페이스 명세 불일치, 보안 취약점
동적 테스팅이 찾기 좋은 결함: 데이터 종속적 결함, 런타임 오류, 성능 병목, 메모리 누수
연습 Q6
다음 중 정적 테스팅으로 식별하기 더 적합한 결함 유형은?
정답: b)
요구사항 결함(모호성, 모순, 누락)은 정적 테스팅(리뷰)을 통해 더 효과적으로 식별된다. 나머지는 실행이 필요한 동적 테스팅 영역이다.

Pair 10: 인스펙션 vs 워크스루 vs 기술 리뷰

3가지 비교
특성인스펙션워크스루기술 리뷰
주도자훈련된 중재자저자중재자
공식성가장 공식적비교적 비공식적중간 수준
주요 목적이상 사항 최대한 발견, 메트릭 수집교육, 합의 도출, 이상 사항 발견기술 문제 합의, 이상 사항 발견
저자 역할리뷰 리더/서기 불가리더로 참여참여자로 참여
개별 리뷰필수선택일반적으로 수행
시험 함정: "인스펙션에서 저자가 서기를 맡았다" → 오답! 인스펙션에서 저자는 리뷰 리더나 서기가 될 수 없다.

Pair 11: 리뷰 역할 비교

역할 정리
역할주요 책임
관리자 (Manager)리뷰할 내용을 결정하고 필요한 자원(사람, 시간)을 제공

저자 (Author)리뷰 대상 작업 산출물을 작성하고, 발견된 결함을 수정

리뷰 리더참여자 결정, 리뷰 일정/장소 협의 등 전반적 책임
기억법: 관리자는 "What"을 결정, 리뷰 리더는 "Who/When/Where"를 결정, 중재자는 회의 "진행"을 담당
연습 Q7
리뷰 회의에서 시간 관리를 하고 모든 참가자가 자유롭게 발언할 수 있는 환경을 조성하는 역할은?
정답: c)
중재자(퍼실리테이터)는 중재, 시간 관리, 안전한 리뷰 환경 조성 등 리뷰 회의의 효과적인 운영을 담당한다.

4. 테스트 기법

Pair 12: 동등 분할(EP) vs 경계값 분석(BVA)

VS

동등 분할 (EP)

  • 데이터를 분할(파티션) 단위로 나눔
  • 각 분할의 대표값 하나만 테스트
  • 유효 분할 + 비유효 분할
  • 분할은 서로 겹치지 않아야 함

경계값 분석 (BVA)

  • 분할의 경계에 있는 값에 초점
  • 2-value BVA: 경계값 + 인접 분할의 가장 가까운 값
  • 3-value BVA: 경계값 + 양쪽 이웃 값
  • 정렬된 분할에만 사용 가능
실전 예제: 입력 범위가 1~100인 경우
• EP: 유효 분할(1~100)에서 대표값 50 / 비유효 분할에서 대표값 -5, 150
• 2-value BVA: 경계값 쌍 = (0,1), (100,101)
• 3-value BVA: 경계값 + 이웃 = (0,1,2), (99,100,101)
시험 함정: BVA는 EP를 보완하는 기법이다. BVA를 적용하려면 먼저 EP가 전제되어야 한다. 또한 3-value BVA가 2-value BVA보다 더 엄격하다.
연습 Q8
입력 필드가 18~65 범위의 나이를 허용한다. 2-value BVA를 적용할 때 테스트해야 하는 경계값의 총 개수는?
정답: b)
2-value BVA에서 각 경계는 2개의 커버리지 항목을 가진다. 경계가 2개(18, 65)이므로: (17,18)과 (65,66) = 총 4개 값이다.

Pair 13: 구문 커버리지 vs 분기 커버리지

VS

구문 커버리지 (Statement)

  • 실행 가능한 구문이 실행되었는가?
  • 커버리지 항목 = 실행 가능한 구문
  • 각 구문을 최소 1회 실행
  • 모든 분기를 보장하지 않음

분기 커버리지 (Branch)

  • 모든 분기(T/F)가 실행되었는가?
  • 커버리지 항목 = 분기
  • 조건문의 참/거짓 모두 실행
  • 구문 커버리지를 포함
핵심 관계: 100% 분기 커버리지 → 100% 구문 커버리지 (자동 달성)
역은 성립하지 않음! 100% 구문 커버리지로는 100% 분기 커버리지를 보장할 수 없다.
시험 함정: "100% 구문 커버리지를 달성하면 100% 분기 커버리지도 달성된다" → 거짓! 분기 커버리지가 구문 커버리지를 포함하는 것이지, 그 반대가 아니다.
연습 Q9
다음 코드에서 100% 구문 커버리지 달성에 필요한 최소 테스트 케이스 수는?
if (x > 0) { print("positive"); }
print("done");
정답: a)
x=1일 때 "positive"와 "done" 모두 실행되어 100% 구문 커버리지를 달성한다. 단, 100% 분기 커버리지를 위해서는 x=-1도 필요하다(if문의 거짓 분기).

Pair 14: 블랙박스 vs 화이트박스 vs 경험 기반

3가지 비교
특성블랙박스화이트박스경험 기반
기반명세 (사양)내부 구조 (코드)테스터의 지식/경험
별칭명세 기반 기법구조 기반 기법-
내부 구조 참조참조하지 않음반드시 참조필요에 따라
대표 기법EP, BVA, 결정 테이블, 상태 전이구문 테스팅, 분기 테스팅오류 추정, 탐색적, 체크리스트
테스트 생성 시점구현 전/후 모두 가능구현 완료 후언제든 가능
핵심: 세 가지 기법 유형은 서로 보완하는 관계이다. 어느 하나만으로 충분하지 않으며 함께 사용해야 효과적이다.

Pair 15: 오류 추정 vs 탐색적 테스팅 vs 체크리스트 기반

3가지 비교
특성오류 추정탐색적 테스팅체크리스트 기반
접근 방식경험으로 결함 위치 예측학습+설계+실행 동시 진행미리 정의된 목록 기반
구조화 수준결함 목록/공격 목록 사용테스트 차터로 가이드체크리스트로 가이드
효과성 좌우 요소테스터의 과거 경험테스터의 기술, 호기심, 창의성체크리스트의 품질
재현 가능성낮음 (개인 경험에 의존)낮음 (매번 다른 경로)중간 (같은 목록 사용하나 실행은 달라짐)
기억법: 오류 추정 = "어디서 깨질지 감(感)으로 안다", 탐색적 = "하면서 배우면서 테스트한다", 체크리스트 = "목록 보고 하나씩 확인한다"
연습 Q10
테스터가 테스트 차터를 기반으로 시간을 정해 놓고, 테스트 대상을 배워가면서 동시에 테스트를 설계하고 실행하는 기법은?
정답: b)
탐색적 테스팅은 학습, 설계, 실행을 동시에 수행하며, 세션 기반 접근법에서는 테스트 차터를 지침으로 사용하고 시간을 정해 놓고 수행한다.

Pair 16: 제품 리스크 vs 프로젝트 리스크

VS

제품 리스크 (Product Risk)

  • 소프트웨어 품질 특성과 관련
  • 예: 기능 결함, 성능 문제, 보안 취약점, 열악한 UX
  • 사용자 불만족, 매출 손실 등 초래
  • 테스팅으로 완화 가능

프로젝트 리스크 (Project Risk)

  • 프로젝트 관리 및 제어와 관련
  • 예: 일정 지연, 인력 부족, 예산 축소, 도구 문제
  • 일정, 예산, 범위에 영향
  • 프로젝트 목표 달성 능력에 영향
구분법: "이 문제가 발생하면 소프트웨어 품질이 떨어지나?" → 제품 리스크 / "프로젝트 진행이 어려워지나?" → 프로젝트 리스크
시험 함정: "응답 시간이 요구사항을 충족하지 못할 리스크" = 제품 리스크 (품질 특성), "핵심 개발자가 퇴사할 리스크" = 프로젝트 리스크 (인력 문제). 어느 쪽인지 정확히 구분해야 한다.
연습 Q11
다음 중 제품 리스크에 해당하는 것은?
정답: c)
보안 취약점은 소프트웨어 품질 특성과 관련된 제품 리스크이다. a), b), d)는 모두 프로젝트 관리와 관련된 프로젝트 리스크에 해당한다.

Pair 17: 시작 조건 vs 완료 조건

VS

시작 조건 (Entry Criteria)

  • 활동을 수행하기 위한 전제조건
  • 애자일: 준비의 정의(Definition of Ready)
  • 예: 자원 가용성, 테스트웨어 가용성, 테스트 대상 초기 품질 수준
  • 충족하지 못하면 시간/비용 증가 위험

완료 조건 (Exit Criteria)

  • 활동 종료를 선언하기 위한 달성 사항
  • 애자일: 완료의 정의(Definition of Done)
  • 예: 커버리지 수준, 미해결 결함 수, 모든 테스트 실행 여부
  • 시간/예산 소진도 유효한 완료 조건
기억법: 시작 조건 = "시작해도 되나?" / 완료 조건 = "끝내도 되나?"
시험 함정: 애자일에서 "준비의 정의(DoR)"는 시작 조건, "완료의 정의(DoD)"는 완료 조건에 해당한다. 이 매핑을 혼동하지 말 것!

Pair 18: 테스트 피라미드 vs 테스팅 사분면

VS

테스트 피라미드

  • 테스트 자동화 비율과 세분화 수준 모델
  • 아래층: 많고, 작고, 빠름 (단위 테스트)
  • 위층: 적고, 크고, 느림 (E2E 테스트)
  • 테스트 노력 할당 가이드

테스팅 사분면

  • 비즈니스/기술 x 팀 지원/제품 평가 2x2 매트릭스
  • Q1: 기술+팀지원 (컴포넌트 테스트, 자동화)
  • Q2: 비즈니스+팀지원 (기능 테스트, 수동/자동)
  • Q3: 비즈니스+평가 (탐색적/UAT, 수동)
  • Q4: 기술+평가 (성능/비기능, 자동화)
핵심 차이: 피라미드는 "얼마나 많이/어느 레벨에서 자동화할까"에 초점, 사분면은 "어떤 유형의 테스트를 누가 왜 하는가"에 초점
연습 Q12
테스트 피라미드에서 가장 아래층의 테스트 특성으로 올바른 것은?
정답: b)
피라미드의 아래층은 수가 많고, 작고, 독립적이며, 실행이 빠른 테스트(컴포넌트/단위 테스트)로 구성된다.

6. 시험 핵심 구분 포인트

"올바르지 않은 것" 문제 패턴

ISTQB 시험에서 가장 자주 등장하는 유형이다. "다음 중 올바르지 않은 것은?" 또는 "다음 중 해당하지 않는 것은?" 형태로 출제된다.

자주 출제되는 거짓 진술들
  • "테스팅은 결함이 없음을 증명할 수 있다" → 원리 1 위반
  • "완벽한 테스팅이 가능하다" → 원리 2 위반
  • "결함은 항상 장애를 유발한다" → 실행되지 않는 코드의 결함
  • "100% 구문 커버리지 = 100% 분기 커버리지" → 역은 성립하지 않음
  • "테스팅과 QA는 같다" → 별개의 개념
  • "시프트 레프트는 후반 테스팅을 생략한다" → 초기에 더 하되 후반도 유지

K1 vs K2 문제 구분법

VS

K1 (기억/인식)

  • "다음 중 ~인 것은?"
  • "다음 중 ~에 해당하는 것은?"
  • 용어, 정의, 목록을 기억하는 문제
  • 정의를 정확히 외우면 풀 수 있음

K2 (이해/설명)

  • "다음 중 ~를 가장 잘 설명한 것은?"
  • "시나리오에서 적용되는 원리는?"
  • 개념을 이해하고 구별하는 문제
  • 시나리오 기반으로 적용 능력 측정

절대적 표현 = 보통 오답

선택지에서 주의할 키워드

선택지에 "항상", "절대", "반드시", "모든 경우에", "유일한" 같은 절대적 표현이 있으면 높은 확률로 오답이다. ISTQB 실러버스는 대부분 "~할 수 있다", "~하는 경우가 많다", "일반적으로" 같은 유연한 표현을 사용하기 때문이다.

예외: 7가지 테스팅 원리 등 명확히 정의된 원칙은 절대적 표현이 맞을 수 있다.

최종 비교 요약 체크리스트

시험 직전 최종 점검
  • Error → Defect → Failure 체인 방향 확인
  • Verification(사양 충족) vs Validation(사용자 충족) 구분
  • Testing(발견) vs Debugging(수정) 주체 구분
  • Confirmation(수정 확인) vs Regression(부작용 확인) 목적 구분
  • Static(비실행) vs Dynamic(실행) 찾기 좋은 결함 유형 구분
  • Statement Coverage ⊂ Branch Coverage 포함 관계
  • Product Risk(품질) vs Project Risk(관리) 분류 기준
  • Entry Criteria(DoR) vs Exit Criteria(DoD) 매핑
  • Test Pyramid(자동화 비율) vs Testing Quadrants(유형 분류)
