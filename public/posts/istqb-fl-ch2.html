2.1 SDLC에서의 테스팅

소프트웨어 개발수명주기(SDLC) 모델은 개발 프로세스의 여러 단계와 활동이 논리적, 시간상으로 어떻게 연관되는지 정의한다. SDLC 모델의 선택은 테스팅 활동의 범위, 시기, 자동화 수준 등에 직접적인 영향을 미친다.

2.1.1 SDLC가 테스팅에 미치는 영향 (FL-2.1.1, K2)

SDLC 모델의 선택은 다음 항목에 영향을 미친다:

  • 테스트 활동 범위 및 시기 (테스트 레벨 및 테스트 유형)
  • 테스트 문서 상세화 수준
  • 테스트 기법 및 접근법 선택
  • 테스트 자동화 범위
  • 테스터의 역할과 책임
SDLC 모델 예시 테스팅 특징
순차적 모델 폭포수, V-모델 초기에 리뷰/분석 참여, 동적 테스팅은 후반에 수행
반복적/점진적 모델 나선형, 프로토타이핑, 통합 프로세스 반복 주기마다 모든 테스트 레벨 수행, 빠른 피드백과 광범위한 리그레션 테스팅 필요
애자일 스크럼, 칸반, XP 가벼운 작업 산출물, 테스트 자동화 선호, 경험 기반 테스트 기법 활용

2.1.2 좋은 테스팅 프랙티스 (FL-2.1.2, K1)

선택한 SDLC 모델에 상관없이 적용되는 좋은 테스팅 프랙티스는 다음과 같다:

  • 모든 소프트웨어 개발 활동에 상응하는 테스트 활동을 둔다
  • 테스트 레벨마다 구체적이면서 독립적인 테스트 목적을 설정한다
  • 특정 테스트 레벨의 분석과 설계를 해당 개발 단계에서 시작해 조기 테스팅 원칙을 준수한다
  • 테스터가 작업 산출물 초안 리뷰에 즉시 참여해 시프트 레프트 전략을 지원한다
시험 포인트 "모든 SDLC 활동에는 상응하는 테스트 활동이 있다"는 순차적/반복적/점진적 모든 모델에 적용된다. 특정 모델에만 적용된다는 보기는 오답이다.

2.1.3 테스트 우선 접근법: TDD, ATDD, BDD (FL-2.1.3, K1)

코드 작성 전에 테스트를 먼저 정의하는 접근법으로, 조기 테스팅 원리와 시프트 레프트를 실천한다.

접근법 핵심 개념 주요 특징
TDD
(테스트 주도 개발)		 테스트 케이스로 코딩을 주도 테스트 작성 → 코드 작성 → 리팩토링 순서. 주로 단위 테스트 수준, 코드 지향적
ATDD
(인수 테스트 주도 개발)		 인수 조건에서 테스트 도출 설계 프로세스 중 인수 조건을 기반으로 테스트 작성. 애플리케이션 개발 전에 테스트 작성
BDD
(행위 주도 개발)		 기대 동작을 자연어로 표현 Given/When/Then 형태. 이해관계자가 이해하기 쉬운 자연어 사용. 자동 실행 가능한 테스트로 변환
ATDD vs BDD 구분법 ATDD는 "인수 조건 기반으로 테스트를 도출"하는 데 초점이 있고, BDD는 "이해관계자가 이해하기 쉬운 자연어(Given/When/Then)"로 기대 동작을 표현하는 데 초점이 있다. TDD는 단위 수준에서 코드를 주도하는 개발자 중심 접근법이다.

2.1.4 DevOps와 테스팅 (FL-2.1.4, K2)

DevOps는 개발(테스팅 포함)과 운영이 협력해 공통된 목표를 달성하기 위한 조직 차원의 접근법이다.

테스팅 관점의 이점 리스크와 어려움
  • 코드 품질에 대한 빠른 피드백
  • CI가 시프트 레프트 테스팅을 장려
  • CI/CD를 통한 안정적 테스트 환경 구축
  • 비기능 품질 특성에 대한 가시성 향상
  • 반복적 수동 테스팅 필요성 감소
  • 자동 리그레션 테스트로 리그레션 리스크 최소화
  • 배포 파이프라인 정의 및 설정 필요
  • CI/CD 도구 도입 및 유지보수 필요
  • 테스트 자동화를 위한 추가 자원 필요
주의 DevOps는 높은 수준의 테스트 자동화를 동반하지만, 수동 테스팅(특히 사용자 관점에서)은 여전히 필요하다.

2.1.5 시프트 레프트 접근법 (FL-2.1.5, K2)

시프트 레프트는 테스트를 SDLC 초기에 수행하도록 하는 접근법이다. 단, SDLC 후반의 테스트를 무시해도 된다는 의미는 아니다.

시프트 레프트의 실천 사례:

  • 코드 작성 전에 테스트 케이스를 작성하고 테스트 하네스에서 실행
  • 지속적 통합/배포(CI/CD) 적용 - 소스 코드와 함께 자동 컴포넌트 테스트 제출
  • 동적 테스팅 전 또는 자동화 프로세스 일부로 정적 분석 완료
  • 가능한 한 컴포넌트 테스트부터 비기능 테스팅 수행
시프트 레프트가 아닌 것 형상 관리 프로세스를 구축하기 전에 테스트 스크립트를 작성하는 것은 시프트 레프트가 아니다. 테스트 스크립트는 형상 관리 대상이므로, 형상 관리가 먼저 구축되어야 의미가 있다.

시프트 레프트는 프로세스 초기에 훈련, 공수, 비용이 추가로 들지만, 후반의 공수와 비용 절감을 기대할 수 있다.

2.1.6 회고 및 프로세스 개선 (FL-2.1.6, K2)

회고는 프로젝트/반복 주기가 끝날 때, 릴리스 마일스톤에서, 또는 필요시 진행한다. 참가자는 테스터, 개발자, 설계자, 제품 소유자, 비즈니스 분석가 등이다.

논의 사항:

  • 무엇이 성공적이었고 유지해야 할 것은 무엇인가?
  • 무엇이 부족했고 개선할 수 있는 점은 무엇인가?
  • 향후 개선 사항을 도입하고 성공 요소를 유지하려면 어떻게 해야 하는가?

테스팅 관점의 이점:

  • 테스트 효과성/효율성 향상
  • 테스트웨어 품질 향상
  • 팀의 결속 및 학습 향상
  • 테스트 베이시스 품질 개선
  • 개발과 테스팅 간의 협업 개선
회고의 핵심 포인트 회고는 제품이 아닌 프로세스에 대한 피드백을 수집하는 것이 목적이다. 팀 내부 활동이며 최종 사용자 대표를 포함하지 않는다. 결과는 기록하여 테스트 완료 보고서에 포함하는 경우가 많다.

2.2 테스트 레벨과 테스트 유형

테스트 레벨은 함께 구성하고 관리하는 테스트 활동 집합이다. 테스트 유형은 특정 품질 특성과 관련된 테스트 활동의 집합으로, 대부분 모든 테스트 레벨에서 수행할 수 있다.

2.2.1 테스트 레벨 (FL-2.2.1, K2)

테스트 레벨 별칭 테스트 초점 수행자/환경
컴포넌트 테스팅 단위 테스팅 컴포넌트를 개별적으로 테스트 개발자가 자신의 개발 환경에서 수행. 테스트 하네스/단위 테스트 프레임워크 필요
컴포넌트 통합 테스팅 단위 통합 테스팅 컴포넌트 간 인터페이스와 상호 작용 상향식/하향식/빅뱅 등 통합 전략에 따라 달라짐
시스템 테스팅 - 전체 시스템의 전반적 동작과 기능 (엔드투엔드) 독립된 테스트팀 수행 가능. 실제 환경을 대변하는 테스트 환경
시스템 통합 테스팅 - 다른 시스템/외부 서비스와의 인터페이스 운영 환경과 유사한 테스트 환경
인수 테스팅 - 밸리데이션과 배포 준비 확인 (비즈니스 요구 충족) 사용자/고객 참여

테스트 레벨은 다음 속성으로 구분한다: 테스트 대상, 테스트 목적, 테스트 베이시스, 결함과 장애, 접근법과 역할

장애 유형과 테스트 레벨 매핑 코드 내 논리 결함 → 단위 테스팅 / 컴포넌트 간 통신 실패 → 단위 통합 테스팅 / 비즈니스 규칙 오류 → 시스템 테스팅 / 사용자 비즈니스 요구와 다른 동작 → 인수 테스팅

2.2.2 테스트 유형 (FL-2.2.2, K2)

테스트 유형 평가 대상 주요 목적
기능 테스팅 시스템이 '무엇을' 해야 하는지 기능 성숙도(완전성), 기능 정확성, 기능 타당성(적합성) 확인
비기능 테스팅 시스템이 '얼마나 잘' 동작하는지 수행 효율성, 호환성, 유용성, 신뢰도, 보안, 유지보수성, 이동성, 안전성 (ISO/IEC 25010)
블랙박스 테스팅 명세 기반 - 내부 구조 무관 명세와 비교해 시스템의 동작을 확인
화이트박스 테스팅 구조 기반 - 코드/아키텍처/워크플로우 내부 구조를 충분히 커버하는 것

4가지 테스트 유형은 모든 테스트 레벨에서 사용할 수 있지만, 레벨마다 관심 대상은 다를 수 있다.

2.2.3 확인 테스팅 vs 리그레션 테스팅 (FL-2.2.3, K2)

구분 확인 테스팅 (Confirmation Testing) 리그레션 테스팅 (Regression Testing)
목적 원래 결함이 성공적으로 수정됐는지 확인 변경으로 인한 부정적 영향(부작용)이 없는지 확인
대상 수정된 부분 (결함으로 실패한 테스트 케이스 재실행) 변경되지 않은 부분 (같은 시스템 내 다른 컴포넌트, 연결된 시스템, 환경)
자동화 필요에 따라 자동화에 매우 적합 (반복 실행, 케이스 수 증가)
범위 파악 결함 보고서 기반 영향도 분석 수행 권장
확인 vs 리그레션 구분 예시 TC1이 실패 후 수정되어 다시 실행하면 → 확인 테스트. TC2가 이전에 합격했지만 다른 결함 수정 후 영향 확인을 위해 다시 실행하면 → 리그레션 테스트. 핵심은 "이전에 실패했던 것을 재확인"이면 확인 테스트, "이전에 합격했던 것이 여전히 합격하는지 확인"이면 리그레션 테스트다.

2.3 유지보수 테스팅

2.3.1 유지보수 테스팅과 유발 요인 (FL-2.3.1, K2)

유지보수 테스팅은 운영 중인 시스템에 대해 수행하며, 변경사항의 성공적 구현 검증과 변경되지 않은 부분의 리그레션 확인을 모두 포함한다.

유지보수 테스팅의 범위는 변경의 리스크 수준, 기존 시스템의 크기, 변경사항의 크기에 따라 달라진다.

유발 요인(트리거):

유발 요인 설명
계획된 개선/수정 릴리스 기반 개선사항, 수정을 위한 변경, 핫픽스
운영 환경 변경 업그레이드나 마이그레이션 (플랫폼 변경). 새로운 환경 관련 테스트와 데이터 변환 테스트 필요 가능
단종(수명 종료) 시스템 단종 시 데이터 보관(archiving) 테스팅, 보관 이후 복원 및 복구 절차 테스팅 필요 가능
영향도 분석의 중요성 유지보수에서는 변경 전에 영향도 분석을 수행해 시스템의 다른 영역에 미칠 잠재적 영향을 파악하는 것이 중요하다. 이 분석은 리그레션 테스팅에 적절한 테스트 케이스를 선택하는 데도 도움이 된다.

핵심 용어 정리

2장에서 반드시 숙지해야 할 핵심 용어들이다.

Acceptance Testing
인수 테스팅 - 시스템이 사용자 비즈니스 요구를 충족하는지 확인
Black-box Testing
블랙박스 테스팅 - 명세 기반, 내부 구조와 무관한 테스팅
Component Integration Testing
컴포넌트 통합 테스팅 - 컴포넌트 간 인터페이스와 상호 작용 테스트
Component Testing
컴포넌트 테스팅 - 개별 컴포넌트를 독립적으로 테스트
Confirmation Testing

문제 1 (FL-2.1.3, K1)

다음 중 인수 테스트 주도 개발(ATDD) 접근법을 가장 잘 설명한 것은?

  1. 인수 조건을 일반적으로 Given/When/Then 형식으로 작성한다
  2. 테스트 케이스가 주로 단위 테스팅 수준에서 작성되며 코드 지향적이다
  3. 관련 소프트웨어 개발을 주도하기 위해 인수 조건을 기반으로 테스트를 작성한다
  4. 테스트가 소프트웨어의 기대 동작을 기반으로 하므로 팀원들이 이를 더 쉽게 이해할 수 있다
정답: c
a) Given/When/Then은 BDD에서 더 자주 사용된다.
b) 단위 수준 코드 지향적 테스트는 TDD에 대한 설명이다.
c) ATDD에서 테스트는 설계 프로세스 중 인수 조건을 기반으로 작성된다.
d) 팀원이 쉽게 이해하는 자연어 기반은 BDD의 특징이다.
문제 2 (FL-2.2.1, K2)

1. 사용자 비즈니스 요구와 다른 시스템 동작   2. 컴포넌트 간 통신 실패   3. 코드 내 논리 결함   4. 올바르게 구현되지 않은 비즈니스 규칙

A. 단위 테스팅   B. 단위 통합 테스팅   C. 시스템 테스팅   D. 인수 테스팅

  1. 1D, 2B, 3A, 4C
  2. 1D, 2B, 3C, 4A
  3. 1B, 2A, 3D, 4C
  4. 1C, 2B, 3A, 4D
정답: a (1D, 2B, 3A, 4C)
사용자 비즈니스 요구 → 인수 테스팅(D) / 컴포넌트 간 통신 → 단위 통합 테스팅(B) / 코드 내 논리 → 단위 테스팅(A) / 비즈니스 규칙 → 시스템 테스팅(C)
문제 3 (FL-2.2.3, K2)

TC1~TC3으로 세 번의 테스트를 실행했다. 결함 수정 후 새 버전에서 재실행한다.

첫 번째 실행두 번째 실행세 번째 실행
TC1(1) 실패(4) 합격(7) 합격
TC2(2) 합격(5) 실패(8) 합격
TC3(3) 실패(6) 실패(9) 합격

다음 중 리그레션 테스트로 수행된 테스트는?

  1. 4, 7, 8, 9
  2. 5, 7
  3. 4, 6, 8, 9
  4. 5, 6
정답: b (5, 7)
TC1, TC3이 1차에서 실패 → (4), (6)은 확인 테스트.
TC2, TC3이 2차에서 실패 → (8), (9)는 확인 테스트.
TC2가 1차에서 합격했는데 2차에서 다시 실행 → (5)는 리그레션 테스트.
TC1이 2차에서 합격했는데 3차에서 다시 실행 → (7)은 리그레션 테스트.
핵심: 이전에 합격한 테스트를 변경 후 재실행하면 리그레션, 이전에 실패한 것을 수정 후 재확인하면 확인 테스트다.
