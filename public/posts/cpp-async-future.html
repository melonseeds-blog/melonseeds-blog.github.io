비동기(async)란

지금 호출하지만 결과는 나중에 받는다. 이게 비동기의 한 줄 정의다. 호출과 결과 수신이 시간적으로 분리된다.

방식호출자 동작결과 수신 시점
동기 (sync)호출 후 결과 나올 때까지 멈춤호출이 반환되는 즉시
비동기 (async)호출 후 즉시 다음 코드 진행나중에 핸들로 조회

전통적인 방법은 std::thread를 직접 만들고 join()으로 끝나길 기다린 뒤, 공유 변수에서 결과를 꺼내는 것이다. 이 패턴은 동기화 코드(뮤텍스, 조건 변수)가 너무 많이 필요하다. future/promise는 이 패턴을 한 줄로 줄여준다.

// 동기 호출 — 30초 동안 멈춤
int result = heavyComputation();
useResult(result);

// 비동기 호출 — 즉시 반환
auto fut = std::async(std::launch::async, heavyComputation);
doOtherWork();          // 계산이 도는 동안 다른 일 진행
int result = fut.get(); // 필요할 때 결과 받기 (필요시 대기)

4개 도구 한눈에 보기

C++ 표준 비동기 도구는 4개로 구성된다. 모두 <future> 헤더에 있다.

도구역할한 줄 요약
std::future<T>결과를 받는 쪽비동기 작업의 결과 핸들
std::promise<T>결과를 넣는 쪽수동으로 future에 값 전달
std::packaged_task<F>함수 + future 묶음호출 가능 객체를 future로 감쌈 (다음 글)
std::async모든 걸 한 방에함수 호출 → future 자동 생성
관계 요약 future는 결과 수신 핸들이고, 그걸 만드는 방법이 세 가지(promise, packaged_task, async)다. 이 글은 future, async, promise를 다루고 packaged_task는 다음 글에서 자세히 살펴본다.

std::future<T>

비동기 작업의 결과를 받는 핸들이다. 결과는 아직 없을 수도 있고, 이미 준비됐을 수도 있다.

주요 멤버 함수

std::future<int> fut = std::async(std::launch::async, []{ return 42; });

// 결과를 가져온다 (필요시 대기). 한 번만 호출 가능.
int v = fut.get();

// 대기만 하고 결과는 안 가져간다
fut.wait();

// 시간 제한 대기
auto status = fut.wait_for(std::chrono::milliseconds(500));
auto status2 = fut.wait_until(std::chrono::system_clock::now() + std::chrono::seconds(1));

// future가 유효한 상태인지 (get() 호출 후엔 false)
if (fut.valid()) { /* ... */ }

future_status

값의미
future_status::ready결과가 준비됨. get() 즉시 반환
future_status::timeout대기 시간 안에 준비 못 됨
future_status::deferred아직 실행 안 됨 (deferred policy)
auto status = fut.wait_for(std::chrono::milliseconds(100));
if (status == std::future_status::ready)
{
    int v = fut.get();
}
else if (status == std::future_status::timeout)
{
    // 아직 실행 중. 나중에 다시 시도
}
같은 future에 get()을 두 번 호출하면 UB get()은 결과를 "옮겨"가기 때문에 한 번만 호출할 수 있다. 두 번째 호출은 미정의 동작이다. 여러 곳에서 결과를 받아야 한다면 std::shared_future를 사용한다.

std::async와 launch policy

가장 쉬운 비동기 실행 도구다. 함수와 인자를 넘기면 future를 돌려준다.

int compute(int a, int b) { return a + b; }

auto fut = std::async(std::launch::async, compute, 3, 4);
int result = fut.get();  // 7

launch policy 비교

정책동작실행 스레드
std::launch::async호출 즉시 새 스레드에서 실행새 스레드
std::launch::deferredget()/wait() 호출까지 미룸 (lazy)호출자 스레드
둘 다 (기본값)구현이 알아서 결정구현 의존
launch policy 기본값을 쓰지 말자 std::async(func)는 async | deferred가 기본이라 새 스레드를 만들 수도, 안 만들 수도 있다. 동작이 구현에 의존하므로 항상 명시적으로 std::launch::async 또는 std::launch::deferred를 지정하자.

임시 future 함정

std::async가 반환한 future는 특별 규칙을 따른다. 소멸자에서 비동기 작업이 끝날 때까지 블로킹 join한다. 즉 future를 변수로 받지 않으면 즉시 join이 일어나 사실상 동기 호출이 된다.

// 실수: 임시 future. 한 줄이 끝나면 소멸자에서 join 대기.
std::async(std::launch::async, longTask);  // ← 동기 호출과 같음!
std::async(std::launch::async, anotherTask); // longTask 끝난 뒤에야 시작

// 올바름: future를 변수로 받기
auto f1 = std::async(std::launch::async, longTask);
auto f2 = std::async(std::launch::async, anotherTask);
// 두 작업이 병렬 실행
std::async의 future는 소멸자에서 블로킹된다 일반 future(promise나 packaged_task에서 얻은)는 소멸자가 블로킹하지 않는다. 오직 std::async가 만든 future만 이 특별 규칙이 있다. 표준이 std::async를 위해 만든 예외다.

std::promise<T>

한쪽에서 값/예외를 set하고, 다른 쪽에서 future로 받는다. std::async가 함수 결과를 자동으로 future에 넣어준다면, promise는 사용자가 직접 넣는다.

#include <future>
#include <thread>

int compute() { return 42; }

std::promise<int> p;
std::future<int> f = p.get_future();  // future를 먼저 꺼낸다

std::thread t([&p]{
    int v = compute();
    p.set_value(v);  // 결과 전달
});

int result = f.get();  // 워커가 set_value할 때까지 대기
t.join();

주요 멤버 함수

함수역할
get_future()대응되는 future 반환 (한 번만)
set_value(v)결과 값 설정
set_exception(eptr)예외 설정. get() 시 다시 throw됨
언제 promise를 쓸까 결과 전달 시점이 함수 반환과 일치하지 않을 때다. 예: 콜백 기반 라이브러리, 이벤트 루프, 작업이 여러 단계에 나뉘어 있을 때 등.

std::shared_future<T>

여러 곳에서 같은 결과를 받고 싶을 때 사용한다. get()을 여러 번 호출할 수 있고, 여러 스레드에서 안전하게 읽을 수 있다.

std::promise<int> p;
std::shared_future<int> sf = p.get_future().share();  // future → shared_future

// 여러 워커가 같은 결과를 사용
std::thread w1([sf]{ std::cout << "worker1: " << sf.get() << "\n"; });
std::thread w2([sf]{ std::cout << "worker2: " << sf.get() << "\n"; });

p.set_value(100);
w1.join();
w2.join();
fan-out 패턴에 유용 한 번 계산한 결과를 N개의 워커가 각자 사용해야 할 때 shared_future가 깔끔하다. 한 번 share()하면 복사가 자유로워진다.

예외 전파

비동기 함수에서 throw된 예외는 future에 저장됐다가 get() 호출 시 다시 throw된다. 멀티스레드 예외 처리가 한결 깔끔해진다.

auto fut = std::async(std::launch::async, []{
    throw std::runtime_error("worker failed");
    return 0;
});

try
{
    int v = fut.get();  // 여기서 runtime_error가 다시 throw됨
}
catch (const std::exception& e)
{
    std::cerr << "caught: " << e.what() << "\n";
}

promise에서는 set_exception으로 명시적으로 예외를 전달한다.

std::promise<int> p;
auto f = p.get_future();

std::thread t([&p]{
    try
    {
        if (badInput()) throw std::runtime_error("bad input");
        p.set_value(compute());
    }
    catch (...)
    {
        p.set_exception(std::current_exception());
    }
});

try { int v = f.get(); }
catch (const std::exception& e) { /* ... */ }
t.join();

항목std::asyncstd::thread

예외 전파future가 자동 전달수동 (놓치면 std::terminate)
자동 joinfuture 소멸자에서 (특별 규칙)없음. 명시적 join/detach 필수
스레드 생성 비용구현이 풀링할 수 있음매번 새 스레드 생성
제어 수준낮음 (위임)높음 (직접 제어)
적합한 용도결과 받는 단발성 작업장기 실행, 세밀한 제어

흔한 함정

1. 임시 future 무시 → 동기 호출

앞서 다룬 함정. std::async 결과를 변수로 안 받으면 즉시 블로킹된다.

2. 같은 future에 get() 두 번 → UB

int v1 = fut.get();
int v2 = fut.get();  // UB! valid()는 false가 됨

3. deferred일 때 wait_for가 영원히 timeout 아님

deferred 정책의 future는 get()/wait() 호출 전엔 실행되지 않으므로 wait_for가 즉시 deferred 상태를 반환한다.

auto fut = std::async(std::launch::deferred, longTask);

// 잘못된 폴링 — 영원히 ready가 안 됨!
while (fut.wait_for(std::chrono::milliseconds(10)) != std::future_status::ready)
{
    // 무한 루프
}

// 올바른 처리 — deferred 먼저 체크
auto status = fut.wait_for(std::chrono::seconds(0));
if (status == std::future_status::deferred)
{
    fut.get();  // 여기서 실행됨
}
wait_for 폴링 전엔 항상 deferred 체크 deferred future는 호출자가 깨워주지 않으면 영원히 잠들어 있다. 폴링 루프를 돌리기 전에 반드시 status를 확인하자.

4. launch policy 기본값 사용

std::async(func)는 표준이 async | deferred를 허용하므로 동작이 구현마다 다르다. 항상 명시적으로 정책을 지정한다.

5. future를 두 곳에서 get

std::future는 복사 불가, 이동만 가능하다. 두 스레드에서 같은 결과를 받아야 한다면 shared_future로 변환한다.

C++20 stop_token / jthread (미리보기)

C++20부터 협력적 취소를 위한 std::stop_token과 자동 join되는 std::jthread가 추가됐다.

#include <thread>
#include <stop_token>

std::jthread worker([](std::stop_token st){
    while (!st.stop_requested())
    {
        doWork();
    }
});

// ... 어딘가에서 ...
worker.request_stop();  // 협력적 취소 요청
// jthread 소멸자에서 자동 join
jthread는 thread의 안전판 std::thread는 join을 잊으면 std::terminate가 호출된다. jthread는 소멸자에서 자동 stop + join하므로 RAII 안전성이 높다. 새 코드에선 가능한 jthread를 쓰자. 다음 글에선 thread pool 구현으로 이어진다.

언제 std::async를 쓸까

적합한 경우피해야 할 경우
한 번성 짧은 비동기 작업대량의 작업 분배 (스레드 풀이 낫다)
결과를 받아야 하는 백그라운드 계산장기 실행 워커 (jthread가 낫다)
예외 전파가 필요한 병렬 호출스레드 수를 정밀 제어해야 할 때
동기 코드를 빠르게 비동기로 전환실시간 우선순위 제어가 필요할 때
본격적인 작업 분배는 thread pool std::async는 매번 스레드를 만들 수 있어(구현 의존) 수백~수천 작업엔 부적합하다. 다음 글에서 std::packaged_task로 thread pool을 만드는 방법을 다룬다.
