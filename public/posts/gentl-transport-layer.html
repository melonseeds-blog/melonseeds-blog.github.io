GenTL이란?

GenTL(GenICam Transport Layer)은 GenICam 표준의 하부 구조로, 카메라와 호스트 PC 간의 데이터 전송을 추상화하는 표준 인터페이스이다. GigE Vision, USB3 Vision, Camera Link 등 물리적 전송 방식이 달라도, GenTL 인터페이스를 통해 동일한 API로 카메라를 제어하고 이미지를 수신할 수 있다.

GenTL의 핵심 목표

카메라 SDK 개발자나 사용자가 전송 계층(Transport Layer)의 세부 구현을 몰라도, 표준화된 함수 집합만으로 카메라 열거, 연결, 이미지 스트리밍, 해제를 수행할 수 있도록 하는 것이다. 마치 OS의 파일 시스템이 HDD/SSD/USB 드라이브의 차이를 숨기듯, GenTL은 GigE/USB3/CameraLink의 차이를 숨긴다.

GenICam과 GenTL의 관계 GenICam은 카메라 제어(레지스터 접근, 노드맵)를 표준화하고, GenTL은 데이터 전송(카메라 발견, 연결, 이미지 수신)을 표준화한다. 둘은 서로 보완적이며, 대부분의 머신비전 SDK는 GenICam + GenTL 조합으로 동작한다.

GenTL 계층 구조

GenTL은 5단계 계층으로 구성된다. 각 계층은 상위 계층의 핸들(Handle)을 통해 접근하며, 트리 형태로 확장된다.

그림 1. GenTL 5단계 계층 구조 — System, Interface, Device, DataStream, Buffer

각 계층의 역할

System (TL) GenTL 라이브러리의 최상위 진입점. TLOpen()으로 시작하며, 하위 Interface 목록을 제공한다. 하나의 CTI 파일 = 하나의 System.
Interface 물리적 통신 포트를 추상화한다. GigE Vision의 경우 NIC(네트워크 카드), USB3 Vision의 경우 USB 호스트 컨트롤러에 해당한다.
Device 개별 카메라를 나타낸다. Interface를 통해 발견(Enumerate)되며, 카메라에 연결하면 Device 핸들을 얻는다.
DataStream 이미지 데이터를 수신하는 논리적 채널. 대부분의 카메라는 하나의 DataStream을 가지지만, 멀티 스트림 카메라는 여러 개를 가질 수 있다.
Buffer 실제 이미지가 저장되는 메모리 블록. 애플리케이션이 할당(Announce)하고 큐(Queue)에 등록하면, GenTL이 이미지를 채워준다.

Producer와 Consumer

GenTL의 핵심 아키텍처는 Producer-Consumer 모델이다. 전송 계층의 구현과 사용을 분리하여, 서로 다른 제조사의 카메라와 SDK가 호환될 수 있게 한다.

GenTL Producer (CTI 파일)

  • 제공자 — 카메라 제조사 또는 SDK 벤더
  • 형태 — .cti 확장자의 DLL 파일
  • 역할 — 실제 전송 프로토콜 구현
  • 예시 — mvGenTLProducer.cti (MVTec), VwGigETL.cti (Vieworks)
  • 위치 — 환경변수 GENICAM_GENTL64_PATH로 지정

GenTL Consumer (SDK/앱)

  • 제공자 — SDK 또는 최종 애플리케이션
  • 형태 — SDK 라이브러리 또는 사용자 앱
  • 역할 — GenTL 표준 API를 호출하여 카메라 제어
  • 예시 — HALCON, VieworksSDK, Pylon 등
  • 특징 — Producer가 바뀌어도 코드 변경 불필요
그림 2. GenTL Producer-Consumer 아키텍처
CTI 파일 경로 설정 주의 GenTL Consumer(SDK)는 환경변수 GENICAM_GENTL64_PATH (64비트) 또는 GENICAM_GENTL32_PATH (32비트)에 등록된 경로에서 CTI 파일을 검색한다. 여러 SDK를 동시에 설치하면 경로가 꼬여 카메라를 못 찾는 문제가 발생할 수 있다.

카메라 연결 흐름

GenTL을 통한 카메라 사용은 열거(Discovery) - 연결 - 스트리밍 - 해제 4단계를 따른다. 각 단계에서 호출되는 GenTL 함수와 흐름을 정리한다.

1단계: 열거 (Discovery)

카메라 열거 과정

CTI 파일을 로드하여 System을 열고, Interface 목록을 가져온 뒤, 각 Interface에서 연결된 Device(카메라)를 검색한다. GigE Vision의 경우 네트워크 브로드캐스트, USB3 Vision의 경우 USB 디바이스 열거를 수행한다.

// 1. GenTL 초기화 및 카메라 열거 의사 코드
GCInitLib();                           // GenTL 라이브러리 초기화
hTL = TLOpen();                        // System 핸들 획득

TLUpdateInterfaceList(hTL);            // Interface 목록 갱신
n = TLGetNumInterfaces(hTL);            // Interface 개수 조회

for (i = 0; i < n; i++) {
    id = TLGetInterfaceID(hTL, i);      // Interface ID 획득
    hIF = TLOpenInterface(hTL, id);    // Interface 열기

    IFUpdateDeviceList(hIF);            // Device(카메라) 검색
    m = IFGetNumDevices(hIF);           // 발견된 카메라 수
    // 카메라 목록 표시...
}

2단계: 연결 (Connect)

발견된 Device ID로 카메라에 연결하고, DataStream을 열어 이미지 수신을 준비한다.

// 2. 카메라 연결 및 DataStream 준비
devID = IFGetDeviceID(hIF, 0);        // 첫 번째 카메라 ID
hDev = IFOpenDevice(hIF, devID);      // Device 열기 (카메라 연결)

nDS = DevGetNumDataStreams(hDev);       // DataStream 개수 조회
hDS = DevOpenDataStream(hDev, 0);      // DataStream 열기

3단계: 스트리밍 (Streaming)

Buffer를 할당하여 큐에 등록한 뒤, Acquisition을 시작하면 카메라가 이미지를 전송하기 시작한다.

// 3. 버퍼 할당 및 스트리밍 시작
for (i = 0; i < NUM_BUFFERS; i++) {
    pBuf = malloc(bufSize);             // 버퍼 메모리 할당
    hBuf = DSAnnounceBuffer(hDS, pBuf); // GenTL에 버퍼 등록
    DSQueueBuffer(hDS, hBuf);           // 수신 대기 큐에 추가
}

DSStartAcquisition(hDS);               // 스트리밍 시작
// + GenICam으로 AcquisitionStart 명령 전송

// 이미지 수신 루프
while (running) {
    hBuf = DSGetBufferID(hDS, timeout); // 채워진 버퍼 대기
    // 이미지 처리...
    DSQueueBuffer(hDS, hBuf);           // 처리 완료 후 재등록
}

4단계: 해제 (Release)

역순으로 자원을 해제한다. 순서를 지키지 않으면 핸들 누수나 크래시가 발생할 수 있다.

// 4. 역순 해제 (반드시 순서 준수)
DSStopAcquisition(hDS);                // 스트리밍 중지
DSFlushQueue(hDS);                      // 큐 비우기

for (i = 0; i < NUM_BUFFERS; i++) {
    DSRevokeBuffer(hDS, hBuf[i]);       // 버퍼 등록 해제
    free(pBuf[i]);                       // 메모리 해제
}

DSClose(hDS);                            // DataStream 닫기
DevClose(hDev);                          // Device 닫기
IFClose(hIF);                            // Interface 닫기
TLClose(hTL);                            // System 닫기
GCCloseLib();                             // 라이브러리 종료
해제 순서가 중요한 이유 GenTL의 각 핸들은 상위-하위 관계를 가진다. Buffer를 해제하지 않고 DataStream을 닫거나, Device를 닫지 않고 Interface를 닫으면 내부적으로 핸들 누수가 발생한다. 실제로 DeviceReset 같은 예외 상황에서 이 순서가 깨지면 대량의 핸들/메모리 누수로 이어진다.

주요 GenTL 함수

GenTL 표준은 약 50개의 함수를 정의한다. 아래는 실무에서 가장 자주 사용되는 핵심 함수들이다.

계층 함수 역할
공통 GCInitLib / GCCloseLib GenTL 라이브러리 초기화 / 종료
System TLOpen / TLClose System 핸들 열기 / 닫기
System TLUpdateInterfaceList Interface 목록 갱신
System TLGetInterfaceID 인덱스로 Interface ID 조회
Interface IFUpdateDeviceList 연결된 카메라 검색 (Discovery)
Interface IFGetDeviceID 인덱스로 Device ID 조회
Device DevOpenDataStream DataStream 열기
Device DevGetPort GenICam 레지스터 접근용 포트 획득
DataStream DSAnnounceBuffer 이미지 버퍼 등록
DataStream DSQueueBuffer 버퍼를 수신 대기 큐에 추가
DataStream DSStartAcquisition 이미지 수신 시작
DataStream DSRevokeBuffer 등록된 버퍼 해제
Buffer DSGetBufferInfo 버퍼 상태/정보 조회 (크기, 타임스탬프 등)
GenTL 함수 네이밍 규칙 함수 이름의 접두사가 계층을 나타낸다. TL = System, IF = Interface, Dev = Device, DS = DataStream, GC = Global (공통). 이 규칙만 알면 처음 보는 함수도 어떤 계층에서 사용되는지 바로 파악할 수 있다.

실무에서 GenTL 이해가 필요한 이유

대부분의 머신비전 SDK가 GenTL을 내부적으로 감싸고 있어 직접 GenTL 함수를 호출할 일은 드물다. 그러나 다음과 같은 상황에서 GenTL 구조를 이해하고 있으면 문제 해결이 훨씬 빨라진다.

디버깅과 핸들 누수 추적

핸들 누수의 근본 원인

GenTL의 5단계 계층은 각각 핸들을 생성한다. DeviceReset이나 비정상 종료 시 해제 순서가 깨지면, 하위 핸들(DataStream, Buffer)이 해제되지 않아 누수가 발생한다. 계층 구조를 이해해야 "어디서 누수가 시작되는지" 추적할 수 있다.

핸들 누수 Device를 닫지 않으면 DataStream/Buffer 핸들이 남아 프로세스 핸들 수가 계속 증가한다. 장시간 운용 시 시스템 불안정의 원인.
메모리 누수 Buffer를 Revoke하지 않으면 이미지 메모리(수 MB~수십 MB)가 회수되지 않는다. 고해상도 카메라일수록 심각.
이벤트 누수 등록된 이벤트 핸들러를 해제하지 않으면 콜백이 계속 호출되거나, 닫힌 핸들에 접근하여 크래시 발생.

멀티 Producer 충돌

여러 SDK(예: HALCON + Pylon + 자체 SDK)를 동시에 설치하면, 각 SDK가 자체 CTI Producer를 등록한다. 이때 GENICAM_GENTL64_PATH에 여러 경로가 등록되어 다음과 같은 문제가 발생할 수 있다.

흔한 증상

  • 카메라가 두 번 표시됨 (중복 열거)
  • 다른 SDK의 Producer가 카메라를 점유
  • 연결 시 "Device already opened" 에러
  • SDK 업데이트 후 카메라 인식 불가

해결 방법

  • 환경변수에서 사용하지 않는 CTI 경로 제거
  • SDK 초기화 시 특정 CTI 파일만 로드
  • Producer 목록을 로그로 출력하여 확인
  • 필요 시 GenTL Producer 경로를 코드에서 직접 지정

버퍼 관리 최적화

GenTL의 Buffer 관리 방식을 이해하면 이미지 수신 성능을 크게 개선할 수 있다.

버퍼 수 결정 기준 버퍼 수가 너무 적으면 처리 지연 시 프레임 드롭이 발생하고, 너무 많으면 메모리를 낭비한다. 일반적으로 3~10개를 권장하며, 이미지 처리 시간과 카메라 프레임레이트를 고려하여 결정한다. 고속 촬영(100fps 이상)에서는 큐에 항상 여유 버퍼가 있도록 넉넉하게 할당해야 한다.
Announce vs Allocate DSAnnounceBuffer는 사용자가 미리 할당한 메모리를 GenTL에 등록하는 방식이고, DSAllocAndAnnounceBuffer는 GenTL이 직접 할당하는 방식이다. 전자는 메모리 정렬(Alignment)을 직접 제어할 수 있어 DMA 전송 최적화에 유리하다.
