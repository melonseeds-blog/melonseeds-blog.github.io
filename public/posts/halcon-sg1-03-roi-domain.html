이 문서의 목적 ROI는 머신비전 시스템에서 가장 자주 다루지만, 가장 자주 잘못 설계되는 영역입니다. Region/Domain/Alignment를 한 흐름으로 묶어 시스템 안정성 관점에서 정리한 문서입니다.

ROI의 진짜 의미

ROI는 단순히 "이미지 일부만 처리하는 최적화"가 아닙니다. 실무에서는 ROI가 검사 안정성, 오검출 방지, 파라미터 단순화, 성능 최적화를 동시에 담당합니다.

HALCON에서 ROI와 관련해 자주 등장하는 개념은 다음입니다.

개념 의미
Region 픽셀들의 집합
Domain image에서 operator가 유효하게 처리할 영역
ROI 검사/처리 대상 관심 영역
Reduced image domain이 제한된 image
Mask 연산을 제한하는 binary 영역

HALCON에서는 많은 operator가 image의 domain을 고려합니다. 따라서 reduce_domain을 잘 활용하면 전체 이미지 데이터를 유지하면서 처리 대상만 제한할 수 있습니다.

ROI 기본 흐름

ROI 생성
→ 필요 시 ROI 이동/회전/확장
→ image domain 제한
→ ROI 내부에서 operator 수행
→ 결과 feature 추출

ROI 생성 방법은 크게 세 가지입니다.

  1. 수동/고정 geometry
    rectangle, circle, polygon 등
  2. segmentation 기반
    threshold, morphology, connected region
  3. matching/alignment 기반
    기준 pose를 찾고 ROI를 동적으로 변환

Region 생성

Region은 HALCON에서 binary object처럼 다루는 핵심 자료구조입니다.

3.1 Geometry 기반 region

  • rectangle
  • rotated rectangle
  • circle
  • ellipse
  • polygon

장점:

  • 빠름
  • 안정적
  • 시스템 구조화 쉬움

단점:

  • 위치 변화에 약함
  • 객체 형상이 복잡하면 불필요 영역 포함

3.2 Segmentation 기반 region

이미지에서 threshold나 edge를 통해 region을 생성합니다.

Image
→ threshold
→ connection
→ select_shape
→ final ROI/objects

주의할 점은 segmentation 기반 ROI는 조명에 의존한다는 것입니다. ROI 생성 자체가 흔들리면 이후 검사가 전부 흔들립니다.

3.3 Model 기반 ROI

가장 실무적인 구조입니다.

Reference model 생성
→ 현재 이미지에서 model 찾기
→ pose(row, column, angle, scale) 획득
→ reference ROI를 현재 pose로 변환
→ 변환된 ROI 내부 검사

이 방식은 위치 편차가 있는 라인에서 매우 중요합니다.

Domain 제어

HALCON image는 전체 matrix를 가지지만, domain을 통해 operator 처리 영역을 제한할 수 있습니다.

4.1 reduce_domain

reduce_domain은 image와 region을 결합해 처리 domain이 제한된 image를 만듭니다.

장점:

  • 원본 gray value 유지
  • operator 처리 영역 제한
  • 속도 향상
  • 오검출 감소

4.2 Domain과 crop의 차이

항목 reduce_domain crop
좌표계 원본 좌표 유지 좌표가 잘릴 수 있음
데이터 image object + domain 실제 image size 축소
후속 좌표 해석 쉬움 offset 관리 필요
속도 operator에 따라 개선 data size 감소로 개선

실무에서는 좌표 해석을 위해 reduce_domain을 선호하고, 메모리/속도 최적화가 중요하면 crop을 고려합니다.

ROI Alignment

고정 ROI만으로 가능한 시스템은 제한적입니다. 대부분의 부품은 위치, 회전, 스케일 편차를 가집니다.

5.1 Alignment가 필요한 이유

  • 컨베이어 위치 편차
  • 지그 유격
  • 부품 회전
  • 카메라 설치 오차
  • 제품 모델 변경
  • robot pick 위치 편차

5.2 기본 패턴

기준 객체 위치 검출
→ homography 또는 rigid transform 계산
→ reference ROI 변환
→ 변환된 ROI로 검사

5.3 Transform 종류

Transform 설명 사용 상황
Translation x/y 이동 위치만 변함
Rigid 이동 + 회전 일반 부품 정렬
Similarity 이동 + 회전 + scale 크기 변화 가능
Affine shear 포함 카메라 각도/왜곡 일부
Homography perspective 평면 물체의 원근 변화

ROI Shape과 성능

ROI는 단순히 작을수록 빠른 것이 아닙니다. 형태와 operator 특성에 따라 성능 차이가 납니다.

6.1 Rectangle ROI

  • 생성/관리 쉬움
  • memory locality 좋음
  • 불필요 영역 포함 가능

6.2 Arbitrary Region ROI

  • 실제 object shape에 밀착 가능
  • 오검출 감소
  • operator에 따라 처리 overhead 있음

6.3 Multiple ROI

여러 부품 검사에서 흔히 사용합니다.

global alignment
→ ROI1 검사
→ ROI2 검사
→ ROI3 검사
→ 결과 통합

이때 ROI별 threshold/feature 기준을 다르게 가져갈 수 있습니다.

ROI 재사용

반복 검사에서는 매 frame마다 복잡한 ROI를 새로 만들지 않는 것이 좋습니다.

재사용 가능한 항목:

  • reference ROI
  • transformed ROI template
  • measure object
  • metrology model
  • shape model
  • calibration map

동적으로 변환해야 하는 경우에도 reference ROI는 저장해두고 transform만 적용하는 구조가 좋습니다.

ROI와 Blob/Measuring/Matching 연결

8.1 Blob에서 ROI

Blob 분석에서는 ROI를 통해 배경 noise와 주변 object를 제거합니다.

reduce_domain
→ threshold
→ connection
→ select_shape

8.2 Measuring에서 ROI

1D measure는 측정 ROI 위치가 조금만 틀어져도 잘못된 edge를 잡을 수 있습니다. 따라서 matching 기반 alignment 후 measure ROI를 이동하는 것이 안정적입니다.

8.3 Matching에서 ROI

matching의 search space를 줄이면 속도가 크게 개선됩니다.

  • search ROI 제한
  • angle range 제한
  • scale range 제한
  • pyramid level 조정

실무 실패 케이스

문제 원인 대응
ROI가 대상에서 벗어남 alignment 없음 기준 matching 추가
ROI 내부에 배경 포함 ROI shape 부정확 ROI 축소 또는 mask 적용
속도 개선 안 됨 operator가 domain 최적화 미흡 crop 또는 algorithm 변경
threshold가 frame마다 흔들림 조명/노출 문제 acquisition 안정화
부분 가림 시 ROI 실패 기준점 하나에 의존 다중 기준 또는 fallback

설계 권장 패턴

10.1 고정 지그 검사

fixed ROI
→ threshold/blob/measure
→ rule-based decision

10.2 위치 편차 있는 부품 검사

shape matching
→ pose 획득
→ ROI transform
→ ROI 내부 blob/measure
→ decision

10.3 다품종 검사

product ID
→ recipe load
→ model/ROI/threshold parameter load
→ alignment
→ inspection

핵심 정리

  • ROI는 속도 최적화보다 안정성 구조로 이해해야 합니다.
  • HALCON의 domain 개념은 매우 중요합니다.
  • 고정 ROI보다 alignment 기반 동적 ROI가 실무에서 강합니다.
  • ROI shape, 재사용, transform 관리가 시스템 품질을 결정합니다.
  • ROI 설계가 좋아지면 segmentation과 matching parameter가 훨씬 단순해집니다.
