GigE Vision 네트워크 기본 — IP 설정

GigE Vision 카메라는 표준 이더넷(UDP/IP)을 사용하므로, PC의 NIC와 카메라가 같은 서브넷에 있어야 통신이 가능하다. IP 할당 방식에 따라 연결 안정성과 초기 검색 속도가 달라진다.

IP 할당 방식 비교

방식 설명 장점 단점 권장 환경
고정 IP (Persistent) 카메라에 IP를 직접 지정 부팅 즉시 연결, 안정적 수동 관리 필요 생산 라인 (권장)
DHCP DHCP 서버에서 자동 할당 관리 편리 서버 의존, IP 변경 가능 개발/테스트 환경
LLA (Link-Local) 169.254.x.x 대역 자동 할당 설정 불필요 할당 느림 (수십 초), 불안정 임시 연결/디버깅
실무 권장: 고정 IP 사용 생산 환경에서는 반드시 고정 IP를 사용한다. DHCP는 서버 장애 시 카메라 연결이 끊기고, LLA는 IP 할당에 30초 이상 걸릴 수 있어 라인 가동 시간에 영향을 준다. 고정 IP는 전원 인가 후 1~2초 내에 연결이 가능하다.

서브넷 분리 원칙

GigE Vision 카메라 트래픽은 대역폭을 거의 전부 사용하므로, 카메라 전용 NIC를 별도 서브넷으로 분리해야 한다. 사무용 네트워크와 같은 서브넷을 쓰면 패킷 충돌과 프레임 손실이 발생한다.

그림 1. 카메라 전용 서브넷과 사무 네트워크를 물리적으로 분리한 구성
같은 서브넷에 카메라와 사무망을 혼용하지 말 것 GigE Vision 카메라는 최대 1Gbps에 가까운 트래픽을 지속적으로 발생시킨다. 사무용 트래픽과 혼합되면 스위치 버퍼 오버플로우로 패킷 손실이 발생하고, 이미지가 깨지거나 프레임 드롭이 일어난다.

NIC 최적화

GigE Vision은 소프트웨어 기반 프로토콜(UDP)을 사용하므로, NIC 설정이 성능에 직접적인 영향을 준다. 기본 설정 그대로 사용하면 CPU 부하가 높고 패킷 손실이 발생할 수 있다.

Jumbo Frame (9000 바이트)

Jumbo Frame이란?

일반 이더넷 프레임의 MTU(Maximum Transmission Unit)는 1500 바이트다. Jumbo Frame은 이를 9000 바이트까지 확대하여, 동일한 데이터를 전송할 때 패킷 수를 약 1/6로 줄인다. 패킷당 헤더 처리 오버헤드가 감소하므로 CPU 부하가 크게 낮아진다.

기본 MTU 1500

  • 1MB 이미지 전송 시 약 700개 패킷
  • 패킷당 인터럽트 발생 가능
  • CPU 사용률 20~40%

Jumbo Frame 9000

  • 1MB 이미지 전송 시 약 120개 패킷
  • 인터럽트 횟수 대폭 감소
  • CPU 사용률 5~15%

설정 방법: 장치 관리자 > 네트워크 어댑터 > 카메라 연결 NIC > 속성 > 고급 > Jumbo Packet 또는 Jumbo Frame > 9014 Bytes 선택

카메라도 Jumbo Frame을 지원해야 한다 PC NIC만 설정하면 안 된다. 카메라의 SCPS(Stream Channel Packet Size) 레지스터를 9000으로 설정해야 양쪽 모두 Jumbo Frame으로 통신한다. 대부분의 GigE Vision SDK에서 카메라 연결 시 자동으로 최대 패킷 크기를 협상한다.

인터럽트 조절 (Interrupt Moderation)

NIC는 패킷을 수신할 때마다 CPU에 인터럽트를 걸어 처리를 요청한다. GigE Vision처럼 초당 수만 개의 패킷이 들어오면 인터럽트 자체가 CPU 병목이 된다.

설정 값 설명
Interrupt Moderation Rate Extreme 또는 Off 인터럽트를 모아서 처리 (또는 폴링 모드)
Receive Buffers 최대값 (2048~4096) 수신 버퍼를 늘려 패킷 드롭 방지
Receive Side Scaling Enabled 수신 처리를 여러 CPU 코어에 분산
Flow Control Disabled GigE Vision은 자체 흐름 제어 사용
수신 버퍼(Receive Buffers) 크기 설정이 가장 중요하다 기본값(보통 256)으로는 고해상도 카메라의 데이터를 처리하기에 부족하다. 최대값(2048 이상)으로 설정하면 버스트 트래픽 시에도 패킷 손실을 방지할 수 있다.

필터 드라이버

GigE Vision Filter Driver란?

Windows의 표준 네트워크 스택(Winsock)을 우회하여, NIC에서 직접 GigE Vision 패킷을 수신하는 커널 모드 드라이버다. Windows 네트워크 스택의 오버헤드(방화벽 검사, QoS, TCP/IP 처리 등)를 제거하여 CPU 부하를 크게 줄이고 패킷 손실을 방지한다.

그림 2. 표준 네트워크 스택 vs Filter Driver 경로 비교

성능 비교

항목 표준 소켓 (Winsock) Filter Driver
CPU 사용률 (1 카메라) 20~40% 3~8%
패킷 손실률 부하 시 0.1~5% 거의 0%
최대 카메라 수 (1 NIC) 1~2대 3~4대
방화벽 영향 받음 받지 않음

설치 방법

대부분의 GigE Vision SDK(MVS, Pylon, eBUS 등)에 Filter Driver가 포함되어 있다. SDK 설치 시 자동으로 설치되거나, 별도 유틸리티로 설치한다.

MVS (HIKROBOT) MVS 설치 시 자동 포함. MVS > 도구 > GigE 설정에서 확인 가능.
Pylon (Basler) Pylon GigE Configurator에서 NIC별로 Filter Driver 설치/제거.
eBUS (Pleora) eBUS Universal Pro Driver로 설치. 성능 최적화 모드 선택 가능.
수동 확인 장치 관리자 > NIC 속성 > 네트워킹 탭에서 Filter Driver 항목 확인.
서로 다른 SDK의 Filter Driver를 동시에 설치하지 말 것 MVS와 Pylon의 Filter Driver를 같은 NIC에 동시 설치하면 충돌이 발생한다. 하나의 NIC에는 하나의 Filter Driver만 설치해야 한다.

멀티 카메라 구성

여러 대의 GigE Vision 카메라를 하나의 시스템에 연결할 때는 대역폭 분배와 네트워크 토폴로지를 신중하게 설계해야 한다.

전용 NIC vs 스위치 구성

카메라당 전용 NIC

  • 카메라 1대 = NIC 1개
  • 대역폭 100% 보장
  • 가장 안정적, 패킷 손실 없음
  • PC에 NIC 슬롯 필요 (PCIe)
  • 권장: 고속/고해상도 카메라

GigE 스위치 공유

  • 여러 카메라를 스위치 1대에 연결
  • 대역폭을 카메라끼리 공유
  • 비용 절감, 설치 간편
  • 동시 트리거 시 대역폭 부족 가능
  • 권장: 저해상도, 순차 트리거

대역폭 계산

카메라의 필요 대역폭은 다음 공식으로 계산한다:

// 대역폭 계산 공식
대역폭(MB/s) = 가로(px) x 세로(px) x 비트심도(byte) x FPS

// 예시: 5MP 카메라, Mono8, 20fps
2448 x 2048 x 1 x 20 = 100,270,080 B/s = ~95.6 MB/s

// GigE 1Gbps 실효 대역폭: ~115 MB/s
// 위 카메라 1대는 GigE 1포트로 전송 가능 (95.6 / 115 = 83%)

// 예시: 같은 카메라 2대를 스위치에 연결하면?
95.6 x 2 = 191.2 MB/s > 115 MB/s = 대역폭 초과!
// -> 전용 NIC 2개 필요 또는 FPS/해상도 조정
경험 법칙: NIC 대역폭의 80%까지만 사용 프로토콜 오버헤드, 재전송 등을 고려하면 GigE 1Gbps NIC의 실사용 가능 대역폭은 약 90~100 MB/s다. 카메라 대역폭 합계가 이를 초과하면 반드시 NIC를 분리해야 한다.

IP 대역 분리 전략

NIC 서브넷 카메라 용도
NIC1 192.168.1.0/24 Camera 1~2 상면 검사 (Top View)
NIC2 192.168.2.0/24 Camera 3~4 측면 검사 (Side View)
NIC3 192.168.3.0/24 Camera 5 라인스캔 (Line Scan)
온보드 NIC 10.0.0.0/24 - 사무망 / PLC 통신

방화벽/보안 설정

GigE Vision은 UDP 프로토콜을 사용하며, 카메라 검색(Discovery)과 스트리밍에 서로 다른 포트 범위를 사용한다. Windows 방화벽이 이 포트를 차단하면 카메라가 검출되지 않는다.

GigE Vision 포트 범위

기능 프로토콜 포트 설명
Device Discovery UDP 3956 카메라 검색 (GVCP)
Control Channel UDP 3956 카메라 제어 명령 (GVCP)
Stream Channel UDP 동적 (49152~65535) 이미지 데이터 전송 (GVSP)
Message Channel UDP 동적 이벤트 알림

Windows 방화벽 예외 설정

가장 확실한 방법은 GigE Vision SDK 실행 파일을 방화벽 예외에 등록하는 것이다. 포트 단위보다 프로그램 단위 예외가 관리하기 쉽다.

# PowerShell (관리자 권한) - 프로그램 예외 등록
New-NetFirewallRule -DisplayName "GigE Vision App" `
    -Direction Inbound -Program "C:\MyApp\VisionApp.exe" `
    -Action Allow -Protocol UDP

# 또는 포트 범위 직접 열기
New-NetFirewallRule -DisplayName "GigE Vision GVCP" `
    -Direction Inbound -LocalPort 3956 `
    -Action Allow -Protocol UDP

New-NetFirewallRule -DisplayName "GigE Vision GVSP" `
    -Direction Inbound -LocalPort 49152-65535 `
    -Action Allow -Protocol UDP
Filter Driver 사용 시 방화벽 설정 불필요 Filter Driver는 Windows 네트워크 스택을 우회하므로 방화벽 규칙의 영향을 받지 않는다. Filter Driver가 설치된 NIC에서는 별도 방화벽 설정 없이 카메라 통신이 가능하다.
카메라 전용 NIC에서 Windows 방화벽 프로필을 "비공개"로 설정 새 NIC가 연결되면 Windows가 "공용 네트워크"로 인식하여 방화벽 규칙이 더 엄격해진다. 카메라 전용 NIC의 네트워크 프로필을 비공개(Private)로 변경하면 불필요한 차단을 줄일 수 있다.

트러블슈팅

GigE Vision 카메라 연결에서 자주 발생하는 문제와 해결 방법을 정리한다.

카메라 미검출

서브넷 불일치 PC NIC와 카메라가 다른 서브넷에 있으면 검색 안 됨. SDK의 Force IP 기능으로 카메라 IP를 임시 변경하여 확인.
방화벽 차단 UDP 3956 포트 차단 시 Discovery 패킷 무시됨. 일시적으로 방화벽을 끄고 테스트하여 원인 확인.
IP 충돌 두 카메라에 같은 IP가 설정된 경우 하나만 검출됨. 각 카메라의 MAC 주소로 개별 IP 재설정.
링크 속도 확인 Cat5e 미만 케이블이나 불량 포트는 100Mbps로 연결될 수 있음. NIC 상태에서 1.0 Gbps인지 확인.

패킷 손실 / 이미지 깨짐

Jumbo Frame 미설정 MTU 1500에서 고해상도 카메라 사용 시 CPU 부하로 패킷 드롭. Jumbo Frame 9000 설정.
수신 버퍼 부족 NIC Receive Buffers가 기본값(256)이면 버스트 트래픽에 드롭 발생. 최대값으로 증가.
다른 트래픽 간섭 사무용 트래픽이 같은 NIC/스위치에 있으면 대역폭 경쟁. 카메라 전용 네트워크 분리.
Inter-Packet Delay 카메라의 GevSCPD (Stream Channel Packet Delay)를 조정하여 패킷 간격을 넓힘. 스위치 환경에서 효과적.

대역폭 부족

FPS 줄이기 카메라의 AcquisitionFrameRate를 낮춰 대역폭 요구량 감소.
ROI 설정 관심 영역(Region of Interest)만 전송하여 데이터량 감소. Width, Height, OffsetX/Y 조정.
픽셀 포맷 변경 Mono12 → Mono8로 변경하면 데이터량 33% 감소. 컬러는 BayerRG8이 RGB8보다 1/3 대역폭.
NIC 추가 PCIe GigE NIC 카드를 추가하여 카메라별 전용 포트 확보. 가장 근본적인 해결책.
그림 3. GigE Vision 카메라 문제 해결 순서 — 물리 연결부터 네트워크 재설계까지
Wireshark로 패킷 분석하기 원인 불명의 패킷 손실이 발생할 때는 Wireshark에서 gvsp 필터로 GigE Vision 스트림 패킷을 캡처하면 어느 시점에서 패킷이 유실되는지 정확히 확인할 수 있다. 카메라의 GevSCPSPacketSize와 실제 캡처된 패킷 크기가 일치하는지도 확인하자.
