6.1 테스팅 지원 도구

테스트 도구는 다양한 테스트 활동을 지원하고 촉진한다. 실러버스에서는 다음과 같은 도구 유형을 소개한다.

도구 유형 분류

도구 유형설명예시
테스트 관리 도구SDLC, 요구사항, 테스트, 결함, 형상 관리를 용이하게 해서 테스트 프로세스 효율성을 높인다Jira, TestRail, Azure DevOps
정적 테스팅 도구테스터의 리뷰와 정적 분석 수행을 지원한다SonarQube, ESLint, PMD
테스트 설계 및 구현 도구테스트 케이스, 테스트 데이터, 테스트 절차 생성을 용이하게 한다Tricentis, 모델 기반 도구
테스트 실행 및 커버리지 도구자동 테스트 실행 및 커버리지 측정을 지원한다Selenium, JUnit, JaCoCo
비기능 테스팅 도구수동으로 수행하기 어렵거나 불가능한 비기능 테스트를 수행할 수 있게 한다JMeter, LoadRunner, Gatling
데브옵스 도구배포 파이프라인, 작업 흐름 추적, 자동 빌드, CI/CD 등을 지원한다Jenkins, GitHub Actions, GitLab CI
협업 도구원활한 커뮤니케이션을 지원한다Slack, Confluence, Teams
확장성 및 배포 표준화 도구가상 머신, 컨테이너화 도구 등Docker, Kubernetes, Vagrant
FL-6.1.1 학습 목표 (K2) 다양한 유형의 테스트 도구가 어떻게 테스팅을 지원하는지 설명할 수 있어야 한다. 각 도구 유형의 목적과 활용 장면을 구분해서 이해하는 것이 중요하다.
시험 팁: 도구 분류 매칭 문제 시험에서는 도구 설명과 도구 분류를 연결하는 매칭 문제가 자주 출제된다. 각 도구 유형의 핵심 키워드를 정확히 암기해두자. 예를 들어, "배포 파이프라인" = 데브옵스 도구, "리뷰 지원" = 정적 테스팅 도구, "가상 머신" = 확장성 및 배포 표준화 도구와 같이 연결하면 된다.

6.2 테스트 자동화의 효과와 리스크

도구를 도입한다고 성공이 보장되는 것은 아니다. 실질적이고 지속적인 효과를 얻기 위해서는 도구 도입, 유지보수, 교육 등에 추가 노력이 필요하다.

테스트 자동화의 효과 (이점)

  • 시간 절약: 반복적 수작업(리그레션 테스트 실행, 데이터 반복 입력, 예상/실제 결과 비교, 코딩 표준 확인)을 줄여준다
  • 일관성 및 재현성 향상: 사람의 단순 실수를 방지한다. 도구가 같은 주기 및 동일한 순서로 테스트를 실행한다
  • 객관적 평가: 커버리지 등 복잡한 측정치를 제공하며, 사람이 판단하기 어려운 데이터를 도출한다
  • 테스팅 정보 접근 용이: 테스트 진행률, 실패율, 실행 기간 등 통계/그래프/집계 데이터를 제공한다
  • 테스트 실행 시간 단축: 결함 조기 식별, 빠른 피드백, 출시 시간 단축이 가능해진다
  • 테스터의 시간 확보: 테스터가 새로운, 더 심층적이며 효과적인 테스트를 설계할 시간을 확보할 수 있다

테스트 자동화의 잠재적 리스크

  • 비현실적인 기대: 도구의 효과(기능 및 사용 편의성)에 대한 과도한 기대
  • 부정확한 추정: 도구 도입, 스크립트 유지보수, 기존 프로세스 변경에 필요한 시간/비용/노력을 잘못 예측
  • 부적절한 적용: 수동 테스팅이 더 적합한 곳에 테스트 도구를 사용
  • 도구 과의존: 사람의 비판적 사고의 필요성을 무시하고 도구에 지나치게 의존
  • 공급업체 종속성: 폐업, 지원 중단, 매각, 열악한 지원 등의 문제 발생 가능
  • 오픈소스 리스크: 지원이 중단되거나 잦은 업데이트가 필요할 수 있음
  • 플랫폼 비호환: 자동화 도구가 개발 플랫폼과 호환되지 않을 수 있음
  • 규제 부적합: 규제 요건이나 안전 표준을 준수하지 않는 부적합한 도구의 선택
FL-6.2.1 학습 목표 (K1) 테스트 자동화의 효과와 리스크를 상기할 수 있어야 한다. K1 수준이라 정확한 항목을 기억하고 있어야 한다.

자동화 vs 수동 테스트 판단 기준

자동화가 적합한 경우수동 테스트가 적합한 경우
반복적으로 실행하는 리그레션 테스트탐색적 테스팅(예측 불가 시나리오)
대량의 데이터 조합 테스트사용성(Usability) 테스팅
성능/부하 테스트한 번만 실행하는 테스트
CI/CD 파이프라인에서 자동 실행해야 하는 테스트요구사항이 자주 변경되는 초기 단계 테스트

핵심 용어 정리

Test Automation
테스트 자동화 - 도구를 사용하여 테스트 활동의 일부 또는 전부를 자동으로 수행하는 것
Test Management Tool
테스트 관리 도구 - 테스트 계획, 실행, 추적, 보고를 통합 관리하는 도구
Static Analysis Tool
정적 분석 도구 - 코드를 실행하지 않고 코드 품질을 분석하는 도구
CI/CD
지속적 통합/지속적 배포 - 코드 변경을 자동으로 빌드, 테스트, 배포하는 프로세스
Coverage Tool
커버리지 도구 - 테스트가 코드나 요구사항을 얼마나 커버하는지 측정하는 도구
Performance Testing Tool
성능 테스팅 도구 - 시스템의 응답 시간, 처리량 등을 측정하는 도구

연습 문제

문제 1 (FL-6.1.1, K2)
도구에 대한 설명이 다음과 같이 주어지고:
1. 작업 흐름 추적 지원
2. 의사소통 촉진
3. 가상 머신
4. 리뷰 지원

테스트 도구 분류는 다음과 같다:
A. 정적 테스팅 도구
B. 확장성 및 배포 표준화 지원 도구
C. 데브옵스 도구
D. 협업 도구

다음 중 도구 설명과 도구 분류를 가장 적절하게 연결한 것은?
  • a) 1A, 2B, 3C, 4D
  • b) 1B, 2D, 3C, 4A
  • c) 1C, 2D, 3B, 4A
  • d) 1D, 2C, 3A, 4B
정답: c)
1-C: 작업 흐름 추적은 데브옵스 도구의 기능이다.
2-D: 의사소통 촉진은 협업 도구의 목적이다.
3-B: 가상 머신은 확장성 및 배포 표준화 지원 도구에 해당한다.
4-A: 리뷰 지원은 정적 테스팅 도구의 기능이다.
문제 2 (FL-6.2.1, K1)
다음 중 테스트 자동화의 이점으로 가장 적절한 것은?
  • a) 사람이 도출하기에는 너무 복잡한 커버리지 측정치를 제공해준다
  • b) 테스팅에 대한 책임을 도구 공급업체와 공유하게 한다
  • c) 테스트 결과 분석 시 비판적 사고를 하지 않아도 되게 한다
  • d) 프로그램 코드를 분석해서 테스트 케이스를 작성한다
정답: a)
테스트 자동화는 사람이 도출하기에는 너무 복잡한 측정치를 제공할 수 있다.
b) 도구를 사용한다고 테스팅 책임이 공급업체에 이전되지 않는다.
c) 자동화되더라도 결과 해석에는 여전히 비판적 사고가 필요하다.
d) 테스터나 도구 모두 코드 분석만으로 테스트 케이스를 작성할 수 없다.
