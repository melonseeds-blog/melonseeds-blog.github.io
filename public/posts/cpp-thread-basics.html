C++11 이전 vs 이후

C++11 이전에는 표준 스레드 API가 없었다. POSIX 환경에서는 pthread, Windows에서는 CreateThread 같은 OS별 API를 직접 호출해야 했다. 같은 로직을 플랫폼마다 다르게 작성하거나 Boost.Thread 같은 외부 라이브러리에 의존해야 했다.

C++11부터 <thread>, <mutex>, <condition_variable>, <atomic>, <future> 헤더가 표준에 들어왔다. 이제 한 번 작성한 동시성 코드가 모든 플랫폼에서 동일하게 동작한다.

시기스레드동기화
~ C++03OS별 API (pthread, WinAPI)OS별 mutex/critical section
C++11std::threadstd::mutex, condition_variable
C++17—scoped_lock, shared_mutex
C++20std::jthread, stop_tokenstd::latch, barrier, semaphore

std::thread 기본

스레드 생성은 호출 가능한 객체와 그 인자를 생성자에 넘기는 방식이다. 함수, 함수 객체, 람다, 멤버 함수 모두 가능하다.

#include <thread>
#include <iostream>

void worker(int id, const std::string& name)
{
    std::cout << "thread " << id << " : " << name << std::endl;
}

struct Functor
{
    void operator()(int x) const { /* ... */ }
};

class Camera
{
public:
    void Grab() { /* ... */ }
};

int main()
{
    // 일반 함수
    std::thread t1(worker, 1, "alpha");

    // 함수 객체
    std::thread t2(Functor{}, 42);

    // 람다
    std::thread t3([]{ std::cout << "lambda" << std::endl; });

    // 멤버 함수 (첫 인자는 객체 포인터)
    Camera cam;
    std::thread t4(&Camera::Grab, &cam);

    t1.join(); t2.join(); t3.join(); t4.join();
}

join 또는 detach는 필수다

std::thread 객체가 소멸될 때 joinable() 상태이면 std::terminate가 호출되어 프로그램이 즉시 종료된다. 반드시 소멸 전에 join() 또는 detach()를 호출해야 한다.

std::thread t([]{ /* work */ });

if (t.joinable())
{
    t.join();   // 종료 대기 (대부분 이걸 쓴다)
    // t.detach();  // 분리 — 부모와 무관하게 실행
}
join/detach 누락 = 즉시 terminate 예외 경로까지 포함해 모든 흐름에서 join이 보장되어야 한다. 수동 관리는 RAII 래퍼나 C++20 std::jthread로 대체하는 것이 안전하다.

스레드 식별과 코어 수

std::thread t([]{
    std::cout << "id = " << std::this_thread::get_id() << std::endl;
});

std::thread::id tid = t.get_id();
unsigned n = std::thread::hardware_concurrency();  // 논리 코어 수 (힌트, 0 가능)

t.join();
C++20 std::jthread 소멸자에서 자동으로 request_stop()과 join()을 호출한다. stop_token을 인자로 받아 협력적 취소(cooperative cancellation)도 지원한다. C++20을 쓸 수 있다면 std::thread 대신 거의 항상 std::jthread가 정답이다. 
#include <thread>

// jthread는 소멸 시 자동 join
std::jthread jt([](std::stop_token st)
{
    while (!st.stop_requested())
    {
        // do work
    }
});
// 스코프 끝에서 stop 요청 + join 자동 수행

인자 전달 시 주의

스레드 생성자에 넘긴 인자는 기본적으로 값 복사된다. 함수가 참조로 받더라도 마찬가지다. 이 동작은 안전하지만 종종 의도와 다르다.

void process(Image& img);          // 참조로 받는 함수
void consume(std::unique_ptr<Buffer> b);

Image img;

// 잘못된 예 — 컴파일 에러 (참조에 복사본이 바인딩 못함)
// std::thread t(process, img);

// 올바른 예 — std::ref로 참조 전달
std::thread t1(process, std::ref(img));

// 이동 전용 객체는 std::move
auto buf = std::make_unique<Buffer>();
std::thread t2(consume, std::move(buf));

t1.join(); t2.join();
참조 전달의 수명에 주의 std::ref로 넘긴 객체가 스레드보다 먼저 파괴되면 dangling 참조가 된다. 부모 함수가 끝나도 스레드가 살아 있을 수 있는 detach 패턴에서 특히 위험하다.

데이터 레이스와 동기화

두 개 이상의 스레드가 동시에 같은 메모리 위치에 접근하고, 그 중 적어도 하나가 쓰기이며, 동기화가 없다면 데이터 레이스다. C++ 표준은 데이터 레이스를 정의되지 않은 동작(UB)으로 규정한다.

int counter = 0;

auto inc = []
{
    for (int i = 0; i < 100000; ++i)
        ++counter;   // 데이터 레이스!
};

std::thread a(inc), b(inc);
a.join(); b.join();

std::cout << counter << std::endl;  // 200000이 아닐 가능성 높음

++counter는 read-modify-write로 분해되며, 두 스레드의 단계가 섞이면서 일부 증가가 사라진다. 해결책은 뮤텍스로 보호하거나 std::atomic<int>를 사용하는 것이다.

std::mutex

가장 기본적인 상호 배제(mutual exclusion) 도구다. 한 시점에 하나의 스레드만 잠글 수 있다.

#include <mutex>

std::mutex m;
int counter = 0;

void inc()
{
    m.lock();
    ++counter;
    m.unlock();
}
직접 lock/unlock은 금지 중간에 예외가 던져지면 unlock이 호출되지 않아 영구 데드락이 된다. 항상 RAII 래퍼를 사용한다.

RAII 락 가드

std::mutex m;

void safeInc()
{
    std::lock_guard<std::mutex> lock(m);   // C++11
    ++counter;
}                                          // 스코프 끝에서 자동 unlock

// C++17 CTAD로 타입 생략 가능
void safeInc17()
{
    std::lock_guard lock(m);
    ++counter;
}
락 가드도입특징사용 시점
lock_guardC++11가장 가볍다. 잠금만, 해제 불가단순 보호 구간
unique_lockC++11defer/try/timed lock, 중간 unlock 가능condition_variable, 유연한 제어
scoped_lockC++17여러 뮤텍스를 데드락 없이 한 번에2개 이상의 뮤텍스
shared_lockC++14shared_mutex의 read 잠금reader-writer의 reader 측
여러 뮤텍스가 필요하면 무조건 scoped_lock std::scoped_lock lock(m1, m2); 한 줄이면 데드락 방지 알고리즘으로 두 뮤텍스를 안전하게 잠근다. 직접 순서를 맞추려는 시도보다 안전하고 명확하다.

데드락과 회피

두 스레드가 서로가 가진 자원을 기다리면 영원히 멈춘다. 가장 흔한 형태는 두 뮤텍스를 다른 순서로 잠그는 경우다.

std::mutex m1, m2;

// 위험한 패턴
void threadA()
{
    std::lock_guard l1(m1);
    std::lock_guard l2(m2);  // m2 대기 중...
}
void threadB()
{
    std::lock_guard l1(m2);
    std::lock_guard l2(m1);  // m1 대기 중... 데드락!
}
여러 뮤텍스를 잡을 때는 항상 동일한 순서로 규약으로 강제하기 어렵다면 std::scoped_lock 또는 std::lock(m1, m2)로 한 번에 잠가야 한다. 둘 다 데드락 회피 알고리즘을 내부적으로 사용한다. 
// C++17 — 가장 깔끔
void threadSafe()
{
    std::scoped_lock lock(m1, m2);
    // critical section
}

// C++11 호환 방식
void threadSafe11()
{
    std::unique_lock l1(m1, std::defer_lock);
    std::unique_lock l2(m2, std::defer_lock);
    std::lock(l1, l2);   // 데드락 없이 동시 잠금
}

recursive_mutex, shared_mutex

recursive_mutex

같은 스레드가 여러 번 잠글 수 있다. 잠근 횟수만큼 풀어야 해제된다. 보통 잠긴 함수 안에서 같은 객체의 다른 잠긴 함수를 호출해야 할 때 쓴다.

recursive_mutex가 필요한 상황은 대부분 설계 문제 public 메서드와 내부 헬퍼를 분리해서 헬퍼 쪽은 잠금 없이 호출하도록 리팩토링하는 편이 깔끔하다.

shared_mutex (C++17)

reader-writer 락이다. 여러 reader가 동시에 들어갈 수 있고, writer는 단독으로 잠근다. 읽기가 압도적으로 많고 쓰기가 드문 자료구조(설정 캐시, 조회 테이블 등)에 유용하다.

#include <shared_mutex>

std::shared_mutex sm;
std::map<int, std::string> cache;

std::string Read(int key)
{
    std::shared_lock lock(sm);    // 다중 reader 허용
    return cache.at(key);
}

void Write(int key, std::string val)
{
    std::unique_lock lock(sm);    // 단독 writer
    cache[key] = std::move(val);
}

std::condition_variable

"어떤 조건이 만족될 때까지 대기"를 표현하는 도구다. 단순한 sleep과 달리 다른 스레드의 통지(notify)에 즉시 깨어난다. 반드시 std::unique_lock<std::mutex>과 함께 써야 한다.

#include <condition_variable>
#include <queue>

std::mutex m;
std::condition_variable cv;
std::queue<int> q;
bool done = false;

// Producer
void produce(int v)
{
    {
        std::lock_guard lock(m);
        q.push(v);
    }
    cv.notify_one();   // 대기자 한 명 깨움
}

// Consumer
void consume()
{
    std::unique_lock lock(m);
    cv.wait(lock, []{ return !q.empty() || done; });

    if (!q.empty())
    {
        int v = q.front(); q.pop();
        lock.unlock();
        // process v ...
    }
}
wait는 반드시 predicate 형태로 cv.wait(lock, predicate)는 spurious wakeup(이유 없이 깨어나는 현상)을 자동으로 방어한다. 술어가 false면 다시 대기에 들어간다. 인자 없는 wait(lock)은 거의 쓸 일이 없다.
메서드동작사용 시점
notify_one대기자 중 한 명 깨움큐에 항목 1개 푸시 등
notify_all모든 대기자 깨움종료 신호, 상태 일괄 변경
condition_variable은 mutex 없이 쓰면 UB wait 호출 시점에 mutex가 잠긴 상태여야 하고, 술어가 검사하는 변수도 같은 mutex로 보호되어야 한다. 그렇지 않으면 통지를 놓치거나 race가 발생한다.

shared_ptr의 thread-safety 오해 제어 블록의 참조 카운트는 atomic이지만, shared_ptr 인스턴스 자체나 가리키는 객체에 대한 접근은 보호되지 않는다. 같은 shared_ptr을 여러 스레드에서 동시에 reset/= 하면 race다.

실전: 이미지 acquisition 파이프라인

카메라가 프레임을 grab하는 producer 스레드와, 프레임을 처리하는 consumer 스레드를 큐로 연결한 구조다. condition_variable로 빈 큐 대기를 효율적으로 처리한다.

#include <thread>
#include <mutex>
#include <condition_variable>
#include <queue>

class FrameQueue
{
public:
    void Push(Image f)
    {
        {
            std::lock_guard lock(m_mutex);
            m_queue.push(std::move(f));
        }
        m_cv.notify_one();
    }

    bool Pop(Image& out)
    {
        std::unique_lock lock(m_mutex);
        m_cv.wait(lock, [this]{ return !m_queue.empty() || m_done; });

        if (m_queue.empty())
            return false;   // done 신호로 깨어남

        out = std::move(m_queue.front());
        m_queue.pop();
        return true;
    }

    void Stop()
    {
        {
            std::lock_guard lock(m_mutex);
            m_done = true;
        }
        m_cv.notify_all();
    }

private:
    std::mutex m_mutex;
    std::condition_variable m_cv;
    std::queue<Image> m_queue;
    bool m_done = false;
};

void RunPipeline(Camera& cam, Processor& proc)
{
    FrameQueue queue;

    std::jthread producer([&](std::stop_token st)
    {
        while (!st.stop_requested())
        {
            Image frame;
            if (cam.Grab(frame) == ErrorCode::Ok)
                queue.Push(std::move(frame));
        }
        queue.Stop();
    });

    std::jthread consumer([&]
    {
        Image f;
        while (queue.Pop(f))
            proc.Process(f);
    });
    // 스코프 종료 시 producer에 stop 요청, 둘 다 자동 join
}
다음 글로 이어진다 프레임마다 스레드를 새로 만들면 비용이 크다. 고정된 워커들이 작업을 가져가는 스레드 풀로 발전시키는 것이 다음 주제다. std::async, std::future, std::packaged_task도 함께 살펴본다.
