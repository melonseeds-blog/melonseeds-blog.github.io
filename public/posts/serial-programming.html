시리얼 통신 프로그래밍 개요

산업 현장에서 장비를 제어하려면 시리얼 통신을 직접 구현해야 하는 경우가 많다. 조명 컨트롤러, 모터 드라이버, PLC, 바코드 리더 등 대부분의 산업 장비는 RS-232 또는 RS-485 기반 시리얼 통신을 지원하며, 제조사에서 제공하는 SDK가 없거나 커스텀 프로토콜을 사용하는 경우가 빈번하다.

왜 직접 구현해야 하는가?

장비 제조사가 제공하는 통신 라이브러리가 없거나, 특정 언어만 지원하거나, 다수의 장비를 하나의 프로그램에서 통합 제어해야 할 때 직접 구현이 필요하다. 시리얼 통신의 원리를 이해하면 어떤 장비든 매뉴얼의 프로토콜 문서만으로 연동할 수 있다.

그림 1. 시리얼 통신 프로그래밍의 기본 흐름

포트 설정

시리얼 통신에서 가장 중요한 것은 송신 측과 수신 측의 통신 파라미터가 정확히 일치해야 한다는 점이다. 하나라도 다르면 데이터가 깨지거나 통신 자체가 불가능하다.

핵심 파라미터 5가지

파라미터 설명 일반적인 값
Baud Rate 초당 전송되는 심볼(비트) 수. 양측이 반드시 동일해야 함 9600, 19200, 38400, 115200
Data Bits 한 프레임에 포함되는 실제 데이터 비트 수 8 (가장 보편적)
Parity 오류 검출용 비트. None/Even/Odd 중 선택 None (N)
Stop Bits 프레임 종료를 알리는 비트 수 1
Flow Control 데이터 흐름 제어. 수신 버퍼 오버플로우 방지 None (산업용 대부분)
8N1 표기법 "9600, 8N1"이라 하면 9600 baud, 8 data bits, No parity, 1 stop bit를 의미한다. 산업 장비 매뉴얼에서 가장 흔히 볼 수 있는 표기법이며, 대부분의 장비가 이 설정을 기본값으로 사용한다.

Flow Control 상세

Hardware (RTS/CTS)

  • RTS(Request to Send), CTS(Clear to Send) 핀 사용
  • 수신 버퍼가 차면 CTS를 Low로 내려 송신 중단 요청
  • 고속 통신이나 대량 데이터 전송 시 안정적
  • 별도 신호선 필요 (RS-232에서만 사용 가능)

Software (XON/XOFF)

  • 데이터 라인으로 제어 문자 전송 (XON=0x11, XOFF=0x13)
  • 별도 핀 불필요, 3선식 연결 가능
  • 바이너리 데이터에 XON/XOFF 값이 포함되면 오동작
  • 산업용에서는 거의 사용하지 않음

C++ 구현 (Windows)

Windows에서 시리얼 통신은 Win32 API를 사용한다. 시리얼 포트를 파일처럼 다루며, CreateFile로 열고 ReadFile/WriteFile로 데이터를 주고받는다.

포트 열기 및 설정

// 시리얼 포트 열기 및 설정 (Win32 API)
#include <windows.h>
#include <stdio.h>

HANDLE OpenSerialPort(const char* portName, DWORD baudRate)
{
    // 1. 포트 열기 ("\\\\.\\COM3" 형식 사용 - COM10 이상도 지원)
    HANDLE hSerial = CreateFileA(
        portName,
        GENERIC_READ | GENERIC_WRITE,
        0,              // 공유 모드 없음 (독점 열기)
        NULL,
        OPEN_EXISTING,
        0,              // 동기 모드 (FILE_FLAG_OVERLAPPED 시 비동기)
        NULL
    );

    if (hSerial == INVALID_HANDLE_VALUE) {
        printf("포트 열기 실패: %d\n", GetLastError());
        return INVALID_HANDLE_VALUE;
    }

    // 2. DCB 구조체로 통신 파라미터 설정
    DCB dcb = { 0 };
    dcb.DCBlength = sizeof(DCB);
    GetCommState(hSerial, &dcb);

    dcb.BaudRate = baudRate;       // 예: CBR_9600, CBR_115200
    dcb.ByteSize = 8;             // Data Bits
    dcb.Parity   = NOPARITY;      // No Parity
    dcb.StopBits  = ONESTOPBIT;    // 1 Stop Bit
    dcb.fDtrControl = DTR_CONTROL_ENABLE;
    dcb.fRtsControl = RTS_CONTROL_ENABLE;

    if (!SetCommState(hSerial, &dcb)) {
        printf("DCB 설정 실패: %d\n", GetLastError());
        CloseHandle(hSerial);
        return INVALID_HANDLE_VALUE;
    }

    // 3. 타임아웃 설정 (밀리초 단위)
    COMMTIMEOUTS timeouts = { 0 };
    timeouts.ReadIntervalTimeout         = 50;   // 바이트 간 최대 간격
    timeouts.ReadTotalTimeoutMultiplier  = 10;   // 바이트당 추가 시간
    timeouts.ReadTotalTimeoutConstant    = 100;  // 전체 읽기 최소 대기
    timeouts.WriteTotalTimeoutMultiplier = 10;
    timeouts.WriteTotalTimeoutConstant   = 100;
    SetCommTimeouts(hSerial, &timeouts);

    return hSerial;
}

데이터 송수신

// 데이터 송신
BOOL SendData(HANDLE hSerial, const char* data, DWORD length)
{
    DWORD bytesWritten = 0;
    if (!WriteFile(hSerial, data, length, &bytesWritten, NULL)) {
        printf("송신 실패: %d\n", GetLastError());
        return FALSE;
    }
    printf("송신 완료: %d bytes\n", bytesWritten);
    return TRUE;
}

// 데이터 수신
int ReceiveData(HANDLE hSerial, char* buffer, DWORD bufSize)
{
    DWORD bytesRead = 0;
    if (!ReadFile(hSerial, buffer, bufSize, &bytesRead, NULL)) {
        printf("수신 실패: %d\n", GetLastError());
        return -1;
    }
    return (int)bytesRead;
}

// 사용 예시
int main()
{
    HANDLE hPort = OpenSerialPort("\\\\.\\COM3", CBR_9600);
    if (hPort == INVALID_HANDLE_VALUE) return 1;

    // 명령 송신
    SendData(hPort, "GET_STATUS\r\n", 12);

    // 응답 수신
    char buf[256] = { 0 };
    int len = ReceiveData(hPort, buf, sizeof(buf));
    if (len > 0) printf("응답: %s\n", buf);

    CloseHandle(hPort);
    return 0;
}
COM 포트 이름 주의 COM10 이상의 포트를 열 때는 반드시 "\\\\.\\COM10" 형식을 사용해야 한다. "COM10"만 쓰면 CreateFile이 실패한다. 안전을 위해 항상 "\\\\.\\COMx" 형식을 사용하는 것이 좋다.

Python 구현

Python에서는 pyserial 라이브러리를 사용하면 OS에 관계없이 간결하게 시리얼 통신을 구현할 수 있다. 프로토타이핑이나 테스트 도구 제작에 매우 유용하다.

기본 사용법

# pip install pyserial
import serial
import time

# 포트 열기 및 설정
ser = serial.Serial(
    port='COM3',          # 포트 이름 (Linux: '/dev/ttyUSB0')
    baudrate=9600,         # Baud Rate
    bytesize=serial.EIGHTBITS,   # Data Bits = 8
    parity=serial.PARITY_NONE,    # Parity = None
    stopbits=serial.STOPBITS_ONE,  # Stop Bits = 1
    timeout=1              # 읽기 타임아웃 (초)
)

# 데이터 송신
ser.write(b'GET_STATUS\r\n')
print(f"송신 완료")

# 데이터 수신 (줄 단위)
response = ser.readline()
print(f"응답: {response.decode('ascii').strip()}")

# 포트 닫기
ser.close()

실무 패턴: 명령-응답 루프

import serial
import time

class SerialDevice:
    def __init__(self, port, baudrate=9600, timeout=1):
        self.ser = serial.Serial(port, baudrate, timeout=timeout)
        time.sleep(0.5)  # 포트 안정화 대기
        self.ser.reset_input_buffer()  # 수신 버퍼 초기화

    def send_command(self, cmd):
        """명령 송신 후 응답 수신"""
        self.ser.reset_input_buffer()
        self.ser.write((cmd + '\r\n').encode('ascii'))

        # 응답 대기 (최대 3초)
        response = b''
        start = time.time()
        while time.time() - start < 3:
            if self.ser.in_waiting > 0:
                response += self.ser.read(self.ser.in_waiting)
                if b'\r\n' in response:
                    break
            time.sleep(0.01)

        return response.decode('ascii').strip()

    def close(self):
        self.ser.close()

# 사용 예시
dev = SerialDevice('COM3', 9600)
status = dev.send_command('GET_STATUS')
print(f"장비 상태: {status}")
dev.close()
포트 안정화 대기 시리얼 포트를 열고 나서 바로 데이터를 보내면 첫 번째 명령이 무시되는 경우가 있다. time.sleep(0.5)로 포트가 안정화될 시간을 주고, reset_input_buffer()로 쓰레기 데이터를 제거하는 것이 좋다.

RS-485 멀티드롭 통신

RS-485는 하나의 버스에 최대 32개 장비를 연결할 수 있는 멀티드롭(Multi-drop) 방식이다. 반이중(Half-Duplex)이므로 한 번에 하나의 장비만 송신할 수 있으며, 일반적으로 마스터-슬레이브 구조를 사용한다.

DE/RE 방향 제어

DE/RE 핀이란?

RS-485 트랜시버(예: MAX485)에는 DE(Driver Enable)와 RE(Receiver Enable) 핀이 있다. DE를 High로 하면 송신 모드, RE를 Low로 하면 수신 모드가 된다. 보통 DE와 RE를 하나의 GPIO에 연결하여 동시에 전환한다.

그림 3. RS-485 멀티드롭 버스 구조와 마스터-슬레이브 통신

주소 기반 프로토콜 설계

RS-485 멀티드롭에서는 각 슬레이브를 구분하기 위한 주소(Address)가 필요하다. 자체 프로토콜을 설계할 때 일반적인 프레임 구조는 다음과 같다.

그림 4. 주소 기반 RS-485 프로토콜 프레임 구조
STX / ETX 패킷의 시작(0x02)과 끝(0x03)을 표시. 수신 측에서 프레임 경계를 판별하는 데 사용.
ADDR 대상 슬레이브 주소. 0x00은 브로드캐스트, 0x01~0xFF는 개별 장비.
CMD 명령 코드. 예: 0x01=읽기, 0x02=쓰기, 0x03=상태 요청 등.
CRC ADDR~DATA 영역의 CRC-16 체크섬. 전송 중 데이터 손상 감지용.
Modbus RTU 산업 현장에서 가장 널리 쓰이는 RS-485 프로토콜은 Modbus RTU다. 직접 프로토콜을 설계하기보다 Modbus를 채택하면 다양한 장비와의 호환성을 확보할 수 있다. Python에서는 pymodbus 라이브러리를 사용하면 된다.

디버깅 도구

시리얼 통신은 눈에 보이지 않기 때문에 디버깅이 까다롭다. 적절한 도구를 활용하면 문제를 빠르게 파악할 수 있다.

소프트웨어 도구

시리얼 터미널

  • Tera Term — 무료, 가볍고 빠름. 매크로 지원
  • PuTTY — SSH뿐 아니라 시리얼도 지원
  • RealTerm — HEX 표시, 바이너리 송수신에 강점
  • CoolTerm — macOS/Linux 크로스 플랫폼

시리얼 모니터 (스니퍼)

  • Free Serial Port Monitor — 포트 트래픽 캡처
  • Serial Port Monitor (Eltima) — 상용, 강력한 필터
  • Wireshark + USBPcap — USB-Serial 트래픽 분석
  • 기존 프로그램을 수정하지 않고 통신 내용 확인 가능

하드웨어 도구

루프백 테스트 TX와 RX 핀을 직접 연결하여 보낸 데이터가 그대로 돌아오는지 확인. 가장 기본적인 하드웨어 검증 방법. RS-232에서는 2번(TX)과 3번(RX) 핀을 점퍼로 연결한다.
오실로스코프 실제 전기 신호를 확인. Baud Rate 불일치, 노이즈, 신호 레벨 문제를 시각적으로 진단할 수 있다. RS-232는 +/-3~15V, RS-485는 차동 신호를 확인한다.
로직 분석기 디지털 신호를 캡처하여 프로토콜 수준에서 분석. 저가형(Saleae Logic 클론)으로도 시리얼 디코딩이 가능하다. 타이밍 문제 파악에 유용하다.
USB-Serial 컨버터 FTDI FT232, CP2102, CH340 칩 기반. 드라이버 호환성과 안정성에서 FTDI가 가장 우수하며, 산업용에서는 FTDI 칩을 권장한다.

실무 트러블슈팅

시리얼 통신에서 자주 발생하는 문제와 해결 방법을 정리한다. 대부분의 문제는 설정 불일치나 하드웨어 연결 오류에서 비롯된다.

깨진 데이터 (문자 깨짐)

원인 증상 해결 방법
Baud Rate 불일치 완전한 쓰레기 문자 출력 양측 Baud Rate를 동일하게 맞춤. 오실로스코프로 실제 속도 확인
Parity/Stop Bits 불일치 일부 문자만 깨지거나 간헐적 오류 장비 매뉴얼 확인 후 정확한 설정 적용
전기 노이즈 간헐적 비트 오류, CRC 에러 차폐 케이블 사용, RS-485 전환, 종단 저항(120ohm) 확인
GND 미연결 RS-232에서 불안정한 통신 반드시 GND(Pin 5)를 연결. 기준 전위가 없으면 신호 판별 불가

타임아웃 / 무응답

TX/RX 뒤바뀜 가장 흔한 실수. TX→RX, RX→TX로 교차 연결해야 한다. Null Modem 케이블 사용 여부 확인.
포트 번호 오류 장치 관리자에서 실제 COM 포트 번호 확인. USB-Serial 컨버터는 뽑았다 꽂을 때 번호가 바뀔 수 있다.
포트 이미 사용 중 다른 프로그램이 포트를 점유하고 있으면 열기 실패. 시리얼 터미널이나 이전 프로그램 인스턴스를 종료한다.
RS-485 방향 제어 DE/RE 전환 타이밍 오류. 송신 완료 후 충분한 지연 시간을 두고 수신 모드로 전환해야 한다.
종단 문자 누락 많은 장비가 CR(0x0D), LF(0x0A), 또는 CR+LF를 명령 종료 구분자로 사용. 누락 시 장비가 명령을 인식하지 못한다.

포트 충돌 및 기타 문제

USB-Serial 드라이버 문제 CH340 칩 기반 컨버터는 저가형이지만 드라이버 충돌이 잦다. FTDI FT232R 또는 Silicon Labs CP2102가 안정적이다. Windows 업데이트 후 드라이버가 깨지는 경우도 있으므로, 장치 관리자에서 느낌표 표시가 없는지 확인한다.
체계적 디버깅 순서 1. 루프백 테스트로 하드웨어 정상 확인 → 2. 시리얼 터미널로 수동 명령 테스트 → 3. 시리얼 모니터로 실제 송수신 데이터 캡처 → 4. 코드 레벨 디버깅. 이 순서를 지키면 문제의 위치(하드웨어 vs 소프트웨어)를 빠르게 좁힐 수 있다.
