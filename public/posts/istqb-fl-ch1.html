1.1 테스팅이란 무엇인가?

소프트웨어는 우리 일상에 깊이 관여하고 있다. 올바르게 동작하지 않는 소프트웨어는 금전적 손실, 시간 낭비, 비즈니스 평판 하락은 물론 심각한 경우 인명 피해까지 초래할 수 있다. 소프트웨어 테스팅은 소프트웨어의 품질을 평가하고, 운영 중 발생할 수 있는 장애의 위험을 줄이기 위한 활동이다.

테스팅은 단순히 프로그램을 실행하고 결과를 확인하는 행위(테스트 실행)에 국한되지 않는다. 테스팅에는 계획, 분석, 설계, 구현, 실행, 완료 등 다양한 활동이 포함되며, 이 활동들은 소프트웨어 개발수명주기(SDLC)에 걸쳐 수행된다.

또한 테스팅은 기술적 활동만이 아니다. 적절한 계획, 관리, 추정, 모니터링, 제어도 필요하며, 테스터의 전문 지식과 분석 기술을 활용하는 지적인 활동이라는 점을 기억해야 한다.

테스트 목적 (FL-1.1.1 K1)

일반적인 테스트 목적은 다음과 같다:

  • 요구사항, 사용자 스토리, 설계, 소스 코드 등 작업 산출물 평가
  • 장애 유발 및 결함 식별
  • 테스트 대상에 대해 요구된 커버리지 확보
  • 이해관계자가 정보에 입각한 결정을 내리는 데 필요한 정보 제공
  • 테스트 대상의 품질에 대한 자신감 획득
  • 테스트 대상의 완성 여부와 이해관계자의 기대 충족 여부를 확인하는 밸리데이션
Verification vs Validation 베리피케이션(Verification)은 시스템이 주어진 요구사항(명세)을 올바르게 충족하는지 확인하는 것이다. "제품을 올바르게 만들고 있는가?(Are we building the product right?)"
밸리데이션(Validation)은 시스템이 운영 환경에서 사용자의 실제 필요를 만족하는지 확인하는 것이다. "올바른 제품을 만들고 있는가?(Are we building the right product?)"

테스팅은 동적 테스팅(소프트웨어를 실행)과 정적 테스팅(소프트웨어를 실행하지 않음)으로 나뉜다. 정적 테스팅에는 리뷰와 정적 분석이 포함되고, 동적 테스팅에는 다양한 테스트 기법과 접근법이 사용된다.

테스팅과 디버깅 (FL-1.1.2 K2)

테스팅과 디버깅은 별개의 활동이다.

  • 테스팅: 소프트웨어 결함으로 인한 장애를 유발하거나(동적 테스팅), 테스트 대상에 있는 결함을 직접 찾아내는(정적 테스팅) 활동
  • 디버깅: 장애의 원인(결함)을 찾고, 분석하고, 제거하는 활동

동적 테스팅이 장애를 유발하면, 디버깅 프로세스가 시작된다:

  1. 장애 재현
  2. 분석 (결함 찾기)
  3. 결함 수정

수정 후에는 확인 테스팅(confirmation testing)으로 수정이 올바른지 검증하고, 리그레션 테스팅(regression testing)으로 수정이 다른 부분에 영향을 주지 않았는지 확인한다.

시험 포인트 테스팅은 결함을 "발견"하는 활동이고, 디버깅은 결함을 "수정"하는 활동이다. 이 차이를 명확히 구분해야 한다. 정적 테스팅에서는 장애를 유발하지 않고 직접 결함을 식별하므로 재현/분석 단계가 필요 없다.

1.2 테스팅이 왜 필요한가?

테스팅은 품질 제어 활동의 일환으로, 정해진 범위, 시간, 품질, 예산 내에서 합의된 테스트 목표를 달성하는 데 도움을 준다. 테스팅이 성공에 기여하는 것은 테스트팀만의 활동이 아니며, 모든 이해관계자가 테스트 기술을 활용해 프로젝트 성공에 기여할 수 있다.

테스팅이 성공에 기여하는 방법 (FL-1.2.1 K2)

테스팅은 결함을 식별하는 비용 효율적인 방법이다. 식별한 결함을 디버깅을 통해 제거하면 테스트 대상의 품질이 간접적으로 향상된다.

  • SDLC 여러 단계에서 테스트 대상의 품질을 직접 평가할 수 있다
  • 평가결과는 다음 단계(릴리스 판단 등)를 위한 의사결정에 기여한다
  • 테스터는 사용자를 간접적으로 대변하며, 사용자 요구사항이 반영되도록 한다
  • 계약/법적 요구사항 충족 또는 규제 표준 준수를 위해서도 테스팅이 필요하다

테스팅과 품질 보증(QA) (FL-1.2.2 K1)

품질 보증(QA)과 테스팅은 혼용되는 경우가 많지만 같지 않다.

테스팅품질 보증(QA)
초점제품 지향 (교정 접근법)프로세스 지향 (예방 접근법)
속성품질 제어(QC)의 주요 활동프로세스 구현과 개선에 집중
결과 활용결함 수정에 사용개발/테스트 프로세스가 잘 동작하는지 확인하는 피드백
책임주로 테스트팀프로젝트 참여 모두의 책임
시험 포인트 테스팅 = 제품 지향, 교정(corrective) 접근법, 품질 제어(QC)에 속함
QA = 프로세스 지향, 예방(preventive) 접근법

오류, 결함, 장애, 근본 원인 (FL-1.2.3 K2)

사람은 다양한 이유(시간 압박, 복잡성, 피로, 훈련 부족 등)로 오류(error, 실수)를 저지른다. 오류로 인해 작업 산출물에 결함(defect, 버그)이 생기고, 결함이 실행 시 장애(failure)로 이어질 수 있다.

사람의 오류
(Error)
산출물의 결함
(Defect/Bug)
시스템의 장애
(Failure)

다만, 모든 오류가 결함으로, 모든 결함이 장애로 이어지는 것은 아니다. 또한 장애는 결함 외에 환경 조건(방사선, 전자기장, 오염 등)으로 발생하기도 한다.

근본 원인(root cause)은 문제 발생의 근본적인 이유를 말한다. 근본 원인 분석(root cause analysis)을 통해 이를 찾고, 근본 원인을 처리(제거)하면 유사한 장애와 결함의 재발을 예방하거나 빈도를 줄일 수 있다.

주의: 오류 vs 결함 vs 장애 오류는 사람이 저지르는 "행위"이고, 결함은 산출물에 존재하는 "상태"이며, 장애는 시스템 실행 시 나타나는 "현상"이다. 시험에서 이 세 용어의 구분이 빈출된다.

1.3 테스팅의 원리

ISTQB 실러버스에서 제시하는 7가지 테스팅 원리는 모든 테스팅에 적용할 수 있는 포괄적 지침이다. 시험에서 매우 높은 빈도로 출제되므로 반드시 암기해야 한다.

7가지 원리 상세 (FL-1.3.1 K2)

원리 1: 테스팅은 결함의 존재를 밝히는 활동이지, 결함이 없음을 증명하지는 않는다

테스팅으로 테스트 대상에 결함이 있음을 보여줄 수 있지만, 결함이 없다는 것을 증명할 수는 없다. 결함이 전혀 발견되지 않더라도 그 소프트웨어가 완벽하다는 뜻은 아니다. 테스팅은 잔존 결함의 확률을 줄이는 활동이다.

원리 2: 완벽한 테스팅은 불가능하다 (Exhaustive Testing)

매우 간단한 소프트웨어를 제외하면, 모든 입력값과 사전 조건의 조합을 테스트하는 것은 불가능하다. 따라서 완벽하게 테스트하려 하기보다 테스트 기법, 테스트 케이스 우선순위 지정, 리스크 기반 테스팅을 사용해 테스트 노력을 집중해야 한다.

원리 3: 조기 테스팅으로 시간과 비용을 절약할 수 있다 (Early Testing)

프로세스 초기에 결함을 발견해 제거하면, 후속 산출물의 결함으로 이어지지 않는다. SDLC 후반에 발생하는 장애가 줄기 때문에 품질 비용이 절감된다. 정적 테스팅과 동적 테스팅 모두 최대한 이른 시점에 시작해야 한다.

원리 4: 결함은 집중된다 (Defect Clustering)

대부분의 결함은 소수의 시스템 컴포넌트에 집중되어 발생하는 경향이 있다. 운영 장애의 대부분 역시 소수의 컴포넌트에서 발생한다. 이는 파레토 원리(80/20 법칙)의 예이다. 리스크 기반 테스팅에서 결함 집중 영역을 중요하게 고려한다.

원리 5: 테스트 효과는 줄어든다 (Pesticide Paradox)

같은 테스트를 계속해서 반복하면, 결국 해당 테스트로 신규 결함을 식별하는 효과가 점점 줄어든다. 이 효과를 극복하려면 기존 테스트와 테스트 데이터를 변경하고, 새로운 테스트를 작성해야 한다. 다만, 리그레션 테스팅처럼 반복이 유용한 경우도 있다.

원리 6: 테스팅은 정황에 의존적이다 (Testing is context dependent)

테스팅에 "만능" 접근법은 없다. 동일한 테스팅 방법이 모든 상황에 적합하지 않으며, 테스팅은 프로젝트의 정황(사용 도메인, 리스크 수준, 개발 방법론 등)에 따라 수행 방식이 달라진다.

원리 7: 오류 부재의 궤변 (Absence-of-errors fallacy)

많은 결함을 발견하고 수정하더라도, 사용자의 요구나 기대에 못 미치거나 비즈니스 목표 달성에 도움이 되지 않는 시스템이 만들어질 수 있다. 따라서 베리피케이션과 함께 밸리데이션도 반드시 수행해야 한다.

7가지 원리 암기 키워드 1. 결함 존재 O, 부재 증명 X  |  2. 완벽 불가  |  3. 조기 테스팅  |  4. 결함 집중(파레토)  |  5. 살충제 패러독스  |  6. 정황 의존  |  7. 오류 부재 궤변(V&V)

1.4 테스트 활동, 테스트웨어, 테스트 역할

테스팅은 정황에 의존적이지만, 테스트 목적을 달성하기 위해 필요한 보편적인 테스트 활동들이 존재한다. 이런 활동들이 테스트 프로세스(test process)를 구성한다.

테스트 활동과 업무 (FL-1.4.1 K2)

테스트 프로세스의 주요 활동은 순차적으로 보이지만, 실제로는 반복적 또는 병렬로 수행되는 경우가 많다.

활동설명
테스트 계획
(Test Planning)		테스트 목적을 정의하고, 제약 조건 내에서 목적을 가장 잘 달성할 수 있는 접근법을 선택하는 과정
테스트 모니터링과 제어
(Test Monitoring & Control)		모니터링: 모든 테스트 활동을 점검하고 실제 진행을 계획과 비교
제어: 테스트 목적 달성에 필요한 조치 수행
테스트 분석
(Test Analysis)		테스트 베이시스를 분석해 테스트 가능한 기능을 식별하고, 테스트 컨디션을 정의/우선순위 부여. "무엇을 테스트할 것인가?"
테스트 설계
(Test Design)		테스트 컨디션을 테스트 케이스와 기타 테스트웨어로 정교화. "어떻게 테스트할 것인가?"
테스트 구현
(Test Implementation)		테스트 실행에 필요한 테스트웨어(테스트 데이터, 스크립트 등) 생성. 테스트 환경 구축 및 검증
테스트 실행
(Test Execution)		테스트 실행 일정에 따라 테스트를 수행하고, 실제 결과와 기대 결과를 비교. 이상 사항 분석 및 보고
테스트 완료
(Test Completion)		마일스톤(릴리스, 반복 주기 완료 등)에서 수행. 미해결 결함 처리, 테스트웨어 보관/인계, 교훈 파악, 완료 보고서 작성

정황에 따른 테스트 프로세스 (FL-1.4.2 K2)

테스팅은 단독으로 수행되지 않으며, 다양한 정황 요소에 따라 수행 방식이 달라진다:

  • 이해관계자 — 필요, 기대, 요구사항, 협력 의지 등
  • 팀원 — 기술, 지식, 경험 수준, 가용성, 훈련 필요성 등
  • 비즈니스 도메인 — 테스트 대상의 중요도, 식별된 리스크, 시장 요구, 법적 규제 등
  • 기술적 요인 — 소프트웨어 유형, 제품 아키텍처, 사용 기술 등
  • 프로젝트 제약 조건 — 범위, 시간, 예산, 자원 등
  • 조직 요인 — 조직 구조, 정책, 관행 등
  • 소프트웨어 개발수명주기 — 개발 프랙티스, 개발 방법론 등
  • 도구 — 가용성, 사용 용이성, 시장 상황 등

테스트웨어 (FL-1.4.3 K2)

테스트웨어(testware)는 테스트 활동을 통해 생성되는 작업 산출물이다. 활동별 산출물은 다음과 같다:

테스트 활동작업 산출물 (테스트웨어)
테스트 계획테스트 계획서, 테스트 일정, 리스크 관리 대장, 시작/완료 조건
테스트 모니터링/제어테스트 진행 상황 보고서, 제어 지침 문서, 리스크 정보
테스트 분석(우선순위가 지정된) 테스트 컨디션, 결함 보고서
테스트 설계(우선순위가 지정된) 테스트 케이스, 테스트 차터, 커버리지 항목, 테스트 데이터/환경 요구사항
테스트 구현테스트 절차, 수동/자동 테스트 스크립트, 테스트 스위트, 테스트 데이터, 테스트 실행 일정, 테스트 환경 요소(스텁, 드라이버, 시뮬레이터 등)
테스트 실행테스트 로그, 결함 보고서
테스트 완료테스트 완료 보고서, 개선 실천 항목, 교훈 기록, 변경 요청서

추적성 (FL-1.4.4 K2)

효과적인 테스트 모니터링과 제어를 위해서는 테스트 베이시스의 개별 요소, 관련 테스트웨어(테스트 컨디션, 리스크, 테스트 케이스), 테스트 결과, 결함 간의 추적성(traceability)을 구축하고 유지하는 것이 중요하다.

추적성이 제공하는 가치:

  • 커버리지 평가 지원
  • 테스트 목적 달성 정도를 나타내는 핵심 성과 지표(KPI)로 활용
  • 비즈니스 목표 대비 제품 품질, 프로세스 역량, 프로젝트 진행 상황 평가 정보 제공
  • 이해관계자에게 이해하기 쉬운 테스트 보고서 작성 지원
  • IT 거버넌스의 감사 요구사항 충족

테스팅에서의 역할 (FL-1.4.5 K2)

실러버스는 두 가지 주요 테스팅 역할을 다룬다:

테스트 관리 역할테스팅 역할
책임테스트 프로세스, 테스트팀, 테스트 활동 리더십 전반테스팅의 공학(기술)적 측면 전반
주요 활동테스트 계획, 모니터링, 제어, 테스트 완료테스트 분석, 설계, 구현, 실행
수행자 예팀 리더, 테스트 관리자, 개발 관리자테스터, 테스트 분석가, 테스트 자동화 엔지니어

상황에 따라 한 사람이 두 역할을 동시에 수행하는 경우도 있다.

1.5 테스팅의 필수 기술 및 모범 사례

기술(skill)이란 어떤 일을 잘 해내는 능력으로, 그 사람의 지식, 경험, 적성에서 비롯된다.

테스팅에 보편적으로 필요한 기술 (FL-1.5.1 K2)

  • 테스팅 지식 — 테스트 기법 활용 등을 통해 테스팅 효과를 높이기 위해
  • 철저함, 신중함, 호기심, 세부사항에 대한 주의력, 체계적 접근 — 찾기 어려운 결함 식별을 위해
  • 우수한 의사소통 기술, 경청, 팀플레이 — 이해관계자와 효과적 상호작용, 정보 전달, 결함 보고/논의를 위해
  • 분석적 사고, 비판적 사고, 창의성 — 테스팅 효과를 높이기 위해
  • 기술 지식 — 적절한 테스트 도구를 사용한 테스팅 효율성 향상
  • 도메인 지식 — 최종 사용자/비즈니스 담당자를 이해하고 소통하기 위해

테스터는 종종 좋지 않은 소식을 전해야 하는데, 이때 확증 편향(confirmation bias) 때문에 부정적 결과를 받아들이기 어려워하는 사람들이 있다. 따라서 결함 정보를 건설적인 방법으로 전달하는 것이 중요하다.

전체 팀 접근법 (FL-1.5.2 K1)

전체 팀 접근법(Whole Team Approach)은 익스트림 프로그래밍(XP)에서 시작된 개념으로, 필요한 지식과 기술을 갖춘 팀원이라면 누구나 어떤 작업이라도 수행할 수 있고, 모든 팀원이 품질에 대한 책임을 진다.

  • 같은 공간 사용(co-location)을 통해 의사소통과 상호작용 용이
  • 팀 활력 증대, 의사소통과 협업 강화, 다양한 기술 활용으로 시너지 창출
  • 테스터는 비즈니스 담당자와 협력해 적절한 인수 테스트를 작성하고, 개발자와 협력해 테스트 전략과 자동화 접근법을 결정
주의 정황에 따라 전체 팀 접근법이 적절하지 않을 수도 있다. 예를 들어, 안전이 치명적인(safety-critical) 경우에는 높은 수준의 테스트 독립성이 더 필요할 수 있다.

테스팅의 독립성 (FL-1.5.3 K2)

일정 수준의 독립성은 저자와 테스터의 인지 편향(cognitive biases) 차이로 인해 테스터가 결함을 더 효과적으로 식별할 수 있게 해준다.

독립성 수준 (낮음 → 높음):

  1. 저자가 직접 테스트 (독립성 없음)
  2. 저자의 같은 팀 동료가 테스트 (일정 수준의 독립성)
  3. 조직 내 다른 팀이 테스트 (높은 독립성)
  4. 외부 조직이 테스트 (매우 높은 독립성)
독립적 테스팅의 장점독립적 테스팅의 단점
저자와 다른 인지 편향을 가져 다른 유형의 결함/장애 식별 가능개발팀과 차단되어 협업과 의사소통이 어려울 수 있음
이해관계자의 가정을 검증하고 이의 제기/반증 가능개발팀과 적대적 관계로 이어질 수 있음
개발자가 품질에 대한 책임감을 잃을 수 있음
독립적 테스터가 병목이나 출시 지연의 원인으로 비난받을 수 있음

핵심 용어 정리

1장에서 다루는 핵심 용어이다. ISTQB 시험에서 정확한 정의 숙지가 합격의 핵심이다.

Coverage (커버리지)
테스트 대상에 대해 주어진 커버리지 항목이 테스트된 정도
Debugging (디버깅)
장애의 원인(결함)을 찾고, 분석하고, 제거하는 활동
Defect (결함)
작업 산출물에 존재하는 불완전한 부분으로 장애를 유발할 수 있는 상태
Error (오류)
사람이 저지르는 실수로 결함을 발생시키는 행위
Failure (장애)
시스템이 기대한 대로 동작하지 않는 현상
Quality (품질)
컴포넌트나 시스템이 정해진 요구사항과 사용자 기대를 충족하는 정도
Quality Assurance (품질 보증)
프로세스 구현과 개선에 초점을 맞춘 예방적 접근법
Root Cause (근본 원인)
문제 발생의 근본적인 이유. 분석을 통해 유사 결함 예방 가능
Test Basis (테스트 베이시스)
테스트 분석의 기초가 되는 산출물(요구사항, 설계 문서 등)
Test Case (테스트 케이스)
테스트 컨디션을 확인하기 위한 입력값, 실행 조건, 기대 결과의 집합
Test Condition (테스트 컨디션)
테스트 베이시스에서 식별된 테스트 가능한 항목
Testware (테스트웨어)
테스트 활동을 통해 생성되는 모든 작업 산출물
Traceability (추적성)
테스트 베이시스 요소와 테스트웨어 간의 연결 관계
Verification (베리피케이션)
명세/설계대로 올바르게 만들어졌는지 확인 (Are we building the product right?)
Validation (밸리데이션)
사용자 요구를 충족하는지 확인 (Are we building the right product?)
Test Analysis (테스트 분석)
테스트 베이시스를 분석해 테스트 컨디션을 식별하는 활동
Test Design (테스트 설계)
테스트 컨디션을 테스트 케이스로 정교화하는 활동
Test Planning (테스트 계획)
테스트 목적을 정의하고 달성 접근법을 선택하는 활동

연습 문제

ISTQB 공식 샘플 문제 중 Ch1과 관련된 문제 3선이다. 답을 먼저 생각해본 후 정답을 확인하자.

Q1 (FL-1.1.1 K1)
다음 설명 중 유효한 테스트 목적인 것은?
정답: c)
테스팅은 결함과 장애를 찾아서 리스크 수준을 줄이고 테스트 대상의 품질에 대한 자신감을 부여하는 것이 유효한 목적이다.
a)는 원리 1(결함 부재 증명 불가), b)는 원리 7(오류 부재의 궤변), d)는 원리 2(완벽한 테스팅 불가능)와 관련되어 잘못된 선택지이다.
Q2 (FL-1.3.1 K2)
당신은 새로운 시스템을 점진적으로 개발하고 있는 팀의 테스터이다. 여러 번의 반복주기가 진행되는 동안 리그레션 테스트 케이스가 바뀌지 않았고, 새로운 리그레션 결함도 식별되지 않았음을 확인했다. 팀의 관리자는 기뻐하고 있지만, 당신은 그렇지 못하다. 당신이 걱정하고 있는 것을 설명하는 테스팅의 원리는?
정답: a)
원리 5 "테스트 효과는 줄어든다(살충제 패러독스)"에 해당한다. 같은 테스트를 계속 반복하면 신규 결함을 식별하는 효과가 점점 줄어든다. 리그레션 테스트가 변경 없이 반복되면서 새 결함을 찾지 못하는 것은 이 원리의 전형적인 사례이다.
Q3 (FL-1.4.1 K2)
당신은 배달 주문 모바일 애플리케이션을 개발하는 팀에서 일하고 있다. 팀은 현재 반복주기에서 결제 기능을 구현하기로 결정했다. 다음 중 테스트 분석 활동은?
정답: b)
테스트 분석은 "무엇을 테스트할 것인가?"를 결정하는 활동이다. 테스트 컨디션을 정의하는 것이 이에 해당한다.
a)는 테스트 계획(노력 추정), c)는 테스트 설계(테스트 기법 사용), d)는 테스트 실행(결함 보고)에 해당한다.
