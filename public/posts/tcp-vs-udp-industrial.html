TCP와 UDP 기본 개념

네트워크 통신에서 데이터를 전송하는 방법은 크게 연결형(Connection-oriented)과 비연결형(Connectionless)으로 나뉜다. TCP(Transmission Control Protocol)는 연결형, UDP(User Datagram Protocol)는 비연결형 프로토콜의 대표이며, 둘 다 OSI 모델의 전송 계층(Layer 4)에서 동작한다.

TCP (Transmission Control Protocol)

데이터를 보내기 전에 먼저 3-Way Handshake로 연결을 수립한다. 모든 패킷에 순서 번호(Sequence Number)를 부여하고, 수신 측이 ACK(확인 응답)를 보내야 다음 데이터를 전송한다. 패킷 손실 시 자동으로 재전송하므로 데이터의 신뢰성과 순서가 완벽히 보장된다.

UDP (User Datagram Protocol)

연결 수립 없이 데이터그램을 즉시 전송한다. ACK도 없고, 순서 보장도 없으며, 손실된 패킷을 재전송하지 않는다. 그 대신 헤더가 작고(8바이트) 처리가 빠르다. 속도가 중요하고 약간의 데이터 손실이 허용되는 환경에 적합하다.

TCP 3-Way Handshake

그림 1. TCP 3-Way Handshake와 UDP 비연결형 전송 비교
TCP 헤더 vs UDP 헤더 TCP 헤더는 20~60바이트(옵션 포함)로, 순서 번호, ACK 번호, 윈도우 크기, 체크섬 등 다양한 필드를 포함한다. 반면 UDP 헤더는 8바이트(출발지 포트, 목적지 포트, 길이, 체크섬)로 매우 간결하다. 이 차이가 곧 오버헤드 차이로 이어진다.

왜 GigE Vision은 UDP를 쓰는가

머신비전 카메라에서 PC로 영상을 전송할 때, GigE Vision은 TCP가 아닌 UDP를 사용한다. 초당 수십~수백 프레임의 이미지를 실시간으로 전송해야 하는 환경에서 TCP의 재전송 메커니즘은 오히려 성능을 저하시키기 때문이다.

UDP를 선택한 이유

TCP를 쓰면 생기는 문제

  • 재전송 지연 — 패킷 하나가 손실되면 후속 패킷 전체가 대기(Head-of-Line Blocking)
  • 혼잡 제어 — 네트워크 혼잡 시 전송률을 자동으로 낮춤
  • 프레임 지연 — 실시간 영상에서 1프레임 지연은 검사 타이밍 오류로 이어짐
  • CPU 부하 — ACK 처리, 윈도우 관리 등 추가 연산

UDP로 얻는 이점

  • 최소 지연 — 연결 설정 없이 즉시 전송
  • 일정한 전송률 — 혼잡 제어 없이 카메라가 원하는 속도로 전송
  • 멀티캐스트 — 하나의 카메라 영상을 여러 PC에 동시 전송 가능
  • 낮은 CPU 부하 — 단순한 프로토콜 처리

GVSP 자체 재전송 메커니즘

UDP는 패킷 손실을 보장하지 않지만, GigE Vision은 이 문제를 애플리케이션 레벨에서 해결한다. GigE Vision Streaming Protocol(GVSP)은 각 패킷에 block_id와 packet_id를 부여하여 수신 측에서 손실을 감지하고, 필요한 패킷만 선택적으로 재전송을 요청한다.

그림 3. GVSP의 선택적 패킷 재전송 — TCP와 달리 손실된 패킷만 요청하여 효율적으로 복구
TCP의 재전송 vs GVSP의 재전송 TCP는 손실 패킷 이후의 모든 패킷을 중복 전송할 수 있고(Go-Back-N), 혼잡 윈도우를 줄여 전송률이 급락한다. 반면 GVSP는 정확히 손실된 패킷만 재전송하고, 나머지 전송은 영향을 받지 않는다. 이것이 영상 스트리밍에서 UDP + 애플리케이션 레벨 재전송이 더 효율적인 이유다.
Jumbo Frame과 패킷 손실 관계 GigE Vision에서 Jumbo Frame(9000 bytes)을 설정하면 하나의 이미지를 전송하는 데 필요한 패킷 수가 줄어들어 패킷 손실 확률이 낮아진다. 예를 들어 5MP 이미지의 경우 일반 패킷(1500 bytes)으로는 약 3,334개가 필요하지만, Jumbo Frame으로는 약 556개면 충분하다.

TCP가 적합한 경우

모든 데이터가 정확하게, 빠짐없이 전달되어야 하는 상황에서는 TCP가 필수다. 데이터 한 바이트라도 손실되면 시스템 오류로 이어질 수 있는 경우에 해당한다.

Modbus TCP PLC와 PC 간 제어 명령 교환. 레지스터 읽기/쓰기 명령이 손실되면 장비가 오동작할 수 있으므로 신뢰성이 필수. 포트 502 사용.
파일 전송 (FTP/SFTP) 검사 이미지 원본, 로그 파일, 레시피 파일 등을 서버에 저장. 파일이 깨지면 재검사나 이력 추적이 불가능.
검사 결과 전달 OK/NG 판정 결과를 MES(Manufacturing Execution System)에 전송. 결과가 유실되면 불량품이 후공정으로 넘어갈 수 있음.
데이터베이스 연동 검사 데이터를 DB에 INSERT할 때 트랜잭션 무결성이 필요. TCP 기반 DB 프로토콜(MySQL, MSSQL) 사용.
원격 장비 제어 카메라 파라미터 변경, 펌웨어 업데이트 등 제어 채널. GigE Vision도 제어(GVCP)에는 UDP를 쓰지만 ACK 메커니즘을 자체 구현.
Modbus TCP vs Modbus RTU Modbus RTU는 시리얼(RS-485) 기반으로 CRC 에러 체크만 하지만, Modbus TCP는 이더넷 위에서 TCP의 신뢰성을 활용한다. 같은 Modbus 프로토콜이지만 전송 계층이 다르며, TCP 버전이 더 빠르고 네트워크 확장이 용이하다.

UDP가 적합한 경우

실시간성이 데이터 완전성보다 중요하거나, 약간의 손실이 전체 시스템에 치명적이지 않은 경우 UDP가 적합하다. 특히 대용량 데이터를 고속으로 스트리밍해야 하는 상황에서 UDP의 낮은 오버헤드가 결정적 이점이 된다.

영상 스트리밍 GigE Vision(GVSP), IP 카메라(RTP/RTSP). 1프레임 손실보다 전체 스트림의 실시간성이 더 중요.
실시간 센서 데이터 진동 센서, 온도 모니터링 등 주기적 데이터. 이전 값보다 최신 값이 항상 더 중요.
멀티캐스트 전송 하나의 소스에서 여러 수신자에게 동시 전송. TCP는 각 수신자와 개별 연결이 필요하지만 UDP는 한 번에 가능.
장비 디스커버리 GigE Vision의 GVCP Discovery. 네트워크에 브로드캐스트로 카메라 탐색 패킷을 보내고, 카메라가 응답.
NTP 시간 동기화 다수 장비의 시간을 동기화할 때 UDP 포트 123 사용. 빠른 응답이 정확한 동기화의 핵심.
UDP 사용 시 반드시 고려할 점 UDP를 사용하면 애플리케이션에서 직접 패킷 손실 감지, 순서 재정렬, 필요 시 재전송 로직을 구현해야 한다. GigE Vision의 GVSP처럼 프로토콜 자체에 이 기능이 내장되어 있지 않다면, 데이터 무결성은 개발자의 몫이다.

산업 현장 사례

실제 머신비전 검사 시스템에서는 TCP와 UDP를 동시에 사용하는 것이 일반적이다. 각 통신 채널의 특성에 맞게 적절한 프로토콜을 선택하는 것이 핵심이다.

검사 시스템 TCP/UDP 혼용 구성

그림 4. 검사 시스템에서 TCP/UDP 혼용 — 영상은 UDP, 제어/결과/저장은 TCP

프로토콜 선택 요약

통신 채널 프로토콜 선택 이유
카메라 → PC (영상) UDP (GVSP) 고속 스트리밍, 낮은 지연, 자체 재전송
PC → 카메라 (제어) UDP (GVCP) 빠른 응답, 자체 ACK 메커니즘
PC ↔ PLC TCP (Modbus TCP) 제어 명령의 신뢰성 필수
PC → MES TCP (HTTP/DB) 검사 결과 데이터 무결성
PC → File Server TCP (SMB/FTP) 이미지 파일 완전성 보장
카메라 → 모니터링 UDP (멀티캐스트) 하나의 스트림으로 다수 수신
실무 핵심 원칙 "영상은 UDP, 제어와 결과는 TCP" — 이것이 산업용 통신에서 가장 기본적인 프로토콜 선택 원칙이다. 각각의 통신 채널이 요구하는 특성(실시간성 vs 신뢰성)에 맞는 프로토콜을 선택하고, 필요 시 애플리케이션 레벨에서 부족한 기능을 보완하는 것이 최선의 설계다.
