4.1 테스트 기법 개요

테스트 기법은 무엇을 테스트할지(테스트 분석)와 어떻게 테스트할지(테스트 설계)를 지원한다. 적은 수이지만 충분한 테스트 케이스를 체계적으로 개발하는 데 도움을 준다.

이 실러버스는 테스트 기법을 크게 세 가지로 분류한다.

테스트 기법 3가지 분류
분류 별칭 기반 특징
블랙박스 명세 기반 외부 동작/명세 내부 구조를 참조하지 않음. 구현이 바뀌어도 동작이 같으면 테스트 유효
화이트박스 구조 기반 내부 구조/코드 설계나 구현이 끝난 후에 테스트 생성 가능
경험 기반 - 테스터의 지식/경험 블랙박스/화이트박스를 보완, 효과는 테스터 능력에 의존
시험 포인트: 세 기법은 상호 보완적 경험 기반 기법은 블랙박스/화이트박스로 찾지 못하는 결함을 식별할 수 있다. 세 가지를 함께 사용하는 것이 효과적이다.

4.2 블랙박스 테스트 기법

블랙박스 테스트 기법은 테스트 대상의 명세된 동작을 분석해서 테스트 케이스를 도출한다. 소프트웨어 내부 구현에 의존하지 않으므로, 구현이 바뀌어도 요구되는 동작이 동일하면 테스트 케이스는 여전히 유효하다.

4.2.1 동등 분할 (Equivalence Partitioning)

핵심 원리: 하나의 분할에 속한 모든 값은 시스템이 동일한 방식으로 처리한다고 가정한다. 따라서 각 분할에서 하나의 값만 테스트하면 충분하다.

분할 종류:

  • 유효 분할: 시스템이 정상 처리해야 하는 값
  • 비유효 분할: 시스템이 거부하거나 무시해야 하는 값

분할은 서로 겹치지 않아야 하며, 빈 집합일 수 없다.

커버리지: 100% EP 커버리지 = 모든 분할(비유효 포함)을 최소 1회 실행. 커버리지(%) = 실행한 분할 수 / 전체 분할 수 x 100

예제: 나이 입력 필드 (유효 범위: 18~65)
분할 값 범위 유형 대표 테스트 값
EP1 ..., 16, 17 비유효 10
EP2 18, 19, ..., 64, 65 유효 30
EP3 66, 67, ... 비유효 80

100% EP 커버리지를 위한 최소 테스트 케이스 수: 3개 (각 분할에서 1개씩)
참고: 비유효 분할 각각을 별도 테스트 케이스로 만드는 것이 원칙 (결함 마스킹 방지)

이치 초이스 커버리지(Each Choice Coverage) 입력 변수가 여러 개일 때, 모든 분할 집합의 각 분할을 최소 1번 실행하는 것이다. 분할 간 조합은 고려하지 않는 가장 단순한 기준이다.

4.2.2 경계값 분석 (Boundary Value Analysis)

핵심 원리: 개발자가 분할의 경계에 있는 값을 다룰 때 오류를 범할 가능성이 높다. 경계값 분석은 이 경계에 초점을 두는 기법이다. 정렬된 분할에만 사용할 수 있다.

두 가지 유형:

  • 2-value BVA: 경계별 커버리지 항목 2개 (경계값 + 인접 분할의 가장 가까운 값)
  • 3-value BVA: 경계별 커버리지 항목 3개 (경계값 + 양쪽 이웃 값) - 더 엄격
예제: 나이 입력 필드 (유효 범위: 18~65) - BVA 적용

경계는 총 2개 지점: 17|18 구간과 65|66 구간

경계 2-value BVA 테스트 값 3-value BVA 테스트 값
17 | 18 17, 18 17, 18, 19
65 | 66 65, 66 64, 65, 66
합계 4개 테스트 값 6개 테스트 값

3-value BVA가 더 엄격하다. 예: if (x<=10)이 if (x=10)으로 잘못 구현된 경우, 2-value(10,11)로는 못 찾지만 3-value(9)로 찾을 수 있다.

BVA 커버리지 계산법 커버리지(%) = 실행한 경계값(+이웃값) 수 / 식별한 총 경계값(+이웃값) 수 x 100. 시험에서 자주 출제되므로 경계값 개수 세기 연습이 중요하다.

4.2.3 결정 테이블 테스팅 (Decision Table Testing)

핵심 원리: 서로 다른 조건 조합에 따라 달라지는 결과를 표 형태로 정리하고, 각 조합(결정 규칙)을 테스트한다. 비즈니스 규칙 같은 복잡한 논리를 기록하는 데 효과적이다.

구조:

  • 행: 조건들 + 결과 동작
  • 열: 각 결정 규칙 (고유한 조건 조합 + 연관 동작)
  • 표기: T(참), F(거짓), -(무관), N/A(실행 불가), X(동작 발생), 공백(동작 없음)

테이블 유형:

  • 제한-입력(Limited-entry): 모든 값을 부울(T/F)로 표시
  • 확장-입력(Extended-entry): 숫자 범위, 동등 분할 등 복수의 값 사용 가능
예제: 로그인 결과 결정 테이블

조건: 아이디 유효 여부, 비밀번호 유효 여부

R1 R2 R3 R4
조건: 아이디 유효 T T F F
조건: 비밀번호 유효 T F T F
결과: 로그인 성공 X
결과: 오류 메시지 표시 X X X

조건 2개 → 22 = 4개 규칙. 100% 커버리지를 위한 테스트 케이스: 4개 (실현 불가능한 조합 제외 후). 조건이 n개면 최대 2n개 규칙. 조건이 많으면 테이블 최소화나 리스크 기반 접근법을 사용한다.

결정 테이블의 강점 간과했을 수 있는 조건 조합까지 체계적으로 식별할 수 있고, 누락되거나 모순되는 요구사항을 찾는 데 도움이 된다.

4.2.4 상태 전이 테스팅 (State Transition Testing)

핵심 원리: 시스템의 가능한 상태와 유효한 전이를 모델링한 상태 다이어그램/상태 테이블을 기반으로 테스트 케이스를 도출한다. 전이는 이벤트에 의해 발생하며, 가드 조건이 있을 수 있다.

전이 표시 형식: "이벤트 [가드 조건] / 동작"

상태 테이블은 행=상태, 열=이벤트(+가드 조건)로 구성된다. 상태 다이어그램과 달리 비유효 전이를 빈 셀로 명확하게 보여준다.

세 가지 커버리지 기준 (약한 순서대로):

  • 모든 상태 커버리지: 모든 상태를 최소 1번 방문 (가장 약함)
  • 유효 전이 커버리지 (0-스위치): 모든 유효 전이를 최소 1번 실행 (가장 널리 사용)
  • 모든 전이 커버리지: 유효 + 비유효 전이 모두 실행 시도 (가장 강함, 미션 크리티컬 소프트웨어 최소 기준)

포함 관계: 모든 전이 100% ⊃ 유효 전이 100% ⊃ 모든 상태 100%

예제: 문서 상태 전이 모델

문서 관리 시스템의 문서 수명주기를 상태 전이로 모델링한다.

현재 상태 이벤트 다음 상태
초안(Draft) 리뷰 요청 검토 중(Review)
검토 중(Review) 승인 승인됨(Approved)
검토 중(Review) 반려 초안(Draft)
승인됨(Approved) 게시 게시됨(Published)
게시됨(Published) 폐기 폐기됨(Archived)

상태 5개, 유효 전이 5개
유효 전이 100% 커버리지를 위한 테스트 케이스 예시:
TC1: 초안 →[리뷰 요청]→ 검토 중 →[승인]→ 승인됨 →[게시]→ 게시됨 →[폐기]→ 폐기됨 (전이 4개 커버)
TC2: 초안 →[리뷰 요청]→ 검토 중 →[반려]→ 초안 (전이 1개 추가 커버)
최소 2개 테스트 케이스로 유효 전이 100% 달성 가능

비유효 전이 테스트 주의사항 모든 전이 커버리지에서는 비유효 전이(예: "게시됨" 상태에서 "리뷰 요청")도 시도해야 한다. 하나의 테스트 케이스에서 비유효 전이는 하나만 테스트하는 것이 원칙이다 (결함 마스킹 방지).

4.3 화이트박스 테스트 기법

화이트박스 기법은 소스 코드의 내부 구조를 기반으로 테스트를 설계한다. 이 실러버스에서는 두 가지를 다룬다.

4.3.1 구문 테스팅과 구문 커버리지 (Statement Coverage)

커버리지 항목: 실행 가능한 구문(statement)

목적: 코드의 모든 실행 가능한 구문을 적어도 한 번 실행하는 것

구문 커버리지(%) = 실행된 구문 수 / 실행 가능한 구문 총수 x 100

100% 구문 커버리지를 달성해도 모든 결정 논리를 테스트한 것은 아니다. 예를 들어, 코드의 모든 분기가 실행되지 않았을 수 있다.

예제: 구문 커버리지 계산
def calculate_discount(price, is_member): discount = 0 # 구문 1 if price > 100: # 분기점 discount = 10 # 구문 2 if is_member: # 분기점 discount = 20 # 구문 3 return price - discount # 구문 4

실행 가능한 구문: 4개
TC1: price=150, is_member=True → 구문 1,2,3,4 실행 → 구문 커버리지 100% (4/4)
하지만 이 테스트만으로는 price ≤ 100인 경우를 테스트하지 않았다 (분기 커버리지 미달성!)

4.3.2 분기 테스팅과 분기 커버리지 (Branch Coverage)

커버리지 항목: 분기(branch) - 제어 흐름 그래프에서 두 노드 간 제어의 이동

목적: 모든 분기(조건/무조건 분기 포함)를 적어도 한 번 실행하는 것

분기 커버리지(%) = 실행된 분기 수 / 분기 총수 x 100

핵심 관계: 분기 커버리지 ⊃ 구문 커버리지 100% 분기 커버리지를 달성하면 100% 구문 커버리지도 자동 달성된다. 그러나 그 반대는 성립하지 않는다. 위 예제에서 TC1 하나로 구문 100%를 달성했지만, price ≤ 100인 분기(거짓 경로)를 실행하지 않았으므로 분기 커버리지는 100%가 아니다.
예제: 분기 커버리지 달성
def calculate_discount(price, is_member): discount = 0 if price > 100: # 분기: T경로(구문2로) / F경로(구문4로) discount = 10 if is_member: # 분기: T경로(구문3으로) / F경로(구문4로) discount = 20 return price - discount

분기 4개: if(price>100)의 T/F, if(is_member)의 T/F

TC1: price=150, is_member=True → 첫 번째 if T, 두 번째 if T (분기 2개)
TC2: price=50, is_member=False → 첫 번째 if F (분기 1개, 두 번째 if 진입 안 함)
TC3: price=150, is_member=False → 첫 번째 if T, 두 번째 if F (분기 1개 추가)

TC1 + TC2 + TC3 → 분기 4/4 = 분기 커버리지 100%
(참고: TC1 + TC2만으로도 구문 커버리지 100%이지만, 분기 커버리지는 75%)

4.3.3 화이트박스 테스팅의 가치

강점:

  • 전체 소프트웨어 구현을 고려하므로, 명세가 모호하거나 불완전해도 결함 감지 가능
  • 객관적인 커버리지 측정값을 제공하여 추가 테스트 생성 정보를 함께 제공
  • 정적 테스팅(코드 드라이 런, 슈도 코드 검토)에도 사용 가능

약점:

  • 요구사항이 구현되지 않은 경우(누락), 화이트박스 테스팅으로는 그 누락을 식별하지 못할 수 있다
시험 빈출: 화이트박스의 약점 화이트박스 테스팅은 "구현된 것"을 기반으로 하므로, 구현 자체가 누락된 요구사항은 찾을 수 없다. 블랙박스 테스팅으로만 요구사항 대비 동작을 확인할 수 있다.

4.4 경험 기반 테스트 기법

테스터의 지식과 경험을 활용하는 기법이다. 효과는 테스터의 능력에 크게 의존하지만, 블랙박스/화이트박스로 놓칠 수 있는 결함을 발견할 수 있다.

4.4.1 오류 추정 (Error Guessing)

테스터의 지식을 바탕으로 오류, 결함, 장애 발생을 예측하는 기법이다. 다음과 같은 지식이 활용된다.

  • 애플리케이션의 과거 동작
  • 개발자가 범하기 쉬운 오류 유형
  • 유사 애플리케이션에서 발생한 장애 유형

일반적인 오류 영역: 입력(매개변수 누락/오류), 출력(잘못된 형식/결과), 논리(사례 누락, 잘못된 연산자), 계산, 인터페이스, 데이터

결함 공격(Fault Attack): 오류 추정의 구현 방법. 발생 가능한 오류/결함/장애 목록을 작성하여 이를 식별하는 테스트를 설계한다.

4.4.2 탐색적 테스팅 (Exploratory Testing)

테스트의 설계, 실행, 평가를 동시에 수행하는 기법이다. 테스터가 테스트 대상에 대해 배워가면서 점점 깊이 탐색한다.

세션 기반 접근법:

  • 시간을 정해 놓고(타임박스) 수행
  • 테스트 차터(Charter): 테스트 목적이 정의된 지침서 역할
  • 세션 후 보고 및 이해관계자 토론
  • 테스트 세션 시트로 절차와 발견 내용 기록

유용한 상황:

  • 명세가 부족하거나 부적합한 경우
  • 시간 압박이 심할 때
  • 다른 공식 기법을 보완할 때
탐색적 테스팅에 필요한 역량 풍부한 경험, 도메인 지식, 분석 기술, 호기심, 창의성이 있을수록 효과적이다. 탐색적 테스팅 중에도 동등 분할 등 다른 기법을 함께 사용할 수 있다.

4.4.3 체크리스트 기반 테스팅 (Checklist-based Testing)

체크리스트를 활용해 테스트 컨디션을 확인하는 기법이다. 체크리스트는 경험, 사용자 관심사, 소프트웨어 실패 패턴을 바탕으로 작성한다.

체크리스트 작성 원칙:

  • 각 항목은 개별적이면서 직접 확인 가능해야 한다
  • 자동 점검 가능한 항목, 너무 일반적인 항목은 제외
  • 질문 형식으로 표현하는 경우가 많다
  • 정기적으로 업데이트해야 한다 (결함 분석 기반)

체크리스트가 상위 수준이면 커버리지는 높아지지만 재현 가능성은 떨어질 수 있다.

4.5 협업 기반 테스트 접근법

앞의 기법들이 결함 식별에 초점을 두는 반면, 협업 기반 접근법은 협업과 커뮤니케이션을 통한 결함 예방에도 초점을 둔다.

4.5.1 협업 기반 사용자 스토리 작성

사용자 스토리의 3C:

  • 카드(Card): 사용자 스토리를 설명하는 매체
  • 대화(Conversation): 소프트웨어 사용 방법에 대한 설명
  • 확인(Confirmation): 인수 조건

형식: "[역할]로서 [목표]를 달성해 [비즈니스 가치]를 얻기를 원한다."

INVEST 원칙 - 좋은 사용자 스토리의 조건:

  • Independent (독립적)
  • Negotiable (협상 가능)
  • Valuable (가치 있는)
  • Estimable (추정 가능)
  • Small (작은)
  • Testable (테스트 가능)
테스트 가능(Testable)이 핵심 이해관계자가 사용자 스토리를 테스트하는 방법을 모른다면, 그 스토리가 명확하지 않거나 사용자에게 중요한 내용이 반영되지 않았다는 신호이다.

4.5.2 인수 조건 (Acceptance Criteria)

이해관계자가 사용자 스토리 구현 결과를 승인하기 위해 충족해야 하는 조건이다. 인수 조건은 테스트 컨디션으로 볼 수 있다.

인수 조건 작성법 두 가지:

  • 시나리오 기반: Given/When/Then 형식 (BDD에서 사용)
  • 규칙 기반: 입력-출력 매핑을 목록이나 표로 표현

인수 조건의 용도: 범위 정의, 합의 도출, 긍정/부정 시나리오 설명, 인수 테스팅 베이시스 제공, 정확한 계획 및 추정

4.5.3 인수 테스트 주도 개발 (ATDD)

핵심: 테스트 우선 접근법. 사용자 스토리 구현 전에 테스트 케이스를 만든다. 고객, 개발자, 테스터 등 서로 다른 관점을 가진 팀원들이 함께 참여한다.

ATDD 프로세스:

  1. 명세 워크숍: 사용자 스토리와 인수 조건을 분석, 토론, 작성 (불완전성, 모호성, 결함 해결)
  2. 테스트 케이스 작성: 인수 조건 기반, 팀 전체 또는 테스터 개별 수행
  3. 구현 및 자동화: 테스트 자동화 프레임워크 형식으로 작성하면 인수 테스트가 실행 가능한 요구사항이 됨

테스트 케이스 작성 순서:

  1. 긍정/유효 테스트: 정상 동작 확인 (먼저 작성)
  2. 비유효/부정 테스트: 예외/오류 상황
  3. 비기능 테스트: 성능, 사용성 등
예제: ATDD - Given/When/Then 형식

사용자 스토리: "쇼핑몰 회원으로서, 장바구니에 담긴 상품의 총 금액이 5만원 이상이면 무료 배송을 받아 배송비를 절약하고 싶다."

# 긍정 테스트 케이스 Given 회원이 로그인한 상태이고 And 장바구니에 총 금액이 55,000원인 상품이 담겨 있을 때 When 결제 페이지로 이동하면 Then 배송비가 0원으로 표시된다 # 부정 테스트 케이스 Given 회원이 로그인한 상태이고 And 장바구니에 총 금액이 45,000원인 상품이 담겨 있을 때 When 결제 페이지로 이동하면 Then 배송비 3,000원이 추가된다 And "5만원 이상 구매 시 무료 배송" 안내 메시지가 표시된다
ATDD 테스트 작성 원칙 테스트 케이스는 사용자 스토리의 모든 특성을 다뤄야 하지만 스토리를 벗어나면 안 된다. 또한, 두 개 이상의 테스트 케이스가 같은 특성을 중복 설명해서는 안 된다.

핵심 용어 정리

Ch4에서 반드시 알아야 하는 핵심 용어들이다.

Equivalence Partitioning
동등 분할 - 동일하게 처리되는 값의 그룹으로 나누어 각 그룹에서 하나씩 테스트
Boundary Value Analysis
경계값 분석 - 분할 경계의 값에 초점을 두는 기법 (2-value, 3-value)
Decision Table Testing
결정 테이블 테스팅 - 조건 조합에 따른 결과를 표로 정리하여 테스트
State Transition Testing
상태 전이 테스팅 - 상태와 전이를 모델링하여 테스트 케이스 도출
Statement Coverage
구문 커버리지 - 실행 가능한 모든 구문을 최소 1번 실행했는지 측정
Branch Coverage
분기 커버리지 - 모든 분기(T/F)를 최소 1번 실행했는지 측정. 구문 커버리지를 포함
Coverage Item
커버리지 항목 - 테스트 기법에서 커버리지 측정의 기본 단위
Error Guessing
오류 추정 - 테스터의 경험으로 오류 발생을 예측하는 기법
Exploratory Testing
탐색적 테스팅 - 설계/실행/평가를 동시에 수행. 차터와 세션 기반
Checklist-based Testing
체크리스트 기반 테스팅 - 체크리스트로 테스트 항목을 확인하는 기법
ATDD
인수 테스트 주도 개발 - 구현 전 인수 조건 기반 테스트를 먼저 작성하는 접근법
Acceptance Criteria
인수 조건 - 이해관계자가 스토리 구현을 승인하기 위한 조건 (Given/When/Then 또는 규칙 기반)

연습 문제

실제 시험과 유사한 형태의 연습 문제이다. 직접 풀어본 후 정답을 확인하자.

1 학생의 최종 성적 시스템을 테스트한다. 규칙: 0~50점 탈락, 51~60점 보통, 61~70점 만족, 71~80점 좋음, 81~90점 매우 좋음, 91~100점 우수. 다음 테스트 케이스 세트의 2-value BVA 커버리지는? TC1: 91(우수), TC2: 50(탈락), TC3: 81(매우 좋음), TC4: 60(보통), TC5: 70(만족), TC6: 80(좋음)
정답: a. 50%
2-value BVA에서 경계값과 인접값이 커버리지 항목이다. 경계값은 0, 1, 50, 51, 60, 61, 70, 71, 80, 81, 90, 91 총 12개이다. TC가 커버하는 값은 50, 60, 70, 80, 81, 91로 6개이므로 6/12 = 50%.
2 100% 구문 커버리지를 달성한 테스트 스위트에 대한 설명으로 올바른 것은?
정답: a
100% 구문 커버리지란 모든 실행 가능한 구문을 적어도 한 번 실행했다는 뜻이다. 결함이 있는 구문도 포함되므로 결함 존재를 식별할 장애가 나타났어야 한다(단, 데이터 종속 결함은 놓칠 수 있음). b)는 틀림 - 테스트 케이스 수가 아닌 어떤 경로를 실행하느냐가 중요. c)는 경로 커버리지의 개념. d)는 완벽한 테스팅의 개념.
3 다음 중 화이트박스 테스팅에 대한 설명으로 옳지 않은 것은?
정답: d
화이트박스 테스팅의 약점은 구현 누락(요구사항이 구현되지 않은 경우)을 식별하지 못한다는 점이다. a), b), c)는 모두 화이트박스 테스팅의 올바른 특징이다. 요구사항과의 차이를 식별하는 것은 블랙박스(명세 기반) 테스팅의 역할이다.
4 프로젝트에서 출시가 지연되고 테스트가 늦게 시작되었다. 요구사항 전체 목록이 아직 공유되지 않았지만, 당신은 상세한 도메인 지식과 우수한 분석 기술을 보유하고 있다. 가장 적합한 테스트 기법은?
정답: c. 탐색적 테스팅
탐색적 테스팅은 명세가 부족하거나 시간 압박이 심할 때 유용하다. 테스터가 풍부한 도메인 지식과 분석 기술을 가지고 있을 때 더욱 효과적이다. 설계, 실행, 평가를 동시에 수행하므로 별도 명세 없이도 바로 테스트를 시작할 수 있다.
