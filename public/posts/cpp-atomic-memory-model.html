왜 atomic인가

두 스레드가 같은 변수 counter를 동시에 증가시키면 어떤 일이 벌어질까. 단순한 counter++도 사실은 읽기 → 더하기 → 쓰기의 세 단계로 쪼개진다. 두 스레드가 같은 시점에 같은 값을 읽고 똑같이 +1 하면 결과는 +2가 아니라 +1이 된다. 이것이 데이터 레이스(data race)다.

int counter = 0;

// 스레드 A, B가 동시에 실행
void Worker()
{
    for (int i = 0; i < 100000; ++i)
        counter++;  // read-modify-write, 데이터 레이스
}
// 기대값: 200000, 실제 결과: 매번 다름

해결책은 두 가지다. 첫째는 std::mutex로 감싸는 것이고, 둘째는 std::atomic을 쓰는 것이다. 단일 변수에 대한 단순 연산이라면 mutex는 과한 비용이다. 일반적으로 mutex 락/언락은 수십 ns에서 수백 ns가 들지만, atomic은 하드웨어 명령 한두 개로 끝나 수 ns 수준이다.

방식비용적용 범위
std::mutex수십~수백 ns여러 변수, 복잡한 임계 영역
std::atomic수 ns단일 변수의 원자적 연산
아무것도 안 함0 nsUB (정의되지 않은 동작)

std::atomic 기본 사용법

<atomic> 헤더에 정의되어 있고 임의의 trivially-copyable 타입을 감쌀 수 있다. 정수/포인터 같은 기본 타입은 하드웨어 atomic 명령으로 구현된다.

#include <atomic>

std::atomic<int> counter{0};

counter.fetch_add(1);             // +1, 이전 값 반환
counter.fetch_sub(1);             // -1
counter.load();                   // 읽기
counter.store(42);                // 쓰기
counter.exchange(100);            // 교체, 이전 값 반환

int expected = 100;
counter.compare_exchange_weak(expected, 200);  // CAS

// 연산자 오버로드도 가능
++counter;
counter += 1;
int x = counter;  // 암묵적 load

lock-free 여부 확인

모든 std::atomic<T>가 진짜 락 없이 구현되는 것은 아니다. 큰 구조체를 감싸면 내부에서 mutex를 쓸 수도 있다.

std::atomic<int> a;
std::atomic<LargeStruct> b;

a.is_lock_free();                                  // 런타임 체크 (보통 true)
std::atomic<int>::is_always_lock_free;             // 컴파일타임 상수 (true/false)
std::atomic<LargeStruct>::is_always_lock_free;     // 보통 false
is_always_lock_free를 우선 확인하자 static_assert(std::atomic<T>::is_always_lock_free)로 컴파일 시점에 보장할 수 있다. lock-free가 아니면 atomic의 의미가 퇴색된다.

데이터 레이스와 메모리 모델

C++11부터 표준은 메모리 모델을 명시적으로 정의한다. 핵심 두 관계만 이해하면 된다.

happens-before

연산 A가 연산 B보다 happens-before 관계라면, A의 효과가 B에서 반드시 보인다. 같은 스레드 안에서 코드 순서는 그대로 happens-before다(프로그램 순서). 스레드 사이에서는 동기화 지점을 거쳐야만 happens-before가 성립한다.

synchronizes-with

한 스레드의 release 연산이 다른 스레드의 acquire 연산과 짝을 이루면 synchronizes-with 관계가 성립한다. 이 지점을 통해 스레드 간 happens-before가 전파된다. mutex의 unlock-lock도, atomic의 release-acquire도 모두 이 메커니즘이다.

데이터 레이스는 곧 UB다 — 예외 없음 한 스레드가 비-atomic 변수에 쓰는 동안 다른 스레드가 그 변수를 읽거나 쓰면 Undefined Behavior다. "운 좋게 되는 경우"란 없다. 컴파일러는 데이터 레이스가 없다고 가정하고 최적화하므로, 결과가 어떻게 망가질지 예측할 수 없다.

memory_order 6종류

모든 atomic 연산은 두 번째 인자로 std::memory_order를 받는다. 생략하면 가장 강한 seq_cst가 기본값이다.

memory_order의미비용
seq_cst모든 스레드가 동일한 전역 순서 관찰가장 비쌈
acquire이 load 이후 코드가 위로 올라가지 못함중간
release이 store 이전 코드가 아래로 내려가지 못함중간
acq_relRMW 연산용 — acquire + release중간
consume데이터 의존성 기반 acquire (사실상 폐기)이론상 저렴
relaxed원자성만 보장, 순서 제약 없음가장 저렴
memory_order_consume은 쓰지 말자 의도와 달리 모든 주류 컴파일러가 내부적으로 acquire로 처리한다. 표준위원회도 재정의를 논의 중이다. 명시적으로 acquire를 사용하자.

acquire-release 페어 동기화

이 패턴이 atomic 학습의 절반이다. producer가 데이터를 준비한 뒤 플래그를 release로 켜면, 그 플래그를 acquire로 본 consumer는 producer가 플래그 이전에 한 모든 쓰기를 볼 수 있다.

std::atomic<bool> ready{false};
int data = 0;  // 비-atomic이지만 안전

// Producer 스레드
void Produce()
{
    data = 42;                                       // (1)
    ready.store(true, std::memory_order_release);    // (2)
}

// Consumer 스레드
void Consume()
{
    while (!ready.load(std::memory_order_acquire))   // (3)
        ;
    std::cout << data;                              // (4) 반드시 42
}
Producer Consumer data = 42 (1) while (!ready.load(acquire)) (3) ready.store(release) (2) ~~~~~> read data (4) ← 42 보장 관계: (1) happens-before (2) synchronizes-with (3) happens-before (4)

핵심은 (2)의 release가 (3)의 acquire와 페어를 이뤄 (1)의 결과가 (4)까지 전파되는 것이다. 이 다리가 없으면 (4)에서 data가 0으로 보일 수도 있다.

release는 "위쪽 쓰기를 모두 발행", acquire는 "아래쪽 읽기를 모두 수령" 외향(outgoing)/내향(incoming) 데이터의 봉인 같은 비유로 기억하면 편하다. release 위의 쓰기들이 봉인되어 흘러가고, acquire 아래에서 풀린다.

Compare-And-Swap (CAS)

CAS는 lock-free 알고리즘의 가장 중요한 빌딩 블록이다. "값이 expected와 같으면 desired로 바꾸고, 아니면 expected에 현재 값을 채워준다"는 한 번의 원자 연산이다.

std::atomic<int> value{0};

int expected = 0;
bool ok = value.compare_exchange_weak(expected, 10);
// value == 0 이었으면: value=10, expected 그대로 0, ok=true
// value != 0 이었으면: value 그대로, expected에 현재값 들어감, ok=false

CAS 루프 — 사용자 정의 RMW

CAS는 보통 루프로 쓴다. fetch_add처럼 단순 연산이 아닌, 복잡한 갱신 로직을 원자적으로 수행할 때 쓴다.

// 현재 값을 읽어 두 배로 만드는 원자 연산
std::atomic<int> v{1};

int cur = v.load(std::memory_order_relaxed);
while (!v.compare_exchange_weak(
           cur, cur * 2,
           std::memory_order_acq_rel,
           std::memory_order_relaxed))
{
    // 실패 시 cur에 현재 값이 자동으로 들어와 있음
}

weak vs strong

버전특징권장 사용처
compare_exchange_weakspurious failure 허용 (값이 같아도 실패할 수 있음)CAS 루프 — 어차피 다시 돔
compare_exchange_strongspurious failure 없음단발성 CAS — 한 번 성공/실패 판단

lock-free 자료구조 맛보기

lock-free 카운터

다른 스레드와 데이터 동기화가 필요 없는 단순 카운터라면 relaxed로 충분하다. 누적된 총합만 정확하면 된다.

std::atomic<uint64_t> hitCount{0};

void OnRequest()
{
    hitCount.fetch_add(1, std::memory_order_relaxed);
}

uint64_t Read()
{
    return hitCount.load(std::memory_order_relaxed);
}

lock-free 플래그 (스피너)

한 스레드가 다른 스레드에 "준비됨" 신호를 보내려면 release/acquire가 필요하다.

std::atomic_flag started = ATOMIC_FLAG_INIT;

// 시작 알림
started.test_and_set(std::memory_order_release);

// 시작 대기
while (!started.test(std::memory_order_acquire))
    ;
본격적인 lock-free 큐/스택은 매우 어렵다 ABA 문제, 메모리 회수(hazard pointer, RCU), false sharing 등 복합적인 문제가 얽힌다. 검증된 라이브러리(boost::lockfree, moodycamel::ConcurrentQueue) 사용을 권장한다. 직접 구현은 충분한 학습과 검증 후에 시도하자.

memory fence

std::atomic_thread_fence는 변수에 묶이지 않은 순수 순서 제약이다. 여러 비-atomic 쓰기를 한꺼번에 게이트로 감싸고 싶을 때 사용한다.

int a = 0, b = 0;
std::atomic<bool> flag{false};

// Producer
a = 1;
b = 2;
std::atomic_thread_fence(std::memory_order_release);
flag.store(true, std::memory_order_relaxed);

// Consumer
while (!flag.load(std::memory_order_relaxed))
    ;
std::atomic_thread_fence(std::memory_order_acquire);
// 이 시점에 a=1, b=2 확정

실무에서는 atomic 연산 자체에 memory_order를 붙이는 편이 더 명확하다. fence는 라이브러리 내부 구현이나 특수 패턴에서만 등장한다.

언제 어떤 memory_order를 쓸까

상황권장 order비고
잘 모르겠음seq_cst기본값, 가장 안전
단순 카운터/통계relaxed합계만 정확하면 됨
producer 데이터 준비 → consumer 확인release / acquire가장 흔한 패턴
CAS 루프 (lock-free RMW)acq_rel / relaxed (실패)성공 시 acq_rel, 실패 시 relaxed
참조 카운트 감소acq_rel 또는 release + fenceshared_ptr 패턴
의심스러우면 seq_cst Herb Sutter의 조언이다. relaxed로 성능을 짜내는 건 프로파일링 후에 해도 늦지 않다. 잘못된 relaxed는 디버깅이 거의 불가능하다.

흔한 함정

큰 구조체의 atomic은 lock-free가 아니다 std::atomic<LargeStruct>는 내부적으로 mutex를 쓸 수 있다. is_always_lock_free로 반드시 확인하자.
relaxed 남용은 데이터 손실을 부른다 producer가 데이터를 쓰고 플래그를 relaxed로 store하면, consumer가 플래그를 본 뒤에도 데이터가 안 보일 수 있다. 동기화 의도가 있다면 반드시 release/acquire를 쓰자.
volatile ≠ atomic volatile은 컴파일러 최적화를 막을 뿐이다. 멀티스레드 동기화 보장은 전혀 없다. 메모리 매핑 I/O용이지 스레드용이 아니다.
shared_ptr의 참조 카운트는 atomic, 포인터 자체는 atomic이 아니다 여러 스레드가 같은 shared_ptr 객체에 동시에 reset/대입하면 데이터 레이스다. 각자 복사본을 가지면 안전하지만, 공유된 한 인스턴스를 동시에 수정하는 건 위험하다. C++20 이전에는 std::atomic_load(&sp) 자유함수를, C++20부터는 std::atomic<std::shared_ptr<T>>를 쓰자.

C++20 추가 기능

std::atomic_ref

일반 변수에 임시로 atomic 인터페이스를 부여한다. 기존 자료구조를 바꾸지 않고 일부 시점에만 atomic 접근이 필요할 때 유용하다.

int rawCounter = 0;

void Worker()
{
    std::atomic_ref<int> ref(rawCounter);
    ref.fetch_add(1, std::memory_order_relaxed);
}
atomic_ref가 가리키는 동안엔 비-atomic 접근 금지 같은 객체를 한쪽에선 raw로, 한쪽에선 atomic_ref로 접근하면 데이터 레이스다. 모든 동시 접근이 atomic_ref를 거쳐야 한다.

std::atomic<std::shared_ptr<T>>

표준이 atomic shared_ptr을 직접 지원한다. 내부적으로 split reference count 같은 기법을 쓰며, lock-free일 수도 있고 아닐 수도 있다.

wait/notify

atomic도 condition variable처럼 동작한다. 내부적으로 OS의 futex 같은 메커니즘을 사용하므로 스피닝보다 효율적이다.

std::atomic<bool> ready{false};

// 대기 스레드
ready.wait(false);  // ready != false가 될 때까지 블록

// 알림 스레드
ready.store(true);
ready.notify_one();

하드웨어 관점

같은 memory_order 코드라도 CPU 아키텍처에 따라 생성되는 명령어가 완전히 다르다.

아키텍처메모리 모델acquire/release 비용
x86 / x64강함 (TSO)relaxed와 거의 같음 (대부분 무료)
ARM / AArch64약함실제로 메모리 배리어 명령 추가
PowerPC매우 약함더 무거운 배리어 필요
x86에서 잘 돌아도 ARM에서 깨질 수 있다 x86은 release/acquire 시맨틱을 거의 공짜로 제공하므로 잘못된 relaxed도 운 좋게 돌아간다. 같은 코드를 ARM 서버나 Apple Silicon에 옮기면 엉뚱하게 동작한다. 코드는 표준이 보장하는 메모리 모델 기준으로 작성하자. 실제 배리어가 박힌 ARM에서도 정확해야 한다.

정리하면, atomic은 단순한 "락 없는 변수"가 아니라 메모리 모델 위에서 동작하는 동기화 프리미티브다. 단일 변수 카운터로 시작해서 release/acquire 페어로 데이터 전달, CAS 루프로 lock-free RMW까지 단계적으로 익히면 된다. 그 너머의 본격 lock-free 자료구조는 다음 글에서 다룰 주제다.
