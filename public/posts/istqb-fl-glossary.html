1. 용어 사전 사용법

이 용어 사전은 ISTQB CTFL v4.0.1 (한국어판 v1.1.1) 실러버스에 수록된 모든 공식 용어를 챕터별로 정리한 것이다. 시험 준비에 바로 활용할 수 있도록 각 용어에 K-레벨, 정의, 소속 섹션을 표기했다.

K-레벨이란?

ISTQB는 학습 목표를 K1~K3의 인지 수준으로 분류한다. 용어에 표시된 K-레벨은 해당 용어가 가장 높은 수준으로 요구되는 학습 목표의 레벨이다.

K1
기억(Remember)
용어와 개념을 인식/상기
K2
이해(Understand)
개념을 설명/구별/비교
K3
적용(Apply)
기법을 실제로 적용/도출
활용 팁 K3 용어는 시험에서 계산/도출 문제로 출제될 가능성이 높다. K1 용어도 선택지에서 혼동을 유도하는 데 자주 사용되므로 정확한 정의를 반드시 암기하자. 주황 왼쪽 테두리가 있는 카드는 샘플 시험에 자주 등장하는 핵심 용어이다.

2. Ch1. 테스팅의 기초 용어 29개

커버리지Coverage K2
테스트 스위트가 커버리지 항목을 실행한 정도. 백분율로 표시하며, 테스트 완전성을 측정하는 핵심 지표이다.
1.1 / 1.4.4
디버깅Debugging K2
장애의 원인(결함)을 찾고, 분석하고, 제거하는 활동. 테스팅과 디버깅은 별개의 활동이다.
1.1.2
결함Defect K2
작업 산출물에 존재하는 불완전함 또는 결점. 결점, 버그라고도 한다. 오류(실수)에 의해 발생하며, 장애를 유발할 수 있다.
1.2.3
오류Error K2
부정확한 결과를 만드는 인간의 행위(실수). 오류는 결함을 만들어 내며, 시간 압박, 복잡성, 피로 등 다양한 원인에 의해 발생한다.
1.2.3
장애Failure K2
컴포넌트나 시스템이 기대 결과와 다르게 동작하는 사건. 결함에 의해 유발되지만, 환경 조건 등 결함 없이도 발생 가능하다.
1.2.3
품질Quality K2
컴포넌트나 시스템이 명시된/묵시된 요구사항을 충족시키는 정도.
1.2
품질 보증Quality Assurance (QA) K2
프로세스의 구현과 개선에 초점을 맞춘 프로세스 중심의 예방 접근법. 테스팅(교정 접근법)과 구분해야 한다.
1.2.2
근본 원인Root Cause K2
문제 발생의 근본적인 이유. 근본 원인 분석을 통해 찾으며, 이를 처리하면 유사한 결함과 장애를 예방할 수 있다.
1.2.3
테스트 분석Test Analysis K2
테스트 베이시스를 분석하여 테스트 가능한 기능과 테스트 컨디션을 식별하는 활동. "무엇을 테스트할 것인가?"를 결정한다.
1.4.1
테스트 베이시스Test Basis K2
테스트 분석과 설계의 기반이 되는 지식의 원천. 요구사항 명세, 설계 문서, 코드 등이 해당한다.
1.4.1 / 1.4.4
테스트 케이스Test Case K2
사전 조건, 입력, 기대 결과를 포함하는 테스트 조건의 집합. 테스트 설계 단계에서 도출된다.
1.4.1
테스트 완료Test Completion K2
프로젝트 마일스톤에서 수행되는 활동. 테스트웨어 보관, 테스트 환경 종료, 교훈 도출, 완료 보고서 작성 등을 포함한다.
1.4.1
테스트 컨디션Test Condition K2
테스트 베이시스에서 식별한 테스트 가능한 측면. 테스트 분석 단계에서 정의되며 우선순위가 매겨진다.
1.4.1
테스트 제어Test Control K2
테스트 목적을 달성하기 위해 필요한 수정 조치를 취하는 활동. 테스트 모니터링 정보를 기반으로 한다.
1.4.1 / 5.3
테스트 데이터Test Data K2
테스트 실행에 필요한 데이터. 테스트 구현 단계에서 만들어지거나 획득된다.
1.4.1
테스트 설계Test Design K2
테스트 컨디션을 테스트 케이스와 기타 테스트웨어로 구체화하는 활동. "어떻게 테스트할 것인가?"를 결정한다.
1.4.1
테스트 실행Test Execution K2
테스트 실행 일정에 따라 테스트를 수행하는 활동. 실제 결과를 기대 결과와 비교하고, 이상 사항을 분석한다.
1.4.1
테스트 구현Test Implementation K2
테스트 실행에 필요한 테스트웨어(데이터, 스크립트, 환경 등)를 생성하거나 획득하는 활동.
1.4.1
테스트 모니터링Test Monitoring K2
지속적으로 모든 테스트 활동을 점검하고 실제 진행 상황을 계획과 비교하는 활동.
1.4.1 / 5.3
테스트 대상Test Object K2
테스트의 대상이 되는 작업 산출물. 소프트웨어 코드, 요구사항 명세, 사용자 가이드 등이 될 수 있다.
1.1
테스트 목적Test Objective K1
테스팅을 수행하는 이유. 결함 식별, 커버리지 달성, 품질 평가, 밸리데이션 등이 있다.
1.1.1
테스트 계획Test Planning K2
테스트 목적을 정의하고, 제약 조건 내에서 목적을 가장 잘 달성할 수 있는 접근법을 선택하는 과정.
1.4.1 / 5.1
테스트 절차Test Procedure K2
테스트 케이스의 실행 순서. 테스트 스위트로 조합되며, 실행 일정으로 정리된다.
1.4.1
테스트 결과Test Result K2
테스트 실행의 산출물. 기대 결과와 실제 결과를 비교하여 합격/불합격을 판정한다.
1.4.1
테스팅Testing K2
소프트웨어 결함을 식별하고 산출물의 품질을 평가하는 일련의 활동. 계획, 분석, 설계, 구현, 실행, 완료를 포함한다.
1.1
테스트웨어Testware K2
테스트 활동을 지원하기 위해 생성된 작업 산출물. 테스트 케이스, 스크립트, 데이터, 환경 등을 포함한다.
1.4
추적성Traceability K2
테스트 베이시스, 테스트웨어, 테스트 결과, 결함 간의 연결 관계. 커버리지 평가를 지원하며 유지하는 것이 중요하다.
1.4.4
밸리데이션Validation K2
시스템이 운영 환경에서 사용자/이해관계자의 요구를 만족하는지 확인하는 활동. "올바른 제품을 만들고 있는가?"
1.1
베리피케이션Verification K2
시스템이 주어진 요구사항을 충족하는지 확인하는 활동. "제품을 올바르게 만들고 있는가?"

3. Ch2. SDLC와 테스팅 용어 17개

인수 테스팅Acceptance Testing K2
시스템이 사용자의 비즈니스 요구를 충족하는지 밸리데이션하는 테스트 레벨. UAT, 운영 인수, 규제 인수, 알파/베타 테스팅 등이 있다.
2.2.1
블랙박스 테스팅Black-box Testing K2
테스트 대상의 내부 구조를 참조하지 않고 명세를 기반으로 테스트를 도출하는 테스팅. 명세와 비교하여 동작을 확인한다.
2.2.2
컴포넌트 통합 테스팅Component Integration Testing K2
컴포넌트 간 인터페이스와 상호작용을 테스트하는 레벨. 단위 통합 테스팅이라고도 하며, 상향식/하향식/빅뱅 전략에 따라 달라진다.
2.2.1
컴포넌트 테스팅Component Testing K2
컴포넌트를 개별적으로 테스트하는 레벨. 단위 테스팅이라고도 하며, 일반적으로 개발자가 자신의 개발 환경에서 수행한다.
2.2.1
확인 테스팅Confirmation Testing K2
원래 결함이 성공적으로 수정되었는지 확인하는 테스팅. 이전에 실패한 테스트를 재실행하거나 새 테스트를 추가한다.
2.2.3
기능 테스팅Functional Testing K2
시스템이 수행해야 하는 기능("무엇을")을 평가하는 테스트 유형. 기능 성숙도, 정확성, 타당성을 확인한다.
2.2.2
통합 테스팅Integration Testing K2
컴포넌트 또는 시스템 간의 인터페이스와 상호작용을 테스트하는 활동. 컴포넌트 통합과 시스템 통합으로 나뉜다.
2.2.1
유지보수 테스팅Maintenance Testing K2
운영 중인 시스템의 변경사항(수정, 업그레이드, 마이그레이션, 단종)을 테스트하는 활동. 영향도 분석이 선행된다.
2.3
비기능 테스팅Non-functional Testing K2
시스템이 "얼마나 잘 동작하는지" 테스트. 성능, 호환성, 유용성, 신뢰도, 보안, 유지보수성, 이동성, 안전성 등을 평가한다.
2.2.2
회귀 테스팅Regression Testing K2
변경으로 인한 부정적 영향이 없는지 확인하는 테스팅. 자동화에 매우 적합하며, 확인 테스팅과 구분해야 한다.
2.2.3
시프트 레프트Shift Left K2
SDLC 초기에 테스팅을 수행하는 접근법. 조기 테스팅으로 결함을 일찍 발견해 수정 비용을 줄인다.
2.1.5
시스템 통합 테스팅System Integration Testing K2
다른 시스템 또는 외부 서비스와 테스트 대상 시스템의 인터페이스를 테스트하는 레벨. 운영 환경과 유사한 환경을 사용한다.
2.2.1
시스템 테스팅System Testing K2
전체 시스템의 전반적 동작과 기능에 중점을 둔 테스트 레벨. 엔드투엔드 기능 테스팅과 비기능 테스팅을 포함한다.
2.2.1
테스트 레벨Test Level K2
함께 구성하고 관리하는 테스트 활동의 집합. 컴포넌트, 컴포넌트 통합, 시스템, 시스템 통합, 인수의 5개 레벨이 있다.
2.2.1
테스트 대상Test Object K2
테스트의 대상이 되는 작업 산출물. (Ch1에서도 정의됨) 테스트 레벨에 따라 그 범위가 달라진다.
2.2
테스트 유형Test Type K2
특정 품질 특성과 관련된 테스트 활동의 집합. 기능, 비기능, 블랙박스, 화이트박스의 4가지가 있으며, 모든 레벨에 적용 가능하다.
2.2.2
화이트박스 테스팅White-box Testing K2
시스템의 내부 구조(코드, 아키텍처, 워크플로우, 데이터 플로우)에서 테스트를 도출하는 테스팅.

4. Ch3. 정적 테스팅 용어 10개

Ch3 리뷰 기법 안내 실러버스 Ch3에서는 리뷰 기법(체크리스트 기반, 시나리오 기반, 역할 기반, 관점 기반 읽기)이 본문에서 설명되지만, 공식 용어 목록에 포함되지 않은 것도 있다. 시험에 출제 가능성을 고려해 공식 목록의 용어를 중심으로 수록한다.
이상 사항Anomaly K1
기대 결과와 다른 상태 또는 사건. 리뷰 중 식별되며, 반드시 결함인 것은 아니다. 분석과 논의를 통해 상태를 판단한다.
3.2.2
동적 테스팅Dynamic Testing K2
소프트웨어를 실행하여 결함을 발견하는 테스팅. 장애를 유발한 후 분석을 통해 결함을 찾는다. 정적 테스팅과 대비된다.
3.1.3
공식 리뷰Formal Review K2
문서화된 절차와 규칙을 따르는 리뷰. 인스펙션이 가장 공식적인 유형이며, 감사 기록이 필요한 경우 사용한다.
3.2.4
비공식 리뷰Informal Review K2
정의된 프로세스를 따르지 않고 공식적 결과 문서도 요구하지 않는 리뷰. 주요 목적은 이상 사항 식별이다.
3.2.4
인스펙션Inspection K2
가장 공식적인 리뷰 유형. 이상 사항을 최대한 많이 찾는 것이 주요 목적이며, 저자는 리뷰 리더나 서기가 될 수 없다.
3.2.4
리뷰Review K2
작업 산출물을 수동으로 평가하는 정적 테스팅 유형. 계획, 리뷰 착수, 개별 리뷰, 논의/분석, 수정/보고의 활동으로 구성된다.
3.2.2
정적 분석Static Analysis K2
도구를 사용해 소프트웨어를 실행하지 않고 코드나 산출물을 분석하는 기법. 코딩 결함, 보안 취약점, 유지보수성 문제를 식별한다.
3.1
정적 테스팅Static Testing K2
소프트웨어를 실행하지 않고 작업 산출물을 평가하는 테스팅. 리뷰와 정적 분석을 포함하며, SDLC 초기 결함 식별에 효과적이다.
3.1
기술 리뷰Technical Review K2
기술적 자격을 갖춘 리뷰어가 수행하고 중재자가 리더가 되는 리뷰. 기술 문제에 대한 합의 도출과 결정이 목적이다.
3.2.4
워크스루Walkthrough K2
저자가 리더가 되어 진행하는 리뷰 유형. 품질 평가, 리뷰어 교육, 합의 도출, 이상 사항 발견 등 다양한 목적을 달성한다.
3.2.4

5. Ch4. 테스트 분석과 설계 용어 18개

Ch4는 K3가 가장 많은 챕터! 동등 분할, 경계값 분석, 결정 테이블, 상태 전이, ATDD는 모두 K3 수준이다. 시험에서 직접 테스트 케이스를 도출하는 계산 문제가 출제되므로 반드시 연습이 필요하다.
인수 조건Acceptance Criteria K2
컴포넌트나 시스템이 사용자, 고객 또는 기타 이해관계자가 수용하기 위해 충족해야 하는 조건.
4.5.2
인수 테스트 주도 개발Acceptance Test-Driven Development (ATDD) K3
개발자, 테스터, 비즈니스 담당자가 협업해 인수 조건에서 테스트를 먼저 작성하고, 이를 기반으로 개발하는 테스트 우선 접근법.
4.5.3
블랙박스 테스트 기법Black-box Test Technique K2
내부 구조를 참조하지 않고 테스트 대상의 명시된 동작 분석을 기반으로 하는 기법. 명세 기반 기법이라고도 한다.
4.1 / 4.2
경계값 분석Boundary Value Analysis (BVA) K3
동등 분할의 경계에서 테스트 값을 선택하는 기법. 2개 선택(경계+인접)과 3개 선택(경계+양쪽 인접)이 있다.
4.2.2
분기 커버리지Branch Coverage K2
코드의 모든 분기(결정 결과의 참/거짓)를 실행한 비율. 100% 분기 커버리지는 100% 구문 커버리지를 보장한다.
4.3.2
체크리스트 기반 테스팅Checklist-based Testing K2
체크리스트에 기록된 테스트 조건을 기반으로 테스트를 설계/구현하는 경험 기반 기법. 체크리스트는 경험으로부터 구축된다.
4.4.3
협업 기반 테스트 접근법Collaboration-based Test Approach K2
개발자, 테스터, 비즈니스 담당자가 협업해 사용자 스토리와 인수 조건을 작성하는 접근법. ATDD가 대표적이다.
4.5
커버리지Coverage K2
(Ch1에서도 정의됨) Ch4에서는 각 테스트 기법별 커버리지 항목과 100% 커버리지 달성 조건을 구체적으로 다룬다.
4.2 / 4.3
커버리지 항목Coverage Item K2
테스트 기법에 의해 식별된, 커버리지 측정의 기본 단위. 기법에 따라 분할, 경계값, 열, 상태, 구문, 분기 등이 된다.
4.2
결정 테이블 테스팅Decision Table Testing K3
조건 조합과 그에 따른 결과를 테이블로 표현해 테스트를 도출하는 기법. 복잡한 비즈니스 규칙 테스트에 효과적이다.
4.2.3
동등 분할Equivalence Partitioning (EP) K3
데이터를 동일하게 처리될 것으로 예상되는 그룹(분할)으로 나누는 기법. 각 분할에서 하나의 값만 테스트하면 충분하다.
4.2.1
오류 추정Error Guessing K2
테스터의 경험과 지식을 활용해 오류, 결함, 장애가 발생할 가능성이 높은 영역을 예측하는 경험 기반 기법.
4.4.1
경험 기반 테스트 기법Experience-based Test Technique K2
테스터의 지식과 경험을 활용하는 기법. 블랙박스/화이트박스 기법을 보완하며, 효과는 테스터 능력에 크게 의존한다.
4.1 / 4.4
탐색적 테스팅Exploratory Testing K2
테스트 설계, 실행, 학습을 동시에 수행하는 경험 기반 기법. 테스트 차터를 사용해 가이드하며, 시간 제한(타임박스)을 둔다.
4.4.2
상태 전이 테스팅State Transition Testing K3
상태 다이어그램이나 상태 테이블을 기반으로 상태 변화와 전이를 테스트하는 기법. 모든 상태, 유효 전이, 모든 전이 커버리지가 있다.
4.2.4
구문 커버리지Statement Coverage K2
코드의 실행 가능한 구문(statement) 중 테스트로 실행된 비율. 100% 구문 커버리지가 100% 분기 커버리지를 보장하지는 않는다.
4.3.1
테스트 기법Test Technique K2
테스트 분석과 설계를 지원하는 절차. 블랙박스, 화이트박스, 경험 기반의 세 가지로 분류된다.
4.1
화이트박스 테스트 기법White-box Test Technique K2
테스트 대상의 내부 구조와 처리에 대한 분석을 기반으로 하는 기법. 구조 기반 기법이라고도 한다. 구문/분기 커버리지가 대표적이다.
4.1 / 4.3

6. Ch5. 테스트 관리 용어 20개

결함 관리Defect Management K3
결함을 발견부터 종료까지 처리하는 프로세스. 기록, 분석, 분류, 대응 결정, 종료의 워크플로우로 구성된다.
5.5
결함 보고서Defect Report K3
테스트에서 발견된 결함(이상 사항)을 문서화한 보고서. 고유 ID, 제목, 재현 절차, 기대/실제 결과, 심각도, 우선순위 등을 포함한다.
5.5
시작 조건Entry Criteria K2
특정 활동을 수행하기 위한 전제 조건. 자원 가용성, 테스트웨어 가용성, 초기 품질 수준 등이 포함된다. 애자일에서는 "준비의 정의"라고도 한다.
5.1.3
완료 조건Exit Criteria K2
특정 활동의 종료를 선언하기 위해 달성해야 하는 사항. 커버리지 수준, 미해결 결함 수, 실패 케이스 수 등이 포함된다. 애자일에서는 "완료의 정의"라고도 한다.
5.1.3
제품 리스크Product Risk K2
제품 품질 특성과 관련된 리스크. 누락된 기능, 부정확한 계산, 성능 문제, 보안 취약점 등이 해당한다.
5.2.2
프로젝트 리스크Project Risk K2
프로젝트 관리 및 제어와 관련된 리스크. 인도 지연, 부정확한 추정, 기술 부족, 도구 지원 부족 등이 해당한다.
5.2.2
리스크Risk K1
발생 시 부정적 영향을 미칠 수 있는 잠재적 사건/위험/위협. 리스크 발생 가능성과 리스크 영향의 두 가지 요소로 특징지어진다.
5.2.1
리스크 분석Risk Analysis K2
리스크 식별과 리스크 평가로 구성되는 활동. 포괄적 리스크 목록을 생성하고, 분류/수준 결정/우선순위 지정을 수행한다.
5.2.3
리스크 평가Risk Assessment K2
식별된 리스크를 분류하고, 발생 가능성/영향/수준을 결정하고, 우선순위를 정하고, 조치 방법을 제안하는 활동.
5.2.3
리스크 기반 테스팅Risk-based Testing K2
리스크 분석과 제어를 기반으로 테스트 활동을 선택하고 우선순위를 정하는 테스트 접근법.
5.2
리스크 제어Risk Control K2
식별 및 평가된 리스크에 대응해 취하는 모든 조치. 리스크 완화와 리스크 모니터링으로 구성된다.
5.2.4
리스크 식별Risk Identification K1
포괄적인 리스크 목록을 생성하는 활동. 브레인스토밍, 워크숍, 인터뷰, 원인-결과 다이어그램 등의 기법을 사용한다.
5.2.3
리스크 수준Risk Level K1
리스크를 측정한 값. 리스크 발생 가능성과 리스크 영향(피해)의 두 요소로 표현된다. 수준이 높을수록 조치가 중요하다.
5.2.1
리스크 완화Risk Mitigation K2
리스크 평가 때 제안된 조치를 실행하여 리스크 수준을 낮추는 활동. 적절한 테스트 기법, 커버리지 적용 등이 해당한다.
5.2.4
리스크 모니터링Risk Monitoring K2
리스크 완화 조치의 효과를 확인하고, 추가 정보를 확보하고, 새롭게 나타난 리스크를 식별하는 활동.
5.2.4
테스트 접근법Test Approach K2
특정 프로젝트에서 테스트를 수행하는 방법에 대한 구현 수준의 결정. 테스트 레벨, 유형, 기법, 시작/완료 조건 등을 포함한다.
5.1.1
테스트 완료 보고서Test Completion Report K2
특정 테스트 활동을 요약하는 보고서. 테스트 요약, 품질 평가, 계획 대비 편차, 미해결 결함, 교훈 등을 포함한다.
5.3.2
테스트 진행 보고서Test Progress Report K2
테스트 기간, 진행 상황, 방해 요소, 메트릭, 신규 리스크, 다음 주기 예정 테스팅 등을 포함하는 정기 보고서.
5.3.2
테스트 피라미드Test Pyramid K1
테스트의 세분화 수준이 다를 수 있음을 보여주는 모델. 아래층은 작고 빠른 테스트가 많고, 위층은 크고 느린 테스트가 적다.
5.1.6
테스팅 사분면Testing Quadrants K2
테스트를 기술/비즈니스 축과 팀 지원/제품 평가 축으로 분류하는 4사분면 모델. 테스트 유형과 레벨의 관계를 시각화한다.
5.1.7

7. Ch6. 테스트 도구 용어 1개

테스트 자동화Test Automation K1
테스트 활동을 도구를 사용해 자동으로 수행하는 것. 반복 작업 감소, 일관성 향상, 빠른 피드백 등의 효과가 있으나, 도입/유지보수 비용과 신뢰성 리스크도 고려해야 한다.
6.1 / 6.2
Ch6 시험 포인트 Ch6은 용어가 1개뿐이지만, 테스트 자동화의 "효과"와 "리스크"를 구분하는 문제가 자주 출제된다. 효과(시간 절약, 일관성, 빠른 피드백)와 리스크(비현실적 기대, 유지보수 비용, 도구 의존성)를 명확히 구분하자.
