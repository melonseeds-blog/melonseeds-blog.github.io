콜백이 필요한 상황

호출자가 결과를 즉시 받을 수 없거나, 라이브러리가 사건 발생 시점에 사용자 코드를 호출해야 할 때 콜백이 등장한다. 산업 SDK에서는 다음과 같은 시나리오가 일반적이다.

  • 비동기 통보 — 카메라 grab 완료, 에러 발생, 디바이스 disconnect
  • GUI 이벤트 — 버튼 클릭, 마우스 이동
  • 알고리즘 진행률 — long-running task의 percent 보고, cancel 요청
  • Dual 통보 시스템 — 함수가 ErrorCode를 즉시 반환하면서, 등록된 전역 콜백에도 동시에 보고

공통점은 "콜러가 호출 시점에 결정한 행동을, 라이브러리가 나중에 자기 컨텍스트에서 실행한다"는 점이다.

세 가지 콜백 표현 방식

C++에서 콜백을 표현하는 주요 수단은 셋이다. 각자 트레이드오프가 다르므로 상황에 맞게 고른다.

방식상태 보유타입 안전성능ABI주 용도
함수 포인터불가 (void* 별도)약함가장 빠름C 호환DLL 경계, C API
인터페이스가능강함가상 호출 1회C++ 한정구조화된 핸들러
std::function가능 (캡처)강함indirection + SBOC++ 한정람다, 멤버, 함수 객체 통합

인터페이스 방식은 핸들러가 여러 메서드를 가진 객체일 때 자연스럽다. 반면 단발성 람다를 받으려면 std::function이 편하다.

함수 포인터 + 사용자 데이터 (C 스타일)

C ABI 호환을 위해 DLL 경계에서 가장 많이 쓰이는 형태다. 상태는 void* 사용자 데이터로 들고 다닌다.

// C-style callback signature
typedef void (*ErrorCb)(int code, const char* msg, void* userData);

ErrorCode SetErrorCallback(ErrorCb cb, void* userData);

// 사용
struct MyContext { Logger* log; };
MyContext ctx{ &myLogger };

SetErrorCallback(
    [](int code, const char* msg, void* user) {
        auto* c = static_cast<MyContext*>(user);
        c->log->Error("err {}: {}", code, msg);
    },
    &ctx);

장점은 명확하다. 가벼운 함수 포인터 1개와 void* 1개만 ABI를 넘나든다. 단점은 void* 캐스팅의 타입 안전성 부재와, 캡처 람다는 변환되지 않는다는 점이다 (capture-less 람다만 함수 포인터로 변환 가능).

DLL 경계라면 함수 포인터가 정답 std::function은 표준이지만 ABI가 컴파일러/STL 버전에 묶인다. 외부에 SDK 헤더를 공개한다면 함수 포인터 + void* 패턴이 가장 안전하다.

std::function 사용

같은 라이브러리 내부에서, 또는 C++ 헤더 only 인터페이스에서는 std::function이 표현력이 가장 좋다.

#include <functional>
#include <string>

using ErrorCallback = std::function<void(int code, const std::string& msg)>;

ErrorCode SetErrorCallback(ErrorCallback cb);

// 람다 등록 (캡처로 상태 보유)
Logger logger;
SetErrorCallback([&logger](int code, const std::string& msg) {
    logger.Error("err {}: {}", code, msg);
});

// 멤버 함수 등록
class Handler
{
public:
    void OnError(int code, const std::string& msg);
};

Handler h;
SetErrorCallback([&h](int c, const std::string& m) { h.OnError(c, m); });

람다 캡처로 컨텍스트를 자연스럽게 들고 다닌다. std::bind 대신 람다로 멤버 함수를 감싸는 편이 가독성이 좋다.

std::function의 비용

편의에는 비용이 따른다. std::function 내부는 type-erasure이고, 호출 시 가상 함수와 비슷한 indirection을 거친다.

  • SBO (Small Buffer Optimization) — 작은 callable은 내부 버퍼에 저장
  • SBO 한계 (보통 2~3 포인터 크기) 초과 시 힙 할당
  • 호출마다 type-erased thunk 경유 → 인라이닝 거의 불가능
  • 생성/복사 비용도 있음 (특히 힙 할당이 발생할 때)
std::function은 hot path에서 신중하게 프레임당 수백만 회 호출되는 inner loop에서 쓰면 indirection이 누적된다. 이런 곳은 템플릿 콜백을 쓰는 편이 빠르다 (아래 성능 섹션 참고).

이벤트 디스패처 설계

여러 구독자가 하나의 이벤트를 듣는 구조라면 디스패처가 필요하다. 기본 골격은 다음과 같다.

enum class EventType { GrabDone, Error, Disconnect };

struct Event { /* payload */ };

using Token = std::uint64_t;

class EventDispatcher
{
public:
    using Handler = std::function<void(const Event&)>;

    Token Subscribe(EventType t, Handler h);
    void Unsubscribe(Token t);
    void Dispatch(const Event& e, EventType t);

private:
    struct Entry { Token id; Handler fn; };
    std::unordered_map<EventType, std::vector<Entry>> m_handlers;
    Token m_nextId{ 1 };
    std::mutex m_mutex;
};

구독 해제는 Token(증가하는 ID)으로 식별한다. iterator를 반환하면 vector 재할당 시 무효화되므로, 안정적인 식별자가 더 안전하다.

Dispatch 도중 Subscribe/Unsubscribe — vector 무효화 핸들러 안에서 다시 Subscribe나 Unsubscribe를 호출하면 순회 중인 vector가 변경되어 iterator가 무효화된다. delayed list를 두거나, dispatch 중에는 추가 작업을 큐에 모아 두고 끝난 뒤 적용하는 방식으로 보호한다.
멀티스레드 동시 Subscribe/Dispatch — 데이터 레이스 std::vector는 동시 read/write에 thread-safe하지 않다. 최소한 mutex로 보호하거나, copy-on-dispatch (락 안에서 vector 복사 후 락 풀고 호출) 패턴을 쓴다. lock-free가 필요하면 shared_ptr<const vector> 스왑 방식이 흔하다.

약한 참조 콜백 (lifetime 안전성)

콜백 안에서 this를 캡처했는데 객체가 먼저 소멸하면 dangling이다. 디스패처는 핸들러 vector를 잡고 있을 뿐, 핸들러가 가리키는 객체의 수명에는 관여하지 않는다.

해결책은 std::weak_ptr를 캡처하고 호출 직전에 lock()으로 살아있는지 확인하는 것이다.

class MyClass : public std::enable_shared_from_this<MyClass>
{
public:
    void Register(EventDispatcher& d)
    {
        auto self = std::weak_ptr<MyClass>(shared_from_this());
        d.Subscribe(EventType::GrabDone,
            [self](const Event& e)
            {
                if (auto p = self.lock())
                {
                    p->OnDone(e);
                }
                // 객체가 죽었으면 그냥 무시
            });
    }

    void OnDone(const Event& e);
};
weak_ptr 캡처는 lifetime 안전성의 표준 패턴 shared_ptr를 캡처하면 디스패처가 객체 수명을 강제로 연장시켜 의도치 않은 순환 참조가 생길 수 있다. weak_ptr가 보통 더 좋은 기본값이다.

에러 콜백 + ErrorCode dual 통보 패턴

산업 SDK에서 자주 쓰는 패턴이다. 함수는 ErrorCode를 즉시 반환하여 호출 컨텍스트가 처리할 수 있게 하고, 동시에 등록된 전역 콜백에도 통보하여 로깅/UI 업데이트가 가능하게 한다.

ErrorCode ImageProcessor::ApplyBlur(Image& img, int kernel)
{
    if (img.IsEmpty())
    {
        const ErrorCode ec = ErrorCode::InvalidArgument;
        // 1) 등록된 콜백에 통보 (전역 로깅, GUI)
        m_errorBus->Notify(ec, "image is empty");
        // 2) 즉시 반환 (호출자 컨텍스트 처리)
        return ec;
    }
    // ... 정상 처리
    return ErrorCode::Ok;
}
한 에러가 두 곳에 보고된다 호출자도 ErrorCode로 처리하고, 글로벌 핸들러도 받는다. 양쪽이 idempotent해야 한다 (동일 에러를 두 번 로깅하지 않도록 한쪽만 사용자에게 표시하는 정책 등).

성능 고려

콜백이 hot path에 있다면 std::function의 indirection이 누적된다. 컴파일 타임에 콜백 타입을 알 수 있으면 템플릿 매개변수로 받는 편이 빠르다.

// 매 호출마다 std::function indirection
void ProcessFrames(std::function<void(const Frame&)> cb);

// 람다 타입 그대로 인스턴스화 → 인라이닝 가능
template <class Cb>
void ProcessFrames(Cb cb)
{
    for (auto& f : m_frames)
    {
        cb(f);  // 인라인됨
    }
}

API 경계에서는 std::function으로 받고, 성능이 결정적인 inner loop는 템플릿으로 분리하는 하이브리드도 흔하다.

흔한 함정

this 캡처 후 객체 소멸 → dangling [this] 캡처는 raw pointer와 같다. 객체보다 콜백이 오래 살면 즉시 UB. weak_ptr 또는 명시적 unsubscribe로 보호한다.
콜백 안에서 dispatcher.Unsubscribe(self) 디스패처가 vector 순회 중인데 그 vector를 수정하는 격이다. 무효화된 iterator로 다음 핸들러를 호출하려 하면 crash. 자기 자신을 해제하려면 pending 큐를 사용해 dispatch 종료 후에 적용한다.
콜백에서 throw → 호출자 의도 파괴 디스패처가 예외 처리를 안 한다면 다음 구독자들이 호출되지 않는다. 라이브러리 콜백은 가능하면 noexcept를 권장하거나, 디스패처가 try/catch로 감싸 다른 구독자를 보호한다.
shared_ptr 강한 참조 캡처 → 순환 참조 [self = shared_from_this()]로 캡처하면 디스패처가 살아있는 동안 객체도 살아남는다. 등록자가 자기 객체의 멤버 디스패처에 자기를 구독하면 사이클. 가능한 한 weak_ptr.

C++23 std::move_only_function

std::function은 보관하는 callable이 복사 가능해야 한다. 그래서 std::unique_ptr를 캡처한 람다는 std::function에 못 넣는다.

auto resource = std::make_unique<Resource>();
std::function<void()> f = [r = std::move(resource)]() { /* ... */ };  // 컴파일 에러

C++23부터 std::move_only_function이 표준에 들어와, move-only callable도 보관할 수 있다.

std::move_only_function — 한 번만 호출하는 작업에도 유용 thread pool에 넣을 task처럼 unique_ptr/이동만 가능한 자원을 캡처하는 callable을 자연스럽게 다룰 수 있다. 컴파일러/STL이 C++23 지원하는지 확인 필요.

이미지 acquisition 시스템 적용 예

카메라 grabber에 세 종류의 콜백을 노출하는 그림이다.

class CameraGrabber
{
public:
    using FrameCallback    = std::function<void(const Frame&)>;
    using ErrorCallback    = std::function<void(int code, const std::string&)>;
    using ProgressCallback = std::function<void(int percent)>;

    ErrorCode SetFrameCallback(FrameCallback cb);
    ErrorCode SetErrorCallback(ErrorCallback cb);
    ErrorCode SetProgressCallback(ProgressCallback cb);

    ErrorCode Start();
    ErrorCode Stop();

private:
    void onFrameReceived(const Frame& f)
    {
        if (m_frameCb) m_frameCb(f);
    }
    void onErrorOccurred(int code, const std::string& msg)
    {
        if (m_errorCb) m_errorCb(code, msg);
    }

    FrameCallback    m_frameCb;
    ErrorCallback    m_errorCb;
    ProgressCallback m_progressCb;
};

// 사용
CameraGrabber cam;
cam.SetFrameCallback([&](const Frame& f)   { display.Show(f); });
cam.SetErrorCallback([&](int c, auto& m)  { logger.Error("{}: {}", c, m); });
cam.SetProgressCallback([&](int p)         { progressBar.Set(p); });
cam.Start();

외부 SDK 헤더로 노출할 때는 함수 포인터 + void*로 바꾸고, 내부에서 std::function으로 어댑터를 두는 구조도 흔하다. ABI 안정성과 표현력을 모두 잡는 절충안이다.
