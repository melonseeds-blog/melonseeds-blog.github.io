Modbus란?

Modbus는 1979년 Modicon(현 Schneider Electric)이 PLC 간 통신을 위해 개발한 시리얼 통신 프로토콜이다. 공개된 표준으로 라이선스 비용 없이 누구나 구현할 수 있어, 40년이 넘은 지금까지도 산업 자동화의 사실상(de facto) 표준으로 널리 사용되고 있다.

왜 Modbus가 아직도 쓰이나?

단순한 구조, 무료 사양 공개, 거의 모든 산업용 장비(PLC, 인버터, 센서, 미터 등)가 지원하기 때문이다. 새로운 프로토콜(EtherCAT, PROFINET 등)이 등장했지만, 복잡한 실시간 제어가 필요하지 않은 모니터링/설정 용도에서는 Modbus가 가장 간단하고 확실한 선택이다.

Master-Slave 구조

Modbus는 Master-Slave(Client-Server) 구조로 동작한다. Master(주로 PC나 HMI)가 요청(Request)을 보내면, Slave(PLC, 센서 등)가 응답(Response)을 돌려준다. Slave는 스스로 데이터를 전송하지 않으며, 반드시 Master의 요청이 있어야만 응답한다. 하나의 버스에 최대 247대의 Slave를 연결할 수 있다(주소 1~247).

Modbus Organization 현재 Modbus 사양은 Modbus Organization(modbus.org)에서 관리하며, 누구나 무료로 사양서를 다운로드할 수 있다. 최신 사양에서는 Master/Slave 대신 Client/Server 용어를 사용하지만, 현장에서는 여전히 Master/Slave가 더 많이 쓰인다.

Modbus 데이터 모델

Modbus는 4가지 데이터 영역을 정의한다. 각 영역은 0번부터 시작하는 주소 공간을 가지며, 장비마다 어떤 주소에 어떤 값이 매핑되는지는 제조사 매뉴얼을 참고해야 한다.

그림 1. Modbus의 4가지 데이터 영역과 특성
영역 주소 범위 크기 접근 용도 예시
Coil 00001 ~ 09999 1 bit 읽기/쓰기 릴레이 ON/OFF, 모터 기동/정지
Discrete Input 10001 ~ 19999 1 bit 읽기 전용 리밋 스위치, 센서 감지 상태
Input Register 30001 ~ 39999 16 bit 읽기 전용 온도, 압력, 전류 측정값
Holding Register 40001 ~ 49999 16 bit 읽기/쓰기 설정 온도, PID 파라미터, 속도 설정
주소 표기 주의: 0-based vs 1-based Modbus 사양서에서 주소는 0부터 시작(0-based)하지만, 많은 장비 매뉴얼과 소프트웨어는 1부터 표기(1-based)한다. 예를 들어, Holding Register 0번은 "40001"로 표기된다. 장비마다 다를 수 있으므로 반드시 매뉴얼을 확인해야 한다.

Modbus RTU vs ASCII vs TCP

Modbus 프로토콜은 전송 방식에 따라 3가지 모드로 나뉜다. 각 모드는 동일한 데이터 모델과 Function Code를 사용하지만, 프레임 구조와 물리 계층이 다르다.

항목 Modbus RTU Modbus ASCII Modbus TCP
물리 계층 RS-232/422/485 RS-232/422/485 Ethernet (TCP/IP)
데이터 인코딩 바이너리 (8-bit) ASCII 문자 (HEX) 바이너리 (8-bit)
오류 검출 CRC-16 LRC TCP 체크섬
프레임 구분 3.5 문자 시간 간격 시작(:) / 끝(CR LF) MBAP 헤더 길이 필드
전송 효율 높음 (바이너리) 낮음 (2배 용량) 높음 (바이너리)
속도 9600~115200 bps 9600~19200 bps 10/100 Mbps+
최대 거리 1,200m (RS-485) 1,200m (RS-485) 100m (스위치로 확장)
주요 용도 현장 계기, PLC 연동 느린 장비, 디버깅 SCADA, HMI, 원격 모니터링

RTU vs ASCII 선택 기준 실무에서는 거의 대부분 RTU 모드를 사용한다. ASCII 모드는 데이터를 HEX 문자로 변환하므로 전송량이 2배가 되어 비효율적이다. ASCII가 쓰이는 경우는 텍스트 터미널로 수동 디버깅할 때 정도로, 현대 장비에서는 RTU가 표준이다.

주요 Function Code

Function Code는 Master가 Slave에게 어떤 동작을 요청하는지 나타내는 1바이트 값이다. 가장 많이 사용되는 코드는 다음과 같다.

코드 이름 대상 영역 설명
01 (0x01) Read Coils Coil 1~2000개 Coil 상태 읽기
02 (0x02) Read Discrete Inputs Discrete Input 1~2000개 입력 상태 읽기
03 (0x03) Read Holding Registers Holding Register 1~125개 레지스터 읽기
04 (0x04) Read Input Registers Input Register 1~125개 레지스터 읽기
05 (0x05) Write Single Coil Coil Coil 1개 ON/OFF 쓰기
06 (0x06) Write Single Register Holding Register 레지스터 1개 값 쓰기
15 (0x0F) Write Multiple Coils Coil 여러 개 Coil 동시 쓰기
16 (0x10) Write Multiple Registers Holding Register 여러 개 레지스터 동시 쓰기

FC 03 요청/응답 예시

Slave #1의 Holding Register 주소 0x0000부터 2개를 읽는 예시(RTU 모드):

요청 (Master -> Slave)

01 03 00 00 00 02 C4 0B

01 Slave 주소 (1번)
03 Function Code (Read Holding Registers)
00 00 시작 주소 (0x0000 = 40001)
00 02 읽을 레지스터 수 (2개)
C4 0B CRC-16 체크섬
응답 (Slave -> Master)

01 03 04 00 64 00 C8 FA 33

01 Slave 주소 (1번)
03 Function Code (echo)
04 바이트 수 (4 bytes = 2 registers x 2)
00 64 Register #0 값 = 100 (0x0064)
00 C8 Register #1 값 = 200 (0x00C8)
FA 33 CRC-16 체크섬

예외 응답 (Exception Response)

Slave가 요청을 처리할 수 없을 때, Function Code의 최상위 비트를 1로 설정(+0x80)하여 예외 응답을 보낸다.

예외 코드 이름 의미
01 Illegal Function 지원하지 않는 Function Code
02 Illegal Data Address 유효하지 않은 레지스터 주소
03 Illegal Data Value 요청 데이터 값이 범위를 벗어남
04 Slave Device Failure Slave 내부 오류

Modbus TCP 구조

Modbus TCP는 기존 RTU 프레임을 이더넷(TCP/IP)으로 감싼 것이다. Slave 주소와 CRC가 빠지고, 대신 MBAP(Modbus Application Protocol) 헤더가 추가된다. 기본 포트는 502번이다.

MBAP 헤더 구조

그림 4. Modbus TCP 프레임 구조 (MBAP 헤더 + PDU)
Transaction ID 요청/응답을 매칭하는 식별자. Master가 지정하며, Slave는 동일한 값을 응답에 포함.
Protocol ID 항상 0x0000 (Modbus 프로토콜). 다른 값이면 Modbus가 아님.
Length Unit ID + PDU의 바이트 수. 수신 측에서 메시지 경계를 파악하는 데 사용.
Unit ID RTU의 Slave Address에 해당. TCP 직접 연결 시 보통 0x01 또는 0xFF.

Modbus 게이트웨이

TCP-RTU 게이트웨이

기존 RS-485 Modbus RTU 장비를 이더넷 네트워크에 연결할 때 사용한다. PC(Modbus TCP Master)가 게이트웨이에 TCP 요청을 보내면, 게이트웨이가 RTU 프레임으로 변환하여 시리얼 버스의 Slave에 전달한다. Unit ID가 RTU Slave의 주소로 매핑된다.

동시 연결과 타임아웃 Modbus TCP는 여러 Client가 동시에 하나의 Server에 접속할 수 있다. 하지만 게이트웨이 뒤의 RTU 버스는 한 번에 하나의 요청만 처리하므로, 게이트웨이가 요청을 큐잉(Queuing)한다. 타임아웃 설정이 너무 짧으면 큐 대기 중 오류가 발생할 수 있으므로 주의해야 한다.

실무 예시

PLC 레지스터 읽기/쓰기

시나리오: 온도 설정값 읽기 및 변경

PLC의 Holding Register 40100에 현재 설정 온도(단위: 0.1도)가 저장되어 있다. 현재 값을 읽고, 25.0도로 변경하는 과정:

1. 읽기 (FC03) 01 03 00 63 00 01 74 12 — Slave#1, 주소 99(=40100-40001), 1개 읽기
2. 응답 01 03 02 00 C8 ... — 값 200 = 20.0도
3. 쓰기 (FC06) 01 06 00 63 00 FA 78 1A — 주소 99에 250(=25.0도) 쓰기
4. 응답 01 06 00 63 00 FA 78 1A — 쓰기 성공 (요청과 동일한 echo)

조명 컨트롤러 제어

시나리오: 4채널 조명 밝기 일괄 설정

조명 컨트롤러의 Holding Register 40001~40004에 채널 1~4의 밝기(0~255)가 매핑되어 있다. FC16(Write Multiple Registers)으로 4개 채널 밝기를 한 번에 설정:

요청 (FC16) 02 10 00 00 00 04 08 00 80 00 C0 00 FF 00 40 ...
해석 Slave#2, 시작주소 0, 4개 레지스터, 8바이트 데이터: CH1=128, CH2=192, CH3=255, CH4=64
실무 팁: 폴링 주기 설계 Modbus는 Master가 주기적으로 Slave를 폴링(Polling)하는 구조다. 폴링 주기가 너무 짧으면 버스가 포화되고, 너무 길면 모니터링 지연이 생긴다. RS-485에서 다수 Slave를 연결할 때는 각 Slave 응답 시간 + 턴어라운드 딜레이를 고려하여 전체 스캔 타임을 계산해야 한다.
바이트 순서(Byte Order) 주의 Modbus 표준은 16-bit 레지스터를 Big-Endian(MSB first)으로 전송한다. 하지만 32-bit 실수(Float)나 32-bit 정수를 2개 레지스터에 나눠 저장할 때, 레지스터 순서(Word Order)는 장비마다 다를 수 있다. AB CD / CD AB / BA DC / DC BA 등 4가지 조합이 가능하므로, 반드시 장비 매뉴얼을 확인해야 한다.
